# 1. Information Disclosure

## What is Information Disclosure?  
Information disclosure (or information leakage) occurs when a website unintentionally reveals sensitive data. This can include:  
- User data (e.g., usernames, financial details)  
- Business or commercial data  
- Technical details about the website  

Even leaking technical details can expose vulnerabilities, helping attackers craft complex exploits.  

## How Does Information Disclosure Happen?  
Sensitive information can be leaked due to:  
- **Internal content left in public files** (e.g., developer comments in HTML)  
- **Insecure configurations** (e.g., enabling debugging or verbose error messages)  
- **Application design flaws** (e.g., distinct error responses revealing valid usernames)  

## Examples of Information Disclosure  
- Exposed directory structure via `robots.txt` or directory listing  
- Source code leaks through temporary backup files  
- Error messages revealing database schema  
- Hard-coded credentials in source code  
- Exposing API keys, IP addresses, or sensitive resources  

## Impact of Information Disclosure  
The severity depends on what information is exposed:  
- **High impact** – Leaking user data (e.g., credit card details)  
- **Potentially dangerous** – Exposing technical details (e.g., frameworks, file paths)  
- **Context matters** – Some information is harmless unless attackers exploit it  

## Assessing Severity  
The impact depends on exploitability:  
- If leaked information helps attackers exploit a vulnerability, it's severe  
- If the disclosed data is already secure (e.g., patched software versions), the risk is low  

## Prevention Best Practices  
- Educate developers about sensitive data risks  
- Audit code for accidental leaks (e.g., developer comments, debug info)  
- Use generic error messages to avoid revealing system details  
- Disable debugging and diagnostic features in production  
- Review third-party configurations to disable unnecessary features  


# 2. Testing for Information Disclosure Vulnerabilities  

## General Approach  
Avoid "tunnel vision" when testing—sensitive data can appear in unexpected places. A key skill is recognizing valuable information whenever you find it.  

## Key Testing Techniques  

### 1. Fuzzing  
- Submit unexpected data types and fuzz strings to observe application behavior.  
- Look for explicit or subtle differences in responses (e.g., response time, error messages).  
- Use **Burp Intruder** to:  
  - Automate high-volume input testing with wordlists.  
  - Compare HTTP status codes, response times, and content.  
  - Search for keywords like `error`, `invalid`, `SELECT`, `SQL`, etc.  
  - Extract useful data from responses.  
- Use **Logger++ (BApp Store extension)** for logging and advanced filtering.  

### 2. Using Burp Scanner *(Pro version)*  
- Automatically detects leaked data like private keys, email addresses, and backup files.  
- Supports live scanning while browsing or scheduled scans.  

### 3. Burp's Engagement Tools  
Access via **right-click > Engagement tools** in Burp Suite. Useful tools include:  
- **Search** – Find keywords, use regex, and perform negative searches.  
- **Find Comments** – Extract developer comments from responses.  
- **Discover Content** – Find hidden directories and files not listed on the website.  

### 4. Engineering Informative Responses  
- Error messages can reveal useful details. Study how they change based on input.  
- In some cases, you can manipulate errors to extract arbitrary data.  
- Example: Submitting an invalid parameter may trigger a debug response with sensitive info.  


# 3. Common Sources of Information Disclosure  

Information disclosure can occur in various parts of a website. Below are some key areas where sensitive data might be exposed.  

## 1. Files for Web Crawlers  
- **/robots.txt** and **/sitemap.xml** often list directories that should be avoided by crawlers, which may contain sensitive information.  
- These files are usually not linked within the site but can be accessed manually.  

## 2. Directory Listings  
- Some web servers list directory contents if no index page exists.  
- This can reveal sensitive files like temporary files or crash dumps.  
- While not a direct vulnerability, it can expose restricted resources.  

## 3. Developer Comments  
- In-line HTML comments may be left behind in production.  
- These can contain hints about hidden directories or application logic.  
- Use Burp Suite or browser developer tools to inspect them.  

## 4. Error Messages  
- **Verbose error messages** often disclose:  
  - Expected input types and exploitable parameters.  
  - Technology stack details (e.g., database type, server version).  
  - Default configurations or known exploits.  
- Differences in error responses can reveal application behavior (e.g., SQL injection, username enumeration).  

> The idea is to poking the web and see how it handle invalid parameters, values. If not hadnle properly, we might be able to see the **Error message** which can contain information about the target.

## 5. Debugging Data  
- Debug messages can contain:  
  - Session variables, hostnames, credentials, file paths, and encryption keys.  
- Debug logs stored separately may reveal the application's runtime state.  

> There a chance we can find the Debug page, which contain useful infromation, in the page source code **commend**, or by fuzzing directories, files.

## 6. User Account Pages  
- Normally, users can only access their own data.  
- Logic flaws may allow access to other users' information (e.g., IDOR vulnerabilities).  
- Example:  `GET /user/personal-info?user=carlos`
- If poorly implemented, changing `user=carlos` could expose another user's data.  

## 7. Source Code Disclosure via Backup Files  
- Attackers gaining access to source code can:  
- Understand the application’s logic.  
- Extract hard-coded secrets (e.g., API keys, database credentials).  
- Backup files may be accessible due to misconfigurations (e.g., `.php~`, `.bak` files).  

> Usualy invol fuzzign directoies and files

## 8. Insecure Configuration  
- Misconfigured third-party technologies can introduce vulnerabilities.  
- Debugging options might be enabled in production (e.g., HTTP TRACE method).  
- **TRACE** responses can expose internal authentication headers.  

> For example, if the tagert is **misconfigure to accept TRACE method**, when using **TRACE** method to it, we might be able to see some **customized** `HEADER`, which can be come handy.

## 9. Version Control History  
- Some websites accidentally expose **.git** directories.  
- If accessible, an attacker can:  
- Download the `.git` folder and reconstruct version history.  
- View logs, code changes, and potentially sensitive data.  

> If the **.git** directory is exposed, we can download it to our local git environmetn. (If using linux, try command `wget` to download). Once download, we can investigate further (If using linux, try tool called **git-cola** - UI friendly, then try to access version control history, revoke some commits to see more information, ...)
