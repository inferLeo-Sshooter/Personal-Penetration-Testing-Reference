## DNS

| Component | Description |
|-----------|-------------|
| **DNS Purpose** | Resolves domain names (e.g., hackthebox.com) to IP addresses without central database |
| **Server Types** | DNS root, Authoritative/Non-authoritative name servers, Caching, Forwarding, Resolver |
| **Distribution** | Globally distributed across thousands of name servers like distributed phone books |
| **Security Risk** | Mainly unencrypted; vulnerable to local WLAN and ISP interception |
| **Encryption Solutions** | **DoT**: DNS over TLS<br>**DoH**: DNS over HTTPS<br>**DNSCrypt**: Computer-to-nameserver encryption |
| **Additional Data** | Stores service information beyond IP mapping (email servers, name servers, etc.) |

## DNS Server Types
1. **Root Server**: Handles top-level domains (TLDs) like .com, .org. There are 13 globally managed root servers.
2. **Authoritative Name Server**: Responsible for answering queries for its domain zone. Data is considered final.
3. **Non-authoritative Name Server**: Relays cached information or forwards the request using recursion.
4. **Caching Server**: Temporarily stores results to reduce external DNS lookups.
5. **Forwarding Server**: Forwards queries to another DNS server.
6. **Resolver**: Local client that performs DNS queries, typically within a device or router.

## Footprinting the Service

DNS server footprinting occurs through our query requests. First, we can query a DNS server for known name servers using NS records and the `@` character to specify the target server. This reveals other DNS servers we can query for records. These additional servers may have different configurations and serve different zones.

- **DIG - NS Query**: `dig ns inlanefreight.htb @10.129.14.128`
- **DIG - Version Query**: `dig CH TXT version.bind 10.129.120.85`
- **DIG - ANY Query**: `dig any inlanefreight.htb @10.129.14.128`

The **SOA** record is located in a domain's zone file and specifies who is responsible for the operation of the domain and how DNS information for the domain is managed.

- `dig soa www.inlanefreight.com`

Zone transfer is the process of transferring DNS zones between servers, typically over TCP port 53, abbreviated as `AXFR` (Asynchronous Full Transfer Zone). Since DNS failures can severely impact companies, zone files are maintained identically across multiple name servers. When changes occur, all servers must synchronize to ensure data consistency.

The original zone data resides on a `primary` name server. For reliability and load distribution, `secondary` name servers are deployed (some TLDs mandate at least two servers for Second Level Domains). DNS modifications only occur on the primary server, either manually or through dynamic database updates.

In the master-slave relationship, a `master` serves as the direct synchronization source, while a `slave` obtains zone data from masters. Primaries are always masters; secondaries can be both slaves and masters. Slaves periodically fetch the zone's `SOA` record from masters (typically hourly) and compare serial numbers. Mismatched serial numbers indicate data inconsistency, triggering synchronization.

- **DIG - AXFR Zone Transfer**: `dig axfr inlanefreight.htb @10.129.14.128`
- **DIG - AXFR Zone Transfer - Internal**: `dig axfr internal.inlanefreight.htb @10.129.14.128`
- **Subdomain Brute Forcing**: `for sub in $(cat /opt/useful/seclists/Discovery/DNS/subdomains-top1million-110000.txt);do dig $sub.inlanefreight.htb @10.129.14.128 | grep -v ';\|SOA' | sed -r '/^\s*$/d' | grep $sub | tee -a subdomains.txt;done`
- **DNSenum**: `dnsenum --dnsserver 10.129.14.128 --enum -p 0 -s 0 -o subdomains.txt -f /opt/useful/seclists/Discovery/DNS/subdomains-top1million-110000.txt inlanefreight.htb`

## Domain Takeovers & Subdomain Enumeration

### Domain Takeover

Registering expired/non-existent domains to control another domain. Attackers can host malicious content or launch phishing campaigns using the claimed domain.

### Subdomain Takeover

Occurs via DNS CNAME records that map subdomains to third-party services (AWS, GitHub, Akamai, Fastly, CDNs).

**Example:**
```shell-session
sub.target.com.   60   IN   CNAME   anotherdomain.com
```

**Vulnerability:** If `anotherdomain.com` expires while the CNAME record remains, anyone registering `anotherdomain.com` gains full control over `sub.target.com` until the DNS record is updated.

**Attack Vector:** Organizations using third-party hosting often create subdomains pointing to external services, creating takeover opportunities when those external domains expire.

### Subdomain Enumeration

### Tools

**Subfinder** - Scrapes subdomains from open sources (e.g., DNSdumpster):

**Subbrute** - DNS brute-forcing with custom resolvers (ideal for internal pentests without Internet access):

## DNS Spoofing (DNS Cache Poisoning)

Altering legitimate DNS records to redirect traffic to fraudulent websites.

### Attack Methods

- **MITM Attack:** Intercept user-DNS server communication to redirect traffic
- **DNS Server Exploitation:** Compromise DNS server to modify records directly

### Local DNS Cache Poisoning

Use MITM tools like **Ettercap** or **Bettercap** on local networks.

**Ettercap Setup:**

1. **Edit DNS mapping** (`/etc/ettercap/etter.dns`):
```shell-session
inlanefreight.com      A   192.168.225.110
*.inlanefreight.com    A   192.168.225.110
```

2. **Scan and target hosts:**
   - `Hosts > Scan for Hosts`
   - Target1: Victim IP (e.g., `192.168.152.129`)
   - Target2: Gateway IP (e.g., `192.168.152.2`)

3. **Activate attack:**
   - `Plugins > Manage Plugins > dns_spoof`

**Result:** Victims from target machine visiting `inlanefreight.com` are redirected to attacker's IP (`192.168.225.110`):

```cmd-session
C:\>ping inlanefreight.com
Pinging inlanefreight.com [192.168.225.110] with 32 bytes of data:
Reply from 192.168.225.110: bytes=32 time<1ms TTL=64
```
