## SMTP

## SMTP Protocol Overview

| **Component** | **Description** |
|---|---|
| **Protocol** | Simple Mail Transfer Protocol (SMTP) - for sending emails in IP networks |
| **Architecture** | Client-server based, operates between email clients/servers or between SMTP servers |

## Ports and Security

| **Port** | **Usage** | **Security** |
|---|---|---|
| **25** | Default SMTP port | Unencrypted (plaintext) |
| **587** | Authenticated users | STARTTLS encryption |
| **465** | Encrypted connections | SSL/TLS encryption |

## SMTP Components

| **Component** | **Full Name** | **Function** |
|---|---|---|
| **MUA** | Mail User Agent | Email client that converts emails into headers and body |
| **MSA** | Mail Submission Agent (Relay server) | Validates email origin and authenticity |
| **MTA** | Mail Transfer Agent | Handles sending/receiving, spam checking, storage, DNS lookups |
| **MDA** | Mail Delivery Agent | Delivers emails to recipient mailboxes |

## Key Features

| **Feature** | **Details** |
|---|---|
| **Authentication** | ESMTP with SMTP-Auth to prevent spam |
| **Encryption** | SSL/TLS encryption to protect data transmission |
| **Vulnerability** | Open Relay Attacks due to misconfigurations |
| **Process Flow** | MUA → MSA → MTA → DNS lookup → Destination MTA → MDA → Mailbox |

## SMTP Disadvantages

| **Disadvantage** | **Description** | **Impact** |
|---|---|---|
| **1. No Delivery Confirmation** | SMTP doesn't return usable delivery confirmation; only English error messages with undelivered headers | Poor delivery tracking and user experience |
| **2. No Authentication** | Users aren't authenticated on connection; sender addresses are unreliable | Enables spam, mail spoofing with fake sender addresses |

## Security Solutions

| **Technology** | **Full Name** | **Purpose** |
|---|---|---|
| **DKIM** | DomainKeys Identified Mail | Email authentication and verification protocol |
| **SPF** | Sender Policy Framework | Prevents email spoofing by validating sender IPs |
| **ESMTP** | Extended SMTP | Enhanced SMTP with security features |

## ESMTP Security Process

| **Step** | **Command/Action** | **Result** |
|---|---|---|
| **1. Greeting** | `EHLO` command | Initiates ESMTP session |
| **2. Encryption** | `STARTTLS` command | Establishes SSL-protected connection |
| **3. Authentication** | `AUTH PLAIN` extension | Safe authentication over encrypted channel |

## Additional Security Measures

| **Measure** | **Action** |
|---|---|
| **Spam Detection** | Suspicious emails rejected or moved to quarantine/spam folder |
| **Connection Encryption** | Entire SMTP connection encrypted via TLS after STARTTLS |
| **Modern Usage** | ESMTP is the standard (when people say "SMTP," they usually mean ESMTP) |


## Footprinting the Service

- The default Nmap scripts include smtp-commands, which uses the EHLO command to list all possible commands that can be executed on the target SMTP server: `sudo nmap 10.129.14.128 -sC -sV -p25`
- We can also use the smtp-open-relay NSE script to identify the target SMTP server as an open relay using 16 different tests: `sudo nmap 10.129.14.128 -p25 --script smtp-open-relay -v`
- Use **msfconsole** to enum user: `auxiliary/scanner/smtp/smtp_enum`

## Additional info

Mail servers handle and deliver email over networks/Internet, receiving emails from clients and sending them to other servers or delivering them to client devices.

### Email Protocols

**SMTP (Simple Mail Transfer Protocol):**
- Delivers emails from clients to servers and between servers
- Used when pressing "Send" in email applications

**POP3 (Post Office Protocol 3):**
- Downloads emails from server to local device
- **Default behavior:** Removes downloaded messages from server
- Makes multi-device access difficult (messages stored locally)
- Can be configured to keep server copies

**IMAP4 (Internet Message Access Protocol 4):**
- Downloads emails from server to local device
- **Default behavior:** Keeps messages on server
- Enables easy multi-device email access

### Key Difference
- **POP3:** Best for single-device access (downloads and removes)
- **IMAP4:** Best for multi-device access (downloads but retains on server)

## Email Enumeration

Email servers are complex, often using cloud services (Microsoft 365, G-Suite) or custom implementations. Attack approach varies by service type.

### Identify Mail Servers (MX Records)

**Using host:**
```shell-session
host -t MX hackthebox.eu
host -t MX microsoft.com
```

**Using dig:**
```shell-session
dig mx plaintext.do | grep "MX" | grep -v ";"
dig mx inlanefreight.com | grep "MX" | grep -v ";"
```

**Resolve mail server IP:**
```shell-session
host -t A mail1.inlanefreight.htb
```

### Service Identification

MX records reveal provider types:
- `aspmx.l.google.com` → G-Suite
- `microsoft-com.mail.protection.outlook.com` → Microsoft 365
- `mx.zoho.com` → Zoho
- Custom domains → Self-hosted servers

**Cloud vs. Custom:** Cloud providers use proprietary implementations with modern auth (unique attack vectors). Custom servers may have misconfigurations allowing traditional protocol attacks.

### Common Mail Server Ports

| Port | Service |
|------|---------|
| 25 | SMTP (unencrypted) |
| 110 | POP3 (unencrypted) |
| 143 | IMAP4 (unencrypted) |
| 465 | SMTP (encrypted) |
| 587 | SMTP (encrypted/STARTTLS) |
| 993 | IMAP4 (encrypted) |
| 995 | POP3 (encrypted) |

**Scan with Nmap:**
```shell-session
sudo nmap -Pn -sV -sC -p25,143,110,465,587,993,995 10.129.14.128
```

## Email Misconfigurations

Email services use authentication for sending/receiving. Misconfigurations occur when SMTP allows anonymous authentication or supports user enumeration protocols.

### User Enumeration Commands

**VRFY** - Checks email username validity:
```shell-session
telnet 10.10.110.20 25
VRFY root
252 2.0.0 root

VRFY new-user
550 5.1.1 <new-user>: Recipient address rejected: User unknown
```

**EXPN** - Lists distribution list members:
```shell-session
EXPN support-team
250 2.0.0 carol@inlanefreight.htb
250 2.1.5 elisa@inlanefreight.htb
```

**RCPT TO** - Identifies email recipients:
```shell-session
MAIL FROM:test@htb.com
250 2.1.0 test@htb.com... Sender ok

RCPT TO:john
250 2.1.5 john... Recipient ok

RCPT TO:julio
550 5.1.1 julio... User unknown
```

**POP3 USER** - Enumerates users via POP3:
```shell-session
telnet 10.10.110.20 110
USER john
+OK

USER julio
-ERR
```

### Automated Enumeration

**smtp-user-enum** tool: `(Also, remember to add IP to /etc/hosts file)`
```shell-session
smtp-user-enum -M RCPT -U userlist.txt -D inlanefreight.htb -t 10.129.203.7
```
- `-M`: Mode (VRFY, EXPN, RCPT)
- `-U`: Username list file
- `-D`: Domain
- `-t`: Target IP

Enumerated usernames enable password spraying, brute-forcing, or credential guessing attacks.

## Password Attacks

We can use `Hydra` to perform a password spray or brute force against email services such as `SMTP, POP3, or IMAP4`. First, we need to get a username list and a password list and specify which service we want to attack. Let us see an example for `POP3`.

`[!bash!]$ hydra -L users.txt -p 'Company01!' -f 10.10.110.20 pop3`

If cloud services support SMTP, POP3, or IMAP4 protocols, we may be able to attempt to perform password spray using tools like `Hydra`, but these tools are usually blocked. We can instead try to use custom tools such as `o365spray` or `MailSniper` for Microsoft Office 365 or `CredKing` for Gmail or Okta. Keep in mind that these tools need to be up-to-date because if the service provider changes something (which happens often), the tools may not work anymore.

## Open Relay

An open relay SMTP server allows anyone to send emails through it without authentication, which attackers can exploit for phishing.

### Attack Scenarios
- Send emails as non-existing users
- Spoof legitimate email addresses
- Impersonate company notification addresses with phishing links

### Detection

**Nmap smtp-open-relay script:**
```shell-session
nmap --script smtp-open-relay -p25 <target>
```

Identifies if an SMTP server allows open relay, enabling potential email spoofing and phishing campaigns.

