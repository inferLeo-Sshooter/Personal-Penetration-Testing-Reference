## SNMP

## SNMP Protocol Overview

| **Aspect** | **Details** |
|---|---|
| **Full Name** | Simple Network Management Protocol |
| **Primary Purpose** | Monitor network devices |
| **Secondary Functions** | Handle configuration tasks, change settings remotely |
| **Current Version** | SNMPv3 (increased security and complexity) |

## SNMP-Enabled Devices

| **Device Type** | **Examples** |
|---|---|
| **Network Equipment** | Routers, switches |
| **Infrastructure** | Servers |
| **IoT Devices** | Various smart devices |
| **General** | Any device supporting SNMP standard |

## SNMP Communication

| **Communication Type** | **Port** | **Direction** | **Purpose** |
|---|---|---|---|
| **Standard Queries** | UDP 161 | Client → Server | Information exchange, control commands |
| **SNMP Traps** | UDP 162 | Server → Client | Unsolicited event notifications |

## SNMP Functionality

| **Feature** | **Description** |
|---|---|
| **Information Exchange** | Pure data retrieval and monitoring |
| **Control Commands** | Set specific values, change device options/settings |
| **Agent Communication** | Uses agents for command transmission |
| **Trap Mechanism** | Server sends data packets without explicit client requests |
| **Event-Driven** | Traps triggered by specific server-side events |

## SNMP Requirements

| **Requirement** | **Details** |
|---|---|
| **Object Addressing** | SNMP objects must have unique addresses |
| **Mutual Knowledge** | Addresses must be known by both client and server |
| **Prerequisites** | Absolute requirement for successful data transmission and monitoring |

## Communication Models

| **Model** | **Characteristics** |
|---|---|
| **Classical** | Client actively requests information from server |
| **SNMP Enhanced** | Server can send unsolicited data via traps |


## MIB

## MIB (Management Information Base) Overview

| **Aspect** | **Details** |
|---|---|
| **Full Name** | Management Information Base |
| **Purpose** | Ensure SNMP access works across manufacturers and client-server combinations |
| **Format** | Independent format for storing device information |
| **File Type** | Text file with standardized tree hierarchy |

## MIB Structure & Components

| **Component** | **Description** |
|---|---|
| **File Format** | ASCII text file written in ASN.1 (Abstract Syntax Notation One) |
| **Hierarchy** | Standardized tree structure listing all queryable SNMP objects |
| **Object Listing** | Contains all SNMP objects of a device |

## Object Identifier (OID) Information

| **OID Element** | **Purpose** |
|---|---|
| **Unique Address** | Provides necessary unique identification |
| **Name** | Object naming for identification |
| **Type Information** | Specifies the data type used |
| **Access Rights** | Defines permissions for object access |
| **Description** | Explains the object's function and purpose |

## MIB Functionality

| **Function** | **Details** |
|---|---|
| **Data Storage** | MIBs do not contain actual data |
| **Information Mapping** | Explains where to find specific information |
| **Value Description** | Defines what values are returned for specific OIDs |
| **Data Type Specification** | Indicates which data types are used |
| **Standardization** | Ensures cross-platform compatibility |

## MIB Characteristics

| **Characteristic** | **Benefit** |
|---|---|
| **Manufacturer Independent** | Works across different device manufacturers |
| **Client-Server Flexible** | Compatible with various client-server combinations |
| **Standardized Format** | Consistent structure and syntax |
| **Comprehensive Documentation** | Complete object information and metadata |

## OID

## OID (Object Identifier) Overview

| **Aspect** | **Details** |
|---|---|
| **Definition** | Represents a node in a hierarchical namespace |
| **Identification** | Sequence of numbers uniquely identifies each node |
| **Tree Position** | Numbers allow determination of node's position in tree |
| **Specificity** | Longer chain = more specific information |

## OID Structure & Format

| **Element** | **Description** |
|---|---|
| **Composition** | Consists of integers |
| **Notation** | Usually concatenated by dot notation |
| **Hierarchy** | Tree-based hierarchical structure |
| **Uniqueness** | Each sequence uniquely identifies a single node |

## OID Tree Characteristics

| **Feature** | **Details** |
|---|---|
| **Node Content** | Many nodes contain only references to nodes below them |
| **Empty Nodes** | Some nodes have no data except downward references |
| **Tree Navigation** | Structure allows traversing from general to specific |
| **Chain Length** | Longer chains provide more granular information |

## OID Resources

| **Resource** | **Purpose** |
|---|---|
| **Object Identifier Registry** | Lookup location for MIBs and associated OIDs |
| **MIB Association** | Links OIDs to their corresponding MIB definitions |
| **Reference Database** | Central repository for OID information |

## OID Examples (Concept)

| **Chain Length** | **Information Level** |
|---|---|
| **Short Chain** | General/broad information |
| **Medium Chain** | Intermediate specificity |
| **Long Chain** | Highly specific information |

## SNMPv1

## SNMPv1 Overview

| **Aspect** | **Details** |
|---|---|
| **Full Name** | SNMP version 1 |
| **Version Status** | First version of SNMP protocol |
| **Current Usage** | Still used in many small networks |
| **Primary Purpose** | Network management and monitoring |

## SNMPv1 Capabilities

| **Function** | **Description** |
|---|---|
| **Information Retrieval** | Supports retrieval of information from network devices |
| **Device Configuration** | Allows configuration of network devices |
| **Trap Notifications** | Provides traps (event notifications) |
| **Network Monitoring** | Basic network management functionality |

## SNMPv1 Major Security Flaws

| **Security Issue** | **Impact** | **Risk Level** |
|---|---|---|
| **No Built-in Authentication** | Anyone accessing the network can read and modify data | **High** |
| **No Encryption Support** | All data sent in plain text | **Critical** |
| **Data Interception** | Network traffic can be easily intercepted | **High** |
| **Unauthorized Access** | No mechanism to verify user identity | **High** |

## SNMPv1 Characteristics

| **Characteristic** | **Details** |
|---|---|
| **Protocol Maturity** | Legacy protocol with basic functionality |
| **Network Scope** | Suitable for small, trusted networks only |
| **Security Model** | Community-based (shared secret strings) |
| **Data Transmission** | Unencrypted, plaintext communication |

## SNMPv1 Limitations

| **Limitation** | **Consequence** |
|---|---|
| **Plain Text Communication** | Credentials and data visible to network sniffers |
| **No User Authentication** | Cannot verify legitimate users vs. attackers |
| **No Data Integrity** | Cannot detect if data has been modified in transit |
| **No Access Control** | Limited ability to restrict operations by user |

## SNMPv2

## SNMPv2 Overview

| **Aspect** | **Details** |
|---|---|
| **Current Version** | SNMPv2c (community-based SNMP) |
| **Version History** | Existed in different versions, v2c is the surviving version |
| **Extension Meaning** | "c" stands for community-based SNMP |
| **Functionality** | Extended with additional functions from discontinued party-based SNMP |

## SNMPv2 Security Characteristics

| **Security Aspect** | **Details** | **Comparison** |
|---|---|---|
| **Security Level** | On par with SNMPv1 | No improvement over SNMPv1 |
| **Community String** | Transmitted in plain text | Same vulnerability as SNMPv1 |
| **Encryption** | No built-in encryption | Critical security flaw |
| **Authentication** | Community-based authentication only | Limited security model |

## SNMPv2 Major Security Issues

| **Issue** | **Impact** | **Risk Level** |
|---|---|---|
| **Plain Text Community String** | Security credentials transmitted without encryption | **Critical** |
| **No Built-in Encryption** | All communication vulnerable to interception | **High** |
| **Legacy Security Model** | Same fundamental flaws as SNMPv1 | **High** |

## SNMPv2 Evolution

| **Component** | **Status** | **Details** |
|---|---|---|
| **SNMPv2c** | **Active** | Current version in use |
| **Party-based SNMP** | **Discontinued** | No longer in use |
| **Additional Functions** | **Integrated** | Features from party-based version added to v2c |

## SNMPv2 Protocol Issues

| **Problem Area** | **Specific Issue** |
|---|---|
| **Initial Protocol Design** | Significant security problems from original implementation |
| **Community String Security** | Provides security but transmitted without protection |
| **Backward Compatibility** | Maintains same security weaknesses as SNMPv1 |

## SNMPv3

## SNMPv3 Overview

| **Aspect** | **Details** |
|---|---|
| **Version** | SNMPv3 (latest SNMP version) |
| **Security Improvement** | Enormously increased security compared to previous versions |
| **Trade-off** | Higher security comes with increased complexity |

## SNMPv3 Security Features

| **Security Feature** | **Implementation** | **Benefit** |
|---|---|---|
| **Authentication** | Username and password based | Verifies user identity |
| **Encryption** | Data transmission encryption | Protects data confidentiality |
| **Pre-shared Key** | Key-based encryption method | Secure data transmission |

## SNMPv3 vs SNMPv2c Comparison

| **Aspect** | **SNMPv3** | **SNMPv2c** |
|---|---|---|
| **Security Level** | High (enormous increase) | Low (plain text) |
| **Authentication** | Username/password | Community string only |
| **Encryption** | Built-in encryption | No encryption |
| **Configuration** | Significantly more complex | Simpler configuration |
| **Configuration Options** | Many more options | Limited options |

## SNMPv3 Characteristics

| **Characteristic** | **Details** |
|---|---|
| **Security Enhancement** | Addresses major flaws of SNMPv1 and SNMPv2c |
| **Complexity Increase** | Same extent as security improvement |
| **Configuration Demand** | Requires more detailed setup and management |
| **Industry Standard** | Recommended version for secure environments |

## SNMPv3 Implementation Considerations

| **Consideration** | **Impact** |
|---|---|
| **Setup Complexity** | Requires more technical expertise |
| **Configuration Time** | Longer initial setup compared to v2c |
| **Security Benefit** | Significantly improved network security |
| **Modern Networks** | Essential for security-conscious environments |

## Community Strings

## Community Strings Overview

| **Aspect** | **Details** |
|---|---|
| **Function** | Act as passwords for SNMP access control |
| **Purpose** | Determine whether requested information can be viewed |
| **Security Model** | Basic authentication mechanism for SNMPv1 and SNMPv2c |

## Current Usage & Migration Challenges

| **Challenge** | **Details** | **Impact** |
|---|---|---|
| **SNMPv2 Persistence** | Many organizations still using SNMPv2 | Continued security vulnerabilities |
| **Migration Complexity** | Transition to SNMPv3 is very complex | Delayed security improvements |
| **Service Continuity** | Services must remain active during transition | Cannot afford downtime |
| **Administrator Concerns** | Creates problems administrators want to avoid | Resistance to upgrading |

## Security Vulnerabilities

| **Vulnerability** | **Technical Details** | **Risk Level** |
|---|---|---|
| **No Encryption** | Community strings sent in plain text | **Critical** |
| **Network Interception** | Strings can be intercepted during transmission | **High** |
| **Readable Credentials** | Captured strings are immediately usable | **High** |
| **Persistent Exposure** | Every SNMP transaction exposes credentials | **High** |

## Administrator Knowledge Gaps

| **Knowledge Gap** | **Consequence** |
|---|---|
| **Information Acquisition Methods** | Don't understand how data can be obtained |
| **Attack Methodologies** | Unaware of how attackers exploit SNMP |
| **Risk Assessment** | Approach seems inexplicable due to lack of awareness |
| **Security Implications** | Underestimate the severity of plain text transmission |

## Organizational Impact

| **Impact Area** | **Details** |
|---|---|
| **Security Posture** | Weakened by continued use of vulnerable protocols |
| **Risk Management** | Administrators avoid problems rather than address them |
| **Network Exposure** | Every SNMP communication potentially compromises security |
| **Upgrade Resistance** | Complexity creates barrier to implementing proper security |

## Footprinting the Service

- For footprinting SNMP, we can use tools like `snmpwalk`, `onesixtyone`, and `braa`.
- `Snmpwalk` is used to query the `OIDs` with their information.
- `Onesixtyone` can be used to brute-force the names of the community strings since they can be named arbitrarily by the administrator.
- Since these community strings can be bound to any source, identifying the existing community strings can take quite some time.

- **SNMPwalk**: `snmpwalk -v2c -c public 10.129.14.128`
- If we do not know the community string, we can use **onesixtyone** and **SecLists** wordlists to identify these community strings: `onesixtyone -c /opt/useful/seclists/Discovery/SNMP/snmp.txt 10.129.14.128`
- Once we know a community string, we can use it with `braa` to brute-force the individual OIDs and enumerate the information behind them:

```
$ sudo apt install braa
$ braa <community string>@<IP>:.1.3.6.*   # Syntax
$ braa public@10.129.14.128:.1.3.6.*
```
