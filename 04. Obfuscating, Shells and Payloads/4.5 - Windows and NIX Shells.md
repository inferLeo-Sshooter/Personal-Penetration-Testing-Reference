## Enumerating Windows & Fingerprinting Methods

- We can determine if the host is a Window or not by `ping` and `TTL`. When pinging, a typical TTL response from a Windows host would be `128`
- This value may `not always be exact`, especially if you are `not in the same layer three network` as the target.
- We can utilize this value since most hosts will never be more than 20 hops away from your point of origin, so there is little chance of the TTL counter dropping into the acceptable values of another OS type

## Bats, DLLs, & MSI Files

Windows hosts offer multiple payload options including DLLs, batch files, MSI packages, and PowerShell scripts. All are executable but serve different purposes. Consider your delivery mechanism when selecting payload types.

**Payload Types:**

* **DLLs** - Dynamic Linking Libraries provide shared code for multiple programs. Malicious DLL injection or hijacking can elevate privileges to SYSTEM and bypass User Account Controls.

* **Batch** - Text-based DOS scripts with `.bat` extensions used for automated command-line tasks. Can open ports, connect to attack boxes, and perform enumeration.

* **VBS** - VBScript is a lightweight Visual Basic-based language. Though dated and disabled in modern browsers, it's still used in phishing attacks to execute code via macros or user interaction.

* **MSI** - Installation database files for Windows Installer. Crafted malicious `.msi` files can be executed with `msiexec` to gain elevated access like reverse shells.

* **PowerShell** - Modern shell environment and .NET-based scripting language. Provides extensive options for gaining shells and execution during penetration testing.

Now that we understand what each type of Windows file can be used for let's discuss some basic tools, tactics, and procedures for building our payloads and delivering them onto the host to land a shell.

Beyond web-drive-by, phishing, or dead drops, Windows hosts offer several delivery avenues:

- `Impacket`: Python toolset for network protocol interaction (psexec, smbclient, wmi, Kerberos, SMB server)
- `Payloads All The Things`: Quick oneliners for file transfers

- `Remote execution via MSF`: Automated payload building, staging, and execution through exploit modules
- `Other Protocols`: FTP, TFTP, HTTP/S can provide file upload capabilities

## CMD-Prompt and Power[Shell]s

Windows hosts provide two default shell options. Choose based on your needs at the time.

| Feature | CMD Shell | PowerShell |
|---------|-----------|------------|
| **Origin** | Original MS-DOS shell | Expanded capabilities shell |
| **Purpose** | Basic interaction and IT operations | Advanced operations with .NET integration |
| **Automation** | Simple batch files | Cmdlets and self-sufficient modules |
| **Input/Output** | Text-based | .NET objects |
| **Command History** | No logging (more stealthy) | Keeps command records |
| **Command Support** | MS-DOS commands only | MS-DOS commands + .NET-based commands |
| **Restrictions** | Not affected by Execution Policy or UAC | Subject to Execution Policy and UAC limitations |
| **Windows Compatibility** | All Windows versions (including XP and older) | Windows 7 and newer only |
| **Stealth Factor** | Leaves fewer traces | More detectable due to logging |

**Key Takeaway:** CMD is better for stealth and legacy systems, while PowerShell offers more functionality on modern systems.

| Use CMD when: | Use PowerShell when:|
|-|-|
|You are on an older host that may not include PowerShell.<br>When you only require simple interactions/access to the host.<br>When you plan to use simple batch files, net commands, or MS-DOS native tools.<br>When you believe that execution policies may affect your ability to run scripts or other actions on the host.|You are planning to utilize cmdlets or other custom-built scripts.<br>When you wish to interact with .NET objects instead of text output.<br>When being stealthy is of lesser concern.<br>If you are planning to interact with cloud-based services and hosts.<br>If your scripts set and use Aliases.|

# 2. about Unix/Linux shell

## Common Considerations

Gaining shell access to systems typically involves exploiting application vulnerabilities to deliver payloads. When targeting Unix/Linux systems, consider:

* Linux distribution in use
* Available shells & programming languages  
* System's network function/role
* Hosted applications
* Known vulnerabilities

## Spawning Interactive Shells

After establishing a shell session, we often start with a limited/jail shell. We used Python to spawn a TTY bourne shell for more commands and a proper prompt—a common scenario in practice.

When Python isn't available on systems with limited shells, several methods can spawn interactive shells. Note that `/bin/sh` or `/bin/bash` references can be replaced with whatever shell interpreter binaries exist on the target system. Most Linux systems natively include `bourne shell` (`/bin/sh`) and `bourne again shell` (`/bin/bash`).

**/bin/sh -i:**

- This command will execute the shell interpreter specified in the path in interactive mode (`-i`).

```
/bin/sh -i
sh: no job control in this shell
sh-4.2$
```

---

**Perl:**

- If the programming language `Perl` is present on the system, these commands will execute the shell interpreter specified. **Perl To Shell:**

```
perl —e 'exec "/bin/sh";'
```
or
```
perl: exec "/bin/sh";
```
The command directly above should be run from a script.

---

**Ruby:**

- If the programming language `Ruby` is present on the system, this command will execute the shell interpreter specified:

```
ruby: exec "/bin/sh"
```
The command directly above should be run from a script.

---

**Lua:**

- If the programming language `Lua` is present on the system, we can use the `os.execute` method to execute the shell interpreter specified using the full command below:

```
lua: os.execute('/bin/sh')
```
The command directly above should be run from a script.

---

**AWK:**

- `AWK` is a C-like pattern scanning and processing language present on most UNIX/Linux-based systems, widely used by developers and sysadmins to generate reports. It can also be used to spawn an interactive shell. This is shown in the short awk script below:

```
awk 'BEGIN {system("/bin/sh")}'
```

---

**Find:**

- `Find` is a command present on most Unix/Linux systems widely used to search for & through files and directories using various criteria. It can also be used to execute applications and invoke a shell interpreter.

```
find / -name nameoffile -exec /bin/awk 'BEGIN {system("/bin/sh")}' \;
```

- This use of the find command is searching for any file listed after the `-name` option, then it executes `awk` (`/bin/awk`) and runs the same script we discussed in the awk section to execute a shell interpreter.

---

**Using Exec To Launch A Shell:**

```
find . -exec /bin/sh \; -quit
```

- This use of the find command uses the execute option (`-exec`) to initiate the shell interpreter directly. If `find` can't find the specified file, then no shell will be attained.

---

**VIM:**

- We can set the shell interpreter language from within the popular command-line-based text-editor `VIM`. This is a very niche situation we would find ourselves in to need to use this method, but it is good to know just in case.
- **Vim To Shell:**

```
vim -c ':!/bin/sh'
```

- **Vim Escape:**

```
vim
:set shell=/bin/sh
:shell
```

---

## Execution Permissions Considerations

- In addition to knowing about all the options listed above, we should be mindful of the permissions we have with the shell session's account. We can always attempt to run this command to list the file properties and permissions our account has over any given file or binary:

**Permissions:**

```
ls -la <path/to/fileorbinary>
```

- We can also attempt to run this command to check what `sudo` permissions the account we landed on has: `sudo -l`

```
sudo -l
Matching Defaults entries for apache on ILF-WebSrv:
    env_reset, mail_badpass,
    secure_path=/usr/local/sbin\:/usr/local/bin\:/usr/sbin\:/usr/bin\:/sbin\:/bin

User apache may run the following commands on ILF-WebSrv:
    (ALL : ALL) NOPASSWD: ALL
```

- The sudo -l command above will need a stable interactive shell to run. If you are not in a full shell or sitting in an unstable shell, you may not get any return from it. Not only will considering permissions allow us to see what commands we can execute, but it may also start to give us an idea of potential vectors that will allow us to escalate privileges.
