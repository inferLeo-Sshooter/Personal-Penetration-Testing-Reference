# 1. Reflected Cross-Site Scripting (XSS)

Reflected XSS is a type of Non-Persistent XSS vulnerability where:

- User input is processed by the server and immediately returned in the response without proper sanitization
- The attack is temporary and doesn't persist after page refresh
- Only affects the targeted user, not other visitors
- Typically occurs in features like search functions, error messages, or confirmation messages

**Example:**
```
https://insecure-website.com/search?term=<script>/* Malicious code */</script>
```

When this URL is visited, the server includes the unsanitized script in its response:
```html
<p>You searched for: <script>/* Malicious code */</script></p>
```

The malicious script then executes in the victim's browser in the context of their session with the website.

# 2. Impact of Reflected XSS Attacks

When successful, a reflected XSS attack gives attackers significant control over the victim's browser session. Key impacts include:

* **Complete account compromise** - Attackers can perform any action the victim is authorized to do
* **Data theft** - Access to all information visible to the victim
* **Data manipulation** - Ability to modify any information the victim can change
* **Identity spoofing** - Can initiate interactions with other users that appear to come from the victim

## Attack delivery methods:
* Malicious links on attacker-controlled websites
* Links posted on sites allowing user-generated content
* Links sent via email, social media, or messaging platforms
* Can target specific users or be launched indiscriminately

# 3. How to Find and Test for Reflected XSS

Here's how to test manually:

### 1. **Test Each Input**
Check all input points in HTTP requests — query params, body, URL path, and headers.

### 2. **Inject a Unique Value**
Send a short, random alphanumeric string (e.g. `abc123xy`) to each input. Use **Burp Intruder** to automate this.

### 3. **Look for Reflection**
Check if your string shows up in the server response. This means it’s being reflected.

### 4. **Identify the Context**
See where the value appears in the response — in HTML, JavaScript, attributes, etc. The context affects your payload.

### 5. **Try a Payload**
Based on the context, insert an XSS payload (like `<script>alert(1)</script>`) and see if it triggers. Use **Burp Repeater** for testing.

### 6. **Try Alternatives if Blocked**
If your payload gets filtered, try different techniques depending on how the input is handled.

### 7. **Confirm in Browser**
Test successful payloads in a real browser. Use simple scripts like `alert(document.domain)` to confirm.
