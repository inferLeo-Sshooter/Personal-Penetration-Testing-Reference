# Content Security Policy (CSP)

CSP is a browser security mechanism that mitigates XSS and other attacks by restricting page resources and framing. It's implemented via the `Content-Security-Policy` HTTP header containing directives.

## Mitigating XSS with CSP

**Script restrictions:**
- Allow same-origin scripts: `script-src 'self'`
- Allow specific domain: `script-src https://scripts.normal-website.com`

**Important considerations:**
- Avoid untrusted external domains (like shared CDNs)
- Use security features:
  - Nonces: Random values matching script tags
  - Hashes: Content validation for trusted scripts

**Limitations:**
- Many CSPs block scripts but allow images, enabling CSRF token disclosure
- Some browsers have built-in protections against dangling markup
- Restrictive policies can be bypassed through user interaction

## Bypassing CSP with policy injection

When websites reflect user input into the CSP (typically in the `report-uri` directive), you can inject semicolons `;` to add custom directives. Since `report-uri` is usually last, you must overwrite existing directives to bypass security.

While directly overwriting a `script-src` directive isn't normally possible, Chrome introduced `script-src-elem` which:
- Controls script elements specifically (not events)
- Can overwrite existing `script-src` directives

This creates a vulnerability that allows attackers to bypass script restrictions in affected browsers.

**Example:**

### Lab: Reflected XSS protected by CSP, with CSP bypass (only work on Chrome)

> - On approaching the lab, notice that the web reflect user input using search.
> - However, a payload like: `<img src=x onerror=alert()>` reflected as error image, but no `alert` pop-up.
> - This is because of CSP
> - Inspect the web, and in network tab, we can see the CSP as below:

![image](https://github.com/user-attachments/assets/cbffa39c-0cbb-4d55-bb1c-e3f9acebc26e)

> - Notice that there a parameter `token`.
> - We can control the value of `token` by alter the url: `https://.../?search=asd&token=test`.
> - By leverage this, we can override our own policy by using `;` to escape `token` and `script-src-elem` to add policy - which only happen in Chrome
> - `<script>alert(1)</script>&token=;script-src-elem 'unsafe-inline'`
> - The injection uses the `script-src-elem` directive in CSP. This directive allows you to target just `script` elements. Using this directive, you can overwrite existing `script-src` rules enabling you to inject `unsafe-inline`, which allows you to use inline scripts.
