

# 1. Overview

**Definition & Impact**
File upload vulnerabilities occur when servers allow users to upload files without adequately validating their name, type, contents, or size. These vulnerabilities rank among the most common and dangerous web security issues, frequently receiving High or Critical CVE ratings. The impact depends on which validation aspects fail and what restrictions exist post-upload.

**Key Risk Factors**
- Unauthenticated arbitrary file uploads present the greatest danger
- Weak or missing file validation is the primary cause
- Flawed blacklisting approaches often miss obscure dangerous file types
- Inconsistent validation across website components
- Easily manipulated file properties

**Common Attack Vectors**
- Remote code execution via web shells (worst-case scenario)
- Overwriting critical system files
- Directory traversal combined with uploads
- Denial of Service through excessive file sizes
- XSS or XXE payload delivery

# 2. How web servers handle requests for static files

When a request is made, the server parses the path to identify the file extension, then matches it to a MIME type using a predefined mapping. What happens next depends on the file type and the server's configuration.
- If its Non-executable files (e.g., images, HTML): The server may just send the file's contents to the client in an HTTP response.
- If its Executable files (e.g., PHP) and the server is configured to execute files of this type: then the server executes the file using request data and returns the output.
- If its Executable files, but the server is not configured to execute files of this type, it may return an error—or in misconfigured setups, expose the raw source code.

**Note:**
> - The Content-Type response header may provide clues as to what kind of file the server thinks it has served. If this header hasn't been explicitly set by the application code, it normally contains the result of the file extension/MIME type mapping.
