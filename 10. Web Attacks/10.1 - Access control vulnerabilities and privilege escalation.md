

# 1. Access control

## 1.1 - The basics

Access control restricts who can perform actions or access resources in web applications. It relies on three components:

* **Authentication** - verifies user identity
* **Session management** - tracks user requests  
* **Access control** - authorizes specific actions

Broken access controls are common critical vulnerabilities. Since access control involves complex business, organizational, and legal requirements that humans must translate into technical implementations, errors frequently occur.

**Vertical access controls** restrict sensitive functionality by user type. Administrators might modify any account, while regular users cannot. These enforce business policies like separation of duties and least privilege.

**Horizontal access controls** restrict resource access to specific users. Users can only access their own subset of resources - like viewing their own bank transactions but not others' accounts.

**Context-dependent access controls** restrict actions based on application state or user interaction sequence. For example, preventing shopping cart modifications after payment completion.

# 2. Examples of broken access controls

Broken access control vulnerabilities exist when a user can access resources or perform actions that they are not supposed to be able to.

## 2.1 - Vertical privilege escalation

- Occurs when users gain access to functionality above their permission level. For example, a regular user accessing an admin page to delete accounts represents vertical privilege escalation.

### Unprotected functionality

**Unprotected functionality** is the most basic form of vertical privilege escalation where applications don't enforce protection on sensitive features.

Administrative functions may only be linked from admin welcome pages, but users can still access them by directly browsing to admin URLs like `https://insecure-website.com/admin`. These URLs might be discoverable through:
- `robots.txt` files 
- Wordlist brute-force attacks
- Other disclosed locations

The functionality remains accessible to anyone who finds the URL, regardless of their authorization level.

### Security by obscurity

Security by obscurity attempts to hide sensitive functionality using unpredictable URLs like `https://insecure-website.com/administrator-panel-yb556`. However, this doesn't provide effective access control since URLs can still be discovered.

Applications often leak these hidden URLs through client-side code. For example, JavaScript that conditionally displays admin links based on user roles:

```javascript
var isAdmin = false;
if (isAdmin) {
    var adminPanelTag = document.createElement('a');
    adminPanelTag.setAttribute('href', 'https://insecure-website.com/administrator-panel-yb556');
    adminPanelTag.innerText = 'Admin panel';
}
```

While the link only appears for admin users, the script containing the URL is visible to all users in the browser, exposing the hidden functionality.

### Parameter-based access control methods

**Parameter-based access control** stores user access rights in user-controllable locations after login, such as:
* Hidden fields
* Cookies  
* Query string parameters

The application makes access control decisions based on these submitted values. For example:
`https://insecure-website.com/login/home.jsp?admin=true`
`https://insecure-website.com/login/home.jsp?role=1`

This approach is insecure because users can simply modify these values to gain unauthorized access to administrative functions.

### Broken access control resulting from platform misconfiguration

**Platform misconfiguration** occurs when applications enforce access controls at the platform layer by restricting URLs and HTTP methods based on user roles. For example:

`DENY: POST, /admin/deleteUser, managers`

This rule blocks managers from using POST on `/admin/deleteUser`. However, bypasses can occur when:

Application frameworks support non-standard headers like `X-Original-URL` and `X-Rewrite-URL` that override the original request URL. If front-end controls restrict access based on URL but allow header overrides, attackers can bypass restrictions:

```
POST / HTTP/1.1
X-Original-URL: /admin/deleteUser
```

This request appears to access `/` but actually targets `/admin/deleteUser`, circumventing the platform-level access control.

### HTTP method bypass

**HTTP method bypass** exploits websites that accept different HTTP methods for the same action. If platform-level controls restrict specific URL and method combinations, but the application tolerates alternative methods, attackers can circumvent restrictions.

For example, if access control blocks:
`DENY: POST, /admin/deleteUser, managers`

But the application also accepts GET requests for the same functionality, an attacker can bypass the restriction by using:
`GET /admin/deleteUser`

This works because the platform control only restricts the POST method, while the application accepts multiple methods for performing the same sensitive action.

### Broken access control resulting from URL-matching discrepancies

**URL-matching discrepancies** occur when applications and access control mechanisms handle URL paths differently, creating bypass opportunities:

**Case sensitivity differences**: Access controls may block `/admin/deleteUser` but allow `/ADMIN/DELETEUSER` if the application treats them as equivalent while the security mechanism doesn't.

**File extension handling**: In Spring framework with `useSuffixPatternMatch` enabled (default before v5.3), `/admin/deleteUser.anything` matches `/admin/deleteUser`, potentially bypassing path-based restrictions.

**Trailing slash inconsistencies**: Systems may treat `/admin/deleteUser` and `/admin/deleteUser/` as different endpoints, allowing bypass by adding or removing trailing slashes.

These discrepancies let attackers access restricted functionality when the application's URL matching is more permissive than its access control implementation.

## 2.2 - Horizontal privilege escalation

**Horizontal privilege escalation** occurs when users gain access to resources belonging to other users at the same privilege level, rather than their own resources.

For example, an employee accessing other employees' records instead of just their own represents horizontal privilege escalation.

These attacks often use similar methods to vertical escalation. A user accessing their account via:
`https://insecure-website.com/myaccount?id=123`

Can potentially access another user's account by simply modifying the `id` parameter:
`https://insecure-website.com/myaccount?id=124`

This grants unauthorized access to another user's data and functions without proper authorization checks.
