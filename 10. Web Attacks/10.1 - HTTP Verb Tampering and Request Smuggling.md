# 1. HTTP Verb Tampering

HTTP Verb Tampering exploits web servers that accept multiple HTTP methods by sending malicious requests using unexpected verbs. This can bypass authorization mechanisms and security controls, making it one of several HTTP attacks that target web server configurations through crafted requests.

## 1.1 - Context

HTTP protocol uses various methods (verbs) like `GET` and `POST` to handle requests. While developers typically focus on these common methods, clients can send any HTTP method to test server responses.

If both the web application and server are properly configured to accept only required methods (e.g., `GET`/`POST`), unauthorized methods will trigger error pages - causing poor user experience but not serious security issues.

However, if server configurations aren't restricted and applications don't handle unexpected HTTP methods (like `HEAD`, `PUT`), attackers can exploit this misconfiguration to access unauthorized functionalities or bypass security controls.

## 1.2 - HTTP Verb Tampering

HTTP has `9 different verbs` that can be accepted as HTTP methods by web servers. Other than `GET` and `POST`, the following are some of the commonly used HTTP verbs:

|Verb|	Description|
|-|-|
|`HEAD`|	Identical to a GET request, but its response only contains the headers, without the response body|
|`PUT`|	Writes the request payload to the specified location|
|`DELETE`|	Deletes the resource at the specified location|
|`OPTIONS`|	Shows different options accepted by a web server, like accepted HTTP verbs|
|`PATCH`|	Apply partial modifications to the resource at the specified location|

Some HTTP methods perform sensitive operations like writing (`PUT`) or deleting (`DELETE`) files to the server's webroot directory. If web servers aren't securely configured to handle these methods, attackers can potentially gain server control.

HTTP Verb Tampering attacks are particularly common and critical because they stem from misconfigurations in either the back-end web server or the web application - meaning vulnerabilities can arise from multiple sources, making them more likely to occur.

## 1.3 - Insecure Configurations

The first type of HTTP Verb Tampering vulnerability occurs when web server authentication is configured for specific HTTP methods only, leaving other methods unprotected. 

For example, this configuration requires authentication for particular web pages:

```xml
<Limit GET POST>
    Require valid-user
</Limit>
```

While this protects `GET` and `POST` requests, attackers can use different HTTP methods (like `HEAD`) to bypass authentication entirely. This leads to authentication bypass, allowing unauthorized access to restricted web pages and domains.


## 1.4 - Insecure Coding

The second type of HTTP Verb Tampering vulnerability stems from inconsistent coding practices where developers apply security filters to specific HTTP methods but not others.

For example, when fixing a SQL Injection vulnerability, a developer might apply input sanitization only to GET parameters:

```php
$pattern = "/^[A-Za-z\s]+$/";

if(preg_match($pattern, $_GET["code"])) {
    $query = "Select * from ports where port_code like '%" . $_REQUEST["code"] . "%'";
    ...SNIP...
}
```

The filter validates `$_GET["code"]` for malicious characters, but the actual query uses `$_REQUEST["code"]`, which includes both GET and POST parameters. This creates an inconsistency where attackers can bypass the sanitization filter by sending malicious input via POST while keeping GET parameters clean, still allowing SQL injection attacks to succeed.


**Note:**
> - While both of the above vulnerabilities are found in public, the second one is much more common, as it is due to mistakes made in coding, while the first is usually avoided by secure web server configurations, as documentation often cautions against it.

# 2. Bypassing Basic Authentication

Exploiting HTTP Verb Tampering vulnerabilities is typically straightforward - simply test different HTTP methods to observe server and application responses.

Automated vulnerability scanners reliably detect vulnerabilities from insecure server configurations since bypassing authentication pages is easily identifiable. However, they often miss coding-related vulnerabilities, which require active testing to determine if security filters can be bypassed.

The first vulnerability type stems from insecure web server configurations and can be exploited to bypass HTTP Basic Authentication on protected pages.

## 2.1 - Identify

When we start the exercise at the end of this section, we see that we have a basic `File Manager` web application, in which we can add new files by typing their names and hitting `enter`:

![image](https://github.com/user-attachments/assets/3a456e0a-86f4-4445-b568-8cf6505710ac)


However, suppose we try to delete all files by clicking on the red `Reset` button. In that case, we see that this functionality seems to be restricted for authenticated users only, as we get the following `HTTP Basic Auth` prompt:

![image](https://github.com/user-attachments/assets/f7fedfcb-c5d3-446f-b6fe-f18b52665e8d)

As we do not have any credentials, we will get a 401 Unauthorized page:

![image](https://github.com/user-attachments/assets/b7ab47ed-546a-452e-8e73-da29daa6ac5c)

So, let's see whether we can bypass this with an HTTP Verb Tampering attack. To do so, we need to identify which pages are restricted by this authentication. If we examine the HTTP request after clicking the Reset button or look at the URL that the button navigates to after clicking it, we see that it is at `/admin/reset.php`. So, either the `/admin` directory is restricted to authenticated users only, or only the `/admin/reset.php` page is. We can confirm this by visiting the `/admin` directory, and we do indeed get prompted to log in again. This means that the full `/admin` directory is restricted.

## 2.2 - Exploit

To try and exploit the page, we need to identify the HTTP request method used by the web application. We can intercept the request in Burp Suite and examine it:

![image](https://github.com/user-attachments/assets/65a37067-4b50-4373-bee7-df184e3fef9b)

As the page uses a `GET` request, we can send a `POST` request and see whether the web page allows `POST` requests (i.e., whether the Authentication covers `POST` requests). To do so, we can right-click on the intercepted request in Burp and select `Change Request Method`, and it will automatically change the request into a `POST` request:

![image](https://github.com/user-attachments/assets/9754ea6e-15cb-46c3-9e4c-b70d6f97b205)

Once we do so, we can click `Forward` and examine the page in our browser. Unfortunately, we still get prompted to log in and will get a `401 Unauthorized` page if we don't provide the credentials:

![image](https://github.com/user-attachments/assets/efe0e54a-14a9-4659-9dff-01185d8ee220)

So, it seems like the web server configurations do cover both `GET` and `POST` requests. However, as we have previously learned, we can utilize many other HTTP methods, most notably the `HEAD` method, which is identical to a `GET` request but does not return the body in the HTTP response. If this is successful, we may not receive any output, but the `reset` function should still get executed, which is our main target.

To see whether the server accepts `HEAD` requests, we can send an `OPTIONS` request to it and see what HTTP methods are accepted, as follows:

```
$ curl -i -X OPTIONS http://SERVER_IP:PORT/

HTTP/1.1 200 OK
Date: 
Server: Apache/2.4.41 (Ubuntu)
Allow: POST,OPTIONS,HEAD,GET
Content-Length: 0
Content-Type: httpd/unix-directory
```

As we can see, the response shows `Allow: POST,OPTIONS,HEAD,GET`, which means that the web server indeed accepts `HEAD` requests, which is the default configuration for many web servers. So, let's try to intercept the `reset` request again, and this time use a `HEAD` request to see how the web server handles it:

![image](https://github.com/user-attachments/assets/b3e8d7e1-8946-4729-8445-3e9cc58a62e1)

Once we change `POST` to `HEAD` and forward the request, we will see that we no longer get a login prompt or a `401 Unauthorized` page and get an empty output instead, as expected with a `HEAD` request. If we go back to the `File Manager` web application, we will see that all files have indeed been deleted, meaning that we successfully triggered the `Reset` functionality without having admin access or any credentials:

# 3. Bypassing Security Filters

## 3.1 - Identify

In the `File Manager` web application, if we try to create a new file name with special characters in its name (e.g. `test;`), we get the following message:

![image](https://github.com/user-attachments/assets/0ca894b2-87ce-4086-9d12-a4a7cfcd817d)

This message shows that the web application uses certain filters on the back-end to identify injection attempts and then blocks any malicious requests. No matter what we try, the web application properly blocks our requests and is secured against injection attempts. However, we may try an HTTP Verb Tampering attack to see if we can bypass the security filter altogether.

## 3.2 - Exploit

To try and exploit this vulnerability, let's intercept the request in Burp Suite (Burp) and then use `Change Request Method` to change it to another method:

![image](https://github.com/user-attachments/assets/a5dfbc15-1780-41fd-88ed-aa40716a285c)

This time, we did not get the `Malicious Request Denied!` message, and our file was successfully created:

![image](https://github.com/user-attachments/assets/81955178-bf73-49eb-9d25-a945c628bc60)

To confirm whether we bypassed the security filter, we need to attempt exploiting the vulnerability the filter is protecting: a Command Injection vulnerability, in this case. So, we can inject a command that creates two files and then check whether both files were created. To do so, we will use the following file name in our attack (`file1; touch file2;`):

![image](https://github.com/user-attachments/assets/24a452cb-5820-44df-ae15-5c4100e7cae9)

Then, we can once again change the request method to a `GET` request:

![image](https://github.com/user-attachments/assets/219de088-91d6-4a26-b656-98d11c9be7f4)

Once we send our request, we see that this time both `file1` and `file2` were created:

![image](https://github.com/user-attachments/assets/0b3a5991-4642-4f8e-85be-4abbfd867e62)

This shows that we successfully bypassed the filter through an HTTP Verb Tampering vulnerability and achieved command injection. Without the HTTP Verb Tampering vulnerability, the web application may have been secure against Command Injection attacks, and this vulnerability allowed us to bypass the filters in place altogether.
