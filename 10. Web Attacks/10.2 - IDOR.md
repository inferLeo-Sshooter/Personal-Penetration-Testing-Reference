
# 1. The basics

**Insecure Direct Object References (IDOR)** are common web vulnerabilities that occur when applications expose direct references to objects (files, database resources) that users can manipulate to access unauthorized resources. These vulnerabilities arise from inadequate access control systems and are difficult to detect through automated testing.

For example, if a file download link uses `download.php?file_id=123`, an attacker could potentially access other files by changing the ID to `download.php?file_id=124`. Without proper backend access controls, predictable IDs allow attackers to access resources beyond their permissions. The challenge of building robust access control systems makes IDOR vulnerabilities widespread and often undetected until production.

**What Makes an IDOR Vulnerability**

Direct object references alone aren't vulnerabilities - the real issue is weak access control systems. Many applications restrict access by limiting which pages or APIs users can reach, but fail to verify permissions on the backend when users access resources directly.
IDOR vulnerabilities exist primarily due to lack of backend access control. Without systems like Role-Based Access Control (RBAC) that compare user authentication against resource permissions, attackers can access or modify other users' data through direct object manipulation.
Many developers skip building proper access control, leaving applications protected only by frontend restrictions. This means all users potentially have backend access to all data - only the interface limits what they see. Manual HTTP request manipulation can reveal full data access, making IDOR attacks successful.

**Impact of IDOR Vulnerabilities**

IDOR vulnerabilities can have severe impacts on web applications:

**Information Disclosure**: The most common impact is unauthorized access to private user data like personal files or credit card information. Attackers identify direct references (database IDs, URL parameters) and test patterns to extract data from arbitrary users.

**Data Modification/Deletion**: Beyond viewing data, IDOR can allow attackers to modify or delete other users' information, potentially leading to complete account takeover.

**Privilege Escalation**: IDOR can enable standard users to gain administrator privileges through "Insecure Function Calls." Many applications expose admin-only APIs in frontend code but only disable them on the user interface. If backend systems don't explicitly restrict non-admin access to these functions, attackers can perform unauthorized administrative operations like changing passwords or granting roles, potentially leading to complete application takeover.

The combination of data exposure and potential administrative access makes IDOR vulnerabilities particularly dangerous for web application security.

## 1.2 - Identifying IDORs

### 1: URL Parameters & APIs

The first step in exploiting IDOR vulnerabilities is identifying Direct Object References. When accessing files or resources, examine HTTP requests for object references in URL parameters or APIs (e.g., `?uid=1` or `?filename=file_1.pdf`). These references can also appear in HTTP headers like cookies.

**Basic Testing**: Try incrementing object reference values (`?uid=2` or `?filename=file_2.pdf`) to access other data. Use fuzzing tools to test thousands of variations and identify successful hits to unauthorized files, which indicate IDOR vulnerabilities.

### 2: AJAX Calls

JavaScript frameworks sometimes insecurely expose all function calls in frontend code, using only appropriate ones based on user roles. Admin functions may be disabled but remain discoverable in the JavaScript code.

**Frontend Code Analysis**: Examine JavaScript code for AJAX calls containing direct object references to endpoints or APIs. These hidden functions may not appear through normal HTTP request monitoring but could be vulnerable to IDOR attacks.

**Example**:
```javascript
function changeUserPassword() {
    $.ajax({
        url:"change_password.php",
        type: "post",
        dataType: "json",
        data: {uid: user.uid, password: user.password, is_admin: is_admin},
        success:function(result){
            //
        }
    });
}
```

This function might never execute for non-admin users, but if found in frontend code, it can be tested for IDOR vulnerabilities by attempting to call it directly. The same approach applies to backend code when accessible (e.g., open-source applications).

### 3: Understand Hashing/Encoding

Applications may encode or hash object references instead of using sequential numbers, but these can still be exploitable without proper backend access control.

**Encoded References**: If references use common encoding like base64 (`?filename=ZmlsZV8xMjMucGRm`), decode to reveal the plaintext (`file_123.pdf`), modify the value (`file_124.pdf`), then re-encode (`?filename=ZmlsZV8xMjQucGRm`) to test for IDOR vulnerabilities.

**Hashed References**: References like `download.php?filename=c81e728d9d4c2f636f067f89cc14862c` may appear secure but could be vulnerable if you can determine the hashing method.

**Example**:
```javascript
$.ajax({
    url:"download.php",
    type: "post", 
    dataType: "json",
    data: {filename: CryptoJS.MD5('file_1.pdf').toString()},
    success:function(result){
        //
    }
});
```

Here, the filename is MD5 hashed, allowing you to calculate hashes for other files. Alternatively, use hash identifier tools to determine the algorithm, then hash different filenames to test access. Successfully downloading unauthorized files indicates an IDOR vulnerability.

### 4: Compare User Roles

Advanced IDOR attacks require registering multiple users to compare HTTP requests and object references. This helps understand how parameters and identifiers are calculated, enabling access to other users' data.

**Multi-User Testing**: If User1 can view their salary through this API call:

```json
{
  "attributes" : 
    {
      "type" : "salary",
      "url" : "/services/data/salaries/users/1"
    },
  "Id" : "1",
  "Name" : "User1"
}
```

User2 shouldn't have access to these parameters, but you can test by making the same API call while logged in as User2. If the application only requires a valid session without proper backend access control comparing the caller's session to the requested data, it may return unauthorized information.

**Vulnerability Identification**: Successfully accessing other users' data by replicating API calls indicates an IDOR vulnerability. Even if you can't calculate parameters for other users, identifying weak backend access control reveals potential exploitation paths through other object references.
