# Network Services

During penetration tests, computer networks have services for managing, editing, or creating content. These services run with specific permissions assigned to particular users. Beyond web applications, common services include `FTP`, `SMB`, `NFS`, `IMAP/POP3`, `SSH`, `MySQL/MSSQL`, `RDP`, `WinRM`, `VNC`, `Telnet`, `SMTP`, and `LDAP`.

To manage a Windows server remotely, you need services for system access, command execution, or GUI/terminal access. The most common options are `RDP`, `WinRM`, and `SSH`. SSH is rare on Windows but standard for Linux systems.

These services typically use username/password authentication. While they can be configured for key-based authentication, many retain default settings.

## 1. WinRM

**WinRM**

Windows Remote Management (`WinRM`) is Microsoft's implementation of the Web Services Management Protocol (`WS-Management`). It's an XML web services network protocol using Simple Object Access Protocol (`SOAP`) for remote Windows system management. WinRM handles communication between Web-Based Enterprise Management (`WBEM`) and Windows Management Instrumentation (`WMI`), which can invoke the Distributed Component Object Model (`DCOM`).

For security, WinRM requires manual activation and configuration in Windows 10/11. Its security depends heavily on the domain or local network environment. Most implementations use certificates or specific authentication mechanisms for enhanced security. WinRM uses TCP ports `5985` (HTTP) and `5986` (HTTPS) by default.

`NetExec` is a useful tool for password attacks on WinRM and other protocols like SMB, LDAP, and MSSQL. Review the official documentation to familiarize yourself with this tool.

### NetExec

**Installing NetExec**: `$ sudo apt-get -y install netexec`

**NetExec Menu Options**: `$ netexec -h`

**NetExec Protocol-Specific Help**: `$ netexec smb -h` (Note that we can specify a specific protocol and receive a more detailed help menu of all of the options available to us. NetExec currently supports remote authentication using NFS, FTP, SSH, WinRM, SMB, WMI, RDP, MSSQL, LDAP, and VNC.)

**NetExec Usage**: 
- The general format for using NetExec is as follows: `$ netexec <proto> <target-IP> -u <user or userlist> -p <password or passwordlist>`
- As an example, this is what attacking a WinRM endpoint might look like:

```
$ netexec winrm 10.129.42.197 -u user.list -p password.list

WINRM       10.129.42.197   5985   NONE             [*] None (name:10.129.42.197) (domain:None)
WINRM       10.129.42.197   5985   NONE             [*] http://10.129.42.197:5985/wsman
WINRM       10.129.42.197   5985   NONE             [+] None\user:password (Pwn3d!)
```

The appearance of (`Pwn3d!`) is the sign that we can most likely execute system commands if we log in with the brute-forced user. Another handy tool that we can use to communicate with the WinRM service is `Evil-WinRM`, which allows us to communicate with the WinRM service efficiently.

### Evil-WinRM

- Installing Evil-WinRM: `$ sudo gem install evil-winrm`
- Evil-WinRM Usage: `$ evil-winrm -i <target-IP> -u <username> -p <password>`

```
$ evil-winrm -i 10.129.42.197 -u user -p password

Evil-WinRM shell v3.3

Info: Establishing connection to remote endpoint

*Evil-WinRM* PS C:\Users\user\Documents>
```

If the login was successful, a terminal session is initialized using the Powershell Remoting Protocol (`MS-PSRP`), which simplifies the operation and execution of commands.

## 2. SSH

**SSH**
Secure Shell (SSH) is a secure method for remote host connection, command execution, and file transfer. It runs on TCP port 22 and uses three cryptographic methods: symmetric encryption, asymmetric encryption, and hashing.

**Symmetric Encryption**
Uses the same key for encryption/decryption. Requires secure key exchange via Diffie-Hellman method. Common variants include AES, Blowfish, and 3DES.

**Asymmetric Encryption**
Uses two keys: private (secret) and public. The private key decrypts messages encrypted with the public key. Server uses public key for initialization and authentication; client proves identity by decrypting with private key.

**Hashing**
Converts data into unique values using one-way mathematical algorithms. SSH uses hashing to verify message authenticity.

## 3. Remote Desktop Protocol (RDP)

Microsoft's RDP is a network protocol enabling remote access to Windows systems via TCP port 3389. It connects a terminal server (where work occurs) and terminal client (for remote control). RDP supports image, sound, keyboard, pointing device exchange, plus printing and storage media access. It operates at the application layer using TCP/UDP and is used by Microsoft apps and third-party solutions.

## 4. SMB

Server Message Block (SMB) is a protocol for transferring data between clients and servers in local networks. It provides file/directory sharing and printing services in Windows networks. SMB isn't a file system but functions similarly to NFS for Unix/Linux systems.

SMB is also called Common Internet File System (CIFS), enabling remote connections across Windows, Linux, and macOS platforms. Samba is an open-source SMB implementation. Tools like Hydra can perform username/password attacks against SMB services.

# Spraying, Stuffing, and Defaults

## 1. Password spraying

Password spraying is a brute-force attack using one password against multiple user accounts. It's effective when users have default or common passwords, like `ChangeMe123!` for new accounts that weren't updated.

Different tools target various systems: Burp Suite for web applications, NetExec or Kerbrute for Active Directory environments.

```shell-session
[!bash!]$ netexec smb 10.100.38.0/24 -u <usernames.list> -p 'ChangeMe123!'
```

## 2. Credential stuffing

Credential stuffing is a brute-force attack using stolen credentials from one service to access others. It exploits password reuse across multiple platforms (email, social media, enterprise systems). Various tools can perform these attacks depending on the target system.

For example, using leaked `username:password` credentials against SSH:

```shell-session
[!bash!]$ hydra -C user_pass.list ssh://10.100.38.23
```

## 3. Default Credentials

Many systems (routers, firewalls, databases) ship with default credentials. While administrators should change these during setup, they're often left unchanged, creating security risks.

Online lists of known default credentials exist, plus dedicated automation tools like the [Default Credentials Cheat Sheet](https://github.com/ihebski/DefaultCreds-cheat-sheet):

```shell-session
[!bash!]$ pip3 install defaultcreds-cheat-sheet
```

Once installed, we can use the `creds` command to search for known default credentials associated with a specific product or vendor.

```
[!bash!]$ creds search linksys

+---------------+---------------+------------+
| Product       |    username   |  password  |
+---------------+---------------+------------+
| linksys       |    <blank>    |  <blank>   |
| linksys       |    <blank>    |   admin    |
| linksys       |    <blank>    | epicrouter |
| linksys       | Administrator |   admin    |
| linksys       |     admin     |  <blank>   |
| linksys       |     admin     |   admin    |
| linksys       |    comcast    |    1234    |
| linksys       |      root     |  orion99   |
| linksys       |      user     |  tivonpw   |
| linksys (ssh) |     admin     |   admin    |
| linksys (ssh) |     admin     |  password  |
| linksys (ssh) |    linksys    |  <blank>   |
| linksys (ssh) |      root     |   admin    |
+---------------+---------------+------------+
```

In addition to public lists and tools, default credentials appear in product documentation during setup procedures. While some systems prompt password creation during installation, others use default weak passwords.

After identifying applications on a customer's network and researching their default credentials online, combine them into a `username:password` list for use with hydra commands.

[Router default credentials](https://www.softwaretestinghelp.com/default-router-username-and-password-list/) are also common targets. Though less likely to remain unchanged due to their critical security role, oversights occur—particularly in internal testing environments where routers may retain default settings, providing further network access opportunities.
