# 1. Windows Authentication Process

Windows client authentication uses multiple modules for logon, credential retrieval, and verification. `Kerberos` is among the most widely used and complex authentication mechanisms. The Local Security Authority (`LSA`) is a protected subsystem that authenticates users, manages local logins, handles local security, and translates between usernames and security identifiers (SIDs).

The security subsystem maintains security policies and user accounts. On Domain Controllers, these apply domain-wide and are stored in Active Directory. LSA also provides access control, permission checks, and security audit message generation.

Local interactive logon involves several coordinated components: `WinLogon` (logon process), `LogonUI` (user interface), credential providers, `LSASS` (Local Security Authority Subsystem Service), authentication packages, and either `SAM` (Security Accounts Manager) or Active Directory. Authentication packages are DLLs that perform authentication checks—typically `Msv1_0.dll` for non-domain interactive logins.

`WinLogon` is a trusted system process managing security-related user interactions: launching `LogonUI` for credential prompts, handling password changes, and workstation locking/unlocking. It relies on credential providers (COM objects implemented as DLLs) to obtain user credentials.

WinLogon exclusively intercepts keyboard login requests via RPC messages from `Win32k.sys`. During logon, it launches LogonUI for the graphical interface. After credential providers collect user credentials, `WinLogon` passes them to `LSASS` for authentication.

## LSASS

The Local Security Authority Subsystem Service (LSASS) is a multi-module system governing all authentication processes. Located at `%SystemRoot%\System32\Lsass.exe`, it enforces local security policy, authenticates users, and forwards security audit logs to the Event Log, serving as Windows' security gatekeeper.

**Authentication Packages:**
- `Lsasrv.dll`: LSA Server service enforcing security policies and managing security packages. Contains the Negotiate function selecting NTLM or Kerberos protocols.
- `Msv1_0.dll`: Handles local machine logons without custom authentication requirements.
- `Samsrv.dll`: Security Accounts Manager storing local security accounts, enforcing policies, and supporting APIs.
- `Kerberos.dll`: Security package for Kerberos-based authentication.
- `Netlogon.dll`: Network-based logon service.
- `Ntdsa.dll`: Creates new Windows registry records and folders.

Each interactive logon session creates a separate WinLogon service instance. The Graphical Identification and Authentication (GINA) architecture loads into WinLogon's process area, processes credentials, and invokes authentication interfaces via the LSALogonUser function.

## SAM database

The Security Account Manager (`SAM`) is a Windows database file storing user account credentials for authenticating local and remote users. It uses cryptographic protections against unauthorized access, storing user passwords as `LM` or `NTLM` hashes in the registry. Located at `%SystemRoot%\system32\config\SAM` and mounted under `HKLM\SAM`, accessing it requires `SYSTEM` privileges.

Windows systems join either workgroups or domains during setup. Workgroup systems handle the SAM database locally, storing all users in this local database. Domain-joined systems require Domain Controller (DC) validation using the Active Directory database (`ntds.dit`) stored at `%SystemRoot%\ntds.dit`.

Microsoft introduced SYSKEY (`syskey.exe`) in Windows NT 4.0 to protect against offline SAM database cracking. When enabled, SYSKEY partially encrypts the SAM file on disk, encrypting all local account password hashes with a system-generated key.

## Credential Manager

Credential Manager is a built-in Windows feature enabling users to store and manage credentials for network resources, websites, and applications. These saved credentials are stored per user profile in the user's Credential Locker, encrypted at:

```powershell-session
PS C:\Users\[Username]\AppData\Local\Microsoft\[Vault/Credentials]\
```

Various methods exist to decrypt Credential Manager-saved credentials, which will be practiced hands-on in this module.

## NTDS

Windows systems commonly join Windows domains for centralized management, allowing administrators to efficiently oversee organizational systems. In these environments, logon requests are sent to Domain Controllers within the same Active Directory forest. Each Domain Controller hosts `NTDS.dit`, synchronized across all Domain Controllers except Read-Only Domain Controllers (RODCs).

`NTDS.dit` is a database file storing Active Directory data including:
* User accounts (username & password hash)
* Group accounts  
* Computer accounts
* Group policy objects

# 2. Attacking SAM, SYSTEM, and SECURITY

With administrative access to a Windows system, we can attempt to quickly dump the files associated with the SAM database, transfer them to our attack host, and begin cracking the hashes offline. Performing this process offline allows us to continue our attacks without having to maintain an active session with the target. 

## Registry hives

- There are three registry hives we can copy if we have local administrative access to a target system, each serving a specific purpose when it comes to dumping and cracking password hashes. A brief description of each is provided in the table below:

|Registry Hive|Description|
|-|-|
|`HKLM\SAM`|Contains password hashes for local user accounts.<br>These hashes can be extracted and cracked to reveal plaintext passwords.|
|`HKLM\SYSTEM`|Stores the system boot key, which is used to encrypt the SAM database.<br>This key is required to decrypt the hashes.|
|`HKLM\SECURITY`|Contains sensitive information used by the Local Security Authority (LSA), including cached domain credentials (DCC2), cleartext passwords, DPAPI keys, and more.|

We can back up these hives using the `reg.exe` utility.

### Using reg.exe to copy registry hives

- By launching `cmd.exe` with administrative privileges, we can use `reg.exe` to save copies of the registry hives. Run the following commands:

```
C:\WINDOWS\system32> reg.exe save hklm\sam C:\sam.save

The operation completed successfully.

C:\WINDOWS\system32> reg.exe save hklm\system C:\system.save

The operation completed successfully.

C:\WINDOWS\system32> reg.exe save hklm\security C:\security.save

The operation completed successfully.
```

For local user hash dumping, only `HKLM\SAM` and `HKLM\SYSTEM` are required. However, saving `HKLM\SECURITY` is often useful as it may contain cached domain user credentials on domain-joined systems and other valuable data. Once saved offline, various methods can transfer these hives to an attack host, such as using `Impacket's smbserver` with basic CMD commands to move copies to an attacker-hosted share.

### Creating a share with smbserver

Create a share by running `smbserver.py -smb2support`, specifying a share name (e.g., `CompData`) and pointing to the local directory where hive copies will be stored (e.g., `/home/ltnbob/Documents`). The `-smb2support` flag ensures compatibility with newer Windows versions, as SMBv1 is disabled by default due to severe vulnerabilities and public exploits.

## DCC2 hashes

## DPAPI

In addition to DCC2 hashes, the machine and user keys for DPAPI were also dumped from `hklm\security`. The Data Protection Application Programming Interface (DPAPI) is a set of Windows APIs used to encrypt and decrypt data blobs on a per-user basis. These blobs are utilized by various Windows features and third-party applications:

**Applications and DPAPI Usage:**
- `Internet Explorer`: Password form auto-completion data (username and password for saved sites)
- `Google Chrome`: Password form auto-completion data (username and password for saved sites)  
- `Outlook`: Passwords for email accounts
- `Remote Desktop Connection`: Saved credentials for connections to remote machines
- `Credential Manager`: Saved credentials for accessing shared resources, joining wireless networks, VPNs and more

# 3. Attacking LSASS

In addition to acquiring copies of the SAM database to extract and crack password hashes, targeting the Local Security Authority Subsystem Service (LSASS) can also be valuable during authorized security assessments. LSASS is a core Windows process responsible for enforcing security policies, handling user authentication, and storing sensitive credential material in memory.

Upon initial logon, LSASS will:
* Cache credentials locally in memory
* Create access tokens
* Enforce security policies
* Write to Windows' security log

## Dumping LSASS process memory

Like attacking the SAM database, we should first create a copy of LSASS process memory by generating a memory dump. This dump file allows offline credential extraction on our attack host. Offline attacks provide more flexibility in attack speed and reduce time on the target system. We'll cover techniques using built-in Windows tools, though numerous methods exist for creating memory dumps.

### Task Manager method

With access to an interactive graphical session on the target, we can use task manager to create a memory dump. This requires us to:

1. Open `Task Manager`
2. Select the `Processes` tab
3. Find and right click the `Local Security Authority Process`
4. Select `Create dump file`

A file called `lsass.DMP` is created and saved in `%temp%`. This is the file we will transfer to our attack host. 

### Rundll32.exe & Comsvcs.dll method

An alternative method to dump LSASS process memory through a command-line utility called `rundll32.exe`. This way is faster than the Task Manager method and more flexible because we may gain a shell session on a Windows host with only access to the command line. 

It is important to note that modern anti-virus tools recognize this method as malicious activity.

Before issuing the command to create the dump file, we must determine what process ID (`PID`) is assigned to `lsass.exe`. This can be done from cmd or PowerShell:

**Finding LSASS's PID in cmd:**

From cmd, we can issue the command `tasklist /svc` to find `lsass.exe` and its process ID.

```
C:\Windows\system32> tasklist /svc

Image Name                     PID Services
```

# 4. Attacking Windows Credential Manager

## Windows Vault and Credential Manager

Credential Manager, built into Windows since Server 2008 R2 and Windows 7, allows users and applications to securely store credentials for other systems and websites. Credentials are stored in encrypted folders under user and system profiles:

* `%UserProfile%\AppData\Local\Microsoft\Vault\`
* `%UserProfile%\AppData\Local\Microsoft\Credentials\`
* `%UserProfile%\AppData\Roaming\Microsoft\Vault\`
* `%ProgramData%\Microsoft\Vault\`
* `%SystemRoot%\System32\config\systemprofile\AppData\Roaming\Microsoft\Vault\`

Each vault folder contains a `Policy.vpol` file with DPAPI-protected AES keys (AES-128 or AES-256) used to encrypt credentials. Newer Windows versions use Credential Guard to further protect DPAPI master keys in secured memory enclaves.

Microsoft often refers to the protected stores as `Credential Lockers` (formerly `Windows Vaults`). Credential Manager is the user-facing feature/API, while the actual encrypted stores are the vault/locker folders. The following table lists the two types of credentials Windows stores:

|Name	|Description|
|-|-|
|Web Credentials	|Credentials associated with websites and online accounts. This locker is used by Internet Explorer and legacy versions of Microsoft Edge.|
|Windows Credentials	|Used to store login tokens for various services such as OneDrive, and credentials related to domain users, local network resources, services, and shared directories.|

# 5. Attacking Active Directory and NTDS.dit

Active Directory (AD) is a critical directory service in enterprise Windows networks that we'll encounter frequently. Since Windows organizations typically use AD for system management, attacking AD is an extensive topic covered across multiple modules.

This section focuses on extracting credentials via `dictionary attacks` against `AD accounts` and `dumping hashes` from the `NTDS.dit` file.

Like previous attacks, targets must be network-reachable, typically requiring an established foothold on the internal network. Some organizations may use port forwarding for protocols like RDP (`3389`) through edge routers, but most methods here simulate post-compromise scenarios with internal network access.

Before exploring attack methods, we'll examine the authentication process when Windows systems join a domain, helping us understand AD's significance and its susceptibility to password attacks.

Once a Windows system joins a domain, it no longer defaults to the SAM database for logon validation. Instead, authentication requests are sent to the domain controller for validation before allowing user logon.

The SAM database remains usable for local accounts by specifying the hostname before the username (`WS01\nameofuser`) or typing `.\` in the Username field with direct device access.

This is important because we must consider which system components our attacks impact. It also provides additional attack vectors when targeting Windows desktop or server systems with physical access or network connectivity.

## Dictionary attacks against AD accounts using NetExec

Dictionary attacks use computer power to guess usernames and/or passwords from customized lists. These attacks can be `noisy` (easily detected) over networks due to high traffic generation, system alerts, and potential blocking from Group Policy login restrictions.

When dictionary attacks are viable, tailoring the approach improves effectiveness. This involves researching the target organization through social media and company websites to find employee directories, providing actual employee names. New employees typically receive usernames as one of their first resources.

Many organizations follow a naming convention when creating employee usernames. Here are some common conventions to consider:

|Username convention|Practical example for `Jane Jill Doe`|
|-|-|
|`firstinitiallastname`|	jdoe|
|`firstinitialmiddleinitiallastname`|	jjdoe|
|`firstnamelastname`|	janedoe|
|`firstname.lastname`|	jane.doe|
|`lastname.firstname`|	doe.jane|
|`nickname`|	doedoehacksstuff|

Often, an email address's structure will give us the employee's username (structure: `username@domain`). For example, from the email address `jdoe`@`inlanefreight.com`, we can infer that `jdoe` is the username.

```
Note: Email structures can often be found by Googling the domain name (e.g., "@inlanefreight.com") to discover valid emails. Scripts can then scrape social media sites to create potential username lists.

Some organizations obfuscate usernames to prevent spraying by using aliases like "a907" that map back to "joe.smith" internally. This allows email delivery while hiding actual usernames.

Google dorks like "inlanefreight.com filetype:pdf" can reveal usernames in PDF properties from graphics editors. This may help discern username patterns for creating combination lists to test for valid accounts.
```

## Capturing NTDS.dit

`NT Directory Services` (`NTDS`) is AD's directory service for finding and organizing network resources. The `NTDS.dit` file is stored at `%systemroot%/ntds` on domain controllers, where `.dit` stands for directory information tree.

This primary AD database file contains all domain usernames, password hashes, and critical schema information. If captured, it could potentially compromise every domain account, similar to `attacking SAM, SYSTEM, and SECURITY files`.

When practicing this technique, consider AD protection importance and brainstorm methods to prevent this attack.

## Cracking hashes and gaining credentials

We can proceed with creating a text file containing all the NT hashes, or we can individually copy and paste a specific hash into a terminal session and use Hashcat to attempt to crack the hash and a password in cleartext.

### Cracking a single hash with Hashcat

```
$ sudo hashcat -m 1000 64f12cddaa88057e06a81b54e73b949b /usr/share/wordlists/rockyou.txt

64f12cddaa88057e06a81b54e73b949b:Password1
```

In many of the techniques we have covered so far, we have had success in cracking hashes we've obtained.

**`What if we are unsuccessful in cracking a hash?`**

## Pass the Hash (PtH) considerations

We can still use hashes to attempt to authenticate with a system using a type of attack called `Pass-the-Hash` (`PtH`). A PtH attack takes advantage of the `NTLM authentication protocol` to authenticate a user using a password hash. Instead of `username:clear-text password` as the format for login, we can instead use `username:password hash`. 

Here is an example of how this would work:

```
Pass the Hash (PtH) with Evil-WinRM Example

$ evil-winrm -i 10.129.201.57 -u Administrator -H 64f12cddaa88057e06a81b54e73b949b
```

# 6. Credential Hunting in Windows

Once we have GUI or CLI access to a target Windows machine, credential hunting provides significant advantages. `Credential hunting` involves performing detailed file system searches and examining various applications to discover stored credentials.

Consider this scenario: we've gained RDP access to an IT admin's Windows 10 workstation. This presents an opportunity to systematically search for additional credentials that could expand our access within the environment.

## Search-centric

Many Windows tools include built-in search functionality that we can leverage during engagements. Users often document passwords somewhere on the system, and default credentials may exist in various files. 

Our credential search should be informed by understanding how the target system is used. Since we have access to an IT admin's workstation, we should ask: `What might an IT admin be doing daily and which tasks may require credentials?`

This targeted approach helps refine our search and reduces random guessing, making credential hunting more efficient and effective.

### Key terms to search for

Whether we end up with access to the GUI or CLI, we know we will have some tools to use for searching but of equal importance is what exactly we are searching for. Here are some helpful key terms we can use that can help us discover some credentials:

* Passwords
* Passphrases
* Keys
* Username
* User account
* Creds
* Users
* Passkeys
* configuration
* dbcredential
* dbpassword
* pwd
* Login
* Credentials

## Search tools

### Windows Search

With access to the GUI, it is worth attempting to use `Windows Search` to find files on the target using some of the keywords mentioned above.

By default, it will search various OS settings and the file system for files and applications containing the key term entered in the search bar.

### LaZagne

Third-party tools like LaZagne can quickly discover credentials stored insecurely by web browsers and other applications. LaZagne uses `modules` targeting different software for password extraction:

| Module | Description |
|--------|-------------|
| browsers | Extracts passwords from Chromium, Firefox, Microsoft Edge, Opera |
| chats | Extracts passwords from chat applications like Skype |
| mails | Searches mailboxes for passwords including Outlook, Thunderbird |
| memory | Dumps passwords from memory, targeting KeePass and LSASS |
| sysadmin | Extracts passwords from sysadmin tool configs like OpenVPN, WinSCP |
| windows | Extracts Windows credentials from LSA secrets, Credential Manager |
| wifi | Dumps WiFi credentials |

**Note:** Web browsers are particularly valuable targets since many offer built-in credential storage. Popular browsers like Google Chrome, Microsoft Edge, and Firefox encrypt stored credentials, but online decryption tools exist (firefox_decrypt, decrypt-chrome-passwords). LaZagne supports 35 different Windows browsers.

It's beneficial to keep a standalone copy of LaZagne on our attack host for quick transfer to targets. `LaZagne.exe` works well for this scenario. We can use our RDP client to copy the file from our attack host - with `xfreerdp`, simply copy and paste into the established RDP session.

Once `LaZagne.exe` is on the target, open Command Prompt or PowerShell, navigate to the upload directory, and execute:

```cmd-session
C:\Users\bob\Desktop> start LaZagne.exe all
```

This will execute LaZagne and run `all` included modules. We can include the option `-vv` to study what it is doing in the background. Once we hit enter, it will open another prompt and display the results.

### findstr

We can also use `findstr` to search from patterns across many types of files. Keeping in mind common key terms, we can use variations of this command to discover credentials on a Windows target:

`C:\> findstr /SIM /C:"password" *.txt *.ini *.cfg *.config *.xml *.git *.ps1 *.yml`

## Additional considerations

Thousands of tools and terms exist for credential hunting on Windows systems. Tool selection depends primarily on the computer's function - Windows Servers require different approaches than Windows Desktops. Always consider system usage to determine where to look. Sometimes credentials can be found by simply navigating and listing file system directories while tools run.

Other important credential hunting locations include:

* Passwords in Group Policy in the SYSVOL share
* Passwords in scripts in the SYSVOL share  
* Passwords in scripts on IT shares
* Passwords in `web.config` files on dev machines and IT shares
* Passwords in `unattend.xml`
* Passwords in AD user or computer description fields
* KeePass databases (if master password can be guessed or cracked)
* Files on user systems and shares
* Files named `pass.txt`, `passwords.docx`, `passwords.xlsx` on user systems, shares, and `SharePoint`
