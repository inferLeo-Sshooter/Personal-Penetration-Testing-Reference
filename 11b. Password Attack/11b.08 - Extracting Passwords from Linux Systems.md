# 1. Linux Authentication Process

Linux distributions commonly use Pluggable Authentication Modules (PAM) for authentication. Key modules like `pam_unix.so` or `pam_unix2.so`, located in `/usr/lib/x86_64-linux-gnu/security/` on Debian systems, handle user authentication, sessions, and password changes. When users run `passwd`, PAM securely manages the password update process.

The `pam_unix.so` module uses system library API calls to update account information in `/etc/passwd` and `/etc/shadow`. PAM supports additional modules for LDAP, mount operations, and Kerberos authentication.

## Passwd file

The `/etc/passwd` file stores user information for all system users and is readable by everyone. Each user entry contains seven colon-separated fields with user data. A typical entry looks like:

```shell-session
htb-student:x:1000:1000:,,,:/home/htb-student:/bin/bash
```

|Field	|Value|
|-|-|
|Username	|`htb-student`|
|Password	|`x`|
|User ID	|`1000`|
|Group ID	|`1000`|
|GECOS	|`,,,`|
|Home directory	|`/home/htb-student`|
|Default shell	|`/bin/bash`|

The `Password` field is most relevant for security assessment. On very old systems, it may contain actual password hashes, but modern systems use `x` to indicate hashes are stored in `/etc/shadow`. Since `/etc/passwd` is world-readable, exposed hashes can be cracked.

If `/etc/passwd` is mistakenly writable, attackers can remove the root password field entirely:

```shell-session
[!bash!]$ head -n 1 /etc/passwd

root::0:0:root:/root:/bin/bash
```

This eliminates the password prompt when switching to `root`:

```shell-session
[!bash!]$ su

root@htb[/htb]#
```

While rare, these misconfigurations occur when inexperienced administrators assign overly broad permissions to `/etc` directory for applications requiring specific folder access.

## Shadow file

The `/etc/shadow` file was introduced to securely store password hashes, as readable hash values risk system compromise. It has similar formatting to `/etc/passwd` but focuses solely on password management. Users listed in `/etc/passwd` without corresponding `/etc/shadow` entries are invalid. Only administrative users can read this file.

The format contains `nine fields`:

```shell-session
htb-student:$y$j9T$3QSBB6CbHEu...SNIP...f8Ms:18955:0:99999:7:::
```

|**Field** | **Value**|
|-|-|
|Username | `htb-student`|
|Password | `$y$j9T$3QSBB6CbHEu...SNIP...f8Ms`|
|Last change | `18955`|
|Min age | `0`|
|Max age | `99999`|
|Warning period | `7`|
|Inactivity period | `-`|
|Expiration date | `-`|
|Reserved field | `-`|

If the `Password` field contains `!` or `*`, Unix password login is disabled, but other authentication methods (Kerberos, key-based) still work. Empty password fields require no password but may cause programs to deny access to certain functions.

The `Password` field follows this format:
* `$<id>$<salt>$<hashed>`

The `ID` specifies the cryptographic hash algorithm:

|**ID** | **Algorithm**|
|-|-|
|`1` | MD5|
|`2a` | Blowfish|
|`5` | SHA-256|
|`6` | SHA-512|
|`sha1` | SHA1crypt|
|`y` | Yescrypt|
|`gy` | Gost-yescrypt|
|`7` | Scrypt|

Many Linux distributions like Debian now default to `yescrypt`. Older systems may use weaker algorithms that are more vulnerable to cracking.

## Opasswd

The PAM library (`pam_unix.so`) prevents password reuse by storing old passwords in `/etc/security/opasswd`. Root privileges are required to read this file.

```shell-session
[!bash!]$ sudo cat /etc/security/opasswd

cry0l1t3:1000:2:$1$HjFAfYTG$qNDkF0zJ3v8ylCOrKB0kt0,$1$kcUjWZJX$E9uMSmiQeRh4pAAgzuvkq1
```

The file contains multiple entries per user, separated by commas. Note the hash types used—`MD5` (`$1$`) is significantly easier to crack than SHA-512. This is crucial for identifying old passwords and recognizing user patterns, as people often reuse similar passwords across services, improving password guessing success rates.

## Cracking Linux Credentials

Once we have root access on a Linux machine, we can gather user password hashes and attempt to crack them using various methods to recover the plaintext passwords. To do this, we can use a tool called `unshadow`, which is included with `John the Ripper (JtR)`. It works by combining the `passwd` and `shadow` files into a single file suitable for cracking.

```
[!bash!]$ sudo cp /etc/passwd /tmp/passwd.bak 
[!bash!]$ sudo cp /etc/shadow /tmp/shadow.bak 
[!bash!]$ unshadow /tmp/passwd.bak /tmp/shadow.bak > /tmp/unshadowed.hashes
```
This "unshadowed" file can now be attacked with either JtR or hashcat.
```
[!bash!]$ hashcat -m 1800 -a 0 /tmp/unshadowed.hashes rockyou.txt -o /tmp/unshadowed.cracked
```
Note: This is the exact scenario that `JtR's single crack mode` was designed for.

# 2. Credential Hunting in Linux

Credential hunting is a crucial first step after gaining system access, providing potential privilege escalation within minutes. This process is part of local privilege escalation and focuses on key approaches rather than comprehensive coverage.

Consider a scenario where you've gained access via a vulnerable web application and obtained a reverse shell. To escalate privileges efficiently, search for passwords or complete credentials for system login. Credential sources fall into four categories:

* `Files` - configs, databases, notes, scripts, source code, cronjobs, SSH keys
* `History` - logs, command-line history  
* `Memory` - cache, in-memory processing
* `Key-rings` - browser stored credentials

Enumerating these categories increases the probability of finding user credentials. Since situations vary greatly, adapt your approach to the environment while considering the system's purpose, role in business logic, and network position. For example, an isolated database server likely won't contain normal user accounts since access is typically restricted to essential personnel only.

## Files

One core Linux principle is that everything is a file. Therefore, systematically search, find, and filter files according to your requirements. Examine these categories individually:

* Configuration files
* Databases  
* Notes
* Scripts
* Cronjobs
* SSH keys

Configuration files control service functionality on Linux and often contain readable credentials. They also reveal how services work and their requirements. Typically marked with extensions `.config`, `.conf`, or `.cnf`, though these files can be renamed, making extensions optional. When recompiling services, basic configuration filenames can change, producing the same effect. While rare, this possibility shouldn't be excluded from searches.

### Searching for notes

Depending on the environment we are in and the purpose of the host we are on, we can often find notes about specific processes on the system. These often include lists of many different access points or even their credentials. However, it is often challenging to find notes right away if stored somewhere on the system and not on the desktop or in its subfolders. This is because they can be named anything and do not have to have a specific file extension, such as `.txt`. Therefore, in this case, we need to search for files including the `.txt` file extension and files that have no file extension at all.

### Enumerating cronjobs

Cronjobs execute commands, programs, and scripts independently. They're divided between system-wide (`/etc/crontab`) and user-specific executions. Applications and scripts often incorrectly include credentials in cronjobs. 

Additional areas are organized by time ranges: `/etc/cron.daily`, `/etc/cron.hourly`, `/etc/cron.monthly`, `/etc/cron.weekly`. On Debian-based distributions, cron scripts and files are also located in `/etc/cron.d/`.


## Memory and cache

### Mimipenguin

Many applications and processes store authentication credentials in memory or files for reuse, such as system credentials for logged-in users or browser-stored credentials. The tool `mimipenguin` simplifies retrieving this information from Linux distributions but `requires administrator/root permissions`.

```shell-session
inferLeo@htb[/htb]$ sudo python3 mimipenguin.py

[SYSTEM - GNOME]	cry0l1t3:WLpAEXFa0SbqOHY
```

### LaZagne

`LaZagne` is a more powerful credential extraction tool that accesses far more resources than mimipenguin. It extracts passwords and hashes from numerous sources including:

* Wifi
* Wpa_supplicant
* Libsecret
* Kwallet
* Chromium-based browsers
* CLI
* Mozilla
* Thunderbird
* Git
* ENV variables
* Grub
* Fstab
* AWS
* Filezilla
* Gftp
* SSH
* Apache
* Shadow
* Docker
* Keepass
* Mimipy
* Sessions
* Keyrings

`Keyrings` provide secure password storage and management on Linux distributions. Passwords are encrypted and protected with a master password, functioning as an OS-based password manager that eliminates the need to remember individual passwords and reduces repeated password entries.
