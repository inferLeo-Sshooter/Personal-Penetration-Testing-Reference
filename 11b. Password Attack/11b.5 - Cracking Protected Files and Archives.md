## Cracking Protected Files

File encryption is often neglected in private and professional settings. Emails with job applications, account statements, or contracts are frequently sent unencrypted, sometimes violating legal requirements. The EU's GDPR mandates encryption of personal data in transit and at rest, yet unprotected transmission of confidential data via email remains common practice, creating opportunities for attackers.

As companies improve IT security through training and awareness programs, employees increasingly encrypt sensitive files. However, encrypted files can still be cracked using appropriate wordlists and tools. Individual files or folders typically use symmetric encryption algorithms like AES-256, where the same key encrypts and decrypts data. For file transmission, asymmetric encryption is preferred, using two keys: senders encrypt with the recipient's public key, and recipients decrypt with their corresponding private key.

### 1. Hunting for Encrypted Files

Many different extensions correspond to encrypted files—a useful reference list can be found on [FileInfo](https://fileinfo.com/filetypes/encoded). As an example, consider this command we might use to locate commonly encrypted files on a Linux system:

```
[!bash!]$ for ext in $(echo ".xls .xls* .xltx .od* .doc .doc* .pdf .pot .pot* .pp*");do echo -e "\nFile extension: " $ext; find / -name *$ext 2>/dev/null | grep -v "lib\|fonts\|share\|core" ;done

File extension:  .xls

File extension:  .xls*

File extension:  .xltx

File extension:  .od*
/home/cry0l1t3/Docs/document-temp.odt
/home/cry0l1t3/Docs/product-improvements.odp
/home/cry0l1t3/Docs/mgmt-spreadsheet.ods
...SNIP...
```

If we encounter file extensions on a system that we are unfamiliar with, we can use search engines to research the technology behind them.

### 2. Hunting for SSH keys

SSH keys lack standard file extensions, but can be identified by their distinctive headers and footers. SSH private keys always begin with `-----BEGIN [...SNIP...] PRIVATE KEY-----`. Tools like `grep` can recursively search for them during post-exploitation:

```shell-session
[!bash!]$ grep -rnE '^\-{5}BEGIN [A-Z0-9]+ PRIVATE KEY\-{5}$' /* 2>/dev/null

/home/jsmith/.ssh/id_ed25519:1:-----BEGIN OPENSSH PRIVATE KEY-----
/home/jsmith/.ssh/SSH.private:1:-----BEGIN RSA PRIVATE KEY-----
/home/jsmith/Documents/id_rsa:1:-----BEGIN OPENSSH PRIVATE KEY-----
<SNIP>
```

Some SSH keys use passphrase encryption. Older PEM formats show encryption status in headers with the method used. Modern SSH keys appear identical whether encrypted or not.

```shell-session
[!bash!]$ cat /home/jsmith/.ssh/SSH.private

-----BEGIN RSA PRIVATE KEY-----
Proc-Type: 4,ENCRYPTED
DEK-Info: AES-128-CBC,2109D25CC91F8DBFCEB0F7589066B2CC

8Uboy0afrTahejVGmB7kgvxkqJLOczb1I0/hEzPU1leCqhCKBlxYldM2s65jhflD
<SNIP>
```

To check if an SSH key is encrypted, use `ssh-keygen` to read it. Unencrypted keys display immediately:

```shell-session
[!bash!]$ ssh-keygen -yf ~/.ssh/id_ed25519 
ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIIpNefJd834VkD5iq+22Zh59Gzmmtzo6rAffCx2UtaS6
```

Password-protected keys prompt for a passphrase:

```shell-session
[!bash!]$ ssh-keygen -yf ~/.ssh/id_rsa
Enter passphrase for "/home/jsmith/.ssh/id_rsa":
```

### 3. Cracking encrypted SSH keys

`JtR` has many different scripts for extracting hashes from files—which we can then proceed to crack. We can find these scripts on our system using the following command:

```
[!bash!]$ locate *2john*

/usr/bin/bitlocker2john
/usr/bin/dmg2john
/usr/bin/gpg2john
/usr/bin/hccap2john
/usr/bin/keepass2john
...<SNIP>...
```

For example, we could use the Python script `ssh2john.py` to acquire the corresponding hash for an encrypted SSH key, and then use JtR to try and crack it:

```
[!bash!]$ ssh2john.py SSH.private > ssh.hash
[!bash!]$ john --wordlist=rockyou.txt ssh.hash

```

### 4. Cracking password-protected documents

Password-protected documents are commonly encountered in professional environments. Most reports and documentation are distributed as Microsoft Office files or PDFs. John the Ripper includes `office2john.py` to extract password hashes from Office documents, which can then be cracked offline using wordlists.

```shell-session
[!bash!]$ office2john.py Protected.docx > protected-docx.hash
[!bash!]$ john --wordlist=rockyou.txt protected-docx.hash
[!bash!]$ john protected-docx.hash --show

Protected.docx:1234

1 password hash cracked, 0 left
```

PDF files follow a similar process using `pdf2john.py`:

```shell-session
[!bash!]$ pdf2john.py PDF.pdf > pdf.hash
[!bash!]$ john --wordlist=rockyou.txt pdf.hash
[!bash!]$ john pdf.hash --show

PDF.pdf:1234

1 password hash cracked, 0 left
```

The main challenge involves generating effective password lists. Standard public wordlists may be insufficient as security mechanisms often recognize them. Files may be difficult or impossible to crack within reasonable timeframes due to users increasingly adopting longer, randomly generated passwords or complex passphrases. However, attempting to crack password-protected documents can be worthwhile as they may contain valuable information for further access.

## Cracking Protected Archives

Beyond standalone files, we commonly encounter password-protected archives and compressed files like ZIP files.

Consider an administrative company where clients request analysis summaries in multiple formats—Excel, PDF, Word, and presentations. Rather than sending files individually, which becomes cumbersome for large organizations managing multiple projects and risks file loss, employees often use archive files. These allow structured organization of documents (typically in subfolders) before compression into a single consolidated file.

Common archive file extensions include `tar`, `gz`, `rar`, `zip`, `vmdb/vmx`, `cpt`, `truecrypt`, `bitlocker`, `kdbx`, `deb`, `7z`, and `gzip`. 

A comprehensive list of archive types is available on FileInfo. Rather than manual enumeration, you can query the data using one-liners, apply filters, and save results to files. Currently, the website lists 365 archive file types.

```
$ curl -s https://fileinfo.com/filetypes/compressed | html2text | awk '{print tolower($1)}' | grep "\." | tee -a compressed_ext.txt

.mint
.zhelp
.b6z
.fzpz
.zst
.apz
.ufs.uzip
.vrpackage
.sfg
.gzip
.xapk
.rar
.pkg.tar.xz
<SNIP>
```

**Note** that not all archive types support native password protection, and in such cases, additional tools are often used to encrypt the files. For example, TAR files are commonly encrypted using `openssl` or `gpg`.

### 5. Cracking ZIP files

The `ZIP` format is often heavily used in Windows environments to compress many files into one file. The process of cracking an encrypted ZIP file is similar to what we have seen already, except for using a different script to extract the hashes.

```
$ zip2john ZIP.zip > zip.hash
$ cat zip.hash 

ZIP.zip/customers.csv:$pkzip2$1*2*2*0*2a*1e*490e7510*0*42*0*2a*490e*409b*ef1e7feb7c1cf701a6ada7132e6a5c6c84c032401536faf7493df0294b0d5afc3464f14ec081cc0e18cb*$/pkzip2$:customers.csv:ZIP.zip::ZIP.zip
```

Once we have extracted the hash, we can use JtR to crack it with the desired password list.

```
$ john --wordlist=rockyou.txt zip.hash

$ john zip.hash --show

ZIP.zip/customers.csv:1234:customers.csv:ZIP.zip::ZIP.zip

1 password hash cracked, 0 left
```

### 6. Cracking OpenSSL encrypted GZIP files

Files aren't always obviously password-protected, especially when the extension corresponds to formats that don't natively support passwords. As discussed, `openssl` can encrypt files in `GZIP` format. The `file` command reveals a file's actual format and contents:

```shell-session
inferLeo@htb[/htb]$ file GZIP.gzip 

GZIP.gzip: openssl enc'd data with salted password
```

Cracking OpenSSL encrypted files presents challenges including numerous false positives or complete failure to identify correct passwords. A more reliable approach uses the `openssl` tool within a `for` loop that attempts direct content extraction, succeeding only when the correct password is found.

The following one-liner may produce several GZIP-related error messages, which can be safely ignored. If the correct password list is used, as in this example, we will see another file successfully extracted from the archive.

```
$ for i in $(cat rockyou.txt);do openssl enc -aes-256-cbc -d -in GZIP.gzip -k $i 2>/dev/null| tar xz;done

gzip: stdin: not in gzip format
tar: Child returned status 1
tar: Error is not recoverable: exiting now

gzip: stdin: not in gzip format
tar: Child returned status 1
tar: Error is not recoverable: exiting now
<SNIP>
```

Once the `for` loop has finished, we can check the current directory for a newly extracted file.

```
$ ls

customers.csv  GZIP.gzip  rockyou.txt
```

### 7. Cracking BitLocker-encrypted drives

BitLocker is Microsoft's full-disk encryption feature for Windows, available since Vista. It uses AES encryption with 128-bit or 256-bit keys. Forgotten passwords can be recovered using a 48-digit recovery key generated during setup.

In enterprise environments, virtual drives often store personal information on company devices to prevent unauthorized access. The `bitlocker2john` script generates four hashes: two for the BitLocker password and two for the recovery key. Since recovery keys are long and randomly generated, they're impractical to crack without partial knowledge. Focus on cracking the password using the first hash (`$bitlocker$0$...`).

```
$ bitlocker2john -i Backup.vhd > backup.hashes
$ grep "bitlocker\$0" backup.hashes > backup.hash
$ cat backup.hash

$bitlocker$0$16$02b329c0453b9273f2fc1b927443b5fe$1048576$12$00b0a67f961dd80103000000$60$d59f37e70696f7eab6b8f95ae93bd53f3f7067d5e33c0394b3d8e2d1fdb885cb86c1b978f6cc12ed26de0889cd2196b0510bbcd2a8c89187ba8ec54f
```

Once a hash is generated, either `JtR` or `hashcat` can crack it. For `hashcat`, the mode for `$bitlocker$0$...` hashes is `-m 22100`. Supply the hash, specify the wordlist, and define the hash mode. Since BitLocker uses strong AES encryption, cracking may take considerable time depending on hardware performance.

After successfully cracking the password, we can access the encrypted drive.

**Mounting BitLocker-encrypted drives in Windows:**

- The easiest method for mounting a BitLocker-encrypted virtual drive on Windows is to double-click the `.vhd` file. Since it is encrypted, Windows will initially show an error. After mounting, simply double-click the BitLocker volume to be prompted for the password.

**Mounting BitLocker-encrypted drives in Linux (or macOS):**

BitLocker-encrypted drives can be mounted in Linux (or macOS) using `dislocker`. First, install the package:

```shell-session
$ sudo apt-get install dislocker
```

Create two mounting folders:

```shell-session
$ sudo mkdir -p /media/bitlocker
$ sudo mkdir -p /media/bitlockermount
```

Configure the VHD as a loop device, decrypt with `dislocker`, and mount the volume:

```shell-session
$ sudo losetup -f -P Backup.vhd
$ sudo dislocker /dev/loop0p2 -u1234qwer -- /media/bitlocker
$ sudo mount -o loop /media/bitlocker/dislocker-file /media/bitlockermount
```

Browse the files:

```shell-session
$ cd /media/bitlockermount/
$ ls -la
```

Unmount when finished:

```shell-session
$ sudo umount /media/bitlockermount
$ sudo umount /media/bitlocker
```
