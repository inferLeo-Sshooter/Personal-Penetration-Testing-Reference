

# 1. Windows Authentication Process

Windows client authentication uses multiple modules for logon, credential retrieval, and verification. `Kerberos` is among the most widely used and complex authentication mechanisms. The Local Security Authority (`LSA`) is a protected subsystem that authenticates users, manages local logins, handles local security, and translates between usernames and security identifiers (SIDs).

The security subsystem maintains security policies and user accounts. On Domain Controllers, these apply domain-wide and are stored in Active Directory. LSA also provides access control, permission checks, and security audit message generation.

Local interactive logon involves several coordinated components: `WinLogon` (logon process), `LogonUI` (user interface), credential providers, `LSASS` (Local Security Authority Subsystem Service), authentication packages, and either `SAM` (Security Accounts Manager) or Active Directory. Authentication packages are DLLs that perform authentication checks—typically `Msv1_0.dll` for non-domain interactive logins.

`WinLogon` is a trusted system process managing security-related user interactions: launching `LogonUI` for credential prompts, handling password changes, and workstation locking/unlocking. It relies on credential providers (COM objects implemented as DLLs) to obtain user credentials.

WinLogon exclusively intercepts keyboard login requests via RPC messages from `Win32k.sys`. During logon, it launches LogonUI for the graphical interface. After credential providers collect user credentials, `WinLogon` passes them to `LSASS` for authentication.

## LSASS

The Local Security Authority Subsystem Service (LSASS) is a multi-module system governing all authentication processes. Located at `%SystemRoot%\System32\Lsass.exe`, it enforces local security policy, authenticates users, and forwards security audit logs to the Event Log, serving as Windows' security gatekeeper.

**Authentication Packages:**
- `Lsasrv.dll`: LSA Server service enforcing security policies and managing security packages. Contains the Negotiate function selecting NTLM or Kerberos protocols.
- `Msv1_0.dll`: Handles local machine logons without custom authentication requirements.
- `Samsrv.dll`: Security Accounts Manager storing local security accounts, enforcing policies, and supporting APIs.
- `Kerberos.dll`: Security package for Kerberos-based authentication.
- `Netlogon.dll`: Network-based logon service.
- `Ntdsa.dll`: Creates new Windows registry records and folders.

Each interactive logon session creates a separate WinLogon service instance. The Graphical Identification and Authentication (GINA) architecture loads into WinLogon's process area, processes credentials, and invokes authentication interfaces via the LSALogonUser function.

## SAM database

The Security Account Manager (`SAM`) is a Windows database file storing user account credentials for authenticating local and remote users. It uses cryptographic protections against unauthorized access, storing user passwords as `LM` or `NTLM` hashes in the registry. Located at `%SystemRoot%\system32\config\SAM` and mounted under `HKLM\SAM`, accessing it requires `SYSTEM` privileges.

Windows systems join either workgroups or domains during setup. Workgroup systems handle the SAM database locally, storing all users in this local database. Domain-joined systems require Domain Controller (DC) validation using the Active Directory database (`ntds.dit`) stored at `%SystemRoot%\ntds.dit`.

Microsoft introduced SYSKEY (`syskey.exe`) in Windows NT 4.0 to protect against offline SAM database cracking. When enabled, SYSKEY partially encrypts the SAM file on disk, encrypting all local account password hashes with a system-generated key.

## Credential Manager

Credential Manager is a built-in Windows feature enabling users to store and manage credentials for network resources, websites, and applications. These saved credentials are stored per user profile in the user's Credential Locker, encrypted at:

```powershell-session
PS C:\Users\[Username]\AppData\Local\Microsoft\[Vault/Credentials]\
```

Various methods exist to decrypt Credential Manager-saved credentials, which will be practiced hands-on in this module.

## NTDS

Windows systems commonly join Windows domains for centralized management, allowing administrators to efficiently oversee organizational systems. In these environments, logon requests are sent to Domain Controllers within the same Active Directory forest. Each Domain Controller hosts `NTDS.dit`, synchronized across all Domain Controllers except Read-Only Domain Controllers (RODCs).

`NTDS.dit` is a database file storing Active Directory data including:
* User accounts (username & password hash)
* Group accounts  
* Computer accounts
* Group policy objects

# 2. Attacking SAM, SYSTEM, and SECURITY

With administrative access to a Windows system, we can attempt to quickly dump the files associated with the SAM database, transfer them to our attack host, and begin cracking the hashes offline. Performing this process offline allows us to continue our attacks without having to maintain an active session with the target. 

## Registry hives

- There are three registry hives we can copy if we have local administrative access to a target system, each serving a specific purpose when it comes to dumping and cracking password hashes. A brief description of each is provided in the table below:

|Registry Hive|Description|
|-|-|
|`HKLM\SAM`|Contains password hashes for local user accounts.<br>These hashes can be extracted and cracked to reveal plaintext passwords.|
|`HKLM\SYSTEM`|Stores the system boot key, which is used to encrypt the SAM database.<br>This key is required to decrypt the hashes.|
|`HKLM\SECURITY`|Contains sensitive information used by the Local Security Authority (LSA), including cached domain credentials (DCC2), cleartext passwords, DPAPI keys, and more.|

We can back up these hives using the `reg.exe` utility.

### Using reg.exe to copy registry hives

- By launching `cmd.exe` with administrative privileges, we can use `reg.exe` to save copies of the registry hives. Run the following commands:

```
C:\WINDOWS\system32> reg.exe save hklm\sam C:\sam.save

The operation completed successfully.

C:\WINDOWS\system32> reg.exe save hklm\system C:\system.save

The operation completed successfully.

C:\WINDOWS\system32> reg.exe save hklm\security C:\security.save

The operation completed successfully.
```

For local user hash dumping, only `HKLM\SAM` and `HKLM\SYSTEM` are required. However, saving `HKLM\SECURITY` is often useful as it may contain cached domain user credentials on domain-joined systems and other valuable data. Once saved offline, various methods can transfer these hives to an attack host, such as using `Impacket's smbserver` with basic CMD commands to move copies to an attacker-hosted share.

### Creating a share with smbserver

Create a share by running `smbserver.py -smb2support`, specifying a share name (e.g., `CompData`) and pointing to the local directory where hive copies will be stored (e.g., `/home/ltnbob/Documents`). The `-smb2support` flag ensures compatibility with newer Windows versions, as SMBv1 is disabled by default due to severe vulnerabilities and public exploits.

## DCC2 hashes

As mentioned, `hklm\security` contains cached domain logon information as DCC2 hashes—local, hashed copies of network credential hashes. An example:


## DPAPI

In addition to DCC2 hashes, the machine and user keys for DPAPI were also dumped from `hklm\security`. The Data Protection Application Programming Interface (DPAPI) is a set of Windows APIs used to encrypt and decrypt data blobs on a per-user basis. These blobs are utilized by various Windows features and third-party applications:

**Applications and DPAPI Usage:**
- `Internet Explorer`: Password form auto-completion data (username and password for saved sites)
- `Google Chrome`: Password form auto-completion data (username and password for saved sites)  
- `Outlook`: Passwords for email accounts
- `Remote Desktop Connection`: Saved credentials for connections to remote machines
- `Credential Manager`: Saved credentials for accessing shared resources, joining wireless networks, VPNs and more

DPAPI encrypted credentials can be decrypted manually with tools like `Impacket's dpapi`, `mimikatz`, or remotely with `DonPAPI`.

# 3. Attacking LSASS

In addition to acquiring copies of the SAM database to extract and crack password hashes, targeting the Local Security Authority Subsystem Service (LSASS) can also be valuable during authorized security assessments. LSASS is a core Windows process responsible for enforcing security policies, handling user authentication, and storing sensitive credential material in memory.

Upon initial logon, LSASS will:
* Cache credentials locally in memory
* Create access tokens
* Enforce security policies
* Write to Windows' security log

## Dumping LSASS process memory

Like attacking the SAM database, we should first create a copy of LSASS process memory by generating a memory dump. This dump file allows offline credential extraction on our attack host. Offline attacks provide more flexibility in attack speed and reduce time on the target system. We'll cover techniques using built-in Windows tools, though numerous methods exist for creating memory dumps.

### Task Manager method

With access to an interactive graphical session on the target, we can use task manager to create a memory dump. This requires us to:

1. Open `Task Manager`
2. Select the `Processes` tab
3. Find and right click the `Local Security Authority Process`
4. Select `Create dump file`

A file called `lsass.DMP` is created and saved in `%temp%`. This is the file we will transfer to our attack host. 

### Rundll32.exe & Comsvcs.dll method

An alternative method to dump LSASS process memory through a command-line utility called `rundll32.exe`. This way is faster than the Task Manager method and more flexible because we may gain a shell session on a Windows host with only access to the command line. 

It is important to note that modern anti-virus tools recognize this method as malicious activity.

Before issuing the command to create the dump file, we must determine what process ID (`PID`) is assigned to `lsass.exe`. This can be done from cmd or PowerShell:

**Finding LSASS's PID in cmd:**

From cmd, we can issue the command `tasklist /svc` to find `lsass.exe` and its process ID.


**Finding LSASS's PID in PowerShell:**

From PowerShell, we can issue the command `Get-Process lsass` and see the process ID in the `Id` field.

```
PS C:\Windows\system32> Get-Process lsass
```

Once we have the PID assigned to the LSASS process, we can create a dump file.

**Creating a dump file using PowerShell:**

With an elevated PowerShell session, we can issue the following command to create a dump file:

```
PS C:\Windows\system32> rundll32 C:\windows\system32\comsvcs.dll, MiniDump 672 C:\lsass.dmp full
```

With this command, we are running `rundll32.exe` to call an exported function of `comsvcs.dll` which also calls the MiniDumpWriteDump (`MiniDump`) function to dump the LSASS process memory to a specified directory (`C:\lsass.dmp`). 

Recall that most modern AV tools recognize this as malicious activity and prevent the command from executing. In these cases, we will need to consider ways to bypass or disable the AV tool we are facing.

If we manage to run this command and generate the `lsass.dmp` file, we can proceed to transfer the file onto our attack box to attempt to extract any credentials that may have been stored in LSASS process memory.

## Using Pypykatz to extract credentials

Once we have the dump file on our attack host, we can use pypykatz to extract credentials from the `.dmp` file. Pypykatz is a Python implementation of Mimikatz that runs on Linux-based attack hosts, unlike Mimikatz which only runs on Windows. This makes pypykatz ideal since we only need the dump file and can run it offline from our Linux host.

LSASS stores credentials for active logon sessions on Windows systems. When we dumped LSASS process memory, we captured a "snapshot" of what was in memory at that time, including any active session credentials. Let's run pypykatz against the dump file to extract them.

### Running Pypykatz

The command initiates the use of `pypykatz` to parse the secrets hidden in the LSASS process memory dump. We use `lsa` in the command because LSASS is a subsystem of the `Local Security Authority`, then we specify the data source as a `minidump` file, proceeded by the path to the dump file stored on our attack host. Pypykatz parses the dump file and outputs the findings:


**MSV:**


MSV is a Windows authentication package that LSA uses to validate logon attempts against the SAM database. Pypykatz extracted the `SID`, `Username`, `Domain`, and password hashes (`NT` & `SHA1`) for the bob user account's logon session stored in LSASS memory. These will be useful for our next attack step.

**WDIGEST:**

```shell-session
	== WDIGEST [14ab89]==
		username bob
		domainname DESKTOP-33E7O54
		password None
		password (hex)

```

`WDIGEST` is an older authentication protocol enabled by default in `Windows XP` - `Windows 8` and `Windows Server 2003` - `Windows Server 2012`. LSASS caches WDIGEST credentials in clear-text, so on systems with WDIGEST enabled, we'll likely see plaintext passwords. Modern Windows systems have WDIGEST disabled by default, and Microsoft has released security updates addressing this issue.

**Kerberos:**

```shell-session
	== Kerberos ==
		Username: bob
		Domain: DESKTOP-33E7O54

```

Kerberos is a network authentication protocol used by Active Directory in Windows Domain environments. Domain users receive tickets upon authentication that allow access to shared network resources without re-entering credentials. LSASS caches Kerberos `passwords`, `ekeys`, `tickets`, and `pins` that can be extracted from process memory and used to access other domain-joined systems.

**DPAPI:**

```shell-session
	== DPAPI [14ab89]==

		key_guid 3e1d1091-b792-45df-ab8e-c66af044d69b
```

Mimikatz and Pypykatz can extract DPAPI `masterkeys` for logged-on users from LSASS process memory. These masterkeys decrypt DPAPI-protected secrets for various applications, capturing credentials for multiple accounts.

### Cracking the NT Hash with Hashcat

We can use Hashcat to crack the NT Hash. In this example, we only found one NT hash associated with the Bob user. After setting the mode in the command, we can paste the hash, specify a wordlist, and then crack the hash.

```
$ sudo hashcat -m 1000 64f12cddaa88057e06a81b54e73b949b /usr/share/wordlists/rockyou.txt

64f12cddaa88057e06a81b54e73b949b:Password1
```
