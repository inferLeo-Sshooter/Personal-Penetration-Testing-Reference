# 1. The basics

## Template Engines

Software that combine pre-defined templates with dynamic data to generate web content. They're commonly used for shared elements like headers and footers across website pages, avoiding code duplication and improving maintainability. Popular examples include `Jinja` and `Twig`.

**Templating:** Template engines require two inputs: a template (string or file) with predefined placeholders, and key-value pairs for dynamic data. The engine inserts values at locations marked with corresponding keys. This process of generating the final string is called `rendering`.

## SSTI

`Server-side Template Injection` (SSTI) occurs when attackers inject malicious templating code into templates. During rendering, the server may execute this code, potentially allowing complete server takeover.

Template engines securely handle user input when provided as values during rendering, since values are inserted without executing code. SSTI occurs when attackers control the template parameter itself, as engines execute code within templates.

SSTI vulnerabilities arise when:
- User input is inserted into templates before rendering
- Applications render the same template multiple times, treating first rendering output as template input
- Users can directly modify or submit templates

Proper implementation always provides user input as values, never as part of the template string.

# 2. Identifying SSTI

Before exploiting SSTI, you must confirm the vulnerability exists and identify the specific template engine in use. Each engine has `different syntax` and `supported functions`, making engine identification crucial for successful exploitation.

## Confirming SSTI

SSTI identification follows the same approach as other injection vulnerabilities like SQL injection. Inject special characters with semantic meaning in template engines and observe the application's response.

**The common test string** `${{<%[%'"}}%\.` contains special characters from popular template engines and should trigger syntax errors if SSTI exists, similar to how a single quote breaks SQL queries in SQL injection vulnerabilities.

**Code context**
Vulnerabilities can occur when user input is placed within template expressions. For example:
`greeting = getQueryParameter('greeting') engine.render("Hello {{"+greeting+"}}", data)`

With URL: `http://vulnerable-website.com/?greeting=data.username`
This renders as: `Hello Carlos`

This context is easily missed since it doesn't show obvious XSS. To test for server-side template injection:

1. First test for direct XSS by injecting HTML:
   `http://vulnerable-website.com/?greeting=data.username<tag>`
   This typically results in blank output, encoded tags, or errors.

2. Try breaking out using template syntax:
   `http://vulnerable-website.com/?greeting=data.username}}<tag>`
   
If this produces an error or blank output, either you're using wrong syntax or injection isn't possible. If it renders correctly with the HTML (`Hello Carlos<tag>`), this indicates a server-side template injection vulnerability.

## Identifying the Template Engine

To enable the successful exploitation of an SSTI vulnerability, we first need to determine the `template engine used by the web application`. We can utilize slight variations in the behavior of different template engines to achieve this. For instance, consider the following commonly used overview containing slight differences in popular template engines:

![image](https://github.com/user-attachments/assets/4e98f720-54fe-4543-936b-2e008dae7275)

We will start by injecting the payload `${7*7}` and follow the diagram from left to right, depending on the result of the injection. Suppose the injection resulted in a successful execution of the injected payload. In that case, we follow the green arrow; otherwise, we follow the red arrow until we arrive at a resulting template engine.

**Note:** the result of payload `{{7*'7'}}` will enable us to deduce the template engine used by the web application. In `Jinja`, the result will be `7777777`, while in `Twig`, the result will be `49`.

Also, submitting invalid syntax often reveals the template engine through error messages. For example, the invalid expression `<%=foobar%>` triggers this response from Ruby's ERB engine:

```txt
(erb):1:in `<main>': undefined local variable or method `foobar' for main:Object (NameError)
from /usr/lib/ruby/2.5.0/erb.rb:876:in `eval'
from /usr/lib/ruby/2.5.0/erb.rb:876:in `result'
from -e:4:in `<main>'
```

This error message reveals both the template engine (ERB) and sometimes the version, making identification straightforward.

# 3. Exploiting SSTI - Jinja2

Assuming that we have successfully identified that the web application uses the `Jinja` template engine. We will only focus on the SSTI exploitation and thus assume that the SSTI confirmation and template engine identification have already been done in a previous step.

`Jinja` is a template engine commonly used in Python web frameworks such as `Flask` or `Django`. This section will focus on a `Flask` web application. The payloads in other web frameworks might thus be slightly different.

## Information Disclosure

We can exploit the SSTI vulnerability to obtain internal information about the web application, including configuration details and the web application's source code. For instance, we can obtain the web application's configuration using the following SSTI payload:

`{{ config.items() }}`

This payload dumps the entire web application configuration, including any used secret keys, we can prepare further attacks using the obtained information.

We can also execute Python code to obtain information about the web application's source code. We can use the following SSTI payload to dump all available built-in functions:

`{{ self.__init__.__globals__.__builtins__ }}`

## Local File Inclusion (LFI)

We can use Python's built-in function `open` to include a local file. However, we cannot call the function directly; we need to call it from the **`__builtins__`** dictionary we dumped earlier. This results in the following payload to include the file `/etc/passwd`:

`{{ self.__init__.__globals__.__builtins__.open("/etc/passwd").read() }}`

## Remote Code Execution (RCE)

To achieve remote code execution in Python, we can use functions provided by the `os` library, such as `system` or `popen`. However, if the web application has not already imported this library, we must first import it by calling the built-in function `import`. This results in the following SSTI payload:

`{{ self.__init__.__globals__.__builtins__.__import__('os').popen('id').read() }}`

# 4. Exploiting SSTI - Twig

Like in the previous section, we will only focus on the SSTI exploitation and thus assume that the SSTI confirmation and template engine identification have already been done in a previous step. `Twig` is a template engine for the `PHP` programming language.

## Information Disclosure

In Twig, we can use the `_self` keyword to obtain a little information about the current template:

`{{ _self }}`

(However, the amount of information is limited compared to Jinja.)

## Local File Inclusion (LFI)

Reading local files (without using the same way as we will use for RCE) is not possible using internal functions directly provided by Twig. However, the PHP web framework `Symfony` defines additional Twig filters. One of these filters is `file_excerpt` and can be used to read local files:

`{{ "/etc/passwd"|file_excerpt(1,-1) }}`

## Remote Code Execution (RCE)

To achieve remote code execution, we can use a PHP built-in function such as `system`. We can pass an argument to this function by using Twig's `filter` function, resulting in any of the following SSTI payloads:

`{{ ['id'] | filter('system') }}`
