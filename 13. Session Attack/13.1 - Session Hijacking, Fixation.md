

# 1. The basics

## 1.1 - Session Hijacking

Session hijacking occurs when the attacker takes advantage of insecure session identifiers, finds a way to obtain them, and uses them to authenticate to the server and impersonate the victim.

An attacker can obtain a victim's session identifier using several methods, with the most common being:
- Passive Traffic Sniffing
- Cross-Site Scripting (XSS)
- Browser history or log-diving
- Read access to a database containing session information
- Weak session identifiers may also be vulnerable to brute force or prediction attacks

## 1.2 - Session Fixation

Session Fixation occurs when attackers can set a valid session identifier and trick victims into using it. Once the victim logs in with this predetermined session ID, the attacker can hijack their session.

This vulnerability typically happens when session identifiers are accepted from URL parameters or POST data rather than being securely generated by the server.

**Three-Stage Attack Process:**

**Stage 1: Obtain valid session identifier**
Attackers get a valid session ID by browsing the application (many apps assign session IDs to unauthenticated users) or creating an account.

**Stage 2: Fixate the session identifier**
The vulnerability exists when:
* Session IDs remain unchanged from pre-login to post-login `and`
* Session identifiers are accepted from URL parameters or POST data

**Stage 3: Trick the victim**
Attackers craft a malicious URL containing their session ID and lure victims into clicking it. When victims log in through this URL, they unknowingly use the attacker's predetermined session identifier, enabling session hijacking.

## 1.3 - Session Fixation Example

**Part 1: Identify the vulnerability**
Navigate to `oredirect.htb.net`. You'll see a URL like:
`http://oredirect.htb.net/?redirect_uri=/complete.html&token=<RANDOM TOKEN VALUE>`

Using Web Developer Tools (Shift+Ctrl+I), notice the `PHPSESSID` cookie value matches the `token` parameter in the URL. This indicates the application accepts session identifiers from URL parameters.

**Part 2: Exploit the vulnerability**
Open a new private window and visit:
`http://oredirect.htb.net/?redirect_uri=/complete.html&token=IControlThisCookie`

Check the `PHPSESSID` cookie - its value is now `IControlThisCookie`, confirming the session fixation vulnerability.

An attacker could send such a URL to victims. When they log in, the attacker knows their session identifier and can hijack their session.

**Alternative Testing Method:** Blindly append the session parameter to any URL:
`http://insecure.exampleapp.com/login?PHPSESSID=AttackerSpecifiedCookieValue`

If the specified value propagates to the application's session cookie, a session fixation vulnerability exists.


**Vulnerable Code Analysis**

The lab's PHP code demonstrates a classic session fixation vulnerability:

```php
<?php
    if (!isset($_GET["token"])) {
        session_start();
        header("Location: /?redirect_uri=/complete.html&token=" . session_id());
    } else {
        setcookie("PHPSESSID", $_GET["token"]);
    }
?>
```

**Code Breakdown:**

1. **No token parameter:** If `token` isn't provided, the code starts a new session and redirects to `/?redirect_uri=/complete.html&token=[session_id]`

2. **Token parameter exists:** If `token` is present, the code directly sets the `PHPSESSID` cookie to whatever value is in the `token` parameter

**The Vulnerability:**
The `else` block allows attackers to control the session identifier by simply providing any value in the `token` parameter. URLs like `http://oredirect.htb.net/?redirect_uri=/complete.html&token=AttackerSpecifiedCookieValue` will set `PHPSESSID` to the attacker's chosen value, enabling session fixation attacks.

This code accepts user-controlled session identifiers without validation, making it trivial for attackers to fixate sessions.

# 2. Obtaining Session Identifiers without User Interaction

There are multiple attacking techniques through which a bug bounty hunter or penetration tester can obtain session identifiers. These attacking techniques can be split into two categories:

- `Session ID-obtaining attacks without user interaction`
- `Session ID-obtaining attacks requiring user interaction`

## 2.1 - Obtaining Session Identifiers via Traffic Sniffing

Traffic sniffing allows attackers to monitor network traffic by connecting to the same local network as victims, often through available ethernet ports. This technique is commonly used by penetration testers for internal network assessments.

**Key Requirements:**
* Attacker and victim must be on the same local network
* Unencrypted HTTP traffic (remote sniffing is impossible)

**Why HTTP is Vulnerable:**
HTTP transmits data unencrypted, allowing attackers to capture usernames, passwords, and session identifiers. Encrypted protocols like HTTPS (SSL) or IPsec make this information much harder or impossible to obtain.

**Tools:**
Wireshark is a popular packet sniffing tool with built-in filtering capabilities for specific protocols (HTTP, SSH, FTP) or source IP addresses, making it easier to analyze captured traffic.

## 2.2 - Obtaining Session Identifiers Post-Exploitation (Web Server Access)

During post-exploitation, attackers can extract session identifiers and data from compromised web servers by accessing disk storage or memory. While server compromise enables many malicious activities, attackers often prefer low-profile operations like session theft to avoid detection and reduce the risk of discovery.

### PHP

Let us look at where PHP session identifiers are usually stored.

The entry `session.save_path` in `PHP.ini` specifies where session data will be stored.

```
$ locate php.ini
$ cat /etc/php/7.4/cli/php.ini | grep 'session.save_path'
$ cat /etc/php/7.4/apache2/php.ini | grep 'session.save_path'
```

![image](https://github.com/user-attachments/assets/ff7161a2-c768-4708-817d-dbe373dc30a4)

In our default configuration case it's `/var/lib/php/sessions`. Now, please note a victim has to be authenticated for us to view their session identifier. The files an attacker will search for use the name convention `sess_<sessionID>`.

How a PHP session identifier looks on our local setup.

![image](https://github.com/user-attachments/assets/eca21052-76d6-4aef-a77a-7e102cc8838e)

The same PHP session identifier but on the webserver side looks as follows.

```
$ ls /var/lib/php/sessions
$ cat //var/lib/php/sessions/sess_s6kitq8d3071rmlvbfitpim9mm
```

![image](https://github.com/user-attachments/assets/b48890c3-cb79-4068-89b8-fdb83d9999db)

As already mentioned, for a hacker to hijack the user session related to the session identifier above, a new cookie must be created in the web browser with the following values:
- cookie name: PHPSESSID
- cookie value: s6kitq8d3071rmlvbfitpim9mm

### Java

According to the Apache Software Foundation:

"The `Manager` element represents the session manager that is used to create and maintain HTTP sessions of a web application.

Tomcat provides two standard implementations of `Manager`. The default implementation stores active sessions, while the optional one stores active sessions that have been swapped out (in addition to saving sessions across a server restart) in a storage location that is selected via the use of an appropriate `Store` nested element. The filename of the default session data file is `SESSIONS.ser`."

### .NET

Session data can be found in:
- The application worker process (aspnet_wp.exe) - This is the case in the InProc Session mode
- StateServer (A Windows Service residing on IIS or a separate server) - This is the case in the OutProc Session mode
- An SQL Server

## 2.3 - Obtaining Session Identifiers Post-Exploitation (Database Access)

In cases where you have direct access to a database via, for example, SQL injection or identified credentials, you should always check for any stored user sessions.

```
show databases;
use project;
show tables;
select * from users;
```

![image](https://github.com/user-attachments/assets/6b04624b-cfe2-4667-ad15-2ea3d2a24c4d)

Here we can see the users' passwords are hashed. We could spend time trying to crack these; however, there is also a "all_sessions" table. Let us extract data from that table.

```
select * from all_sessions;
select * from all_sessions where id=3;
```

![image](https://github.com/user-attachments/assets/bf5b630c-ca96-4003-8d65-8e15b2d86e70)

Here we have successfully extracted the sessions! You could now authenticate as the user "Developer."
