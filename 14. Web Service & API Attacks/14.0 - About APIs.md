# 1. APIs in a Nutshell

**APIs (Application Programming Interfaces)** are sets of rules that let different applications talk to each other and share data in a controlled way.

## How They Work

Think of an API as a waiter in a restaurant:
- You (the application) make a request
- The waiter (API) takes it to the kitchen (server)
- The kitchen prepares your order (processes the request)
- The waiter brings back your food (returns the data)

## Real Example: Twitter API

When you want to get a specific tweet, you send a request like:
```
GET /1.1/statuses/show.json?id=210462857140252672
Host: api.twitter.com
```

The API returns data in **JSON format** - a text-based way to structure data using:
- **Curly brackets `{}`** for objects
- **Square brackets `[]`** for lists  
- **Key-value pairs** like `"name": "Twitter API"`

## Key Concepts

1. **Endpoints** - Specific URLs where you request different resources
2. **Parameters** - Details you include to specify what you want (like the tweet ID)
3. **Authentication** - Most APIs require access tokens or credentials to prove who you are
4. **Response formats** - Usually JSON or XML containing the requested data

**Bottom line**: APIs let your code use features from other applications without building everything from scratch - like plugging Lego blocks together to create something bigger.


# 2. REST APIs in a Nutshell

**REST (Representational State Transfer) APIs** are the most popular type of API structure on the web.

## Key Characteristics

**1. Resource-Based URLs**
REST organizes everything around "resources" (like tweets, users, posts). Each resource has its own endpoint:
- `https://api.twitter.com/1.1/statuses/show` - Get tweet info
- `https://api.twitter.com/1.1/users/show` - Get user info
- `https://api.twitter.com/1.1/statuses/destroy` - Delete a tweet

**2. Predictable Structure**
REST APIs follow consistent patterns, making them easy to learn. Twitter's pattern is:
```
https://api.twitter.com/1.1/RESOURCE/ACTION
```

So if you know one endpoint, you can guess others:
- `/statuses/show` → `/statuses/destroy`
- `/users/show` → `/users/update`

**3. HTTP Methods Define Actions**
REST uses standard HTTP methods to indicate what you want to do:
- **GET** - Retrieve/read data
- **POST** - Create or update data
- **PUT** - Update existing data
- **DELETE** - Remove data

**4. Return Formats**
REST APIs typically return data in JSON or plaintext - simple, readable formats that any programming language can understand.

## Example
```
POST https://api.twitter.com/1.1/statuses/retweet/
```
This **POST** request to the **statuses/retweet** endpoint tells Twitter: "I want to retweet this tweet."

**Bottom line**: REST APIs are like well-organized filing systems where everything has a predictable address and you use simple verbs (GET, POST, PUT, DELETE) to interact with it.


# 3. SOAP APIs in a Nutshell

**SOAP (Simple Object Access Protocol)** is an older, more rigid API architecture. It's less common in modern apps but still used in legacy systems and IoT applications.

## Key Characteristics

**1. XML-Based**
SOAP uses XML (not JSON) to structure all data. Every message has:
- **Header** - Contains metadata and authentication info
- **Body** - Contains the actual request/response data

**2. SOAP Request Example**
```xml
DELETE / HTTPS/1.1
Host: example.s3.amazonaws.com
<DeleteBucket>
  <Bucket>quotes</Bucket>
  <AWSAccessKeyId>AKIAIOSFODNN7EXAMPLE</AWSAccessKeyId>
  <Timestamp>2006-03-01T12:00:00.183Z</Timestamp>
</DeleteBucket>
```
Parameters are passed as XML tags rather than URL parameters.

**3. SOAP Response Example**
```xml
<DeleteBucketResponse>
  <StatusCode>204</StatusCode>
  <Description>No Content</Description>
</DeleteBucketResponse>
```

**4. WSDL - The API Manual**
SOAP APIs come with **WSDL (Web Services Description Language)** files - essentially a blueprint that describes:
- What endpoints are available
- What parameters each endpoint expects
- How to format requests

**Finding WSDL Files:**
- Add `.wsdl` or `?wsdl` to API endpoints
- Search for URLs containing "wsdl"
- Example: `https://api.example.com/service?wsdl`

## REST vs SOAP Quick Comparison

| REST | SOAP |
|------|------|
| JSON/plaintext | XML only |
| Flexible, simple | Rigid, complex |
| Modern apps | Legacy/enterprise systems |
| Lightweight | Heavier messages |

**Bottom line**: SOAP is like sending formal letters with strict formatting rules, while REST is like texting - SOAP is more structured and verbose but still used in older enterprise systems where you need that formal structure.

# 4. GraphQL APIs in a Nutshell

**GraphQL** is a newer, more flexible API technology that lets you request exactly the data you need in a single call.

## Key Characteristics

**1. Single Endpoint**
Unlike REST (which has multiple endpoints), GraphQL uses ONE endpoint for everything:
- `/graphql`
- `/gql`
- `/g`

**2. Request Only What You Need**
You specify the exact fields you want, and GraphQL returns only those fields - no more, no less.

**Example Request:**
```graphql
query {
  shop {
    name
    primaryDomain {
      url
      host
    }
  }
}
```

**Response (JSON):**
```json
{
  "data": {
    "shop": {
      "name": "example",
      "primaryDomain": {
        "url": "https://example.myshopify.com",
        "host": "example.myshopify.com"
      }
    }
  }
}
```

Want less data? Just request fewer fields:
```graphql
query {
  shop {
    name
  }
}
```

**3. Two Main Operations**

- **Queries** - Fetch data (like REST GET)
- **Mutations** - Create, update, delete data (like REST POST/PUT/DELETE)

**Mutation Example:**
```graphql
mutation {
  customerCreate(
    input: {
      firstName: "John",
      lastName: "Tate",
      email: "john@johns-apparel.com"
    })
  {
    customer {
      id
    }
  }
}
```

**4. Introspection - Built-in API Documentation**
GraphQL has a powerful feature called **introspection** that lets you ask the API about itself:

**Get all available types:**
```graphql
{
  __schema {
    types {
      name
    }
  }
}
```

**Get fields of a specific type:**
```graphql
{
  __type(name: "customer") {
    name
    fields {
      name
    }
  }
}
```

This reveals the API's structure - great for developers, but many organizations disable it in production to prevent hackers from mapping their APIs.

## Quick Comparison

| Feature | REST | GraphQL |
|---------|------|----------|
| Endpoints | Multiple | Single |
| Data returned | Everything | Only what you request |
| Fetching related data | Multiple requests | One request |
| Learning curve | Easy | Steeper |

**Bottom line**: GraphQL is like ordering à la carte - you get exactly what you want in one order, whereas REST is like combo meals - you get predefined sets and might need multiple orders for everything you need.
