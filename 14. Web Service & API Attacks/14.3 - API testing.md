

## 1. API recon

- To start API testing, you first need to find out as much information about the API as possible, to discover its attack surface.
- To begin, you should identify API endpoints.

**Finding API Endpoints:**
API endpoints are locations where APIs receive requests for specific resources. For example:
- `GET /api/books HTTP/1.1 Host: example.com` - endpoint `/api/books` retrieves all books
- Another API endpoint might be, for example, `/api/books/mystery` - which would retrieves mystery books specifically

**Interaction Analysis:**
Once endpoints are identified, determine how to interact with them by discovering:
- **Input data** - required and optional parameters
- **Request types** - supported HTTP methods and media formats  
- **Security controls** - rate limits and authentication mechanisms

This information enables constructing valid HTTP requests for effective API testing.

## 2. API Documentation

APIs are typically documented to help developers understand usage and integration.

**Documentation Types:**
- **Human-readable** - Detailed explanations, examples, and usage scenarios for developers
- **Machine-readable** - Structured formats (JSON/XML) for automated processing, integration, and validation

**Reconnaissance Priority:**
API documentation is often publicly available for external developer use. Always begin reconnaissance by reviewing available documentation first, as it provides the fastest path to understanding API structure and functionality.

### 2.1 - Discovering API documentation

**Discovering API Documentation**

Even when documentation isn't publicly available, you can find it through applications using the API.

**Discovery Methods:**
- Use Burp Scanner to crawl the API
- Browse applications manually with Burp's browser
- Look for common documentation endpoints:
  - `/api`
  - `/swagger/index.html`
  - `/openapi.json`

**Base Path Investigation:**
When you find a resource endpoint like `/api/swagger/v1/users/123`, investigate the base paths:
- `/api/swagger/v1`
- `/api/swagger`
- `/api`

This systematic approach helps uncover hidden documentation and API structure. Also, you can use a list of common paths to find documentation using Intruder.
