

## 1. API recon

- To start API testing, you first need to find out as much information about the API as possible, to discover its attack surface.
- To begin, you should identify API endpoints.

**Finding API Endpoints:**
API endpoints are locations where APIs receive requests for specific resources. For example:
- `GET /api/books HTTP/1.1 Host: example.com` - endpoint `/api/books` retrieves all books
- Another API endpoint might be, for example, `/api/books/mystery` - which would retrieves mystery books specifically

**Interaction Analysis:**
Once endpoints are identified, determine how to interact with them by discovering:
- **Input data** - required and optional parameters
- **Request types** - supported HTTP methods and media formats  
- **Security controls** - rate limits and authentication mechanisms

This information enables constructing valid HTTP requests for effective API testing.

## 2. API Documentation

APIs are typically documented to help developers understand usage and integration.

**Documentation Types:**
- **Human-readable** - Detailed explanations, examples, and usage scenarios for developers
- **Machine-readable** - Structured formats (JSON/XML) for automated processing, integration, and validation

**Reconnaissance Priority:**
API documentation is often publicly available for external developer use. Always begin reconnaissance by reviewing available documentation first, as it provides the fastest path to understanding API structure and functionality.

### 2.1 - Discovering API documentation

Even when documentation isn't publicly available, you can find it through applications using the API.

**Discovery Methods:**
- Use Burp Scanner to crawl the API
- Browse applications manually with Burp's browser
- Look for common documentation endpoints:
  - `/api`
  - `/swagger/index.html`
  - `/openapi.json`

**Base Path Investigation:**
When you find a resource endpoint like `/api/swagger/v1/users/123`, investigate the base paths:
- `/api/swagger/v1`
- `/api/swagger`
- `/api`

This systematic approach helps uncover hidden documentation and API structure. Also, you can use a list of common paths to find documentation using Intruder.

### 2.2 - Documentation Analysis Tools

**Burp Suite:**
- Burp Scanner can crawl and audit OpenAPI documentation in JSON or YAML formats
- OpenAPI Parser BApp provides specialized parsing capabilities

**Specialized Testing Tools:**
- **Postman** - Test documented endpoints directly
- **SoapUI** - Alternative tool for endpoint testing

These tools streamline the process of analyzing API documentation and testing the discovered endpoints.

## 3. Identifying API endpoints

Browse applications using the API to gather information, even when documentation exists, as documentation may be inaccurate or outdated.

**Discovery Methods:**
- **Burp Scanner** - Crawl applications and manually investigate attack surface with Burp's browser
- **URL Pattern Analysis** - Look for API endpoint patterns like `/api/` in URL structures
- **JavaScript Analysis** - Review JS files for API endpoint references not directly triggered by browsers

**JavaScript Endpoint Extraction:**
- Burp Scanner automatically extracts some endpoints during crawls
- **JS Link Finder BApp** - Provides more comprehensive endpoint extraction
- **Manual Review** - Examine JavaScript files directly in Burp for additional endpoints

## 4. Interacting with API endpoints

Once endpoints are identified, interact with them using **Burp Repeater** and **Burp Intruder** to observe API behavior and discover additional attack surface.

**Testing Approach:**
- Change HTTP methods and media types to test API responses
- Analyze error messages and responses closely - they often contain information for constructing valid HTTP requests
- Use response details to refine and improve your testing methodology

This interaction phase reveals how the API behaves under different conditions and exposes potential vulnerabilities.

### 4.1 - Identifying supported HTTP methods

HTTP methods specify actions performed on resources:
- **GET** - Retrieves data
- **PATCH** - Applies partial changes  
- **OPTIONS** - Shows supported request methods

**Testing Importance:**
API endpoints may support multiple HTTP methods, so test all potential methods to identify additional functionality and attack surface.

**Example - `/api/tasks` endpoint:**
- `GET /api/tasks` - Retrieves task list
- `POST /api/tasks` - Creates new task
- `DELETE /api/tasks/1` - Deletes specific task

**Automation:**
Use Burp Intruder's built-in **HTTP verbs** list to automatically cycle through different methods for comprehensive testing.

**Note:**
> - When testing different HTTP methods, target low-priority objects. This helps make sure that you avoid unintended consequences, for example altering critical items or creating excessive records.
