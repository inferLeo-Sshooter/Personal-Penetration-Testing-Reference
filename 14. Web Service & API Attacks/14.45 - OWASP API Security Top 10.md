## 1. Broken Object Level Authorization

Web APIs enable users to request data using parameters like `Universally Unique Identifiers` (`UUIDs`), `Globally Unique Identifiers` (`GUIDs`), and integer IDs. Without proper `object-level authorization mechanisms` to verify user ownership and permissions, these requests can expose data and create security vulnerabilities.

A web API endpoint is vulnerable to `Broken Object Level Authorization` (`BOLA`), or `Insecure Direct Object Reference` (`IDOR`), when its authorization checks fail to verify that authenticated users have sufficient permissions to access specific data or perform operations.

Reference: [CWE-639: Authorization Bypass Through User-Controlled Key](https://cwe.mitre.org/data/definitions/639.html).

---

Example:

**Step 1: Authenticate**
- Sign in using provided credentials (`htbpentester1@pentestercompany.com:HTBPentester1`) via `/api/v1/authentication/suppliers/sign-in`
- Obtain and copy the JWT from the response
- Click `Authorize` in Swagger and paste the JWT to authenticate

**Step 2: Identify Your Company**
- Invoke `/api/v1/suppliers/current-user` endpoint
- Note your company ID: `b75a7c76-e149-4ca7-9c55-d9fc4ffa87be`

**Step 3: Check Your Roles**
- Invoke `/api/v1/roles/current-user` endpoint
- Confirm you have the `SupplierCompanies_GetYearlyReportByID` role

**Step 4: Discover the Vulnerable Endpoint**
- Locate `/api/v1/supplier-companies/yearly-reports/{ID}` in the Supplier-Companies group
- Notice it accepts an integer `ID` parameter (not a Guid)

**Step 5: Exploit BOLA**
- Test with `ID=1` - retrieves a report from company `f9e58492-b594-4d82-a4de-16e4f230fce1` (not yours!)
- Try other IDs - successfully access other companies' sensitive yearly reports

**Step 6: Mass Exploitation**
- Use a bash for-loop to fetch multiple reports:

```shell-session
$ for ((i=1; i<= 20; i++)); do
curl -s -w "\n" -X 'GET' \
  'http://[TARGET]/api/v1/supplier-companies/yearly-reports/'$i'' \
  -H 'Authorization: Bearer [JWT_TOKEN]' | jq
done
```

**Result:** Access to 20+ companies' confidential revenue data and C-level comments.

## 2. Broken Authentication

An API has Broken Authentication when its authentication mechanisms can be bypassed or circumvented. This often involves weak password policies that allow brute-force attacks or easily guessable credentials, enabling attackers to hijack legitimate user accounts.

Reference: [CWE-307: Improper Restriction of Excessive Authentication Attempts.](https://cwe.mitre.org/data/definitions/307.html)

---

**Steps to Exploit Broken Authentication:**

**Step 1: Initial Authentication**
- Sign in with provided credentials (`htbpentester3@hackthebox.com:HTBPentester3`) via `/api/v1/authentication/customers/sign-in`
- Obtain JWT and authenticate

**Step 2: Reconnaissance**
- Invoke `/api/v1/customers/current-user` to view your user information
- Check `/api/v1/roles/current-user` - discover roles: `Customers_UpdateByCurrentUser`, `Customers_Get`, `Customers_GetAll`

**Step 3: Discover Weak Password Policy**
- Use `/api/v1/customers` endpoint with `Customers_GetAll` role to retrieve all customer records
- Expand `/api/v1/customers/current-user` PATCH endpoint
- Test password update with 'pass' - rejected (minimum 6 characters required)
- Test with '123456' - accepted! This reveals weak password policy

**Step 4: Identify Brute-Force Target**
- Test `/api/v1/authentication/customers/sign-in` with incorrect credentials
- Note the fail message: 'Invalid Credentials'
- Save high-value target emails to file:
  - `OlawaleJones@yandex.com`
  - `IsabellaRichardson@gmail.com`
  - `WenSalazar@zoho.com`

**Step 5: Execute Brute-Force Attack**
- Use `ffuf` with two wordlists (emails and passwords from SecLists):

```shell-session
$ ffuf -w /opt/useful/seclists/Passwords/xato-net-10-million-passwords-10000.txt:PASS -w customerEmails.txt:EMAIL -u http://[TARGET]/api/v1/authentication/customers/sign-in -X POST -H "Content-Type: application/json" -d '{"Email": "EMAIL", "Password": "PASS"}' -fr "Invalid Credentials" -t 100
```

**Step 6: Account Takeover**
- Discover credentials: `IsabellaRichardson@gmail.com:qwerasdfzxcv`
- Sign in as Isabella using `/api/v1/authentication/customers/sign-in`
- Access all her confidential information

**Additional Attack Vector:**
If password brute-forcing fails due to strong policies, attempt brute-forcing OTPs or security question answers if they have low entropy and lack rate-limiting.

## 3. Broken Object Property Level Authorization

This vulnerability category includes two types:
- **Excessive Data Exposure**: API reveals sensitive data to authorized users who shouldn't access it
- **Mass Assignment**: API allows authorized users to manipulate sensitive object properties beyond their authorized scope (modify, add, or delete values)

---

**Example 1: Excessive Data Exposure**

Reference: [CWE-213: Exposure of Sensitive Information Due to Incompatible Policies.](https://cwe.mitre.org/data/definitions/213.html)

**Steps to Exploit Exposure of Sensitive Information:**

**Step 1: Authenticate as Customer**
- Sign in using credentials (`htbpentester4@hackthebox.com:HTBPentester4`) via `/api/v1/authentication/customers/sign-in`
- Obtain JWT

**Step 2: Check Permissions**
- Invoke `/api/v1/roles/current-user`
- Discover roles: `Suppliers_Get` and `Suppliers_GetAll`

**Step 3: Identify Data Exposure**
- Invoke `/api/v1/suppliers` GET endpoint
- Response includes expected fields (`id`, `companyID`, `name`)
- **But also exposes sensitive fields**: `email` and `phoneNumber`

**Impact:** Customers can bypass the marketplace and contact suppliers directly for discounted purchases, damaging marketplace revenue.

---

**Example 2: Mass Assignment**

Reference: [CWE-915: Improperly Controlled Modification of Dynamically-Determined Object Attributes.](https://cwe.mitre.org/data/definitions/915.html)

**Steps to Exploit Improperly Controlled Object Modification:**

**Step 1: Authenticate as Supplier**
- Sign in using credentials (`htbpentester6@pentestercompany.com:HTBPentester6`) via `/api/v1/authentication/suppliers/sign-in`
- Obtain JWT

**Step 2: Check Permissions**
- Invoke `/api/v1/roles/current-user`
- Discover roles: `SupplierCompanies_Update` and `SupplierCompanies_Get`

**Step 3: Check Current Company Status**
- Invoke `/api/v1/supplier-companies/current-user`
- Note: `isExemptedFromMarketplaceFee` field is set to `0` (false)
- Company 'PentesterCompany' must pay marketplace fees

**Step 4: Identify Mass Assignment Vulnerability**
- Expand `/api/v1/supplier-companies` PATCH endpoint
- Notice it accepts `isExemptedFromMarketplaceFee` field in the request

**Step 5: Exploit the Vulnerability**
- Update `isExemptedFromMarketplaceFee` to `1` via PATCH request
- Receive success message

**Step 6: Verify Exploitation**
- Re-check company info using `/api/v1/supplier-companies/current-user`
- Confirm: `isExemptedFromMarketplaceFee` is now `1` (true)

**Impact:** Supplier companies avoid marketplace fees, increasing their revenue while damaging marketplace stakeholder profits.

## 4. Unrestricted Resource Consumption

An API is vulnerable to Unrestricted Resource Consumption when it fails to limit user-initiated requests that consume resources like network bandwidth, CPU, memory, and storage. Without proper rate-limiting and validation, attackers can exploit file upload/download features to cause financial damage through excessive resource usage or denial-of-service attacks.

Reference: [CWE-400: Uncontrolled Resource Consumption.](https://cwe.mitre.org/data/definitions/400.html)

---

**Steps to Exploit Unrestricted Resource Consumption:**

**Step 1: Authenticate as Supplier**
- Sign in using credentials (`htbpentester8@pentestercompany.com:HTBPentester8`) via `/api/v1/authentication/suppliers/sign-in`
- Obtain JWT

**Step 2: Check Permissions**
- Invoke `/api/v1/roles/current-user`
- Discover roles: `SupplierCompanies_Get` and `SupplierCompanies_UploadCertificateOfIncorporation`

**Step 3: Identify Vulnerable Upload Endpoint**
- Locate `/api/v1/supplier-companies/certificates-of-incorporation` POST endpoint in Supplier-Companies group
- Note: Allows uploading certificate of incorporation as PDF, stored indefinitely

**Step 4: Get Company ID**
- Invoke `/api/v1/supplier-companies/current-user`
- Obtain supplier-company ID: `b75a7c76-e149-4ca7-9c55-d9fc4ffa87be`

**Step 5: Test File Size Limits**
- Generate large random PDF file (30MB):
```shell-session
$ dd if=/dev/urandom of=certificateOfIncorporation.pdf bs=1M count=30
```
- Upload via endpoint - **Success!**
- No file size validation exists
- Repeated uploads without rate-limiting can fill disk storage (DoS)

**Step 6: Test File Type Validation**
- Generate executable file:
```shell-session
$ dd if=/dev/urandom of=reverse-shell.exe bs=1M count=10
```
- Upload .exe file via endpoint - **Success!**
- No file extension validation
- Files stored in `wwwroot/SupplierCompaniesCertificatesOfIncorporations`

**Step 7: Exploit Default ASP.NET Core Behavior**
- Note: API built with ASP.NET Core (per admin)
- Default behavior: Static files in `wwwroot` are publicly accessible
- Download uploaded malicious file:
```shell-session
$ curl -O http://[TARGET]/SupplierCompaniesCertificatesOfIncorporations/reverse-shell.exe
```
- **Success!** File is publicly downloadable

**Impact:**
- **DoS**: Unlimited uploads can consume all disk storage, causing financial losses
- **Malware Distribution**: API can be abused as cloud storage for malware distribution
- **Data Exposure**: Potential access to sensitive files if directory enumeration is possible
- **Social Engineering**: Uploaded executables could compromise admin systems

## 5. Broken Function Level Authorization

An API has Broken Function Level Authorization (BFLA) when it allows unauthorized or unprivileged users to invoke privileged endpoints, accessing sensitive operations or confidential information. Unlike BOLA (where users are authorized but access wrong objects), in BFLA the user shouldn't have access to the endpoint at all.

Reference: [CWE-200: Exposure of Sensitive Information to an Unauthorized Actor.](https://cwe.mitre.org/data/definitions/200.html)

---

**Steps to Exploit Broken Function Level Authorization:**

**Step 1: Authenticate as Customer**
- Sign in using credentials (`htbpentester9@hackthebox.com:HTBPentester9`) via `/api/v1/authentication/customer/sign-in`
- Obtain JWT

**Step 2: Identify Target Endpoint**
- Locate `/api/v1/products/discounts` endpoint in Products group
- Note: Endpoint retrieves all product discounts
- Note: Requires `ProductDiscounts_GetAll` role (per documentation)

**Step 3: Check Your Permissions**
- Invoke `/api/v1/roles/current-user`
- Discover: **No roles assigned** to current user

**Step 4: Test Authorization Bypass**
- Despite having no roles, attempt to invoke `/api/v1/products/discounts`
- **Success!** Endpoint returns all product discount data

**Step 5: Confirm Vulnerability**
- Endpoint responds with sensitive data even though user lacks required `ProductDiscounts_GetAll` role
- Authorization check was not properly implemented

**Impact:** Unauthorized users can access privileged endpoints and retrieve confidential business information (product discounts), despite developers intending role-based access control. This exposes sensitive data to unprivileged accounts.

## 6. Unrestricted Access to Sensitive Business Flows

An API is vulnerable when it exposes operations or data that allows users to abuse business logic and undermine the system (e.g., exploiting discount timing, manipulating pricing). The vulnerability occurs when sensitive business flows lack appropriate access restrictions, enabling users to gain unfair competitive or financial advantages.

---

**Steps to Exploit Unrestricted Access to Sensitive Business Flows:**

**Step 1: Leverage Previous BFLA Vulnerability**
- Use previously exploited `/api/v1/products/discounts` endpoint
- Access comprehensive product discount data (unauthorized)

**Step 2: Analyze Discount Intelligence**
- Review exposed discount schedules and rates
- Example finding: Product ID `a923b706-0aaa-49b2-ad8d-21c97ff6fac7`
  - Discount period: `2023-03-15` to `2023-09-15`
  - Discount rate: **70% off** original price

**Step 3: Plan Business Flow Abuse**
- Identify optimal purchase timing (when discounts begin)
- Calculate potential profit margins

**Step 4: Exploit Combined Vulnerabilities**
- If purchase endpoint lacks rate-limiting (Unrestricted Resource Consumption):
  - Buy entire stock on discount start date
  - Acquire products at 70% discount
  - Resell later at original or higher price after discount ends

**Impact:** 
- **Financial exploitation**: Users can manipulate market timing for profit
- **Inventory manipulation**: Stock depletion through bulk purchasing during discounts
- **Competitive advantage**: Unauthorized knowledge of pricing strategies
- **Revenue loss**: Legitimate customers miss opportunities while attackers profit from reselling

## 7. Server Side Request Forgery

An API is vulnerable to Server-Side Request Forgery (SSRF), also known as Cross-Site Port Attack (XPSA), when it fetches remote or local resources using user-controlled input without validation. This allows attackers to force the application to send crafted requests to unexpected destinations (especially internal resources), bypassing firewalls or VPNs and accessing sensitive local files.

Reference: [CWE-918: Server-Side Request Forgery (SSRF).](https://cwe.mitre.org/data/definitions/918.html)

---

**Steps to Exploit Server Side Request Forgery:**

**Step 1: Authenticate as Supplier**
- Sign in using credentials (`htbpentester10@pentestercompany.com:HTBPentester10`) via `/api/v1/authentication/suppliers/sign-in`
- Obtain JWT

**Step 2: Check Permissions**
- Invoke `/api/v1/roles/current-user`
- Discover roles: `SupplierCompanies_Update` and `SupplierCompanies_UploadCertificateOfIncorporation`

**Step 3: Identify Related Endpoints**
- Locate three relevant endpoints in Supplier-Companies group:
  - `/api/v1/supplier-companies` (PATCH)
  - `/api/v1/supplier-companies/{ID}/certificates-of-incorporation` (GET)
  - `/api/v1/supplier-companies/certificates-of-incorporation` (POST)

**Step 4: Get Company ID**
- Invoke `/api/v1/supplier-companies/current-user`
- Obtain supplier-company ID: `b75a7c76-e149-4ca7-9c55-d9fc4ffa87be`

**Step 5: Upload Legitimate Certificate**
- Expand `/api/v1/supplier-companies/certificates-of-incorporation` POST endpoint
- Upload any PDF file with company ID
- Note response contains `fileURI` field using **file URI Scheme** (e.g., `file:///path/to/file.pdf`)

**Step 6: Verify File URI Storage**
- Re-invoke `/api/v1/supplier-companies/current-user`
- Confirm `certificateOfIncorporationPDFFileURI` now contains uploaded file's URI

**Step 7: Identify Mass Assignment Vulnerability**
- Expand `/api/v1/supplier-companies` PATCH endpoint
- Notice it allows modifying `CertificateOfIncorporationPDFFileURI` field
- This field should only be set by upload endpoint (Mass Assignment flaw)

**Step 8: Exploit SSRF**
- Use PATCH endpoint to update `CertificateOfIncorporationPDFFileURI` to `file:///etc/passwd`
- API doesn't validate the file path

**Step 9: Retrieve Sensitive File**
- Invoke `/api/v1/supplier-companies/{ID}/certificates-of-incorporation` GET endpoint
- Receive base64-encoded contents of `/etc/passwd`

**Step 10: Decode and Verify**
- Use CyberChef or similar tool to decode base64 data
- Successfully retrieve `/etc/passwd` contents from backend server

**Step 11: Further Exploitation**
- Access other critical files like `/etc/shadow`
- Potentially compromise entire system

**Impact:** Attackers can read sensitive local files from the backend server, bypassing security controls and accessing confidential system information that could lead to complete system compromise.

## 8. Security Misconfiguration SQLi

Web APIs are vulnerable to the same security misconfigurations as traditional web applications. Common examples include accepting user-controlled input in SQL queries without validation (enabling SQL injection attacks) and improper HTTP security headers (like insecure CORS policies that can lead to CSRF attacks).

Reference: [CWE-89: Improper Neutralization of Special Elements used in an SQL Command ('SQL Injection').](https://cwe.mitre.org/data/definitions/89.html)

---

**Steps to Exploit Security Misconfiguration (SQL Injection):**

**Step 1: Authenticate as Supplier**
- Sign in using credentials (`htbpentester12@pentestercompany.com:HTBPentester12`) via `/api/v1/authentication/suppliers/sign-in`
- Obtain JWT and authenticate

**Step 2: Check Permissions**
- Invoke `/api/v1/roles/current-user`
- Discover role: `Products_GetProductsTotalCountByNameSubstring`

**Step 3: Identify Target Endpoint**
- Locate `/api/v1/products/{Name}/count` endpoint in Products group
- Endpoint returns total count of products containing a user-provided substring in their name

**Step 4: Test Normal Functionality**
- Use `laptop` as the `Name` parameter
- Response: 18 matching products found
- Endpoint works as expected

**Step 5: Test for SQL Injection**
- Input `laptop'` (with trailing apostrophe)
- **Error message returned** - indicates potential SQL injection vulnerability
- Backend likely concatenating user input directly into SQL query

**Step 6: Exploit SQL Injection**
- Use payload: `laptop' OR 1=1 --`
- Bypass substring filter to retrieve total count of all products
- Response: **720 products** in the Products table
- Successfully exploited SQL injection

**Step 7: Further Exploitation Potential**
- Extract database schema information
- Retrieve sensitive data from other tables
- Potentially compromise entire database

---

**Additional Security Misconfiguration: HTTP Headers**

**Vulnerability:** APIs lacking proper HTTP Security Response Headers face additional risks:
- **Insecure CORS Policy**: Missing or misconfigured `Access-Control-Allow-Origin` header
- **Impact**: Exposes API to Cross-Site Request Forgery (CSRF) attacks, allowing malicious sites to make unauthorized requests on behalf of authenticated users

## 9. Improper Inventory Management

APIs suffer from Improper Inventory Management when outdated or deprecated versions remain accessible without proper security controls. Poor versioning practices create security misconfigurations and expand the attack surface, allowing unauthorized access to legacy endpoints that may expose sensitive data or lack authentication requirements.

---

**Steps to Exploit Improper Inventory Management:**

**Step 1: Discover API Versions**
- Primary interaction has been with `v1` of Inlanefreight E-Commerce Marketplace API
- Check Swagger UI's "Select a definition" drop-down
- **Discovery:** Additional version `v0` exists

**Step 2: Analyze Legacy Version**
- Review `v0` description
- Note: Contains legacy and deleted data
- Marked as unmaintained backup that should be removed
- **Critical finding:** Endpoints show **no lock icons** = no authentication required

**Step 3: Identify Vulnerable Endpoint**
- Locate `/api/v0/customers/deleted` endpoint
- Endpoint accessible without any authentication

**Step 4: Exploit Unauthenticated Access**
- Invoke `/api/v0/customers/deleted` endpoint
- **Success!** API returns deleted customer data including:
  - Customer information
  - **Sensitive password hashes**

**Step 5: Leverage Combined Vulnerabilities**
- **Improper Inventory Management**: Legacy `v0` not removed
- **Broken Authentication**: No authentication required on `v0` endpoints
- **Excessive Data Exposure**: Endpoint exposes password hashes

**Step 6: Post-Exploitation Options**
- Extract all password hashes from deleted customers
- Perform password cracking attacks
- Exploit password reuse:
  - Same customers may have re-registered
  - Likely using same passwords
  - Cracked passwords could compromise active accounts

**Impact:** 
- **Unauthorized data access**: Deleted customer records exposed without authentication
- **Credential compromise**: Password hashes available for offline cracking
- **Account takeover**: Re-registered customers vulnerable if they reused passwords
- **Compliance violations**: Exposure of supposedly deleted customer data

## 10. Unsafe Consumption of APIs

APIs often interact with other APIs to exchange data, creating interconnected service ecosystems. When developers blindly trust data from third-party APIs (especially from reputable sources), they may relax security measures like input validation and data sanitization, introducing significant vulnerabilities in API-to-API communications.

Reference: [CWE-1357: Reliance on Insufficiently Trustworthy Component.](https://cwe.mitre.org/data/definitions/1357.html)

---

**Critical Vulnerabilities in API-to-API Communication:**

**1. Insecure Data Transmission**
- **Issue:** APIs communicating over unencrypted channels (HTTP instead of HTTPS)
- **Impact:** Sensitive data exposed to interception, compromising confidentiality and integrity
- **Attack:** Man-in-the-middle attacks to capture credentials, tokens, or business data

**2. Inadequate Data Validation**
- **Issue:** Failing to validate and sanitize data received from external APIs before processing
- **Impact:** 
  - Injection attacks (SQL, Command, XSS)
  - Data corruption
  - Remote code execution
- **Root cause:** Blind trust in third-party API responses

**3. Weak Authentication**
- **Issue:** Missing or insufficient authentication when communicating with other APIs
- **Impact:** Unauthorized access to sensitive data or critical functionality
- **Examples:** 
  - No API keys required
  - Weak or hardcoded credentials
  - Missing mutual TLS authentication

**4. Insufficient Rate-Limiting**
- **Issue:** No limits on outbound requests to external APIs
- **Impact:** 
  - API can overwhelm downstream services
  - Denial-of-service conditions
  - Cascading failures across service ecosystem
  - Financial costs from excessive API usage

**5. Inadequate Monitoring**
- **Issue:** Insufficient logging and monitoring of API-to-API interactions
- **Impact:** 
  - Difficulty detecting security incidents
  - Delayed incident response
  - Unable to trace attack chains
  - Lack of audit trail for compliance

---

**Key Principle:** Never blindly trust data from external APIs, regardless of reputation. Always implement defense-in-depth strategies including encryption, validation, strong authentication, rate-limiting, and comprehensive monitoring for all API-to-API communications.
