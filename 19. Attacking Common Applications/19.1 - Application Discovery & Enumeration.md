# About Application Discovery & Enumeration

Pentesters must master enumeration starting with minimal information. The typical workflow:

1. Ping sweep → identify live hosts
2. Targeted port scanning
3. Deep service identification
4. Challenge: Managing data from 100s-1000s of hosts manually is inefficient

In large networks with hundreds/thousands of hosts, manual data review becomes **impractical**.

Tools like `EyeWitness` and `Aquatone` automate this process. Both accept Nmap XML output (Aquatone also takes Masscan; EyeWitness accepts Nessus XML) to screenshot web applications and generate browsable reports. This efficiently narrows down hundreds of potential targets into a focused attack list.

- Input: Nmap XML (Aquatone: Masscan; EyeWitness: Nessus XML too)
- Output: Screenshots + browsable HTML reports
- Available on Windows and Linux

## 1. Getting Organized

Choosing and setting up a notetaking system early is crucial for penetration testing. Popular tools include OneNote, Evernote, Notion, and Cherrytree—the choice depends on personal preference. Develop a consistent notetaking methodology and create reusable templates for different assessment types.

```
Options: OneNote, Evernote, Notion, Cherrytree
Pick one and stick with it
Create templates for each assessment type
```

For application discovery, create a dedicated section separate from general enumeration. Include subsections for scope, scans (Nmap, Nessus, Masscan), application screenshots, and notable hosts. Always timestamp every scan, save all output and exact command syntax, and record targeted hosts. This documentation helps address client questions about assessment activity.

```
Application Discovery Organization:

- Separate section from general enumeration
- Key subsections:

  * Scope details
  * Scans (Nmap, Nessus, Masscan, etc.)
  * Application screenshots
  * Interesting/notable hosts for deeper investigation


Critical Documentation Practices:

- Timestamp EVERY scan
- Save all output files
- Record exact command syntax used
- Document targeted hosts
- Purpose: Answer client questions about observed activity
```

Detailed organization from the start streamlines final reporting. Set up the report skeleton and notebook structure at the beginning, allowing you to fill sections while scans run. This saves time at the end, provides more testing time, and ensures thoroughness.

- Example OneNote Structure:

```
External Penetration Test - <Client Name>
├── Scope (IPs, URLs, fragile hosts, timeframes, limitations)
├── Client Points of Contact
├── Credentials
├── Discovery/Enumeration
│   ├── Scans
│   └── Live hosts
├── Application Discovery
│   ├── Scans
│   └── Interesting/Notable Hosts
├── Exploitation
│   ├── <Hostname or IP>
│   └── <Hostname or IP>
└── Post-Exploitation
    ├── <Hostname or IP>
    └── <Hostname or IP>
```

## 2. Initial Enumeration

**Starting Point:**
- Client provides scope list (hostnames + IPs)
- Example domains: app.inlanefreight.local, dev.inlanefreight.local, etc.

**Initial Scan Strategy:**
- Nmap common web ports: `80, 443, 8000, 8080, 8180, 8888, 10000`
- Command: `nmap -p 80,443,8000,8080,8180,8888,10000 --open -oA web_discovery -iL scope_list`
- Run EyeWitness/Aquatone on results
- Follow up with deeper scans (top 10K or all TCP ports)

**Enumeration Philosophy:**
- Iterative process—screenshot after each scan round
- Non-evasive tests: Add Nessus for comprehensive data
- **Critical**: Don't rely solely on automated scanners
- Human testing finds most unique/severe vulns
- Scanners = data inputs for manual validation

**Analyzing Results:**
- Identify OS types from ports (Linux vs. Windows)
- Pay attention to hostnames:
  - `dev` = development environments (untested features, debug mode)
  - `gitlab-dev` = potential credential/sensitive data exposure
  - Generic names (`app.inlanefreight.local`) need further enum

**Interesting Targets to Note:**
- GitLab instances:
  - May allow public repos with sensitive data
  - Self-registration without admin approval
  - Check commit history for credentials

**Service Version Scanning:**
- Use `-sV` flag for detailed service identification
- Example command: `nmap --open -sV 10.129.201.50`
- Identifies specific apps: IIS, Splunk, PRTG Network Monitor
- Reveals ports and versions for targeted exploitation

**Example Findings:**
- Port 80: Microsoft IIS
- Ports 8000/8089: Splunk
- Port 8080: PRTG Network Monitor
- Port 3389: RDP (Windows target)

**Key Takeaway:**
- In large environments, manual host-by-host enum is inefficient
- Risk missing critical applications without systematic approach

## 3. Using EyeWitness

EyeWitness accepts XML output from Nmap and Nessus to create screenshot reports of web applications using Selenium. It categorizes and fingerprints applications, suggests default credentials, and performs DNS resolution. It can work with IP lists, URLs, or scan output, and will prepend `http://` and `https://` to targets with specified ports.

**Installation options:**
- `sudo apt install eyewitness`
- Clone repo and run `Python/setup/setup.sh`
- Docker container
- Windows version via Visual Studio

Run with `--web` flag using Nmap XML as input. The tool screenshots all discovered web services and generates an HTML report that can be opened immediately after completion.

**EyeWitness Capabilities:**
- Accepts: Nmap XML, Nessus XML, IP lists, URLs
- Uses Selenium for screenshots
- Categorizes applications automatically
- Fingerprints applications
- Suggests default credentials based on app type
- Performs DNS resolution for IPs
- Can specify custom port lists

**Basic Usage:**
```bash
eyewitness --web -x web_discovery.xml -d inlanefreight_eyewitness
```

**Command Breakdown:**
- `--web`: Screenshot web services
- `-x`: Input Nmap XML file
- `-d`: Output directory name

**What It Does:**
- Processes all hosts from scan
- Attempts screenshots of each service
- Tries both HTTP and HTTPS
- Screenshots all discovered ports (80, 8000, 8080, etc.)
- Handles timeouts and retries automatically
- Shows progress counter (e.g., "Completed 15 out of 26 services")
- Generates final report in specified directory

**Output:**
- HTML report with all screenshots
- Organized by application type
- Completion time displayed
- Prompt to open report immediately

**Efficiency Benefit:**
- Finished 26 hosts in ~58 seconds
- Much faster than manual browsing
- Visual overview of entire attack surface

**In a Nutshell:**

EyeWitness automates web application reconnaissance by taking Nmap/Nessus XML output and using Selenium to screenshot every discovered web service. It fingerprints applications, categorizes them, and suggests default credentials—all while generating an organized HTML report. Installation is simple via apt or GitHub, and running it against scan results provides a complete visual inventory of the web attack surface in under a minute, dramatically more efficient than manually browsing each host/port combination.

## 4. Using Aquatone

Aquatone is similar to EyeWitness, taking screenshots from `.txt` host lists or Nmap `.xml` files (using `-nmap` flag). Download the precompiled binary, extract it, and optionally move it to a `$PATH` location like `/usr/local/bin` for system-wide access, or keep it in your working directory.

The tool processes Nmap XML input, screenshots all discovered web services, and provides detailed statistics including HTTP response codes (2xx, 3xx, 4xx, 5xx). It also calculates page structures, clusters similar pages, and generates an HTML report. Aquatone displays timing information (start/finish/duration) and success/failure counts for both requests and screenshots.

**Note:** Aquatone is under active development in a new fork with improvements and enhancements.


**Installation:**
```bash
# Download precompiled binary
wget https://github.com/michenriksen/aquatone/releases/download/v1.7.0/aquatone_linux_amd64_1.7.0.zip

# Extract
unzip aquatone_linux_amd64_1.7.0.zip
# Contains: aquatone, README.md, LICENSE.txt
```

**Setup Options:**
- Move to `$PATH` location: `mv aquatone /usr/local/bin/`
- Or keep in working directory
- Pro tip: Build attack VMs with tools in `$PATH` for efficiency

**Input Options:**
- `.txt` file with host lists
- Nmap `.xml` file (use `-nmap` flag)
- Can also accept Masscan XML

**Basic Usage:**
```bash
cat web_discovery.xml | ./aquatone -nmap
```

**Default Settings:**
- Threads: 6
- Ports scanned: 80, 443, 8000, 8080, 8443
- Output: Current directory

**What It Does:**
1. Takes screenshots of all web services
2. Shows HTTP response codes for each target
3. Calculates page structures
4. Clusters similar pages together
5. Generates HTML report

**Output Statistics:**
- **Time**: Start time, finish time, duration
- **Requests**: Successful/failed counts
- **Response codes**: Breakdown by category (2xx, 3xx, 4xx, 5xx)
- **Screenshots**: Success/failure counts
- **Report**: `aquatone_report.html`

**Example Results:**
- 65 targets processed in 33 seconds
- 47 successful (2xx) responses
- 18 forbidden (4xx) responses
- 65 successful screenshots
- 0 failures

**Advanced Features:**
- Page structure calculation
- Similar page clustering (groups duplicate/similar content)
- Organized HTML report output

**Status Note:**
- Active development in new fork
- Check repository for latest installation guide

**In a nutshell**: 

Aquatone is a web reconnaissance tool that screenshots web services from Nmap XML or text host lists. After simple installation (download and extract), pipe your Nmap results with the `-nmap` flag to automatically screenshot all discovered web applications. It provides detailed statistics (HTTP codes, timing, success rates), intelligently clusters similar pages to reduce duplicate review, and generates a comprehensive HTML report—all in about 30 seconds for 65+ targets. Move the binary to your `$PATH` for convenient system-wide access.

## 5. Interpreting the Results

Screenshot reports save significant time, especially in large environments (500-5,000+ hosts). Reports organize findings into categories, with "High Value Targets" listed first. Even massive reports (hundreds of pages) warrant complete review—valuable targets can be buried deep. One example: ManageEngine OpManager with default credentials (`admin:admin`) exposed to the internet led to full domain compromise via a Domain Admin context.

**Key targets to prioritize:**
- **Tomcat**: Test `/manager` and `/host-manager` with default credentials; successful login allows malicious WAR file upload for RCE
- **Custom web applications**: May contain numerous vulnerabilities
- **CMS platforms**: WordPress, Joomla, Drupal
- **Support systems**: osTicket and similar ticketing systems may expose sensitive data or enable social engineering attacks (registering company domain emails for access to other services)

Stay in information gathering mode—don't attack immediately. Rushing into testing creates rabbit holes and risks missing critical findings. During discovery, note URLs, application names/versions, and unusual findings.

**External pentests typically reveal:**
- Custom applications, CMS, Tomcat, Jenkins, Splunk
- Remote access portals (RDS, SSL VPN, OWA, O365)
- Edge network device login pages
- Sometimes improperly exposed applications (e.g., unsecured file upload pages leading to web shell access)

**Internal pentests additionally show:**
- Printer login pages (potential cleartext LDAP credentials)
- ESXi, vCenter, iLO, iDRAC portals
- Network devices, IoT, IP phones
- Code repositories, SharePoint, intranet portals
- Security appliances

Leave no stone unturned—application discovery data can be a treasure trove.
