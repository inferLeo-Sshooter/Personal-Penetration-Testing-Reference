# Attacking Thick Client Applications

**Thick client applications** are locally installed software (unlike thin clients accessed via web browsers) that don't require internet to run and offer better performance through local processing power, memory, and storage. Common in enterprise environments for project management, CRM, inventory management, and productivity tools, they're typically built using Java, C++, .NET, or Microsoft Silverlight.

**Security Technologies:**

**Java:**
* **Sandbox**: Virtual environment isolating untrusted code from system resources
* **Java API restrictions**: Limit what code can access
* **Code Signing**: Verify code authenticity

**.NET:**
* Thick/rich/fat clients perform significant client-side processing
* Better performance, features, and UX than thin clients
* Less server dependency

**Examples:**
- Web browsers, media players, chat software, video games
- Custom enterprise applications delivered by IT departments
- Available via official websites, app stores, or direct deployment

**Characteristics:**
* ✓ Independent software
* ✓ Works offline
* ✓ Stores data locally
* ✗ Less secure than thin clients
* ✗ Consumes more resources
* ✗ More expensive
* ✗ Difficult deployment/maintenance (local patches required)

**Architecture Types:**
- **Two-Tier:**
  * Application → Database (direct communication)
  * Less secure (direct DB access)

- **Three-Tier:**
  * Application → Application Server (HTTP/HTTPS) → Database
  * More secure (no direct DB communication)
  * Server and DB may be on same network or internet

<img width="586" height="419" alt="image" src="https://github.com/user-attachments/assets/62aeb832-0780-4815-9cc8-e045eee3818f" />

**Security Concerns:**
- Downloaded from internet = authenticity risk
- Web vulnerabilities (XSS, CSRF, Clickjacking) don't apply
- Considered less secure than web apps

**Applicable Attacks:**
- Improper Error Handling
- Hardcoded sensitive data
- DLL Hijacking
- Buffer Overflow
- SQL Injection
- Insecure Storage
- Session Management

## Penetration Testing Steps

Thick client applications are considered more complex than others, and the attacking surface can be large. Thick client application penetration testing can be done both using automated tools and manually. The following steps are usually followed when testing thick client applications.

---

### 1. Information Gathering

- In this step, identify the `application architecture`, the `programming languages` and `frameworks` that have been used, and `understand how the application and the infrastructure work`.
- They should also need to identify `technologies` that are used on the client and server sides and find `entry points` and `user inputs`.
- Also look for identifying `common vulnerabilities` like the ones we mentioned earlier. The following tools will help us gather information:
  * [CFF Explorer](https://ntcore.com/explorer-suite/)
  * [Detect It Easy](https://github.com/horsicq/Detect-It-Easy)
  * [Process Monitor](https://learn.microsoft.com/en-us/sysinternals/downloads/procmon)
  * [Strings](https://learn.microsoft.com/en-us/sysinternals/downloads/strings)

### 2. Client Side attacks

Although thick clients perform significant processing and data storage on the client side, they still communicate with servers for various tasks, such as data synchronization or accessing shared resources. This interaction with servers and other external systems can expose thick clients to vulnerabilities similar to those found in web applications, including command injection, weak access control, and SQL injection.

- Sensitive information like **usernames and passwords, tokens, or strings** for communication with other services, might be stored in the `application's local files`.
- **Hardcoded credentials and other sensitive information** can also be found in the `application's source code`
- Static Analysis is a necessary step while testing the application. Using the proper tools, we can **reverse-engineer and examine .NET and Java applications** including **EXE, DLL, JAR, CLASS, WAR, and other file formats**. Dynamic analysis should also be performed in this step, as thick client applications store sensitive information in the memory as well.
  * [Ghidra](https://github.com/NationalSecurityAgency/ghidra)
  * [IDA](https://hex-rays.com/ida-pro)
  * [OllyDbg](https://www.ollydbg.de/)
  * [Radare2](https://www.radare.org/n/)
  * [dnSpy](https://github.com/dnSpy/dnSpy)
  * [x64dbg](https://x64dbg.com/)
  * [JADX](https://github.com/skylot/jadx)
  * [Frida](https://frida.re/)

### 3. Network Side Attacks

If the application is communicating with a local or remote server, network traffic analysis will help us capture sensitive information that might be transferred through HTTP/HTTPS or TCP/UDP connection, and give us a better understanding of how that application is working. Penetration testers that are performing traffic analysis on thick client applications should be familiar with tools like:
  * Wireshark
  * tcpdump
  * TCPView
  * Burp Suite

### 4. Server Side Attacks

Server-side attacks in thick client applications are similar to web application attacks, and penetration testers should pay attention to the most common ones including most of the OWASP Top Ten.

# Exploiting Web Vulnerabilities in Thick-Client Applications

Thick client applications with a three-tier architecture have a security advantage over those with a two-tier architecture since it prevents the end-user from communicating directly with the database server. However, three-tier applications can be susceptible to web-specific attacks like SQL Injection and Path Traversal.

- During penetration testing, it is common for someone to encounter a thick client application that connects to a server to communicate with the database. The following scenario demonstrates a case where the tester has found the following files while enumerating an FTP server that provides `anonymous` user access.
  * fatty-client.jar
  * note.txt
  * note2.txt
  * note3.txt

  * A server has been reconfigured to run on port `1337` instead of `8000`.

  * The client application relies on `Java 8`.
  * The login credentials for login in the client application are `qtc / clarabibi`.
