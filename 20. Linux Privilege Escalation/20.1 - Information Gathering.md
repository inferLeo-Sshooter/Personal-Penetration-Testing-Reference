# Stuff to look

Enumeration is critical for privilege escalation. While helper scripts like [LinEnum](https://github.com/rebootuser/LinEnum) and [LinPEAS](https://github.com/peass-ng/PEASS-ng/tree/master/linPEAS) exist, understanding what to look for and performing manual enumeration is essential. When gaining initial shell access, check these key details:

`OS Version`: Identifies the distribution (Ubuntu, Debian, FreeBSD, etc.) and available tools. May reveal versions with known public exploits.

`Kernel Version`: May have public exploits targeting specific vulnerabilities. Use cautiously—kernel exploits can cause instability or crashes, especially on production systems.

`Running Services`: Identify services running as root. Misconfigured or vulnerable root services (Nagios, Exim, Samba, ProFTPd, etc.) are common privilege escalation vectors. Many have public exploit PoCs, like CVE-2016-9566 affecting Nagios Core < 4.2.4.

`Installed Packages and Versions`: Check for outdated or vulnerable packages that enable easy privilege escalation. Example: Screen version 4.05.00 (a terminal multiplexer like tmux) has a known privilege escalation vulnerability.

`Logged in Users`: Knowing which other users are logged into the system and what they are doing can provide greater insight into possible local lateral movement and privilege escalation paths.

`User Home Directories`: Check if other users' home directories are accessible. They may contain SSH keys for accessing other systems, or scripts and configuration files with credentials. Files with credentials can provide access to other systems or Active Directory environments.

### Home Directory Contents

```

```

We can check individual user directories and check to see if files such as the `.bash_history` file are readable and contain any interesting commands, look for configuration files, and check to see if we can obtain copies of a user's SSH keys.

At the minimum, check the ARP cache to see what other hosts are being accessed and cross-reference these against any useable SSH private keys.

`Sudo Privileges`: Check if the user can run commands as another user or root. Without credentials, sudo permissions may be unusable. However, `NOPASSWD` entries allow commands to run without password prompts. Not all root-executable commands lead to privilege escalation. Users with full sudo privileges can run any command as root—a simple `sudo su` immediately grants a root session.

### Sudo - List User's Privileges

`Configuration Files`: Configuration files can hold a wealth of information. It is worth searching through all files that end in extensions such as `.conf` and `.config`, for usernames, passwords, and other secrets.

`Readable Shadow File`: If the shadow file is readable, you will be able to gather password hashes for all users who have a password set. While this does not guarantee further access, these hashes can be subjected to an offline brute-force attack to recover the cleartext password.

`Password Hashes in /etc/passwd`: Occasionally, you will see password hashes directly in the /etc/passwd file. This file is readable by all users, and as with hashes in the `shadow` file, these can be subjected to an offline password cracking attack. This configuration, while not common, can sometimes be seen on embedded devices and routers.

# Environment Enumeration

**Gaining Situational Awareness**

After gaining access to a Linux host (e.g., via file upload vulnerability during an External Penetration Test) and establishing a reverse shell with persistence, begin gathering system basics.

First, identify the operating system. Enumeration differs between CentOS/Red Hat Enterprise Linux versus Debian-based hosts like Ubuntu. Systems like FreeBSD, Solaris, HP-UX, or AIX use different commands. While commands vary, the principles remain the same.

**Basic orientation commands:**
* `whoami` - current user
* `id` - user's group memberships
* `hostname` - server name (naming convention insights?)
* `ifconfig` or `ip a` - subnet and additional NICs
* `sudo -l` - commands user can run with sudo without password? Sometimes `sudo su` provides immediate root shell.

Include screenshots of this information in client reports as evidence of successful Remote Code Execution (RCE) and clear system identification.

Now let's get into our more detailed, step-by-step, enumeration. We'll start out by checking out what operating system and version we are dealing with.
```
$ cat /etc/os-release

NAME="Ubuntu"
VERSION="20.04.4 LTS (Focal Fossa)"
ID=ubuntu
ID_LIKE=debian
PRETTY_NAME="Ubuntu 20.04.4 LTS"
VERSION_ID="20.04"
HOME_URL="https://www.ubuntu.com/"
SUPPORT_URL="https://help.ubuntu.com/"
BUG_REPORT_URL="https://bugs.launchpad.net/ubuntu/"
PRIVACY_POLICY_URL="https://www.ubuntu.com/legal/terms-and-policies/privacy-policy"
VERSION_CODENAME=focal
UBUNTU_CODENAME=focal
```

The target runs Ubuntu 20.04.4 LTS ("Focal Fossa"). Verify if the version is outdated or maintained. Ubuntu's release cycle shows "Focal Fossa" reaches end-of-life in April 2030. This suggests the internet-facing asset is likely patched, making well-known kernel vulnerabilities less probable—but still check regardless.

Next we'll want to check out our current user's PATH, which Linux uses to locate executables when commands are executed (e.g., `id` located at `/usr/bin/id`). A misconfigured PATH variable for a target user can enable privilege escalation. Document this information in your notetaking tool for later reference.

Environment enumeration includes identifying users on the target system. Users are often created during application and service installations to limit service privileges, maintaining system security. If a service runs with the highest privileges (`root`) and is compromised, the attacker gains complete system control.

All system users are stored in `/etc/passwd`. The file format provides:
1. Username
2. Password
3. User ID (UID)
4. Group ID (GID)
5. User ID info

Check which users have folders under `/home`. Enumerate each to find sensitive data or files containing passwords. Verify if `.bash_history` is readable for interesting commands and examine configuration files. Files with credentials are common and may provide access to other systems or Active Directory environments.

Check SSH keys for all users—these enable persistence, potential privilege escalation, or pivoting and port forwarding into the internal network. At minimum, check the ARP cache for accessed hosts and cross-reference against useable SSH private keys.

Search for "low hanging fruit" like config files and other files containing sensitive information. Configuration files often hold valuable data. Search files with extensions like `.conf` and `.config` for usernames, passwords, and secrets.

Test any gathered passwords against all system users—password reuse is common and may provide quick access.

**Mounted File Systems**

In Linux, sensitive files can be stored in various locations, including mounted file systems. A mounted file system attaches to a specific directory and is accessed through it. Common file systems include ext4, NTFS, and FAT32, each with unique benefits and drawbacks. Some are read-only by the OS; others are read/write for users. Mounting requires root privileges, as does unmounting. These file systems may contain sensitive information, documentation, or applications worth examining.

Three default folders store temporary files, visible and readable by all users, potentially containing temporary logs or script output:

**`/tmp`**: Data automatically deleted after 10 days and immediately upon system restart.

**`/var/tmp`**: Data retained for up to 30 days by default and persists across reboots. Used by programs to store data that must be temporarily kept between reboots.

**Key difference**: `/var/tmp` has longer data retention than `/tmp`.
