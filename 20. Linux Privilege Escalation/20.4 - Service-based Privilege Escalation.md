# Cron Job Abuse

Cron jobs schedule recurring (or one-time) tasks like backups and directory cleanup. Entries in `/var/spool/cron` follow the format: minutes, hours, days, months, weeks, command — e.g., `0 */12 * * * /home/admin/backup.sh` runs every 12 hours.

Although the root crontab requires root/sudo to edit, abuse is still possible. If a world-writable script runs as root, you can append a command (e.g., a reverse shell) to it — even without reading the crontab, execution frequency can often be inferred from the script's behavior. Additionally, some applications place misconfigured cron files in `/etc/cron.d` that non-root users can edit.

For exmaple, let's look around the system for any writeable files or directories. The file `backup.sh` in the `/dmz-backups` directory is interesting and seems like it could be running on a cron job.

# Containers (LXD)

Containers operate at the OS level, sharing the operating system while isolating application processes, whereas VMs operate at the hardware level, allowing multiple OSes to run simultaneously.

This isolation is valuable for resource management and security — it simplifies error monitoring and allows processes that typically require root privileges (like web apps or APIs) to be isolated from the host system, preventing escalation to sensitive resources such as databases.

## Linux Containers

LXC is an OS-level virtualization technique that runs multiple isolated Linux systems on a single host, sharing the host kernel while maintaining separate processes. Compared to VMs, LXC containers are lightweight, use fewer resources, and offer a standard interface for managing multiple containers simultaneously — even across multiple clouds, ensuring consistent application behavior across environments.

LXC's greatest advantage is ease of use, though its widespread adoption is largely credited to Docker, which popularized Linux containers and standardized the entire workflow — from creating and deploying templates to configuring networking and applications.

### Linux Daemon

Linux Daemon (LXD) is similar in some respects but is designed to contain a complete operating system. Thus it is not an application container but a system container. Before we can use this service to escalate our privileges, we must be in either the lxc or lxd group. We can find this out with the following command:
```

```

From here on, there are now several ways in which we can exploit `LXC/LXD`. We can either create our own container and transfer it to the target system or use an existing container. # Docker

Docker is a popular open-source tool that uses containers — lightweight, isolated environments sharing the OS file system and resources — to provide portable, consistent runtime environments. This makes containerization far less resource-intensive than traditional servers or VMs. Applications are packaged into Docker containers that are self-contained, executable, and run on any operating system.

## Docker Architecture

Docker uses a client-server model with two primary components: the Docker client and the Docker daemon.

**Docker Daemon**
The Docker daemon (server) is the powerhouse of Docker, responsible for creating, running, monitoring, and managing containers. It maintains container isolation — each container has its own file system, processes, and network interfaces. It also manages images, pulling them from registries like Docker Hub and storing them locally as the building blocks for containers. Additionally, it handles logging, resource monitoring (CPU, memory, network), container networking (virtual networks, IP addresses, DNS), and storage via Docker volumes, which persist data beyond a container's lifespan.

**Docker Client**
The Docker client is our interface for issuing commands, communicating with the daemon via a RESTful API or Unix socket. Through it, we can create, start, stop, and remove containers, as well as pull, build, and push images — enabling collaboration through remote repositories and Dockerfiles.

In short, the client issues commands while the daemon executes them.

## Docker Images and Containers

A Docker image is a read-only blueprint for creating containers, bundling everything an application needs — code, dependencies, libraries, and configurations — into a self-contained, reproducible package. Images are built using a Dockerfile, which defines the steps for constructing them.

A Docker container is a running instance of an image. It inherits all the image's properties but operates as its own lightweight, isolated environment with its own filesystem, processes, and network interfaces — kept separate from the host system and other containers.

The key distinction: images are immutable and read-only, while containers are mutable at runtime. You can interact with containers, run commands, monitor logs, and modify their filesystem — however, changes are not persisted unless saved as a new image or written to a persistent volume.

## Docker Privilege Escalation

When accessing an environment where users can manage Docker containers, it may be possible to leverage those containers to escalate privileges on the target system.

### Docker Shared Directories

Shared directories (volume mounts) link the host filesystem to the container, enabling data persistence, code sharing, and collaboration. A host path is mapped to a container path, creating a direct connection between the two. They can be mounted as read-only or read-write — read-only mounts prevent changes inside the container from affecting the host, while read-write mounts allow bidirectional changes.

From a security perspective, when we gain access to a container and enumerate its filesystem, we may discover non-standard directories. These shared directories can potentially be exploited to escape the container or escalate privileges, depending on the environment's configuration.

```
root@container:~$ cd /hostsystem/home/cry0l1t3

```
$ ssh cry0l1t3@<host IP> -i cry0l1t3.priv
```

### Docker Sockets

A Docker socket is a special file (Unix or network socket) that bridges the Docker client and daemon — when a CLI command is issued, it travels through the socket to the daemon for execution. Access is typically restricted to specific users or groups to prevent unauthorized interaction.

Docker sockets can also be exposed over a network, enabling remote management of Docker hosts and containers. However, this expanded access introduces security risks: misconfigurations or automated process files associated with the socket can expose useful information, potentially providing a path to escape the container or escalate privileges.

```

### Docker Group

To gain root privileges through Docker, the user we are logged in with must be in the `docker` group. This allows him to use and control the Docker daemon.
```
docker-user@nix02:~$ id

uid=1000(docker-user) gid=1000(docker-user) groups=1000(docker-user),116(docker)
```

Alternatively, Docker may have SUID set, or we are in the Sudoers file, which permits us to run `docker` as root. All three options allow us to work with Docker to escalate our privileges.

Most hosts have a direct internet connection because the base images and containers must be downloaded. However, many hosts may be disconnected from the internet at night and outside working hours for security reasons. However, if these hosts are located in a network where, for example, a web server has to pass through, it can still be reached.

# Kubernetes

Kubernetes (`K8s`) is an open-source container orchestration platform developed by Google that has transformed how applications are deployed and managed. Drawing on over a decade of experience with complex workloads, it has become the industry standard for microservices orchestration under the Cloud Native Computing Foundation.

K8s runs applications in containers isolated from the host system through multiple layers of protection, ensuring stability regardless of host-level changes. Its architecture consists of a `master node` and `worker nodes`, each with distinct roles. The platform's flexibility across environments makes it a go-to for automating, scaling, and managing deployments — and understanding its security aspects is essential, as container access is a common penetration testing objective.

## K8s Concept

At the core of Kubernetes is the **pod** — a group of one or more closely connected containers that functions as its own virtual machine on a node, complete with a unique IP and hostname. Kubernetes simplifies multi-container management through built-in load balancing, service discovery, storage orchestration, and self-healing. To address security and management challenges, K8s includes features like `Role-Based Access Control (RBAC)`, `Network Policies`, and `Security Contexts`.

Differences between K8 and Docker:

|Function|	Docker	|Kubernetes|
|-|-|-|
|Primary	|Platform for containerizing Apps|	An orchestration tool for managing containers|
|Scaling	|Manual scaling with Docker swarm|	Automatic scaling|
|Networking	|Single network	|Complex network with policies|
|Storage	|Volumes	|Wide range of storage options|

## Kubernetes Architecture

Kubernetes is divided into two core components: the **Control Plane** (master node), which manages and coordinates the cluster, and **Worker Nodes** (minions), where containerized applications actually run.

### Nodes

The master node hosts the Control Plane, which maintains the cluster's desired state and issues instructions to minions. Worker nodes execute applications and report back to the Control Plane. Kubernetes supports a wide range of workloads — databases, AI/ML, cloud-native microservices, and edge applications — and is compatible with public clouds (Google Cloud, Azure, AWS) as well as on-premises environments.

### Control Plane

The Control Plane serves as the management layer. It consists of several crucial components, including:

|Service	|TCP |Ports|
|-|-|-|
|`etcd`|	`2379`, `2380`|
|`API server`	|`6443`|
|`Scheduler`|	`10251`|
|`Controller Manager`|	`10252`|
|`Kubelet API`|	`10250`|
|`Read-Only Kubelet API`|	`10255`|

These elements enable the `Control Plane` to make decisions and provide a comprehensive view of the entire cluster.

### Minions

Minions run the containerized applications under Control Plane supervision. The `Scheduler` determines which node a pod should run on based on cluster state, then the `API server` updates `etcd` accordingly. The API server serves as the entry point for all administrative commands — whether from users via `kubectl` or internal controllers — and communicates with `etcd` to fetch and update cluster state.

### K8s Security Measures

Kubernetes security spans four key domains: **cluster infrastructure security**, **cluster configuration security**, **application security**, and **data security** — each requiring careful management across multiple layers by developers and administrators.

## Kubernetes API

The Kubernetes API is the central point of contact for all internal and external interactions within the cluster. Hosted by `kube-apiserver`, it supports declarative control — users define a desired state, and Kubernetes handles the rest. The API processes and verifies RESTful requests for creating, modifying, deleting, and retrieving cluster resources.

API resources are endpoints that house collections of objects (such as Pods, Services, and Deployments), each supporting the following operations:

| Request | Description |
|---------|-------------|
| `GET` | Retrieve a resource or list of resources |
| `POST` | Create a new resource |
| `PUT` | Update an existing resource |
| `PATCH` | Apply partial updates to a resource |
| `DELETE` | Remove a resource |

### Authentication

Kubernetes supports multiple authentication methods — client certificates, bearer tokens, authenticating proxies, and HTTP basic auth — to verify user identity. Once authenticated, access is controlled via `Role-Based Access Control (RBAC)`, which assigns roles with specific permissions to users or processes.

A notable security consideration is the `Kubelet`, which by default permits **anonymous access**. Since anonymous requests are treated as unauthenticated, any process or user that can reach the Kubelet API can make requests and receive responses — potentially exposing sensitive information or enabling unauthorized actions.

# Logrotate

Linux systems generate large volumes of log files, and without management, these can consume all available disk space. `Logrotate` addresses this by archiving or disposing of old logs, keeping the system clean and making log searches more efficient. Logs stored in `/var/log` are essential for administrators to diagnose malfunctions and monitor services — making proper log management a critical system administration task.

`Logrotate` has many features for managing these log files. These include the specification of:
* the `size` of the log file,
* its `age`,
* and the `action` to be taken when one of these factors is reached.

```
[!bash!]$ man logrotate
[!bash!]$ # or
[!bash!]$ logrotate --help

Usage: logrotate [OPTION...] <configfile>
  -d, --debug               Don't do anything, just test and print debug messages
  -f, --force               Force file rotation
  -m, --mail=command        Command to send mail (instead of '/usr/bin/mail')
  -s, --state=statefile     Path of state file
      --skip-state-lock     Do not lock the state file
  -v, --verbose             Display messages during rotation
  -l, --log=logfile         Log file or 'syslog' to log to syslog
      --version             Display version information

Help options:
  -?, --help                Show this help message
      --usage               Display brief usage message
```

The function of the rotation itself consists in renaming the log files. For example, new log files can be created for each new day, and the older ones will be renamed automatically. Another example of this would be to empty the oldest log file and thus reduce memory consumption.

This tool is usually started periodically via `cron` and controlled via the configuration file `/etc/logrotate.conf`. Within this file, it contains global settings that determine the function of `logrotate`.

```
[!bash!]$ cat /etc/logrotate.conf


# see "man logrotate" for details

# global options do not affect preceding include directives

# rotate log files weekly
weekly

# use the adm group by default, since this is the owning group
# of /var/log/syslog.
su root adm

# keep 4 weeks worth of backlogs
rotate 4

# create new (empty) log files after rotating old ones
create

# use date as a suffix of the rotated file
#dateext

# uncomment this if you want your log files compressed
#compress

# packages drop log rotation information into this directory
include /etc/logrotate.d

# system-specific logs may also be configured here.
```

To force a new rotation on the same day, we can set the date after the individual log files in the status file `/var/lib/logrotate.status` or use the `-f/--force` option:

```
[!bash!]$ sudo cat /var/lib/logrotate.status

/var/log/samba/log.smbd" 2022-8-3
/var/log/mysql/mysql.log" 2022-8-3
```

We can find the corresponding configuration files in `/etc/logrotate.d/` directory.

```
[!bash!]$ ls /etc/logrotate.d/

alternatives  apport  apt  bootlog  btmp  dpkg  mon  rsyslog  ubuntu-advantage-tools  ufw  unattended-upgrades  wtmp
```

```
[!bash!]$ cat /etc/logrotate.d/dpkg

/var/log/dpkg.log {
        monthly

## Passive Traffic Capture

If `tcpdump` is installed, unprivileged users may be able to capture network traffic, including, in some cases, credentials passed in cleartext. Several tools exist, such as `net-creds` and `PCredz` that can be used to examine data being passed on the wire. This may result in capturing sensitive information such as credit card numbers and SNMP community strings. It may also be possible to capture Net-NTLMv2, SMBv2, or Kerberos hashes, which could be subjected to an offline brute force attack to reveal the plaintext password. Cleartext protocols such as HTTP, FTP, POP, IMAP, telnet, or SMTP may contain credentials that could be reused to escalate privileges on the host.

## Weak NFS Privileges

Network File System (NFS) allows users to access shared files or directories over the network hosted on Unix/Linux systems. NFS uses TCP/UDP port 2049. Any accessible mounts can be listed remotely by issuing the command `showmount -e`, which lists the NFS server's export list (or the access control list for filesystems) that NFS clients.
```
$ showmount -e 10.129.2.12

Export list for 10.129.2.12:
/tmp             *
/var/nfs/general *
```

When an NFS volume is created, various options can be set:

|Option	|Description|
|-|-|
|`root_squash`|	If the root user is used to access NFS shares, it will be changed to the `nfsnobody` user, which is an unprivileged account. Any files created and uploaded by the root user will be owned by the `nfsnobody` user, which prevents an attacker from uploading binaries with the SUID bit set.|
|`no_root_squash`	|Remote users connecting to the share as the local root user will be able to create files on the NFS server as the root user. This would allow for the creation of malicious scripts/programs with the SUID bit set.|

```
NIX02:~$ cat /etc/exports

# Example for NFSv2 and NFSv3:

#
# Example for NFSv4:

## Hijacking Tmux Sessions

Terminal multiplexers such as `tmux` can be used to allow multiple terminal sessions to be accessed within a single console session. When not working in a `tmux` window, we can detach from the session, still leaving it active (i.e., running an `nmap` scan). For many reasons, a user may leave a `tmux` process running as a privileged user, such as root set up with weak permissions, and can be hijacked. This may be done with the following commands to create a new shared session and modify the ownership.
```
htb@NIX02:~$ tmux -S /shareds new -s debugsess
htb@NIX02:~$ chown root:devs /shareds
```

If we can compromise a user in the `devs` group, we can attach to this session and gain root access.

Check for any running `tmux` processes.

```
htb@NIX02:~$  ps aux | grep tmux

root      4806  0.0  0.1  29416  3204 ?        Ss   06:27   0:00 tmux -S /shareds new -s debugsess
```

Confirm permissions.

```
htb@NIX02:~$ ls -la /shareds 

srw-rw---- 1 root devs 0 Sep  1 06:27 /shareds
```

Review our group membership.

```
htb@NIX02:~$ id

uid=1000(htb) gid=1000(htb) groups=1000(htb),1011(devs)
```

Finally, attach to the `tmux` session and confirm root privileges.

```
htb@NIX02:~$ tmux -S /shareds

id

uid=0(root) gid=0(root) groups=0(root)
```
