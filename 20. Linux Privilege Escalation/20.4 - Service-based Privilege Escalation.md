# Cron Job Abuse

Cron jobs schedule recurring (or one-time) tasks like backups and directory cleanup. Entries in `/var/spool/cron` follow the format: minutes, hours, days, months, weeks, command ‚ÄĒ e.g., `0 */12 * * * /home/admin/backup.sh` runs every 12 hours.

Although the root crontab requires root/sudo to edit, abuse is still possible. If a world-writable script runs as root, you can append a command (e.g., a reverse shell) to it ‚ÄĒ even without reading the crontab, execution frequency can often be inferred from the script's behavior. Additionally, some applications place misconfigured cron files in `/etc/cron.d` that non-root users can edit.

# Containers (LXD)

Containers operate at the OS level, sharing the operating system while isolating application processes, whereas VMs operate at the hardware level, allowing multiple OSes to run simultaneously.

This isolation is valuable for resource management and security ‚ÄĒ it simplifies error monitoring and allows processes that typically require root privileges (like web apps or APIs) to be isolated from the host system, preventing escalation to sensitive resources such as databases.

## Linux Containers

LXC is an OS-level virtualization technique that runs multiple isolated Linux systems on a single host, sharing the host kernel while maintaining separate processes. Compared to VMs, LXC containers are lightweight, use fewer resources, and offer a standard interface for managing multiple containers simultaneously ‚ÄĒ even across multiple clouds, ensuring consistent application behavior across environments.

LXC's greatest advantage is ease of use, though its widespread adoption is largely credited to Docker, which popularized Linux containers and standardized the entire workflow ‚ÄĒ from creating and deploying templates to configuring networking and applications.

### Linux Daemon

Linux Daemon (LXD) is similar in some respects but is designed to contain a complete operating system. Thus it is not an application container but a system container. Before we can use this service to escalate our privileges, we must be in either the lxc or lxd group. We can find this out with the following command:
```
$ id

```

From here on, there are now several ways in which we can exploit `LXC/LXD`. We can either create our own container and transfer it to the target system or use an existing container. Unfortunately, administrators often use templates that have little to no security. This attitude has the consequence that we already have tools that we can use against the system ourselves.

Such templates often do not have passwords, especially if they are uncomplicated test environments. These should be quickly accessible and uncomplicated to use. # Docker

Docker is a popular open-source tool that uses containers ‚ÄĒ lightweight, isolated environments sharing the OS file system and resources ‚ÄĒ to provide portable, consistent runtime environments. This makes containerization far less resource-intensive than traditional servers or VMs. Applications are packaged into Docker containers that are self-contained, executable, and run on any operating system.

## Docker Architecture

Docker uses a client-server model with two primary components: the Docker client and the Docker daemon.

**Docker Daemon**
The Docker daemon (server) is the powerhouse of Docker, responsible for creating, running, monitoring, and managing containers. It maintains container isolation ‚ÄĒ each container has its own file system, processes, and network interfaces. It also manages images, pulling them from registries like Docker Hub and storing them locally as the building blocks for containers. Additionally, it handles logging, resource monitoring (CPU, memory, network), container networking (virtual networks, IP addresses, DNS), and storage via Docker volumes, which persist data beyond a container's lifespan.

**Docker Client**
The Docker client is our interface for issuing commands, communicating with the daemon via a RESTful API or Unix socket. Through it, we can create, start, stop, and remove containers, as well as pull, build, and push images ‚ÄĒ enabling collaboration through remote repositories and Dockerfiles.

In short, the client issues commands while the daemon executes them.

## Docker Images and Containers

A Docker image is a read-only blueprint for creating containers, bundling everything an application needs ‚ÄĒ code, dependencies, libraries, and configurations ‚ÄĒ into a self-contained, reproducible package. Images are built using a Dockerfile, which defines the steps for constructing them.

A Docker container is a running instance of an image. It inherits all the image's properties but operates as its own lightweight, isolated environment with its own filesystem, processes, and network interfaces ‚ÄĒ kept separate from the host system and other containers.

The key distinction: images are immutable and read-only, while containers are mutable at runtime. You can interact with containers, run commands, monitor logs, and modify their filesystem ‚ÄĒ however, changes are not persisted unless saved as a new image or written to a persistent volume.

## Docker Privilege Escalation

When accessing an environment where users can manage Docker containers, it may be possible to leverage those containers to escalate privileges on the target system.

### Docker Shared Directories

Shared directories (volume mounts) link the host filesystem to the container, enabling data persistence, code sharing, and collaboration. A host path is mapped to a container path, creating a direct connection between the two. They can be mounted as read-only or read-write ‚ÄĒ read-only mounts prevent changes inside the container from affecting the host, while read-write mounts allow bidirectional changes.

From a security perspective, when we gain access to a container and enumerate its filesystem, we may discover non-standard directories. These shared directories can potentially be exploited to escape the container or escalate privileges, depending on the environment's configuration.

```

### Docker Sockets

A Docker socket is a special file (Unix or network socket) that bridges the Docker client and daemon ‚ÄĒ when a CLI command is issued, it travels through the socket to the daemon for execution. Access is typically restricted to specific users or groups to prevent unauthorized interaction.

Docker sockets can also be exposed over a network, enabling remote management of Docker hosts and containers. However, this expanded access introduces security risks: misconfigurations or automated process files associated with the socket can expose useful information, potentially providing a path to escape the container or escalate privileges.

```

### Docker Group

To gain root privileges through Docker, the user we are logged in with must be in the `docker` group. This allows him to use and control the Docker daemon.
```
docker-user@nix02:~$ id

uid=1000(docker-user) gid=1000(docker-user) groups=1000(docker-user),116(docker)
```

Alternatively, Docker may have SUID set, or we are in the Sudoers file, which permits us to run `docker` as root. All three options allow us to work with Docker to escalate our privileges.

Most hosts have a direct internet connection because the base images and containers must be downloaded. ### Docker Socket

A case that can also occur is when the Docker socket is writable. Usually, this socket is located in `/var/run/docker.sock`. However, the location can understandably be different. Because basically, this can only be written by the root or docker group. If we act as a user, not in one of these two groups, and the Docker socket still has the privileges to be writable, then we can still use this case to escalate our privileges.

docker-user@nix02:~$ docker -H unix:///var/run/docker.sock run -v /:/mnt --rm -it ubuntu chroot /mnt bash
