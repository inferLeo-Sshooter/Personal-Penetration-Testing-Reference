# Situational Awareness

## Network Enumeration
Key things to check:
- **Dual-homed hosts** — multiple NICs may open paths to previously unreachable network segments
- **Routing tables** — reveals local & adjacent networks
- **Domain info** — DC IP addresses if host is AD-joined
- **ARP cache** (`arp -a`) — shows recent host communication, hints at RDP/WinRM targets admins connect to

> Network info may not directly cause privesc but can guide lateral movement or reveal new targets.

### Interface(s), IP Address(es), DNS Information
```
C:\htb> ipconfig /all
```

### ARP Table
```
C:\htb> arp -a
```

### Routing Table
```
C:\htb> route print
```

## Enumerating Protections

### AV / EDR
- Can **interfere with enumeration** and **block/detect** privesc attempts
- Public PoCs are commonly signatured — may need custom/modified payloads
- Identify what's running **before** deploying tools

### AppLocker
- Whitelisting solution — can block `cmd.exe`, `powershell.exe`, specific file types
- Enumerate policies with:
```powershell
Get-AppLockerPolicy -Effective -Xml
```
- Helps determine if an **AppLocker bypass** is needed before running tools

### Key Takeaway
- Some EDRs even block common binaries like `net.exe`, `tasklist`
- Suspicious activity (e.g., `cmd.exe` on an accountant's machine) may trigger alerts
- Early enumeration of defenses = saves time on non-evasive, **critical** on evasive engagements

### Check Windows Defender Status
```
PS C:\htb> Get-MpComputerStatus
```

### List AppLocker Rules
```
PS C:\htb> Get-AppLockerPolicy -Effective | select -ExpandProperty RuleCollections
```

### Test AppLocker Policy
```
PS C:\htb> Get-AppLockerPolicy -Local | Test-AppLockerPolicy -path C:\Windows\System32\cmd.exe -User Everyone

FilePath                    PolicyDecision MatchingRule
--------                    -------------- ------------
C:\Windows\System32\cmd.exe         Denied c:\windows\system32\cmd.exe
```

# Initial Enumeration

## Escalation Targets
Depending on system config, we can escalate to:
- **NT AUTHORITY\SYSTEM** — highest, more privileges than local admin, runs most Windows services
- **Built-in local Administrator** — often not disabled, frequently reused across systems
- **Local account in Administrators group** — same privileges as built-in admin
- **Standard domain user** in local Administrators group
- **Domain Admin** in local Administrators group

## Why Enumerate Manually?
Tools may be unavailable due to:
- Network restrictions
- No internet access
- AV/EDR blocking

→ Must know manual enumeration via **PowerShell & CMD**

## Key Data Points

### OS Name
- Identifies workstation vs. server and Windows version (7, 10, Server 2008/2012/2016/2019)
- Determines available tools (e.g., PowerShell version — limited on legacy systems)
- Reveals potential **public exploits** for that specific OS

### Version
- Specific build versions may have **known CVEs** with public exploits
- Windows kernel exploits can cause **instability or crashes** — never run blindly on production systems

### Running Services
- Focus on services running as **NT AUTHORITY\SYSTEM** or admin-level accounts
- Misconfigured/vulnerable privileged services = easy privesc win

### Key Data Points to Collect

| Category | What to Look For |
|----------|-----------------|
| OS & Patch Level | Version, build, missing KBs |
| Installed Software | Vulnerable or exploitable apps |
| Current Privileges | What the current user can do |
| Group Memberships | Admin groups, special groups |
| Network Info | Interfaces, routes, ARP cache |
| Running Services | Misconfigured or vulnerable services |
| Scheduled Tasks | Tasks running as SYSTEM |
| Credentials | Stored, cached, or hardcoded |

> Enumeration is the foundation — no privesc without proper recon.

### `set` — Display All Environment Variables

The `set` command prints Windows environment variables, revealing useful host configuration details.

**PATH** is especially important — admins or apps sometimes add folders (e.g., Python, Java) to it. If any such folder is writable, DLL injection may be possible. Also note that Windows searches the CWD first, then PATH left-to-right, so a custom path placed *left* of `C:\Windows\System32` is significantly more dangerous.

**HOME DRIVE** is another useful variable — in enterprises it's typically a file share that may expose accessible directories, sometimes including IT folders with password inventories.

Shares also support **Roaming Profiles**, syncing user files across machines. Placing a file in `USERPROFILE\AppData\Microsoft\Windows\Start Menu\Programs\Startup` means it will execute whenever that user logs into any machine.

## User & Group Information

When systems are well-patched, users are often the weakest link. Key things to enumerate include: users and groups on the system, members of groups with admin-level access, current user privileges, password policy, and any active logged-on users.

### Logged-In Users

Always check who is logged in, whether they're active or idle, and what they may be working on. Active users can sometimes be targeted directly for privilege escalation or further access. On evasive engagements, tread carefully on machines with active users to avoid detection.

```
C:\htb> query user

>administrator         rdp-tcp#2           1  Active          .  3/25/2021 9:27 AM
```

# Communication with Processes

One of the best places to look for privilege escalation is the processes that are running on the system. Even if a process is not running as an administrator, it may lead to additional privileges. The most common example is discovering a web server like IIS or XAMPP running on the box, placing an `aspx/php` shell on the box, and gaining a shell as the user running the web server. Generally, this is not an administrator but will often have the `SeImpersonate` token, allowing for `Rogue/Juicy/Lonely Potato` to provide SYSTEM permissions.

## Access Tokens

Windows access tokens describe the security context of a process or thread, containing a user's identity and privilege information. When a user authenticates, their password is verified and they're assigned a token. A copy of this token is presented every time the user interacts with a process to determine their privilege level.

## Named Pipes

Named pipes are another inter-process communication method — memory-stored files cleared after being read. Cobalt Strike uses them for every non-BOF command: a beacon creates a named pipe, spawns a new process that directs output to it, and the server reads the result. This isolates the beacon from crashes or AV detections of the spawned process.

Cobalt Strike users often rename pipes to impersonate legitimate software (e.g., `mojo` to mimic Chrome). A telling red flag is a `mojo` pipe on a machine without Chrome installed — which in one case turned out to be an internal red team that the blue team had missed, but an external consultant caught.

Named pipes enable inter-process communication via shared memory, with two types: named and anonymous. Windows uses a client-server model where the creating process is the server and the connecting process is the client. Communication can be **half-duplex** (client writes only) or **duplex** (bidirectional). Each new client connection creates a new named pipe instance sharing the same name but using a separate data buffer.
