# I. Why

**Real-World Scenarios**

- **VM Failure**: Losing a VM mid-engagement was no crisis because daily backups and detailed notes meant testing resumed without interruption.
- **Accidental Outage**: When a client blamed the tester for knocking out critical servers, timestamped scan logs and a signed scope document proved no wrongdoing — and prompted a process improvement: always confirm explicit exclusions.
- **Network Slowdown**: Blamed immediately for slowing a network, thorough scan output proved best practices were followed. The real cause (debug mode enabled on all network devices) was uncovered only because documentation forced further investigation.

These scenarios share a common lesson: documentation protects you, supports your client, and prevents costly scrambles to recover lost work.

# II. Preparation

## Notetaking Sample Structure

The structure below is a helpful starting point, but should be adapted to your workflow, project type, and specific circumstances — some categories may not apply to every assessment.

**Attack Path** — Outline the full compromise path using screenshots and command output, making it easier to drop directly into the report later.

**Credentials** — A centralized log of all compromised credentials and secrets gathered during testing.

**Findings** — Create a subfolder per finding containing your narrative and supporting evidence. A summary section helps organize findings for the report.

**Vulnerability Scan Research** — Notes on what you've researched and attempted, so you don't duplicate work.

**Service Enumeration Research** — Track investigated services, failed exploits, and promising vulnerabilities or misconfigurations.

**Web Application Research** — Log interesting applications found through subdomain brute-forcing or port scanning, along with screenshot tool output (e.g., Aquatone/EyeWitness), apps of interest, and credential attempts.

**AD Enumeration Research** — A step-by-step record of Active Directory enumeration performed and areas needing follow-up.

**OSINT** — Track interesting information collected, where applicable.

**Administrative Information** — Store stakeholder contacts, Rules of Engagement objectives, and a running to-do list for test ideas you want to revisit.

**Scoping Information** — Keep in-scope IPs, URLs, and provided credentials here to avoid repeatedly referencing scope documents and prevent scope creep.

**Activity Log** — High-level record of all actions taken during the assessment for event correlation purposes.

**Payload Log** — Track all payloads used, including file hashes and upload locations within the client environment.

## 1. Logging

- Logging all scanning and attack attempts and retaining raw tool output is essential — both for reporting and for responding to client questions when notes fall short.
- Use: `tmux` for `terminal logging` and exploitation attempts.
- Adjust the `.tmux.conf` if need to increase the history limit.
- Do research for the ability to take a screen capture of the current Tmux window or an individual pane. Especially when working with a **split window (2 panes)**.
- Check out the complete `tmux plugins` list to see if others would fit nicely into your workflow.
- Other can `Screenshot` just fine.

### Starting Out - tmux

```bash
tmux new -s htb          # start a named session
tmux attach -t htb       # reattach after disconnect
tmux ls                  # list sessions
```

---

### The Prefix Key
Everything in tmux starts with **`Ctrl+b`** (the prefix), then a command key.

---

### Windows (like browser tabs)

| Action | Keys |
|---|---|
| New window | `Ctrl+b c` |
| Next window | `Ctrl+b n` |
| Previous window | `Ctrl+b p` |
| Switch by number | `Ctrl+b 0-9` |
| Rename window | `Ctrl+b ,` |
| Kill window | `Ctrl+b &` |

---

### Panes (split your terminal)

| Action | Keys |
|---|---|
| Split horizontal | `Ctrl+b "` |
| Split vertical | `Ctrl+b %` |
| Navigate panes | `Ctrl+b ←↑→↓` |
| Resize pane | `Ctrl+b` then hold `Alt+arrow` |
| Zoom pane (fullscreen toggle) | `Ctrl+b z` |
| Kill pane | `Ctrl+b x` |

---

### Logging (critical for pentest documentation)

The guide recommends tmux logging for all terminal activity. Use the **tmux-logging plugin**, or manually:

```bash
# Log entire pane output to file
Ctrl+b, then type:  :pipe-pane -o 'cat >> ~/htb-session.log'

# Stop logging
Ctrl+b :pipe-pane
```

Or with the logging plugin:
```
Ctrl+b Shift+p    # toggle logging on/off
Ctrl+b Alt+p      # save complete pane history
```

---

### Scrollback / Copy Mode

```
Ctrl+b [          # enter scroll/copy mode
q                 # exit copy mode
Space             # start selection (in copy mode)
Enter             # copy selection
Ctrl+b ]          # paste
```

### Recommended Pentest Window Layout

```
Session: htb
├── Window 0: recon      → nmap, enum
├── Window 1: exploit    → active exploitation
├── Window 2: shell      → reverse shells / listeners
└── Window 3: notes      → cat/vim your findings
```

---

### `.tmux.conf` tweaks worth having

```bash
# Increase scrollback history (default is only 2000 lines!)
set -g history-limit 50000

# Mouse support
set -g mouse on

# Better pane splits
bind | split-window -h
bind - split-window -v
```

---

### Quick Tip for HTB

Start your session like this so nothing is ever lost:

```bash
tmux new -s htb
# immediately enable logging
Ctrl+b :pipe-pane -o 'cat >> ~/htb-$(date +%F).log'
```

Every command and output from that point is timestamped in your log file. When you're done, that log serves as raw evidence for your findings notes.

## 2. Artifacts Left Behind

- At minimum, track the following for every `payload used`:
  * when it was deployed
  * which host it was placed on
  * the file path
  * And whether it was cleaned up or requires client cleanup.

## 3. Account Creation & System Modifications

- Any accounts created or system settings changed must be tracked in case they can't be reverted post-assessment. Record the following for each change:
  * Host IP/hostname and timestamp
  * Description and location of the change
  * Application or service affected
  * Account name (and password if applicable)

## 4. Evidence and storage

- **What to Capture:**
  * Every finding needs supporting evidence.
  * It's also worth capturing evidence of unsuccessful tests in case the client questions your thoroughness.
  * For **command-line work**, `Tmux logs` can serve as evidence but are often poorly formatted — so capturing terminal output for significant steps separately alongside your findings is advisable. For everything else, take `screenshots`.

- **Storage:**
  * `Admin` — Scope of Work, kickoff meeting notes, status reports, and vulnerability notifications.
  * `Deliverables` — The report and any supplemental materials (spreadsheets, slide decks) as you build them out.
  * `Findings` — A dedicated folder per finding to keep evidence organized and easy to reference when writing the report.
  * `Scans` — Subdirectories for vulnerability scans, service enumeration (Nmap, Masscan, etc.), web tooling (Burp, ZAP, EyeWitness), and AD enumeration (BloodHound, PowerView, CrackMapExec, Snaffler, etc.).
  * `Notes` — Your working notes files.
  * `OSINT` — Tool output from sources like Intelx or Maltego that doesn't fit neatly into notes.
  * `Wireless` — Output from wireless testing tools, if in scope.
  * `Logging Output` — Tmux, Metasploit, and other logs not covered by the Scans subdirectories.
  * `Misc Files` — Web shells, payloads, custom scripts, and other assessment-relevant files.
  * `Retest` (optional) — If findings are retested post-remediation, mirror the original folder structure here to keep retest evidence cleanly separated.

## 5. Formatting and Redaction

- Credentials and PII must always be redacted in screenshots, along with anything morally objectionable. When preparing screenshots, also consider:
  * Adding arrows or boxes to highlight key items (avoid doing this in MS Word)
  * Adding a minimal border so images stand out against the white page
  * Cropping to show only relevant content
  * Including the address bar to indicate the URL or host in context

**Screenshots**: Prefer terminal text output over screenshots wherever possible — it's easier to redact, looks cleaner, keeps file sizes manageable, and lets clients copy/paste commands to reproduce results.

**Redaction**: Avoid pixelation or blurring — these methods can be reversed. Instead, use solid black bars applied directly to the image, not as a shape overlay in Word (which can simply be deleted)

**Terminal output**: redact credentials using a `<REDACTED>` or `<PASSWORD REDACTED>` placeholder. For `password hashes`, retain the first and last 3–4 characters and strip the middle. Consider `color-coded highlighting` to distinguish the command run from the notable output.

## 6. What Not to Archive

- Avoid viewing, capturing, or extracting sensitive data such as unredacted PII, potentially criminal information, or anything legally discoverable.
- If you encounter a network share with sensitive files, screenshot the directory listing rather than opening individual files — the file names alone are usually sufficient to convey the point.

# III. Writing a Strong Executive Summary

The Executive Summary is critical — it's often read by non-technical stakeholders like C-suite leaders and board members who control budgets. Write it as if your audience has never seen a penetration test before, doesn't know technical jargon, and has a short attention span.

**Do:**
- Use specific numbers, not vague words like "several" or "multiple"
- Keep it to 1.5–2 pages maximum
- Describe what you actually accessed in plain terms (e.g., HR records, banking systems — not "Domain Admin")
- Focus on the process failures behind vulnerabilities, not just the fixes
- Optionally, give a rough sense of remediation effort so leadership sets realistic expectations

**Don't:**
- Recommend specific vendors — suggest technology categories instead
- Use acronyms or technical jargon (even common ones like SNMP or MitM)
- Dwell on low-impact findings at the expense of critical ones
- Reference other sections of the report — the executive summary should stand alone

**Plain-language substitutions:**
Use "secure remote access protocol" instead of VPN/SSH; "offline password decryption" instead of hash cracking; "public data gathering" instead of OSINT; "input manipulation vulnerability" instead of SQL injection/XSS — and so on.

The goal: if a non-technical reader can understand the risk and why it matters, you've done it right.
