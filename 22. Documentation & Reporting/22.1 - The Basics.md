


# 1. Why

**Real-World Scenarios**

- **VM Failure**: Losing a VM mid-engagement was no crisis because daily backups and detailed notes meant testing resumed without interruption.
- **Accidental Outage**: When a client blamed the tester for knocking out critical servers, timestamped scan logs and a signed scope document proved no wrongdoing — and prompted a process improvement: always confirm explicit exclusions.
- **Network Slowdown**: Blamed immediately for slowing a network, thorough scan output proved best practices were followed. The real cause (debug mode enabled on all network devices) was uncovered only because documentation forced further investigation.

These scenarios share a common lesson: documentation protects you, supports your client, and prevents costly scrambles to recover lost work.

# 2. Preparation

## Notetaking Sample Structure

The structure below is a helpful starting point, but should be adapted to your workflow, project type, and specific circumstances — some categories may not apply to every assessment.

**Attack Path** — Outline the full compromise path using screenshots and command output, making it easier to drop directly into the report later.

**Credentials** — A centralized log of all compromised credentials and secrets gathered during testing.

**Findings** — Create a subfolder per finding containing your narrative and supporting evidence. A summary section helps organize findings for the report.

**Vulnerability Scan Research** — Notes on what you've researched and attempted, so you don't duplicate work.

**Service Enumeration Research** — Track investigated services, failed exploits, and promising vulnerabilities or misconfigurations.

**Web Application Research** — Log interesting applications found through subdomain brute-forcing or port scanning, along with screenshot tool output (e.g., Aquatone/EyeWitness), apps of interest, and credential attempts.

**AD Enumeration Research** — A step-by-step record of Active Directory enumeration performed and areas needing follow-up.

**OSINT** — Track interesting information collected, where applicable.

**Administrative Information** — Store stakeholder contacts, Rules of Engagement objectives, and a running to-do list for test ideas you want to revisit.

**Scoping Information** — Keep in-scope IPs, URLs, and provided credentials here to avoid repeatedly referencing scope documents and prevent scope creep.

**Activity Log** — High-level record of all actions taken during the assessment for event correlation purposes.

**Payload Log** — Track all payloads used, including file hashes and upload locations within the client environment.
