## 1. Obfuscation

**Obfuscation:** Making Code Hard to Read  

Obfuscation is a technique used to make code harder for humans to read while keeping its functionality the same. However, it may slow down performance.  

### ⚙️ How Does It Work?  
Obfuscation tools automatically rewrite code to make it difficult to understand.  

### Example:  
- The tool replaces words and symbols with random references.  
- During execution, the code is rebuilt using a hidden dictionary.  

### 🛠 Why Use Obfuscation?  
- **Hide sensitive logic** from attackers.  
- **Prevent reverse engineering** of software.  
- **Make analysis more difficult** for security researchers.  

💡 *Obfuscation doesn’t improve security on its own but adds an extra layer of protection.*  


### 🔍 What Are Interpreted Languages?  
Interpreted languages, like **Python, PHP, and JavaScript**, execute code **without compilation**.  

### 🔒 Why is JavaScript Commonly Obfuscated?  
- **Python & PHP**: Run on the **server-side**, hidden from users.  
- **JavaScript**: Runs in the **browser (client-side)**, meaning the code is **visible** to users.  

Since JavaScript is sent to users in **cleartext**, obfuscation is often used to make it harder to read and understand. 🚀  

### ✅ Why Obfuscate Code?  
1. **Prevent Code Theft** – Hides original code to stop unauthorized reuse or copying.  
2. **Make Reverse Engineering Harder** – Makes it difficult to understand the code’s functionality.  
3. **Add a Security Layer** – Helps obscure authentication or encryption logic (but should **not** replace proper security measures).  

⚠ **Warning:** Authentication or encryption should **never** be handled on the client-side, as it makes the code more vulnerable to attacks.  

### 🚨 Obfuscation in Malicious Activities  
- Attackers use obfuscation to **bypass Intrusion Detection and Prevention Systems (IDS/IPS)**.  
- Malicious scripts are disguised to avoid detection.  

### Basic Obfuscation

**Example:**

`The code` --> `Code minification` - having the entire code in a single (often very long) line (tools: "javascript-minifier", ...) --> `obfuscate` (using tools like "BeautifyTools").

**Note:**
> - The above type of obfuscation is known as "packing", which is usually recognizable from the six function arguments used in the initial function "function(p,a,c,k,e,d)".
> - A packer obfuscation tool usually attempts to convert all words and symbols of the code into a list or a dictionary and then refer to them using the (p,a,c,k,e,d) function to re-build the original code during execution.
> - The (p,a,c,k,e,d) can be different from one packer to another. However, it usually contains a certain order in which the words and symbols of the original code were packed to know how to order them during execution.
> - While a packer does a great job reducing the code's readability, we can still see its main strings written in cleartext, which may reveal some of its functionality. This is why we may want to look for better ways to obfuscate our code.

## 2. Deobfuscation

`Obfuscated code` --> `properly format` (use tools like ` Prettier` or `Beautifier` or use code debugger to align) --> Use deobfuscate tools (like: `UnPacker`, ...)

**Note:**

### 🔍 Guide to Analyzing JavaScript Code  

🛠 **Step 1: Identify the Structure**
- Start by **skimming through the code** to understand its general structure.  
- Look for **functions, variables, loops, and conditions** that control execution.  
- Identify **external references**, such as APIs, file imports, or network requests.  

🔎 **Step 2: Break Down the Code**  
- **Look at variable declarations**: What data types are being used?  
- **Check function definitions**: What parameters do they take? What values do they return?  
- **Identify loops and conditions**: How does the code flow based on inputs?  

🌐 **Step 3: Detect Web Requests (If Any)**  
- Search for **network-related functions** like:  
  - `fetch()`, `XMLHttpRequest()`, `$.ajax()`, or WebSockets.  
- Identify the **target URL**: Is it calling an API? A local file?  
- Check the **HTTP method**: Is it **GET, POST, PUT, or DELETE**?  

🔄 **Step 4: Analyze User Inputs**  
- Does the script process **form inputs, URL parameters, or cookies**?  
- Are there **validations or sanitizations** to prevent security risks?  
- Look for **hardcoded credentials or sensitive information**.  

🔧 **Step 5: Identify Hidden or Unused Code**  
- Check if some functions are **defined but never used**.  
- Look for **commented-out or leftover debugging code** (e.g., `console.log`).  
- Consider whether the code is **meant for future use** or is simply redundant.  

🕵️‍♂️ **Step 6: Test Functionality & Security**  
- Try **manually replicating** network requests to see how the server responds.  
- Test with **different inputs** (unexpected values, long strings, special characters).  
- Look for **possible security vulnerabilities**, like **XSS, SQLi, or CSRF risks**.  


## 🚀 Conclusion  
By following these steps, you can **systematically analyze any JavaScript snippet**, understand its purpose, and uncover potential security flaws or hidden functionalities.
