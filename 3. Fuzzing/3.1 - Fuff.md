## 1. Fuzzing

Fuzzing is a testing method where different types of input are sent to a system to observe its response. 

- **SQL Injection Fuzzing**: Sends special characters to check how a server reacts.
- **Buffer Overflow Fuzzing**: Sends increasingly long strings to see when a program crashes.
- **Web Fuzzing**: Uses wordlists to find hidden links and pages on a website.

Since web servers don’t list all available pages, fuzzing helps discover accessible URLs by testing different paths. If a page doesn’t exist, it returns a **404 Not Found** error.

### Wordlist
**Wordlists** help find existing web pages by testing commonly used directory and page names, similar to a **Password Dictionary Attack**. While not all pages can be discovered, success rates can reach **up to 90%** on some websites.

**Note**: When using ffuf with Pre-built wordlist, use the `-ic` flag in `ffuf` to ignore copyright comments in wordlists and clean results.

## 2. Basic Fuzzing

### 2.1 - Directory fuzzing

The main two options are `-w` for wordlists and `-u` for the URL. We can assign a wordlist to a keyword to refer to it where we want to fuzz. For example, we can pick our wordlist and assign the keyword `FUZZ` to it by adding `:FUZZ` after it:

` $ ffuf -w /opt/useful/seclists/Discovery/Web-Content/directory-list-2.3-small.txt:FUZZ -u http://SERVER_IP:PORT/FUZZ`

ffuf can test for almost 90k URLs in less than 10 seconds. This speed may vary depending on your internet speed and ping if you used ffuf on your machine, but it should still be extremely fast.

We can even make it go faster if we are in a hurry by increasing the number of threads to 200, for example, with `-t 200`, but this is not recommended, especially when used on a remote site, as it may disrupt it, and cause a **Denial of Service**, or bring down your internet connection in severe cases.

### 2.2 - Page fuzzing

#### Extension Fuzzing  

- Sometimes, directories (e.g., `/blog`) exist but show empty pages.  
- To find hidden pages, we need to **fuzz for file extensions** like `.php`, `.html`, `.aspx`, etc.  
- A common way to guess extensions is by checking the **server type** in HTTP headers (e.g., Apache → `.php`, IIS → `.asp`).  
- A more effective method is **fuzzing extensions** using `ffuf`:  

```txt
  ffuf -w /opt/useful/seclists/Discovery/Web-Content/web-extensions.txt:FUZZ -u https://target.com/index.FUZZ
```

**Note**: 
> - Why index.*: Most websites have an index file (e.g., index.php, index.html), making it a good target for extension fuzzing.
> - The pre-built wordlist already includes a dot (.), so we don’t need to add one manually.


#### Page Fuzzing

In **Extension Fuzzing** part, if we do get a couple of hits, but only `.php` gives us a response with code `200`. Then, we now know that this website runs on PHP to start fuzzing for PHP files. We will now use the same concept of keywords we've been using with ffuf, use `.php` as the extension, place our `FUZZ` keyword where the `filename` should be, and use the same wordlist we used for fuzzing directories:
```txt
$ ffuf -w /opt/useful/seclists/Discovery/Web-Content/directory-list-2.3-small.txt:FUZZ -u http://SERVER_IP:PORT/blog/FUZZ.php
```

### 2.3 - Recursive Fuzzing

- Recursive fuzzing **automates** directory and file discovery by scanning **subdirectories** dynamically.
- Helps avoid manually checking each directory, saving **time and effort**.

> Recursive fuzzing is used when encountering a directory with many subdirectories and files. When scanning, it will automatically start another scan for each new directory found and the depth of recursive fuzzing can be customized with the `-recursion flag` and `-recursion-depth` **+ number**.

```txt
$ ffuf -w /opt/useful/seclists/Discovery/Web-Content/directory-list-2.3-small.txt:FUZZ -u http://SERVER_IP:PORT/FUZZ -recursion -recursion-depth 1 -e .php -v
```

| Flag | Description |
|------|------------|
| `-recursion` | Enables recursive scanning. |
| `-recursion-depth <N>` | Limits how deep the scan goes (e.g., `1` scans only direct subdirectories). |
| `-e .php` | Specifies file extensions to search for. |
| `-v` | Displays full URLs for better visibility. |

```txt
ffuf -u https://target.com/FUZZ -w wordlist.txt -recursion -recursion-depth 2
```

- How it works:
1. Scans `https://target.com/` for directories in wordlist.txt.
2. If a directory is found, it automatically fuzzes inside it.
3. It continues up to depth 2 (main directories and subdirectories).

## 3. Domain Fuzzing

### 3.1 - DNS Records

**DNS Records and Resolving Local Domains**
When accessing `/blog`, we received a message stating:  
**"Admin panel moved to academy.htb"**, but visiting `www.academy.htb` in a browser resulted in a **"Can’t connect"** error.

**Why Can't We Connect?**
- Browsers need an **IP address** to connect to a website.
- If a URL is not in the **local `/etc/hosts` file** or public **DNS**, it won't resolve.
- Since **academy.htb** is a local website (inside HTB labs), it isn't a public domain.
- The browser queries a **public DNS (e.g., 8.8.8.8)** but finds no records.

**Solution: Manually Add It to `/etc/hosts`**
To resolve `academy.htb`, we need to **map** it to its corresponding **IP address** in `/etc/hosts`:

**Command to Add Entry:**
```txt
sudo sh -c 'echo "SERVER_IP academy.htb" >> /etc/hosts'
```

Example: If the server’s IP is 10.10.10.10, the entry would be: `10.10.10.10 academy.htb`

**Notes**: 
> - Now We Can Access the Website
> - Don't forget to specify the port in the URL if required.
> - Visiting `http://academy.htb:PORT` should now work.
> - The `/etc/hosts` file is used to manually map hostnames to IPs.
> - This method is useful when testing local environments or CTFs like HTB.
> - Public DNS (e.g., 8.8.8.8, 1.1.1.1) won't resolve private domains.

### 3.2 - Sub-domain Fuzzing

A **sub-domain** is a website that is part of a larger domain.  
For example:  
- `https://photos.google.com` is a sub-domain of `google.com`.

**Why Enumerate Sub-domains?**
- Sub-domains often host additional services or applications.
- They might expose **hidden** or **less secure** parts of a website.
- They have their own **DNS records**, which can be queried.

**Requirements for Scanning**
Before we start enumerating sub-domains, we need:
✅ A **wordlist** (common sub-domain names)  
✅ A **target** domain

```txt
$ ffuf -w /opt/useful/seclists/Discovery/DNS/subdomains-top1million-5000.txt:FUZZ -u http://FUZZ.academy.htb
```

If We see that we **do not get any hits back**. Does this mean that there are no subdomains under `academy.htb`? **No.**

This means that there are no **publicly resolvable** subdomains under `academy.htb`, as it does not have a public DNS record, as previously mentioned. Even though we added `academy.htb` to our `/etc/hosts` file, we only added the **main domain**. 

When `ffuf` searches for subdomains:
- It does **not** find them in `/etc/hosts`.
- It queries the **public DNS**, which obviously does not have them.

**Key Takeaways**
1. **No hits ≠ No subdomains**
   - The target **may** have subdomains, but they are **not publicly discoverable** via DNS.

2. **Why does this happen?**
   - Some organizations do **not** register subdomains in **public DNS**.
   - Internal subdomains may exist but are only accessible **inside a private network**.
   - **/etc/hosts** entries do not affect how `ffuf` or DNS-based scanners work.

3. **How to Proceed if No Public Subdomains Found?**
   - **Check for wildcard DNS records**:
     ```bash
     nslookup randomnotexist.academy.htb
     ```
   - If it still resolves, the domain may have a **catch-all wildcard** (`*.academy.htb`).
   - **Perform virtual host fuzzing with `ffuf`**

### 3.3 - Vhost Fuzzing

In the previous section, we fuzzed **public subdomains** using **public DNS records**. However, when trying to fuzz subdomains that **do not** have a public DNS record or belong to **private websites**, the same method does not work.  

In this section, we will learn how to achieve this using **Vhost Fuzzing**.

**VHosts vs. Subdomains**:

| **VHosts** | **Subdomains** |
|------------|---------------|
| Served on the **same IP** as the main website. | Can have a different IP from the main website. |
| May or may not have a **public DNS record**. | Typically registered in public DNS. |
| Used for hosting multiple websites on **one server/IP**. | Treated as separate websites. |

**Why Can't We Use Regular Subdomain Fuzzing?**
- Some websites have **hidden subdomains** that are **not** registered in public DNS.
- Public subdomain fuzzing tools only discover **publicly registered** subdomains.
- If a subdomain does **not** exist in public DNS, visiting it in a browser will **fail**.

**How Vhost Fuzzing Works**
To discover **both public and non-public subdomains/VHosts**, we:
1. **Target a known IP** (already identified).
2. **Fuzz for VHosts** served on that IP.
3. **Identify websites that are not publicly listed**.

This method allows us to uncover **hidden subdomains** and **virtual hosts** that might not be otherwise accessible.

To scan for VHosts, **without manually** adding the entire wordlist to our `/etc/hosts`, we will be fuzzing HTTP headers, specifically the `Host:` header. To do that, we can use the `-H` flag to specify a header and will use the FUZZ keyword within it, as follows:
```txt
$ ffuf -w /opt/useful/seclists/Discovery/DNS/subdomains-top1million-5000.txt:FUZZ -u http://academy.htb:PORT/ -H 'Host: FUZZ.academy.htb'
```

### 3.4 - Filtering Results



```txt
MATCHER OPTIONS:
  -mc              Match HTTP status codes, or "all" for everything. (default: 200,204,301,302,307,401,403)
  -ml              Match amount of lines in response
  -mr              Match regexp
  -ms              Match HTTP response size
  -mw              Match amount of words in response

FILTER OPTIONS:
  -fc              Filter HTTP status codes from response. Comma separated list of codes and ranges
  -fl              Filter by amount of lines in response. Comma separated list of line counts and ranges
  -fr              Filter regexp
  -fs              Filter HTTP response size. Comma separated list of sizes and ranges
  -fw              Filter by amount of words in response. Comma separated list of word counts and ranges
```


## 4. Parameter Fuzzing

If you found a **hidden page** through fuzzing but got an **"Access Denied"** or **"Forbidden"** message, Parameter Fuzzing could help. Here's why:  

1. Web applications often use **parameters** in URLs, headers, or requests to control access. Some examples:  
- `?admin=true` → Maybe adding `admin=true` in the URL grants admin access.  
- `?debug=1` → Some developers leave debug modes enabled, revealing sensitive info.  
- `?access_key=XYZ` → Some pages require a special access key, and fuzzing might help find it.  

2. Even if you **don’t have access**, the page might behave differently if you send the right **parameter**. For example:  
- A page might return **403 Forbidden** by default, but adding `?user=guest` could allow access.  
- It might be looking for a **specific role**, like `?role=admin`.  
- Some apps check permissions using **session tokens**, so modifying them (`?session=12345`) might reveal useful data.

3. Some websites check permissions using parameters instead of strong authentication. If you try fuzzing for different parameter names, you might bypass security controls:  
- Changing `?view=user` to `?view=admin`.  
- Modifying `?id=1234` to `?id=0001` to see if you can access another user’s data.  

### GET Request Fuzzing:

```txt
$ ffuf -w /opt/useful/seclists/Discovery/Web-Content/burp-parameter-names.txt:FUZZ -u http://admin.academy.htb:PORT/admin/admin.php?FUZZ=key -fs xxx
```
**Note:**
> -  We will get many results back, so we will filter out the default response size we are getting (the -fs xxx part, xxx is the size).

When accessing the page with previous result, if the page show that it's content has been deprecated and appears to be no longer in use, try using **POST** parameter instead. 

### POST Request Fuzzing

The main difference between **POST** requests and **GET** requests is that **POST** requests are not passed with the URL and cannot simply be appended after a `?` symbol. **POST** requests are passed in the **data** field within the HTTP request. (Mentioned in Web Requests)

To fuzz the **data** field with **ffuf**, we can use the `-d` flag, as we saw previously in the output of `ffuf -h`. We also have to add `-X POST` to send **POST** requests.

**Note:**
> -  In PHP, **"POST"** data **"content-type"** can only accept `application/x-www-form-urlencoded`. So, we can set that in "ffuf" with `-H 'Content-Type: application/x-www-form-urlencoded'`.

```txt
$ ffuf -w /opt/useful/seclists/Discovery/Web-Content/burp-parameter-names.txt:FUZZ -u http://admin.academy.htb:PORT/admin/admin.php -X POST -d 'FUZZ=key' -H 'Content-Type: application/x-www-form-urlencoded' -fs xxx
```

### Value Fuzzing

Now, assuming we've found the correct parameter, it could be something like: `id`. However, the value (`key`) is still unknown. So, try FUZZ it with custom/pre-built wordlist.

```txt
$ ffuf -w ids.txt:FUZZ -u http://admin.academy.htb:PORT/admin/admin.php -X POST -d 'id=FUZZ' -H 'Content-Type: application/x-www-form-urlencoded' -fs xxx
```
