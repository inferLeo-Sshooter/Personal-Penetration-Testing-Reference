# Obfuscating attacks using encodings

> - The idea is to evade input filters and inject harmful payloads for a variety of attacks, such as XSS and SQL injection.


### How to Determine Which Obfuscation Method Will Work

When pentesting a website, determining which obfuscation method will work depends on how the application processes and decodes input. While trial and error is sometimes necessary, you can use several techniques to narrow down your approach.

1. Understand How the Application Handles Input
- **Inspect the Source Code** (if available) – Look for how input is sanitized or processed.
- **Check Server-Side Technologies** – Different frameworks and databases handle encoding differently (e.g., PHP, ASP.NET, Python, Java).
- **Review Security Mechanisms** – Web Application Firewalls (WAFs) or input validation libraries may restrict specific encodings.

2. Use Probing Techniques
- **Send Controlled Inputs** – Try simple payloads like `<script>alert(1)</script>` to see if they execute.
- **Test for Partial Filtering** – If `SELECT` is blocked but `SEL ECT` isn’t, obfuscation via spaces or special characters might work.
- **Check Encoding Behavior** – Use payloads with:
  - **URL encoding**: `%3Cscript%3E`
  - **HTML encoding**: `&lt;script&gt;`
  - **Unicode escaping**: `\u003Cscript\u003E`
  - This helps identify how the application decodes input.

3. Observe Responses
- **Error Messages** – Detailed error messages (like SQL errors) can reveal how input is interpreted.
- **Output Reflection** – If input appears encoded (e.g., `&lt;script&gt;` instead of `<script>`), it might be processed but displayed as text.
- **Double Decoding** – If a single encoding doesn’t work, check if double encoding (`%2527` for `%27`) is being processed.

4. Use Debugging Tools
- **Burp Suite Intruder** – Automate testing of different encodings and obfuscation techniques.
- **Payload Testing Frameworks** – Tools like `ffuf`, `wfuzz`, and `sqlmap` can test various input transformations.
- **JavaScript Console / DevTools** – Check how JavaScript processes and executes inputs in client-side vulnerabilities.

5. Check for Encoding Layers
- **Look for Double Encoding Signs** – If `%3Cscript%3E` is blocked but `%253Cscript%253E` works, the server may be decoding twice.
- **Test with Different Contexts** – Some encoding techniques work in JavaScript (`\u0061` for `a`), while others work in SQL (`CHAR(83)` for `S`).

6. Use Known Bypass Techniques
- If a single method doesn’t work, try combining multiple (e.g., HTML encoding + Unicode escaping).
- Sometimes, encoding tricks like adding leading zeros (`%00`) or unnecessary characters (`<!-- -->`) can bypass filters.

### Conclusion
While testing each method is one way to find a bypass, understanding how the target application processes input can help predict which obfuscation methods are more likely to succeed. By using controlled probes, analyzing responses, and leveraging automation tools, you can efficiently determine the best encoding strategy.

## 1. Context-specific decoding

Different parts of a web application **decode data differently** before using it. Attackers take advantage of this behavior to bypass security filters.  

- **Server-side decoding**: Query parameters are usually **URL decoded** by the server.  
- **Client-side decoding**: Browsers may **HTML decode** content inside an HTML element.  

### **💡 Attack Strategy**  
1. **Identify where your payload is being injected.**  
2. **Determine how the input is being decoded** (server-side, client-side, or both).  
3. **Use alternative encodings** to represent the same payload in a way that bypasses filters.

## 2. Decoding discrepancies

Many web applications **block suspicious patterns** like `<script>` in XSS or `UNION SELECT` in SQLi. However, **filters must decode inputs to check them**, and if there’s a difference in how the filter decodes versus how the backend processes data, attackers can **bypass security**.

### **Example of a Decoding Discrepancy Attack**  
1. The web app **filters** `<script>` but **does not normalize encodings**.  
2. The attacker sends an encoded version like `%3Cscript%3E`.  
3. The **filter does not decode it properly**, allowing it to pass.  
4. The **backend or browser decodes it later**, executing the malicious code.  

### 🔒 **Defending Against Encoded Attacks**  
✔ **Normalize input before filtering** – Decode **all** user inputs before checking for malicious content.  
✔ **Use strict input validation** – Allow only expected characters and reject unexpected encodings.  
✔ **Apply Web Application Firewalls (WAFs)** – Use **advanced detection** that can handle multiple encoding layers.  
✔ **Monitor and log traffic** – Look for suspicious encoding patterns in logs.  

## 3. Obfuscation via URL encoding

In URLs, certain characters have special meanings. For example, an ampersand (`&`) is used to separate parameters in a query string. However, user input may also contain these characters, which can create parsing issues.  

To prevent confusion, browsers automatically **URL encode** these characters by replacing them with a `%` followed by their **hex code**.  

### Example:  
If a user searches for `"Fish & Chips"`, the URL becomes: `[...?search=Fish+%26+Chips]`. This ensures that `&` is not mistaken for a delimiter.

### Context of URL Decoding  

Any input passed via a URL is **automatically decoded server-side** before being processed. This means certain encoded sequences are interpreted as specific characters:  

- `%22` → `"`  
- `%3C` → `<`  
- `%3E` → `>`  

This allows attackers to **inject encoded payloads**, which may still be correctly interpreted by the backend application.

### Bypassing Security Filters  

Sometimes, **web application firewalls (WAFs)** and other security mechanisms do not properly decode input before checking for malicious patterns.  

By encoding blacklisted keywords or characters, attackers can **bypass input validation**.  

### Example:  
Instead of sending `SELECT`, an attacker could encode it as: `%53%45%4C%45%43%54`. If the security system does not decode this before filtering, the backend may still interpret it as a valid SQL command.

## 4. Obfuscation via double URL encoding

Some servers **decode URLs twice**, which can be exploited to bypass security mechanisms. If security filters only decode once but the backend decodes twice, an attacker can smuggle malicious input by encoding it twice.

### Example: Injecting an XSS Payload  

A standard XSS payload might be injected like this: `[...?search=%3Cimg%20src%3Dx%20onerror%3Dalert(1)%3E]`. If a **Web Application Firewall (WAF)** decodes this once, it recognizes the payload and blocks it. 

### Bypassing the Filter  

If the input is **double-encoded**, the `%` characters themselves are encoded as `%25`, transforming the URL into: `[...?search=%253Cimg%2520src%253Dx%2520onerror%253Dalert(1)%253E]`. Since the WAF **only decodes once**, it may fail to detect the attack. However, if the backend decodes a second time, the payload is fully restored and executed.

Double URL encoding can help attackers **evade input validation** if security mechanisms do not decode input consistently. To prevent this, security measures should **normalize** input before processing it.

## 5. Obfuscation via HTML encoding

In HTML, certain characters are **encoded** to prevent browsers from misinterpreting them as part of the markup. This is done by replacing characters with their corresponding **HTML entities**, such as:

- `&colon;` for `:`  
- `&#58;` or `&#x3a;` for `:`

### Using HTML Encoding for Obfuscation  

Browsers **automatically decode** these references when rendering a page. Attackers can exploit this behavior to **hide payloads from security filters**.

For example, in an **XSS attack**, security checks might block `alert(1)`. However, if an attacker encodes part of it using HTML entities, it might go undetected: `<img src=x onerror="&#x61;lert(1)">`. When the page loads, the browser decodes the entity, restoring the full payload.

## 6. Leading zeros

When using **decimal or hex-style HTML encoding**, you can insert **leading zeros** in the code points. Some **WAFs and input filters** may fail to properly handle this, allowing attackers to bypass security checks.

### Example of Bypassing Filters  

If a payload is blocked even after HTML encoding, adding **leading zeros** can help evade detection. For instance: `<a href="javascript&#000000000058;alert(1)">Click me</a>`. Here, the `:` character (`&#58;` or &#58; - open source code and see this) is padded with leading zeros, which some filters might not recognize as a match. 

## 7. Obfuscation via XML encoding

XML allows character encoding using **numeric escape sequences**, similar to HTML. This technique can help **bypass WAFs and filters** by hiding attack keywords.

### Why Use XML Encoding?
- It allows **obfuscating** payloads while keeping XML syntax valid.
- Unlike HTML, XML encoding is **decoded by the server**, making it useful for **SQL injection** and **XSS in XML-based inputs**.
- Some security filters may **fail to detect encoded keywords**.

### Example: SQL Injection in XML
```xml
<stockCheck>
    <productId>123</productId>
    <storeId>999 &#x53;ELECT * FROM information_schema.tables</storeId>
</stockCheck>
```

Here, `&#x53;` represents `S`, preventing direct detection of `SELECT`.

## 8. Obfuscation via unicode escaping

Unicode escapes use **`\u` followed by a four-digit hex code** to represent characters. Example:
- `\u003a` → `:` (colon)
- ES6 also allows curly braces: `\u{3a}`

### Why Use Unicode Escaping?
- **Browsers decode unicode escapes**, making them useful for **XSS obfuscation**.
- Can bypass security filters by encoding **key characters** in an attack payload.
- **Remains encoded server-side** until decoded by the browser.

### Example: Bypassing Filters in JavaScript
```js
eval("\u0061lert(1)")
```
Here, `\u0061` represents `a`, so the payload executes as `alert(1)`.

**Note:** 
> Inside a string, you can escape any characters like this. However, outside of a string, escaping some characters will result in a syntax error. This includes opening and closing parentheses, for example.

It's also worth noting that the ES6-style unicode escapes also allow optional leading zeros, so some WAFs may be easily fooled using the same technique we used for HTML encodings. For example: `<a href="javascript:\u{00000000061}alert(1)">Click me</a>`

## 9. Obfuscation via hex escaping

Hex escapes use **`\x` followed by a two-digit hex code** to represent characters. Example:
- `\x61` → `a`

### Why Use Hex Escaping?
- Works like **Unicode escaping** but uses a shorter format.
- **Browsers decode** hex escapes, making them useful for **XSS obfuscation**.
- Can **bypass security filters** by encoding important characters.

Just like unicode escapes, these will be decoded client-side as long as the input is evaluated as a string: `eval("\x61lert")`.

Note that you can sometimes also obfuscate SQL statements in a similar manner using the prefix `0x`. For example, `0x53454c454354` may be decoded to form the `SELECT` keyword.

## 10. Obfuscation via octal escaping

Octal escaping works like **hex escaping**, but it uses a **base-8 numbering system** instead of base-16.

### How It Works
- **Prefix:** A standalone backslash (`\`)
- **Example:** `\141` → `a`

`eval("\141lert(1)")`

### Why Use Octal Escaping?
- **Alternative encoding** for obfuscation.
- Can help **bypass input filters** in web applications.
- **Browsers decode** these characters, making them useful for **XSS payloads**.

## 11. Obfuscation via multiple encodings

You can **combine multiple encodings** to hide payloads behind layers of obfuscation.


1. **HTML Encoding**  
   - The URL starts with **`javascript:`** using an **HTML-encoded backslash (`&bsol;`)**.
   - This results in a **decoded backslash (`\`)**, forming a **Unicode escape sequence**.

   `<a href="javascript:&bsol;u0061lert(1)">Click me</a>`

2. **Unicode Decoding**  
   - The browser then **interprets `\u0061` as 'a'**, making the payload valid.

  `<a href="javascript:\u0061lert(1)">Click me</a>`

3. **Final Decoding**  
   - The payload is fully decoded into a working **JavaScript XSS attack**.

  `<a href="javascript:alert(1)">Click me</a>`

### Why Use Multiple Encodings?
- **Bypasses filters** that block direct injection.
- **Exploits decoding order** in web applications.
- **Evades security mechanisms** like Web Application Firewalls (WAFs).

## 12. Obfuscation via the SQL CHAR() function

The **CHAR() function** in SQL can be used to **bypass keyword blacklists** by converting numeric values into characters.

### How It Works:
- **CHAR()** takes a decimal or hexadecimal code and returns the corresponding character.
- **Hex codes** must be prefixed with **`0x`**.
- Example: `CHAR(83)` or `CHAR(0x53)` returns **S**.
- By concatenating the returned values, you can use this approach to obfuscate blocked keywords. For example, even if `SELECT` is blacklisted, the following injection initially appears harmless:
  `CHAR(83)+CHAR(69)+CHAR(76)+CHAR(69)+CHAR(67)+CHAR(84)`
- However, when this is processed as SQL by the application, it will dynamically construct the `SELECT` keyword and execute the injected query.
