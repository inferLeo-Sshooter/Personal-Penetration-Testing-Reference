
# 1. Understanding Cross-Site Scripting (XSS)

XSS is a common web security vulnerability where attackers inject malicious JavaScript code into websites that don't properly sanitize user input. When other users view the affected page, their browsers unknowingly execute this malicious code.

## How XSS Works
- Vulnerable applications fail to properly sanitize user input in fields like comments or replies
- Malicious code executes in victims' browsers when they view the affected page
- This allows attackers to bypass the same-origin policy that normally separates websites
- Attackers can impersonate victims, perform actions on their behalf, and access their data

Though XSS only executes client-side (not affecting the server directly), its high frequency in web applications makes it a medium risk that developers should actively work to detect and prevent.

## XSS Proof of Concept
To confirm an XSS vulnerability, inject code that executes JavaScript in your browser. The traditional method uses the `alert()` function as it's simple and obvious when triggered. However, in Chrome (from version 92), `alert()` is blocked in cross-origin iframes, so `print()` is recommended as an alternative for testing certain advanced XSS scenarios.

## What Can XSS Attacks Do?
XSS attacks can execute any JavaScript code in the victim's browser, enabling:
- Stealing session cookies and sending them to attacker-controlled servers
- Making unauthorized API calls (like changing passwords)
- Mining cryptocurrencies or displaying unwanted ads
- Other malicious actions within the vulnerable website's domain

While XSS is limited to browser execution and the same-origin policy, it remains dangerous. In rare cases, when combined with browser vulnerabilities, XSS could potentially lead to system-level code execution by breaking out of the browser sandbox.

## Types of XSS

There are three main types of XSS vulnerabilities:

| Type | Description |
|---|---|
| **`Stored (Persistent) XSS`**, where the malicious script comes from the website's database. | The most critical type of XSS, which occurs when user input is stored on the back-end database and then displayed upon retrieval (e.g., posts or comments) |
| **`Reflected (Non-Persistent) XSS`**, where the malicious script comes from the current HTTP request. | Occurs when user input is displayed on the page after being processed by the backend server, but without being stored (e.g., search result or error message) |
| **`DOM-based XSS`**, where the vulnerability exists in client-side code rather than server-side code. | Another Non-Persistent XSS type that occurs when user input is directly shown in the browser and is completely processed on the client-side, without reaching the back-end server (e.g., through client-side HTTP parameters or anchor tags) |


## What Can XSS Be Used For?

XSS vulnerabilities allow attackers to:

- **Impersonate victims** - Execute actions as the compromised user
- **Access sensitive data** - View anything the user has permission to see
- **Steal credentials** - Capture usernames, passwords, or session tokens
- **Deface websites** - Change content visible to the victim
- **Inject malicious code** - Add trojans or other harmful functionality

## Impact Severity

The actual impact of an XSS attack varies based on:

1. **Application type**:
   - Minimal impact on public information sites
   - Severe impact on applications with sensitive data (banking, healthcare, email)

2. **User privileges**:
   - Critical impact when administrative users are compromised
   - Can lead to complete application takeover if privileged accounts are affected

The most dangerous XSS attacks target users with administrative access, as this can compromise the entire application and all its users.


# Stored Cross-Site Scripting (XSS)

Stored XSS (also called persistent or second-order XSS) occurs when:

- A web application stores unsanitized user input in a database or server
- This malicious data is later served to other users as part of the website content
- The victim's browser executes the injected script when viewing the affected page

## Common Entry Points
- Blog comments
- Forum posts
- User profile information
- Customer details in orders
- Third-party content (emails, social media posts)

## Example
On a message board that doesn't sanitize input:
1. Normal user posts: `<p>Hello, this is my message!</p>`
2. Attacker posts: `<p><script>/* Malicious code here... */</script></p>`
3. When any user views the page containing the attacker's post, the malicious script executes in their browser with their privileges

What makes stored XSS particularly dangerous is that the attack persists on the website and can affect multiple victims without requiring them to click on malicious links.


# Reflected Cross-Site Scripting (XSS)

Reflected XSS is the most straightforward type of cross-site scripting vulnerability. It occurs when:

- User input from an HTTP request is immediately returned in the page response
- The application fails to properly sanitize this input before including it in the response
- The malicious script executes in the victim's browser when they load the manipulated URL

## How It Works

1. **Normal functionality**:  
   `https://insecure-website.com/status?message=All+is+well`  
   Displays on page: `Status: All is well.`

2. **Attacker exploitation**:  
   `https://insecure-website.com/status?message=<script>/*+Bad+stuff+here...+*/</script>`  
   Displays and executes: `Status: <script>/* Bad stuff here... */</script>`

3. **Attack delivery**:  
   The attacker tricks victims into clicking the malicious URL through phishing emails, messages, or shortened links

When executed, the malicious script runs with the victim's privileges on the website, allowing the attacker to perform actions as the victim, steal sensitive information, or hijack their session.

# DOM-Based Cross-Site Scripting (XSS)

DOM-based XSS occurs when client-side JavaScript code unsafely handles data and writes it to the page. Unlike other XSS types, the vulnerability exists entirely in the browser - the malicious payload may never be sent to the server.

## How It Works

1. **Unsafe DOM manipulation**: Client-side JavaScript takes user input and inserts it into the page using unsafe methods like `innerHTML`

2. **Example vulnerability**:
   ```javascript
   var search = document.getElementById('search').value;
   var results = document.getElementById('results');
   results.innerHTML = 'You searched for: ' + search;
   ```

3. **Exploitation**: An attacker provides input containing HTML tags with script execution:
   ```
   You searched for: <img src=1 onerror='/* Malicious code here... */'>
   ```

4. **Attack delivery**: Usually through URL parameters that the client-side code reads from the page location, like:
   ```
   https://example.com/search?q=<img src=1 onerror='alert(document.cookie)'>
   ```

DOM-based XSS is unique because the vulnerability exists in the client-side code rather than server-side processing, making it sometimes harder to detect with traditional security testing.


# Finding and Testing for XSS Vulnerabilities

## Automated Testing
- Burp Suite's web vulnerability scanner (or any other scanner) can reliably detect most XSS vulnerabilities

## Manual Testing Methods

### For Reflected and Stored XSS:
1. Submit unique test strings (simple alphanumeric values) into all application input points
2. Identify where your input appears in HTTP responses
3. Test each location with crafted payloads based on the specific context
4. Verify if JavaScript execution is possible

### For DOM-based XSS:
1. **URL parameter testing**: Insert unique values in URL parameters and search the DOM for these values
2. **JavaScript code review**: Required for detecting more complex DOM XSS in:
   - Non-URL inputs (like `document.cookie`)
   - Non-HTML sinks (like `setTimeout`)

Manual testing of complex DOM-based XSS can be extremely time-consuming, making automated tools particularly valuable for these scenarios.

The key to effective XSS testing is understanding the context where your input appears and selecting appropriate payloads based on that context.


# Content Security Policy and Dangling Markup Injection

## Content Security Policy (CSP)

Content Security Policy is a browser security mechanism designed to:
- Mitigate the impact of XSS vulnerabilities
- Control which resources can be loaded and executed
- Restrict how content behaves on a webpage

While CSP provides an additional layer of protection against XSS attacks, it's important to note that many CSP implementations can be circumvented by skilled attackers. CSP should be viewed as a defense-in-depth measure rather than a complete solution to XSS.

## Dangling Markup Injection

This is an alternative attack technique used when full XSS exploitation is blocked:

- Works by injecting incomplete HTML elements ("dangling markup")
- Can capture data across domains where traditional XSS is prevented
- Often used to steal sensitive information visible to users
- Particularly effective for stealing CSRF tokens that protect against cross-site request forgery

The technique exploits how browsers parse and render incomplete HTML, allowing attackers to capture data even when script execution is prevented.
