
# 1. Understanding Cross-Site Scripting (XSS)

XSS is a common web security vulnerability where attackers inject malicious JavaScript code into websites that don't properly sanitize user input. When other users view the affected page, their browsers unknowingly execute this malicious code.

## How XSS Works
- Vulnerable applications fail to properly sanitize user input in fields like comments or replies
- Malicious code executes in victims' browsers when they view the affected page
- This allows attackers to bypass the same-origin policy that normally separates websites
- Attackers can impersonate victims, perform actions on their behalf, and access their data

Though XSS only executes client-side (not affecting the server directly), its high frequency in web applications makes it a medium risk that developers should actively work to detect and prevent.

## XSS Proof of Concept
To confirm an XSS vulnerability, inject code that executes JavaScript in your browser. The traditional method uses the `alert()` function as it's simple and obvious when triggered. However, in Chrome (from version 92), `alert()` is blocked in cross-origin iframes, so `print()` is recommended as an alternative for testing certain advanced XSS scenarios.

## What Can XSS Attacks Do?
XSS attacks can execute any JavaScript code in the victim's browser, enabling:
- Stealing session cookies and sending them to attacker-controlled servers
- Making unauthorized API calls (like changing passwords)
- Mining cryptocurrencies or displaying unwanted ads
- Other malicious actions within the vulnerable website's domain

While XSS is limited to browser execution and the same-origin policy, it remains dangerous. In rare cases, when combined with browser vulnerabilities, XSS could potentially lead to system-level code execution by breaking out of the browser sandbox.

## Types of XSS

There are three main types of XSS vulnerabilities:

| Type | Description |
|---|---|
| `Stored (Persistent) XSS`, where the malicious script comes from the website's database. | The most critical type of XSS, which occurs when user input is stored on the back-end database and then displayed upon retrieval (e.g., posts or comments) |
| `Reflected (Non-Persistent) XSS`, where the malicious script comes from the current HTTP request. | Occurs when user input is displayed on the page after being processed by the backend server, but without being stored (e.g., search result or error message) |
| `DOM-based XSS`, where the vulnerability exists in client-side code rather than server-side code. | Another Non-Persistent XSS type that occurs when user input is directly shown in the browser and is completely processed on the client-side, without reaching the back-end server (e.g., through client-side HTTP parameters or anchor tags) |
