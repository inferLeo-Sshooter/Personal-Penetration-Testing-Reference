

## 1. Stored (Persistent) XSS

**Stored XSS** happens when malicious input is saved by the application (like in a database) and shown to other users later **without proper sanitization**.

### Why It’s Dangerous:
- The attack **persists** on the site.
- **Every user** who views the affected page can be impacted.
- It’s often **hard to remove**, as the payload lives in the back-end.

### Example Scenario:
1. A user submits a comment:
   ```txt
   POST /post/comment
   comment=Great+post!
   ```
2. The comment is stored and later shown like:
   ```txt
   <p>Great post!</p>
   ```
3. An attacker submits:
   ```txt
   comment=<script>alert('Hacked')</script>
   ```
4. Other users who visit the page now see:
   ```txt
   <p><script>alert('Hacked')</script></p>
   ```
   This script runs in their browser, potentially stealing cookies, sessions, or more.

--- 

## 2. Impact of Stored XSS

- **Full user compromise**: Attackers can run scripts in users' browsers, steal data, hijack sessions, or perform actions on their behalf.
- **No user interaction needed**: Unlike reflected XSS, the malicious script is **stored in the app** and triggered automatically when users visit the page.
- **More reliable**: Stored XSS doesn't rely on timing or tricking users into clicking a link — users will be affected just by accessing the vulnerable page.
- **Especially dangerous for logged-in users**, as the attack runs during their active session.

--- 


## 3. How to Find and Test for Stored XSS

### 🛠️ Manual Testing Steps

#### 1. **Identify Entry Points**
Places where user input enters the app:
- URL parameters or body data
- URL path
- HTTP headers
- Third-party sources (emails, tweets, feeds)

#### 2. **Identify Exit Points**
Places where input might appear in responses:
- Any page or feature visible to any user (e.g., logs, comments, dashboards)

#### 3. **Find the Link**
Submit a **unique test value** into each entry point and look for it later in the app’s responses. Focus on areas like:
- Comments
- User profiles
- Search history

Make sure the value persists across requests (not just reflected).

#### 4. **Test Payloads**
Once stored data is found in a response:
- Check **context** (HTML, JavaScript, attributes, etc.)
- Insert **XSS payloads** like `<script>alert(1)</script>` to see if they execute

This step is similar to reflected XSS testing.
