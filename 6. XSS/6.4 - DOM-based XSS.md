# 1. What is DOM-Based XSS

While **reflected XSS** sends the input data to the back-end server through HTTP requests, **DOM XSS** is completely processed on the client-side through JavaScript. DOM XSS occurs when JavaScript is used to change the page source through the **Document Object Model (DOM)**.

**DOM-based XSS** happens when JavaScript on a page takes untrusted data (like from the URL) and sends it to a dangerous location (like `innerHTML` or `eval()`), causing malicious code to run.

## Example:

When we try adding a `test` item, in a `To-Do List web applications`. However, if we open the `Network tab` in the Firefox Developer Tools, and re-add the `test` item, we would notice that **no HTTP requests** are being made. We see that the input parameter in the URL is using a hashtag `#` for the item we added, which means that this is a client-side parameter that is completely processed on the browser. This indicates that the input is being processed at the client-side through JavaScript and never reaches the back-end; hence it is a `DOM-based XSS`.

DOM-based XSS occurs when:
- JavaScript takes data from attacker-controllable sources (usually URLs)
- This data is passed to sinks that can execute code (like `eval()` or `innerHTML`)
- Attackers exploit this to run malicious JavaScript on other users' browsers

To perform an attack:
- Place malicious code in a URL (typically in query strings or fragments)
- Trick victims into clicking the link
- When loaded, the vulnerable page executes the attacker's code

This vulnerability can lead to account hijacking and other security issues.

## Source & Sink

The **`Source`** is the JavaScript object that takes the user input, and it can be any input parameter like a URL parameter or an input field.

On the other hand, the **`Sink`** is the function that writes the user input to a DOM Object on the page. If the **`Sink`** function does not properly sanitize the user input, it would be vulnerable to an XSS attack. 

Some of the commonly used JavaScript functions to write to DOM objects are:
- `document.write()`
- `DOM.innerHTML`
- `DOM.outerHTML`
- `document.writeln()`
- `document.domain`
- `element.innerHTML`
- `element.outerHTML`
- `element.insertAdjacentHTML`
- `element.onevent`

Furthermore, some of the jQuery library functions that write to DOM objects are:
- `add()`
- `after()`
- `append()`
- `animate()`
- `insertAfter()`
- `insertBefore()`
- `before()`
- `html()`
- `prepend()`
- `replaceAll()`
- `replaceWith()`
- `wrap()`
- `wrapInner()`
- `wrapAll()`
- `has()`
- `constructor()`
- `init()`
- `index()`
- `jQuery.parseHTML()`
- `$.parseHTML()`



### 🔁 How DOM-Based XSS Works:
1. **Source**: Attacker-controlled data (e.g., `window.location`).
2. **Sink**: A function or property that executes or renders that data (e.g., `innerHTML`, `eval()`).
3. **Example**: A link like  
   ```
   https://example.com/page?input=<script>alert(1)</script>
   ```
   causes the script to run if the page handles `input` insecurely.

### 💥 Impact:
- JavaScript runs in the victim's browser.
- Can lead to **account hijacking** or other malicious actions.

# 2. Testing for DOM-based XSS

## HTML Sink Testing Process

1. **Insert a test string** into the source (like `location.search`)
2. **Inspect the DOM** using developer tools to find where your string appears
   - Don't use "View source" as it won't show JavaScript-modified HTML
   - Use `Ctrl+F` (or `Cmd+F` on Mac) in developer tools to search for your string

3. **Identify the context** where your string appears in the DOM
4. **Refine your input** based on this context
   - Example: If your string is in a double-quoted attribute, try injecting quotes to break out

**Note:** Browser differences matter:
- Chrome, Firefox, and Safari URL-encode `location.search` and `location.hash`
- Older IE and Edge browsers don't URL-encode these sources
- URL-encoded data is generally less vulnerable to XSS attacks

## Testing JavaScript Execution Sinks for DOM-based XSS

1. **Find source references** in the JavaScript code
   - Use `Ctrl+Shift+F` (or `Cmd+Alt+F` on Mac) in developer tools to search for sources like `location`

2. **Use JavaScript debugger**
   - Add break points where sources are referenced
   - Track how source values flow through the code
   - Follow variable assignments that originate from the source

3. **Inspect values before they reach sinks**
   - Hover over variables in debugger to see their values
   - Identify when source data reaches execution sinks

4. **Refine your input**
   - Test different payloads based on how the data is processed
   - Determine if you can execute arbitrary JavaScript

**Note:** Unlike HTML sinks, input to JavaScript sinks may not be visible in the DOM, making testing more complex.


## Testing for DOM XSS with DOM Invader

### Using Burp's DOM Invader Extension

DOM Invader is a built-in extension in Burp's browser that automates much of the DOM XSS testing process:

- **Simplifies detection** by handling the complex analysis automatically
- **Saves time** by eliminating manual code review of minified JavaScript
- **Automates identification** of sources and sinks in the page

This tool is especially helpful when dealing with:
- Complex JavaScript codebases
- Minified or obfuscated code
- Applications with numerous potential DOM XSS vectors

DOM Invader handles the tedious parts of DOM XSS testing that would otherwise require manual inspection of JavaScript execution paths.

# 3. Exploiting DOM XSS with different sources and sinks


## 3.1. DOM XSS Exploitation Basics:

DOM-based XSS vulnerabilities occur when data flows from a source to a sink without proper validation. Different sinks require different exploitation techniques:

### Key Sinks and Techniques

- **document.write** - Accepts script elements directly:
  ```javascript
  document.write('<script>alert(document.domain)</script>');
  ```
  (You may need to close existing HTML elements depending on context)

- **innerHTML** - Doesn't accept script elements in modern browsers:
  ```javascript
  element.innerHTML='<img src=1 onerror=alert(document.domain)>';
  ```
  (Use alternative elements like img/iframe with event handlers instead)

Website processing and validation must be accounted for when crafting payloads.


## 3.2. Sources and sinks in third-party dependencies

Modern web applications are typically built using a number of third-party libraries and frameworks, which often provide additional functions and capabilities for developers. It's important to remember that some of these are also potential sources and sinks for DOM XSS.

### DOM XSS in jQuery

jQuery functions like `attr()` can create DOM XSS vulnerabilities when handling user input:

**Example:**

- When jQuery modifies DOM element attributes using unvalidated user input:
  ```javascript
  $('#backLink').attr("href", (new URLSearchParams(window.location.search)).get('returnUrl'));
  ```

**Exploitation:**

- Attackers can inject JavaScript URLs into the parameter:
  ```
  ?returnUrl=javascript:alert(document.domain)
  ```
- When the link is clicked, the malicious JavaScript executes

This occurs when data flows from user-controlled sources (like URL parameters) to DOM-modifying jQuery functions without proper sanitization.

### DOM XSS via jQuery Selector Function

**The Vulnerability**

The jQuery `$()` selector function can be a DOM XSS sink when combined with user-controlled inputs:

```javascript
$(window).on('hashchange', function() { 
    var element = $(location.hash); 
    element[0].scrollIntoView(); 
});
```

**Key Points**

- Popular vulnerability occurred when `location.hash` (user-controllable) was passed to `$()`
- Modern jQuery versions block HTML injection when input starts with `#`, but legacy code remains vulnerable
- Similar vulnerabilities exist if you control `$()` input from sources without `#` prefix

**Exploitation**

Triggering without user interaction via iframe:

```html
<iframe src="https://vulnerable-website.com#" onload="this.src+='<img src=1 onerror=alert(1)>'">
```

This loads the page with empty hash, then adds XSS payload to trigger the hashchange event.

### DOM XSS in AngularJS

**Vulnerability**

When a website uses AngularJS (identified by `ng-app` attribute), JavaScript can be executed without traditional XSS vectors.

**Key Points**

- AngularJS processes content within double curly braces `{{ }}`
- These expressions can execute JavaScript code
- No need for angle brackets or event handlers
- Can occur in HTML body or within attributes

**Example**

Any content within `{{ }}` is evaluated as JavaScript when the page uses the AngularJS framework.

## 3.3. DOM XSS Combined with Reflected and Stored Data

### Pure DOM XSS
- Self-contained within a single page
- Data flows directly from client-side source to sink
- Example: Script reads URL data and writes to dangerous sink

### Reflected DOM XSS
- Server echoes request data in response
- Reflected data placed in JavaScript or DOM elements
- Client-side script processes this data unsafely
- Example: `eval('var data = "reflected string"');`

### Stored DOM XSS
- Server stores data from one request
- Data is included in later responses
- Client-side script processes stored data unsafely
- Example: `element.innerHTML = comment.author`

The key difference is where the malicious data originates - directly from the browser, reflected from the server, or retrieved from server storage.
