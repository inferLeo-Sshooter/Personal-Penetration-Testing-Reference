# 1. What is DOM-Based XSS

**DOM-based XSS** happens when JavaScript on a page takes untrusted data (like from the URL) and sends it to a dangerous location (like `innerHTML` or `eval()`), causing malicious code to run.

DOM-based XSS occurs when:
- JavaScript takes data from attacker-controllable sources (usually URLs)
- This data is passed to sinks that can execute code (like `eval()` or `innerHTML`)
- Attackers exploit this to run malicious JavaScript on other users' browsers

To perform an attack:
- Place malicious code in a URL (typically in query strings or fragments)
- Trick victims into clicking the link
- When loaded, the vulnerable page executes the attacker's code

This vulnerability can lead to account hijacking and other security issues.

### 🔁 How It Works:
1. **Source**: Attacker-controlled data (e.g., `window.location`).
2. **Sink**: A function or property that executes or renders that data (e.g., `innerHTML`, `eval()`).
3. **Example**: A link like  
   ```
   https://example.com/page?input=<script>alert(1)</script>
   ```
   causes the script to run if the page handles `input` insecurely.

### 💥 Impact:
- JavaScript runs in the victim's browser.
- Can lead to **account hijacking** or other malicious actions.

# 2. Testing for DOM-based XSS

## HTML Sink Testing Process

1. **Insert a test string** into the source (like `location.search`)
2. **Inspect the DOM** using developer tools to find where your string appears
   - Don't use "View source" as it won't show JavaScript-modified HTML
   - Use `Ctrl+F` (or `Cmd+F` on Mac) in developer tools to search for your string

3. **Identify the context** where your string appears in the DOM
4. **Refine your input** based on this context
   - Example: If your string is in a double-quoted attribute, try injecting quotes to break out

**Note:** Browser differences matter:
- Chrome, Firefox, and Safari URL-encode `location.search` and `location.hash`
- Older IE and Edge browsers don't URL-encode these sources
- URL-encoded data is generally less vulnerable to XSS attacks

## Testing JavaScript Execution Sinks for DOM-based XSS

1. **Find source references** in the JavaScript code
   - Use `Ctrl+Shift+F` (or `Cmd+Alt+F` on Mac) in developer tools to search for sources like `location`

2. **Use JavaScript debugger**
   - Add break points where sources are referenced
   - Track how source values flow through the code
   - Follow variable assignments that originate from the source

3. **Inspect values before they reach sinks**
   - Hover over variables in debugger to see their values
   - Identify when source data reaches execution sinks

4. **Refine your input**
   - Test different payloads based on how the data is processed
   - Determine if you can execute arbitrary JavaScript

**Note:** Unlike HTML sinks, input to JavaScript sinks may not be visible in the DOM, making testing more complex.

# 3. Exploiting DOM XSS with different sources and sinks


## 3.1. DOM XSS Exploitation Basics:

DOM-based XSS vulnerabilities occur when data flows from a source to a sink without proper validation. Different sinks require different exploitation techniques:

### Key Sinks and Techniques

- **document.write** - Accepts script elements directly:
  ```javascript
  document.write('<script>alert(document.domain)</script>');
  ```
  (You may need to close existing HTML elements depending on context)

- **innerHTML** - Doesn't accept script elements in modern browsers:
  ```javascript
  element.innerHTML='<img src=1 onerror=alert(document.domain)>';
  ```
  (Use alternative elements like img/iframe with event handlers instead)

Website processing and validation must be accounted for when crafting payloads.
