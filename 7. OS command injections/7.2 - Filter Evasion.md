# 1. Filter/WAF Detection

For example, we are usign Host Checker web application to testing around. We send our payload: `127.0.0.1; whoami`. However, we get the error message `invalid input`. 

Security mechanism blocked our request. Error appears in the output field, indicating detection by the PHP application itself.

**Note:** If a different page appears showing our IP and request details, this likely means a Web Application Firewall (WAF) intercepted it instead.

Other than the IP (which we know is not blacklisted), we sent:
- A semi-colon character `;`
- A `space` character
- A `whoami` command

So, the web application either detected a blacklisted character or detected a blacklisted command, or both. There for, we will try figure out what is blacklisted by sending the correct input (`127.0.0.1`) with each of element above. (`127.0.0.1;`, `127.0.0.1 `, `127.0.0.1whoami`).

After testing, it result that all 3 elements were blackisted.

# 2. Bypassing Space Filters

## Bypass Blacklisted Operators

In previous example, our payload: `127.0.0.1; whoami`. The `;` was used to seperate 2 command: the original correct `127.0.0.1` and inject command `whoami`. However, `;` is blacklisted. Instead, we can use the `new-line` character `%0a`, which is usually not blacklisted, as it may be needed in the payload itself. We know that the new-line character works in appending our commands both in Linux and on Windows.

Payload: `127.0.0.1%0a` went through and no error message was made.


## Bypass Blacklisted Spaces

### 1 - Using Tabs

Using tabs (`%09`) instead of spaces is a technique that may work, as both Linux and Windows accept commands with tabs between arguments, and they are executed the same. (`127.0.0.1%0a%09`)

### 2 - Using $IFS

Using the (`$IFS`) Linux Environment Variable may also work since its default value is a space and a tab, which would work between command arguments. So, if we use `${IFS}` where the spaces should be, the variable should be automatically replaced with a space, and our command should work. (`127.0.0.1%0a${IFS}`)

### 3 - Using Brace Expansion
There are many other methods we can utilize to bypass space filters. For example, we can use the Bash Brace Expansion feature, which automatically adds spaces between arguments wrapped between braces, as follows:

```
$ {ls,-la}

total 0
drwxr-xr-x 1 21y4d 21y4d   0 Jul 13 07:37 .
drwxr-xr-x 1 21y4d 21y4d   0 Jul 13 13:01 ..
```

As we can see, the command was successfully executed without having spaces in it. We can utilize the same method in command injection filter bypasses, by using brace expansion on our command arguments, like (`127.0.0.1%0a{ls,-la}`)

# 3. Bypassing Other Blacklisted Characters

Common blacklisted characters include injection operators, spaces, and slashes (`/`, `\`) needed for directory paths.

Multiple techniques exist to generate these characters without directly using them, allowing bypass of character filters.


## 1 - LINUX

We can use Linux Environment Variables to obtain special characters like slashes that might be blacklisted. Unlike `${IFS}` (directly replaced with space), we can extract specific characters from variables using substring operations:

* Specify `start` and `length` parameters to extract individual characters
* This works for slashes, semi-colons, and other filtered characters
* Format: `${VARIABLE:position:length}`

For example, if we look at the `$PATH` environment variable in Linux, it may look something like the following:

```
$ echo ${PATH}

/usr/local/bin:/usr/bin:/bin:/usr/games
```

So, if we start at the `0` character, and only take a string of length `1`, we will end up with only the `/` character, which we can use in our payload:

```
$ echo ${PATH:0:1}

/
```

**Note:** When we use the above command in our payload, we will not add `echo`, as we are only using it in this case to show the outputted character.

We can do the same with the `$HOME` or `$PWD` environment variables as well. We can also use the same concept to get a semi-colon character, to be used as an injection operator. For example, the following command gives us a semi-colon:

```
$ echo ${LS_COLORS:10:1}

;
```

It extracts 1 character starting at index 10 from the value of the LS_COLORS variable. Example:

```
echo $LS_COLORS
# dir=01;34:ln=01;36:...

//then,

echo ${LS_COLORS:10:1}
# might output: 3
```

payload: `127.0.0.1${LS_COLORS:10:1}${IFS}`

## 2 - WINDOWS

The same concept works on Windows as well. For example, to produce a slash in Windows Command Line (CMD), we can `echo` a Windows variable (`%HOMEPATH%` -> `\Users\htb-student`), and then specify a starting position (`~6` -> `\htb-student`), and finally specifying a negative end position, which in this case is the length of the username `htb-student` (`-11` -> `\`) :

```
C:\htb> echo %HOMEPATH:~6,-11%

\
```

We can achieve the same thing using the same variables in `Windows PowerShell`. With PowerShell, a word is considered an array, so we have to specify the index of the character we need. As we only need one character, we don't have to specify the start and end positions:

```
PS C:\htb> $env:HOMEPATH[0]

\


PS C:\htb> $env:PROGRAMFILES[10]
PS C:\htb>
```

**Note:**We can also use the Get-ChildItem Env: PowerShell command to print all environment variables and then pick one of them to produce a character we need. Try to be creative and find different commands to produce similar characters.


## 3 - Character Shifting

There are other techniques to produce the required characters without using them, like `shifting characters`. For example, the following Linux command shifts the character we pass by `1`. So, all we have to do is find the character in the ASCII table that is just before our needed character (we can get it with `man ascii`), then add it instead of `[` in the below example. This way, the last printed character would be the one we need:

```
$ man ascii     # \ is on 92, before it is [ on 91
$ echo $(tr '!-}' '"-~'<<<[)

\
```

# Bypassing Blacklisted Commands

## 1 - Linux & Windows

Command shells like Bash and PowerShell ignore certain characters during execution, allowing for obfuscation while maintaining functionality. Common characters that can be inserted without affecting command execution:

* Single-quotes `'`
* Double-quotes `"`
* Various other ignored characters

This technique helps bypass filters while preserving the original command behavior. The easiest to use are quotes, and they work on both Linux and Windows servers. For example, if we want to obfuscate the whoami command, we can insert single quotes between its characters, as follows:

```
21y4d@htb[/htb]$ w'h'o'am'i

21y4d
```

The same works with double-quotes as well:

```
21y4d@htb[/htb]$ w"h"o"am"i

21y4d
```

**Note:** The important things to remember are that we `cannot mix types of quotes` and `the number of quotes must be even`. 

We can try one of the above in our payload (`127.0.0.1%0aw'h'o'am'i`) and see if it works.

## 2 - Linux Only

Bash shell ignores certain characters, allowing command obfuscation while maintaining functionality:

* Backslash character `\`
* Positional parameter `$@`

Unlike quotes, these characters:
* Don't require even numbers (can insert just one)
* Can be placed anywhere within commands

Examples:
```bash
who$@ami
w\ho\am\i
```

Both execute as `whoami` despite obfuscation.

## 3 - Windows Only

There are also some Windows-only characters we can insert in the middle of commands that do not affect the outcome, like a caret (`^`) character, as we can see in the following example:

```
C:\htb> who^ami

21y4d
```

# Advanced Command Obfuscation

Case modification techniques can bypass command blacklists:

* Invert case: `WHOAMI` instead of `whoami`
* Alternate case: `WhOaMi`

Platform differences:
* **Linux**: Case-sensitive, but blacklists may not check all variations
* **Windows**: PowerShell and CMD are case-insensitive, executing commands regardless of case

Example:
```powershell
PS C:\htb> WhOaMi
21y4d
```

This works because filters often search for exact string matches rather than all possible case variations.


For Linux/bash (case-sensitive environments), we can dynamically convert command case using built-in tools:

```bash
$(tr "[A-Z]" "[a-z]"<<<"WhOaMi")
```

(**Note:** if `space` is blackisted, replace the spaces with tabs: `%09`)

This technique:
* Uses `tr` to transform all uppercase letters to lowercase
* Converts `WhOaMi` to `whoami` before execution
* Bypasses filters looking for exact command strings
* Returns normal command output (`21y4d`)

This approach works because the transformation happens at runtime, after any security filters have been applied.

There are many other commands we may use for the same purpose, like the following: `$(a="WhOaMi";printf %s "${a,,}")`
