# 1. Overview

LFI commonly occurs in templating engines where web applications display static content (headers/footers) while dynamically loading page-specific content. When parameters like `/index.php?page=about` are manipulable, attackers can potentially access other server files.

Two common readable files that are available on most back-end servers are `/etc/passwd` on Linux and `C:\Windows\boot.ini` on Windows. So, in a vulnerable scenario, for example, a website that has **language switching parameter is vulnerable**: `/index.php?language=es.php`, we can manipulable the **vuln parameter: language** as follow: 

`/index.php?language=/etc/passwd`, or `/index.php?language=/../../../etc/passwd` and so on.

`/index.php?language=C:\Windows\boot.ini`, or `/index.php?language=/../../../C:\Windows\boot.ini` and so on.

Security Impacts:

- Source code disclosure
- Sensitive data exposure
- Remote code execution (under certain conditions)
- Server compromise

# 2. File Disclosure

## 2.1 - Path Traversal

When web applications append your input to a predefined directory path like:
```php
include("./languages/" . $_GET['language']);
```

A direct path like `/etc/passwd` would fail because the system looks for `./languages//etc/passwd`

### Bypass Using Relative Paths

The solution is using `../` (parent directory reference) to traverse upward in the directory structure:

1. Each `../` moves one directory up toward the root
2. Once you reach the root `/`, you can access any file on the system
3. Example: `../../../../etc/passwd` works because it:
   - Moves up from `/var/www/html/languages/` to the root
   - Then accesses `/etc/passwd`

### Key Points

* Using excessive `../` sequences isn't harmful - if you're already at root, additional `../` are ignored
* For efficiency, calculate the minimum number needed (e.g., 3 for `/var/www/html/`)
* This technique works regardless of whether your input is used directly or appended to a path

## 2.2 - Filename Prefix Challenge

When input is used within a filename prefix pattern:
```php
include("lang_" . $_GET['language']);
```

Standard path traversal (`../../../etc/passwd`) fails because it creates an invalid filename: `lang_../../../etc/passwd`

### Solution
Prefix your payload with `/` to treat the prefix as a directory:
```
language=/../../../etc/passwd
```

This converts `lang_/` into a directory component rather than part of the filename.

**Note:** This approach has limitations:
- May fail if the prefixed directory doesn't exist
- Can interfere with advanced techniques like PHP wrappers or RFI

This approach effectively bypasses the directory restriction by stepping out of the intended directory context.


## 2.3 - Appended Extensions

When the server automatically adds an extension to your input:
```php
include($_GET['language'] . ".php");
```

Basic LFI attempts fail because `/etc/passwd` becomes `/etc/passwd.php`

This is a common practice for:
- Convenience (users don't need to specify extensions)
- Security (limiting inclusion to specific file types)

When the application forces a `.php` extension, direct system file reading becomes more challenging as most system files don't have this extension.

This requires more advanced techniques beyond basic path traversal to bypass.

## Bypassing Appended Extensions

When applications force extensions (e.g., `.php`), two historical techniques may help bypass restrictions on older PHP versions.

### 1. Path Truncation (PHP pre-5.3/5.4)
- Exploits PHP's 4096 character string length limit
- PHP truncates strings exceeding this limit
- Attack method:
  ```
  ?language=non_existing_directory/../../../etc/passwd/./././[REPEATED to reach ~4096 chars]
  ```
- Generation example:
  ```
  echo -n "non_existing_directory/../../../etc/passwd/" && for i in {1..2048}; do echo -n "./"; done
  ```
- The appended `.php` extension gets truncated from the final processed path

### 2. Null Byte Injection (PHP pre-5.5)
- Exploits how PHP handled string termination
- Adding `%00` (null byte) at the end of a string terminates it
- Example: `/etc/passwd%00` causes `.php` to be ignored
- The server processes this as `/etc/passwd%00.php` but treats it as `/etc/passwd`

**Note:** These techniques are ineffective on modern PHP installations but may work on legacy systems.


## 2.4 - Second-Order attack

Second-order LFI attacks occur when applications securely handle direct user input but insecurely use stored values to access files.

**Key characteristics:**
- The payload is stored in one location (e.g., database) then used elsewhere
- Developers often secure direct input but trust database values

**Example scenario:**
- Web app allows downloading avatars via `/profile/$username/avatar.png`
- If username contains `../../../etc/passwd`, accessing the avatar URL may return system files
- The attack succeeds because the application treats database values as trusted

These attacks are particularly effective because the malicious input and the vulnerability exploitation happen at different stages, making them harder to detect during security reviews.

## 2.5 - Non-Recursive Path Traversal Filters

Basic LFI prevention often uses simple string replacement to remove path traversal sequences:

```php
$language = str_replace('../', '', $_GET['language']);
```

**Key vulnerability:** These filters typically run only once on the input string, not recursively on the result.

**Bypass techniques:**
- Using nested traversal patterns that produce valid traversal after filtering
- Example: `....//` becomes `../` after filtering
- Successful bypass: `....//....//....//....//etc/passwd`

**Other effective bypass patterns:**
- `..././` (becomes `../` after filtering)
- `....\/` (using escaped slash)
- `....////` (using multiple slashes)

These work because the filter processes the string only once, allowing specially crafted inputs to produce valid traversal sequences after the filtering operation.

## 2.6 - Encoding Bypass for Path Traversal Filters

Some web applications block specific characters like `.` or `/` to prevent path traversal attacks. These filters can often be bypassed using URL encoding techniques.

**Basic encoding bypass:**
- URL encode the path traversal characters
- `../` becomes `%2e%2e%2f`
- Example: `%2e%2e%2f%2e%2e%2f%2e%2e%2f%2e%2e%2f%65%74%63%2f%70%61%73%73%77%64` to access `/etc/passwd`

**Important notes:**
- Encode ALL characters including dots (some encoders skip them)
- Double encoding (encoding an already encoded string) may bypass more sophisticated filters
- This technique is particularly effective against older PHP versions (pre-5.3.4) but works on newer systems with custom filters

URL encoding transforms blocked characters into their hexadecimal representation, which may pass through filters but get decoded before file inclusion occurs.


## 2.7 - Approved Paths Restrictions

Some applications use regex to limit file inclusion to specific directories:

```php
if(preg_match('/^\.\/languages\/.+$/', $_GET['language'])) {
    include($_GET['language']);
} else {
    echo 'Illegal path specified!';
}
```

**Bypass strategy:**
- Start with the approved path to satisfy the regex check
- Then use path traversal to escape the restricted directory
- Example: `./languages/../../../../etc/passwd`

**Discovery methods:**
- Analyze existing form submissions to identify approved paths
- Fuzz different directories until finding one that passes validation

**Combined defenses:**
- When multiple filters exist, combine bypass techniques
- Example: Start with approved path + URL encoding or recursive patterns

This technique works across various programming languages and frameworks since it exploits the fundamental path resolution mechanism rather than language-specific features.
