# Steps to exploit

Below are steps and what to consider when testing for File Upload Vulnerability

## 1. Validation

- The first thing to consider is `validation`. Usually, there will alway be a validation on what `type of file` can be uploaded.
- Take notice on if the validation is `client-side` or not (by look at the web's response when upload a test file). If it is, this can be bypassed with either with `inspect and delete code` or `Intercept tools` like **Burp**.

### 1.1 - Flawed file type validation

- It is heplful to use `intercept tools` (**Burp**) to view the `POST` request you make.
- By doing so, you can see certain `HEADER` (for example: `Content-Type: ...`)that `validate content-type` of your file.
- If there is such thing, alter its value into what suit your case.

## 2. Web Framework

- Try `Identify Web Framework`. (do `Fingerprinting`)
- The idea is to upload a `shell or reverse shell`. But to do that, we must know the programming language (php, ...) is used by the website to craft the corresponding shell.
- To identify programming language, we can `FUZZ the extension of index file`, or use tools like: `whatweb, Wappalyzer`.

## 3. (if no Filters) Vulnerability Identification and Upload Exploitation

- Assuming that we successfully uploaded a shell to the website. However, when we try to trigger it with stored url: `https://something/storage/shell.php?cmd=id`, nothing happen but a blank or error message. This is because the website might be `not configured to execute files or a certain files`.
- Therefor, we can try FUZZ what other files also get accepted by the website with `file extension lists`. But the shell would be something like this: `<?php echo "This should not be here";?>` (If the web run in PHP ofcause). With this, we will filter the `response` - whatever prompt the word is the vulnerable one.
- From here, you can use your custom shell or shell gen tools to exploit it.
- Shell or reverse shell - depend on your circumstances.
- Resources and tools you can use: `pentestmonkey`, or `Seclists` - PHP reverse shell, `msfvenom` - to gen a shell.

### 3.1 - Preventing file execution in user-accessible directories

- To defend against web shells that bypass upload filters, servers implement a second security layer: preventing script execution in user-accessible directories.
- See more about it in **`9.3, part 2.2`**

## 4. Blacklist Filters 

There are generally two common forms of validating a file extension on the back-end:
- Testing against a `blacklist` of types
- Testing against a `whitelist` of types

Furthermore, the validation may also check the `file type` or the `file content` for type matching by look at the file `metadata`.

The weakest form of validation amongst these is `testing the file extension against a blacklist of extension` to determine whether the upload request should be blocked. (see more in **`9.3 part 2.3`**)

- Since the website simply `BLOCK` certain file extensions in its list. Therefor, we can just `FUZZ out what extension get accepted`. (Lists can be obtained from: `Payloadallthething`, `Seclists`,...)

see more in **`9.3 part 2.3.1`**

## 5. Whitelist Filters

A whitelist is generally more secure than a blacklist. The web server would `only allow` the `specified extensions`.

- We can try FUZZ it as in above to see `what get accepted`, and see `how the web response`. Base on the result, we might get some `clues to bypass` it.

### 5.1 - Bypass with Double Extensions and Reverse Double Extension

- If the code only tests whether the file name contains an `image extension`; a straightforward method of passing the regex test is through `Double Extensions`.
- For example, if the `.jpg` extension was allowed, we can add it in our uploaded file name and still end our filename with `.php` (e.g. `shell.jpg.php`), in which case we should be able to pass the whitelist test, while still uploading a PHP script that can execute PHP code.
- **Reverse Double Extension**: `shell.php.jpg`. This also can happens if website is misconfigured.
- Ofcause, we should also combine this 2 methods with FUZZ.
- However, this bypass method may not always work, as some web applications may use a strict regex pattern.

### 5.2 - Bypass with Character Injection

We can inject several characters before or after the final extension to cause the web application to misinterpret the filename and execute the uploaded file as a PHP script.

The following are some of the characters we may try injecting:
```
%20
%0a
%00
%0d0a
/
.\
.
…
:
```

- Each character has a specific use case that may trick the web application to misinterpret the file extension.
- For example, (`shell.php%00.jpg`) works with `PHP` servers with version `5.X or earlier`, as it causes the PHP web server to end the file name after the (`%00`), and store it as (`shell.php`), while still passing the whitelist.
- The same may be used with web applications hosted on a `Windows` server by injecting a colon (`:`) before the allowed file extension (e.g. `shell.aspx:.jpg`), which should also write the file as (`shell.aspx`).
- Similarly, each of the other characters has a use case that may allow us to upload a PHP script while bypassing the type validation test.

We can write a small bash script that generates all permutations of the file name, where the above characters would be injected before and after both the `PHP` and `JPG` extensions, as follows:

```bash
for char in '%20' '%0a' '%00' '%0d0a' '/' '.\\' '.' '…' ':'; do
    for ext in '.php' '.phps'; do
        echo "shell$char$ext.jpg" >> wordlist.txt
        echo "shell$ext$char.jpg" >> wordlist.txt
        echo "shell.jpg$char$ext" >> wordlist.txt
        echo "shell.jpg$ext$char" >> wordlist.txt
    done
done
```

**Note:** 
> - you can alter the '.php' '.phps' and `.jpg` depend on your circumstances.
> - Also, use a normal picture file (`something.jpg`), then obfuscate it name and ext as above. Then, try access it on the website by using different part of its new gen name, take note of on `what name the picture being rendered`. The idea is to see how the website interact with those `special chars` in file name. 
> - (see more in **`9.3 part 2.3.2 - Obfuscating file extensions`**)


## 6. Type Filters

There are two common methods for validating the file content: `Content-Type Header` or `File Content`. Let's see how we can identify each filter and how to bypass both of them.

### 6.1 - Content-Type

- **Seclists/discovery/web-content/web-all-content-types.txt**.
- Depend on your case, find the corresponding web content type. For example, if your target is a image upload case, you can copy a part from the above list that will suit you.
- ` cat web-all-content-types.txt | grep 'image/' > image-content-types.txt`
- Now try fuzz with the new list.
- **(see more in `9.9 part 2.1 - Flawed file type validation`)**

### 6.2 - MIME-Type

The second and more common type of file content validation is testing the uploaded file's MIME-Type. `Multipurpose Internet Mail Extensions` (MIME) is an internet standard that determines the type of a file through its general format and bytes structure.

This is usually done by inspecting the first few bytes of the file's content, which contain the `File Signature` or `Magic Bytes`. 
- For example, if a file starts with (`GIF87a` or `GIF89a`), this indicates that it is a `GIF image`, while a file starting with plaintext is usually considered a `Text` file.
- If we change the first bytes of any file to the GIF magic bytes, its MIME type would be changed to a GIF image, regardless of its remaining content or extension.
- **(see more in `9.3 part 2.4 - Flawed validation of the file's contents`)**
- we can try using an **Allowed MIME type with a disallowed Content-Type, an Allowed MIME/Content-Type with a disallowed extension, or a Disallowed MIME/Content-Type with an allowed extension, and so on**. 
 
## 7. Limited File Uploads

### 7.1 - XSS

File uploads can introduce Stored XSS vulnerabilities. When applications accept HTML files, attackers can embed JavaScript code that executes when victims view the uploaded page. This creates an attack vector where trusted domains serve malicious content, making social engineering more effective against users who trust the website.

Another example of XSS attacks is web applications that display an image's metadata after its upload. For such web applications, we can include an XSS payload in one of the Metadata parameters that accept raw text, like the `Comment` or `Artist` parameters, as follows:

```
$ exiftool -Comment=' "><img src=1 onerror=alert(window.origin)>' HTB.jpg
$ exiftool HTB.jpg
...SNIP...
Comment                         :  "><img src=1 onerror=alert(window.origin)>
```

We can see that the `Comment` parameter was updated to our XSS payload. When the image's metadata is displayed, the XSS payload should be triggered, and the JavaScript code will be executed to carry the XSS attack. Furthermore, if we change the image's MIME-Type to `text/html`, some web applications may show it as an HTML document instead of an image, in which case the XSS payload would be triggered even if the metadata wasn't directly displayed.

Finally, XSS attacks can also be carried with `SVG` images, along with several other attacks. `Scalable Vector Graphics (SVG)` images are XML-based, and they describe 2D vector graphics, which the browser renders into an image. For this reason, we can modify their XML data to include an XSS payload. For example, we can write the following to `HTB.svg`:
```
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE svg PUBLIC "-//W3C//DTD SVG 1.1//EN" "http://www.w3.org/Graphics/SVG/1.1/DTD/svg11.dtd">
<svg xmlns="http://www.w3.org/2000/svg" version="1.1" width="1" height="1">
    <rect x="1" y="1" width="1" height="1" fill="green" stroke="black" />
    <script type="text/javascript">alert(window.origin);</script>
</svg>
```
Once we upload the image to the web application, the XSS payload will be triggered whenever the image is displayed. 

### 7.2 - XXE

Similar attacks can be carried to lead to XXE exploitation. With SVG images, we can also include malicious XML data to leak the source code of the web application, and other internal documents within the server. The following example can be used for an SVG image that leaks the content of (`/etc/passwd`):
```
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE svg [ <!ENTITY xxe SYSTEM "file:///etc/passwd"> ]>
<svg>&xxe;</svg>
```

When an SVG with XML external entity code is uploaded and viewed, it can access server files (like `/etc/passwd`) and display their contents on the page or in the source code. This technique also works with direct XML document uploads when the application displays that XML data.

This capability is particularly valuable for web penetration testing as it enables access to application source code (whitebox testing). For file upload exploits specifically, examining source files can reveal critical information about:
- Upload directory locations
- Allowed file extensions
- File naming conventions

These discoveries significantly enhance further exploitation potential.

To use XXE to read source code in PHP web applications, we can use the following payload in our SVG image:
```
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE svg [ <!ENTITY xxe SYSTEM "php://filter/convert.base64-encode/resource=index.php"> ]>
<svg>&xxe;</svg>
```

Once the SVG image is displayed, we should get the base64 encoded content of index.php, which we can decode to read the source code.

Using XML data is not unique to SVG images, as it is also utilized by many types of documents, like `PDF, Word Documents, PowerPoint Documents`, among many others. All of these documents include XML data within them to specify their format and structure. Suppose a web application used a document viewer that is vulnerable to XXE and allowed uploading any of these documents. In that case, we may also modify their XML data to include the malicious XXE elements, and we would be able to carry a blind XXE attack on the back-end web server.

### 7.3 - DOS

File upload vulnerabilities can enable several denial-of-service attack vectors:

- **XXE payloads** can trigger DoS conditions as detailed in the Web Attacks module
- **Decompression bombs** use nested ZIP archives that expand exponentially when extracted, potentially consuming petabytes of server storage
- **Pixel flood attacks** manipulate JPG/PNG compression data to declare false dimensions (e.g., 0xffff × 0xffff), causing memory allocation crashes when rendered
- **Oversized files** can fill server storage when upload size limits aren't enforced
- **Directory traversal** in upload functions can place files in critical system locations causing system instability

These techniques target resource exhaustion (storage, memory, CPU) to render the server unresponsive.

## 8. Other Upload Attacks

### 8.1 - Injections in File Name

Malicious filenames can trigger multiple vulnerabilities when reflected or processed by web applications:

**Command Injection:**
Using filenames like `file$(whoami).jpg`, `file`whoami`.jpg`, or `file.jpg||whoami` can execute commands if the application processes uploads with OS commands (e.g., `mv file /tmp`), leading to remote code execution.

**Cross-Site Scripting:**
Filenames containing JavaScript code (`<script>alert(window.origin);</script>`) can execute in users' browsers when displayed.

**SQL Injection:**
Filenames with SQL syntax (`file';select+sleep(5);--.jpg`) can compromise database operations if used directly in queries.

These attacks exploit insufficient sanitization of user-provided filenames in server-side operations.

### 8.2 - Upload Directory Disclosure

When direct access to uploaded files isn't provided, these techniques can help discover upload locations:

- **Directory fuzzing** to systematically probe potential storage paths
- **Leveraging other vulnerabilities** (LFI/XXE) to read application source code and identify storage locations
- **IDOR techniques** from the Web Attacks module to find file storage patterns
- **Forcing error messages** through:
  - Uploading files with duplicate names
  - Sending simultaneous identical requests
  - Using extremely long filenames (5,000+ characters)

These errors often reveal absolute file paths, disclosing the uploads directory and potentially other useful server information for further exploitation.

### 8.3 - Windows-specific Attacks

Several file naming techniques can exploit Windows systems:

**Reserved Characters:**
Using `|`, `<`, `>`, `*`, or `?` in filenames can cause errors when not properly sanitized, potentially revealing upload directories.

**Reserved Names:**
Windows system names like `CON`, `COM1`, `LPT1`, and `NUL` cannot be used as filenames and will trigger errors when attempted.

**8.3 Filename Convention:**
Windows maintains backward compatibility with the 8.3 naming format (8 characters + period + 3 character extension). This can be exploited by:
- Using shortened names like `HAC~1.TXT` to reference `hackthebox.txt`
- Creating files like `WEB~.CONF` to potentially overwrite `web.conf`

These techniques can lead to information disclosure, denial of service, or access to sensitive system files.

### 8.4 - Advanced File Upload Attacks

Beyond basic attacks, sophisticated vulnerabilities exist in automatic file processing operations:

- Video encoding/transcoding processes
- File compression/decompression
- Filename manipulation routines
- Image processing libraries

Well-known library vulnerabilities, like the ffmpeg AVI processing XXE vulnerability, demonstrate how complex file handling can introduce security issues.

Custom code and proprietary libraries require deeper analysis techniques to identify similar weaknesses. These advanced vulnerabilities often have significant impact but require specialized knowledge to detect.

For more examples of sophisticated file upload vulnerabilities, security professionals should review bug bounty reports where such discoveries are documented in detail.
